1. Web 应用的基础词汇

CS615 的大多数题都在问同一件事：用户在浏览器里做一个动作后，请求如何进入服务器，服务器如何处理业务规则，数据如何存取，最后结果如何回到页面。下面这些词是后面所有章节的地基。

词 从零理解 小例子
Client / Browser 用户实际操作的程序。浏览器负责显示网页、运行前端 JavaScript、向服务器发请求。 Chrome 打开购物网站并点击“结账”。
Frontend 用户看得见、点得到的部分，包括页面布局、按钮、表单、交互和部分状态。 React 页面显示订单列表和搜索框。
Server / Backend 用户看不见但真正处理规则的部分。它接收请求、检查权限、运行业务逻辑、读写数据库。 Express 接收 POST /orders 并创建订单。
Database 长期保存数据的系统。服务器重启后，数据库里的订单、用户和商品仍然存在。 PostgreSQL 保存 receipts；MongoDB 保存 JSON-like documents。
Request / Response 浏览器向服务器发 request；服务器处理后发 response 回来。 浏览器请求 GET /products，服务器返回产品 JSON。
API Application Programming Interface。这里通常指前端和后端约定好的“接口地址 + 数据格式”。 GET /api/orders/15 返回第 15 号订单。
JSON 一种轻量数据格式，很像 JavaScript object，常用于前后端传数据。 {"id":15,"total":90}
Business Logic 业务规则，不是页面样式也不是数据库本身。例如折扣、权限、库存、订阅规则。 订单超过 1000 欧给 premium user 打九折。
Framework 帮你组织项目的工具和规则。它规定常见文件结构和写法，减少重复工作。 Rails、Spring Boot、Express、React。

内容理解题（先做 3 题）

核心答案（先确认概念）：Browser request -> backend processing -> database -> response。不要把词孤立背。

为什么这是对的：CS615 的几乎所有题目都从“用户打开浏览器发请求”开始。如果孤立背 Client/Server/Database 的定义，遇到“解释 full-stack 各层职责”这类题就只能堆名词而不会串流程。把词汇串成 Browser→Server→Database→Response 一线，等于提前有了每道解释题的答案骨架——你可以自然地说明请求经过路由、控制器、业务逻辑、数据库后返回给浏览器，每层说一句话，再加一个具体例子就完整了。

题目语境怎么用：遇到 full-stack 解释题，开头写“用户在浏览器点击按钮后，浏览器发 HTTP 请求到服务器”。遇到数据库选择题，先画出 Customers→Receipts→ReceiptItems 的关系链，再判断 SQL/NoSQL。遇到 MVC 题，从路由开始描述 controller 如何协调 model 和 view，而不是从三个词的定义开始。

卷面写法：先写下完整链路：Browser→Route→Controller→Service/Model→Database→View/JSON Response。每条箭头用一句话说明职责，再用一个常见场景（如“用户查看订单详情”）串起来。最后加一句：“CS615 考的就是你对这条链路每一步职责的理解，不是对单个术语的定义。”

常见失分点：只列 Client、Server、Database 三个定义，没有说明它们之间如何通信。或者写了“请求到服务器处理”但不说明处理什么、谁处理、数据在哪存。空泛的“协作”和“交互”不加例子，等于白写。

核心答案（先确认概念）：要说它是前后端约定好的入口，包括 URL、method、数据格式和 response。

为什么这是对的：考试里说 API 是“前后端接口”太模糊——评分标准要看你能不能写出：URL 地址是什么、用哪个 HTTP method、输入什么格式的数据、返回什么格式的结果。比如 GET /api/orders/15 返回 {"id":15,"total":90}，这才是考官想看的完整答案。只说“接口”不提 method 和数据格式，等于只答了半道题。

题目语境怎么用：如果题目问 MVC 中 controller 如何与前端通信：controller 暴露 REST API，前端用 fetch 调 POST /orders 发 JSON body，server 返回 201 Created 和新订单 JSON。如果题目问 CRUD 怎么映射到 Web：Create→POST /orders，Read→GET /orders/:id，Update→PUT 或 PATCH，Delete→DELETE。

卷面写法：用公式：API = URL + HTTP Method + Request Body (JSON) + Response (JSON + Status Code)。写一个具体例子，如“用户创建订单：POST /api/orders，body 为 {items:[...],total:90}，返回 {id:15,status:"created"} 和 201”。补一句 API 的作用是把前端 action 映射为后端操作。

常见失分点：写“API 是前后端交互的约定”就停了，不写 URL、method、数据格式。或者用“接口”和“端点”等词互相解释而不给具体例子。

核心答案（先确认概念）：Business logic 是规则；database 是持久保存数据的地方。规则不应散落在页面或 SQL 字符串里。

为什么这是对的：考试常把这两者混在一起考——比如给一段 controller 代码让你指出“业务逻辑放对位置了吗”。Business logic 是规则：“满 1000 打九折”“非会员不能退款”。Database 是存储：存订单、存用户、保证收据总价和明细一致。规则不是数据——今天的打折策略是九折，明天可能改成满减，改数据库 schema 来适应规则变化是错误设计。

题目语境怎么用：遇到 Sample Rails controller 题：controller 同时算折扣、存订单、发邮件——折扣计算是 business logic，应放入 Service 类；保存订单是 database 操作，应通过 Model/Repository；发邮件是副作用，应放入 Job/Mailer。遇到购物习惯数据题：SQL 的 ACID transaction 保证保存完整，但打折规则、积分规则应写在 application 层。

卷面写法：先定义两者：“Business logic = 业务规则（验证、计算、权限判断）；Database = 数据持久化和一致性保证”。用题面例子说明“这里折算折扣的逻辑应在 OrderService 中，不应直接写在 controller 或 SQL 里，因为规则会变而数据 schema 相对稳定”。

常见失分点：把验证输入格式当成 database 职责；把所有计算塞进 controller 不知道 service 层存在；用数据库触发器/存储过程来写业务规则（在 CS615 语境下通常是反面教材）。

考试题目题（再做 5 题）

标准答案（先写结论）：Frontend 显示界面并收集输入；server 接 request、执行业务规则和权限检查；database 长期保存用户、订单、收据等数据。

为什么这是对的：这道 2025 Summer Q1(a) 题考查的是你能不能把三层架构用最简单的人话讲清楚。Frontend 不是“显示页面”——是“收集用户操作并发请求”。Server 不是“处理请求”——是“检查权限、执行规则、决定返回什么”。Database 不是“存数据”——是“保证数据持久、一致、可查询”。三句话里每一句都在暗示三层之间的边界，这才是 full-stack 题的关键。

题目语境怎么用：给 intern 解释时用具体场景：“你在网页上点查看订单→这就是 frontend。浏览器请求了服务器上的 /api/orders/15→server 接到后去查数据库→database 返回订单信息→server 判断你是订单主人有权限看→把 JSON 返回给 frontend→frontend 渲染成表格。”三句话串起来就是一个完整请求。

卷面写法：第一句写 Frontend：“显示界面，收集用户输入，调用后端 API”。第二句写 Server：“接收请求，执行权限检查和业务规则，读写数据库，选择返回格式”。第三句写 Database：“长期保存订单、用户、收据等数据，通过查询、事务和外键保证数据正确一致”。最后用一句“三者通过 HTTP 和 JSON 串成完整请求链路”收束。

常见失分点：三句话写成三句废话：“Frontend 做前端的事，Server 做后端的事，Database 存数据”。或者反过来，为 intern 写了太多技术名词（React, Express, PostgreSQL）而忽略了解释各层“到底在做什么”。

标准答案（先写结论）：Create 是新增资源，常用 POST；Update 是修改已有资源，常用 PUT 或 PATCH。

为什么这是对的：HTTP 的 method 设计不是随意的——POST 的语义是“创建一个新的从属资源”，PUT 的语义是“完全替换指定资源”。Create 对应 POST 因为新资源还没有 ID，由服务器生成后返回；Update 对应 PUT/PATCH 因为资源已存在，客户端需要指定更新哪个资源——URL 里必须有 ID 如 /orders/15。理解这个语义差异在考试里是基本分。

题目语境怎么用：如果题目给出一个 Web 场景（如购物车），你可以说：“用户提交新订单用 POST /orders（Create），服务器生成 order_id 返回；用户修改收货地址用 PATCH /orders/15（Update），URL 中的 15 就是已有订单号。”PUT 和 PATCH 的区别也可以在答案里提：PUT 一般替换整个资源，PATCH 做局部更新。

卷面写法：Create→POST：“新增资源，URL 不需要 ID，服务器返回新资源 ID 和 201 Created”。Update→PUT/PATCH：“修改已有资源，URL 包含资源 ID，如 PUT /orders/15 替换整个订单，PATCH /orders/15 只改部分字段。”各配一个简短例子。

常见失分点：把 Create 写成 GET 或 PUT；把 Update 写成 POST；只说“增删改查”四个中文词而不对应到 HTTP verbs。另外有些同学只写 POST 不写 PUT/PATCH，漏了 Update 的对应。

标准答案（先写结论）：用收据/订单例子说明 SQL：Customers、Receipts、ReceiptItems 之间有关系且要 transaction；用 clickstream 例子说明 NoSQL：每条行为事件像 JSON document，字段变化快、写入量大。

为什么这是对的：数据库选择不是背“结构化/非结构化”，而是看题面数据的形状。收据/订单类数据有实体间关系、需要一致性保证——选 SQL。点击流/行为日志类数据每条独立、字段可变、写入量大——选 NoSQL。关键是把题面给的具体字段映射到这两个场景中。

题目语境怎么用：看到题面给了 customer、items、price、date、unique receipt number→这是收据系统特征→拆成 Customers、Receipts、ReceiptItems 三张表→说明外键、JOIN、唯一约束、事务的作用。看到题面给了 click、search、scroll、device、timestamp→这是行为事件特征→每条独立 document→字段可变→写入量大→选 MongoDB。

卷面写法：先下判断：“这个场景选 SQL，因为数据像收据系统”。然后列出实体关系：Customers 1→N Receipts 1→N ReceiptItems。解释为什么需要 key、join、constraint、transaction。最后用一句话说出 NoSQL 的适用边界：“如果题目是 document 型行为事件而不是结构化交易记录，NoSQL 才更合理”。

常见失分点：写成“SQL 用于结构化数据，NoSQL 用于非结构化数据”就停笔，不展开任何例子。看到“shopping”就自动选 NoSQL，没有注意到题目给的是收据记录而非点击行为。

标准答案（先写结论）：若交易历史一致性重要、团队小，可推荐 modular monolith + PostgreSQL；价格流压力大时再拆服务。

为什么这是对的：只看 scalability 就选 microservices 是典型的学生思维——题目给的条件是“交易历史必须准确”和“团队小”，这两个条件都更支持 monolith。交易历史的一致性在 monolith 的单数据库 transaction 下是最直接的，microservices 跨服务保证一致性需要 sagas/两阶段提交/eventual consistency，复杂度和出错风险都远高于 monolith。小团队维护 microservices 意味着每个人要同时管多个 service 的部署、监控、日志，运维负担会被严重放大。不回应这两个条件就是离题。

题目语境怎么用：正确的 recommendation 要逐条回应题面条件：“基于三个约束——①交易数据必须强一致（PostgreSQL 的 ACID transaction 比微服务的分布式事务更可靠）；②团队只有 3-5 个开发者（monolith 的调试和部署成本更低）；③价格流有实时性需求但可以解耦处理（用 Redis 缓存 + WebSocket，暂不需拆为独立 service）——推荐 modular monolith + PostgreSQL，价格流用队列和缓存处理，未来用户量增长后再评估是否拆分。”

卷面写法：写 recommendation 的公式：“题面给了 X、Y、Z 三个约束 → 推荐方案 A，因为 A 满足 X 和 Y（解释怎么满足），Z 可以通过……解决 → 方案 B 的代价（具体写）在当前条件下不值得 → 如果条件变化（如团队扩到 15 人），可重新评估 B。”

常见失分点：Recommendation 只写“推荐 modular monolith”而不解释为什么在当前条件下比 microservices 更好。不提“如果条件变化”的演进路径——工程答案永远有条件假设。不逐条回应题面约束。

标准答案（先写结论）：指出 controller 同时算折扣、存订单、发邮件、记 analytics，违反 SRP，应拆 service、model/repository、job/mailer。

为什么这是对的：CI/CD 和 observability 是 Sample 新题里的两个独立概念——考试里最容易把它们当成同一件事。CI/CD 关心“发布前”：CI 是每次 commit 自动 build+test，CD 是通过检查的代码自动部署。Observability 关心“发布后”：代码上线后，通过 logs（发生了什么）、metrics（量有多大）、traces（一个请求经过了哪些服务）来定位问题。两者的联系是：CI 的测试防止错误进入生产，CD 支持快速回滚，observability 把线上发现的 bug 反馈回 CI 的测试用例。

题目语境怎么用：如果题目问“CI/CD 和 observability 如何配合”，你可以这样说：“CI 阶段跑单元测试和集成测试，确保新代码不会引入 SQL injection 或 XSS 漏洞。CD pipeline 用 canary release 先在少量实例上部署，如果 observability 的 metrics 显示 error rate 上升或 latency 增加，CD 自动回滚。Observability 发现的性能瓶颈（如某个 API endpoint 在高峰期响应变慢）反馈回开发团队，在下一次 CI 中加入性能测试。”这个流程把两个概念串起来了。

卷面写法：“CI/CD（持续集成/部署）管 build/test/deploy pipeline——CI 自动构建测试每次 commit，CD 自动部署通过的构建，目标是快速可靠地交付。Observability 管生产环境可观测性——logs 记录离散事件（如错误栈），metrics 记录聚合数据（如 QPS、P99 latency），traces 记录一次请求跨服务的路径。区分关键：CI/CD 关注交付过程的自动化，observability 关注系统运行时的状态和理解。”

常见失分点：把 CI/CD 和 observability 写成同一件事（“都是 DevOps 工具”）。不区分 CI（构建测试）和 CD（部署发布）。observability 只写 monitoring 的内容（只盯着已知指标有没有超阈值），不写 tracing 和未知故障排查。

2. 三层架构和 Full-Stack 的基本分工

Full-stack application 不是“前端加后端”这么简单。考试里要把职责分清楚：前端负责用户界面和交互，服务器负责业务逻辑、认证和 API，数据库负责持久化、查询和一致性。持久化的意思是数据会被长期保存；一致性的意思是系统不会出现互相矛盾的数据，例如一个收据总价和商品明细对不上。三层架构把这些职责整理成 Presentation、Application/Logic、Data 三层。

层 主要职责 常见技术 考试答法关键词
Frontend / Presentation 显示界面、收集用户输入、处理客户端状态、调用 API HTML, CSS, JavaScript, React UI, user interaction, rendering, accessibility
Server / Application Logic 路由请求、执行业务规则、认证授权、调用数据库或外部服务 Node.js/Express, Rails, Spring Boot routes, controllers, services, validation
Database / Data 存储数据、执行查询、保证重要数据不互相矛盾，或提供灵活文档模型 PostgreSQL, MySQL, MongoDB persistence = 长期保存；query = 查询；schema = 数据结构规则

过关问答：先内容理解，再 past/sample paper 题，悬停或点击显示答案

内容理解题（先做 3 题）

核心答案（先确认概念）：不能只列 React、Node、MongoDB。要解释前端、后端、数据库、API、安全和部署如何协作。

为什么这是对的：列技术名（React + Node + MongoDB）最多拿一分——考题问的是“角色和职责”，不是“技术栈”。full-stack 的考点是三层如何协作：前端负责什么交互、后端怎么处理请求和业务、数据库怎么存数据并保持一致性、API 怎么定义通信格式。你必须解释：用户在 React 页面点按钮→发 fetch 到 Express API→Express 调 service 执行业务规则→通过 Mongoose 读写 MongoDB→返回 JSON→React 更新 UI。这才是一份在考官眼里值分的答案。

题目语境怎么用：2025 Summer Q1(a) 让你给 intern 解释 full-stack——这时候你要把三层职责串成一个用户故事：“用户登录后看到订单列表（frontend/React），点击详情时浏览器请求 GET /api/orders/15（API），Express 路由接到后去数据库查询（server/backend），PostgreSQL 返回订单和商品明细（database），服务器把结果打包成 JSON 返回，React 渲染订单详情页”。

卷面写法：开头不要列 MERN。先写三层定义：Presentation 层负责 UI 和交互、Application 层负责业务逻辑和 API、Data 层负责持久化和查询。然后用一个场景串联：用户操作→浏览器发 HTTP→Server 路由到 controller→service 执行业务→model/repository 读写 DB→response 返回→前端刷新。每层配一句技术选型（如“前端用 React 管理组件状态”）但技术名只是补充，不是主体。

3. SQL / NoSQL 的选择取决于数据形状和出错代价

2024 和 2025 都考过 SQL 与 NoSQL。不要把答案写成“SQL 老，NoSQL 新”，也不要只写“看数据形状”。考试里要把题面变成例子：如果是 customer、receipt number、items、price、date，就按收据系统解释 SQL；如果是 click、search、scroll、device、timestamp 这种行为事件，就按点击流解释 NoSQL。

先把数据库术语拆开

术语 不假设基础的解释 为什么考试会用到
Schema 数据的固定格式规则。比如收据必须有 receipt number、date、total，每个 item 必须有 name、quantity、price。 2025 shopping habits 里 receipt 字段固定，所以可以说它支持 SQL table schema。
Relation / 外键 数据之间的连接关系。比如一个 customer 可以有很多 receipts，一个 receipt 可以有很多 items。 查某个顾客买过什么时要连接 Customers、Receipts、ReceiptItems，所以 SQL 更自然。
Transaction / 事务 一组必须一起成功或一起失败的操作。比如创建收据和扣库存，不能只成功一半。 订单、付款、收据这类题目通常要强调事务。
Consistency / 一致性 系统里的数据不互相打架。比如收据总价是 30，但三件商品加起来是 25，这就是不一致。 交易历史、收据、账户余额通常需要强一致性。

Horizontal scaling 不是买一台更强的机器，而是加更多机器一起分担数据和请求。 例如每秒写入大量 clickstream events 时，可以用 NoSQL 解释分布式写入。

ACID 是什么

ACID 是数据库事务的四个安全承诺。它听起来抽象，但可以用“创建一张收据”理解：系统不能保存半张收据，不能让总价和明细对不上，不能让两个操作互相踩踏，保存成功后不能突然丢失。

字母 意思 收据例子
A - Atomicity原子性：一组操作要么全部成功，要么全部失败。收据、商品明细、库存扣减不能只保存其中一部分。
C - Consistency一致性：操作前后都要满足数据规则。receipt total 必须等于 item prices 加总。
I - Isolation隔离性：同时发生的操作不能互相干扰。两个员工同时改同一张收据，不应产生混乱结果。
D - Durability持久性：一旦系统说保存成功，数据就应该留下来。用户付款后，即使服务器重启，收据也不能消失。

扩展 常见做法是升级机器或加读副本；复杂关系查询强 常见做法是把数据分到多台机器；大量写入和分布式场景常见

经典场景：记录顾客购物习惯

题目给出的数据包括 customer、items、price、date、unique receipt number。这些信息天然像订单系统：顾客、收据、商品、价格和日期之间有清晰关系。收据数据还很怕“只存了一半”：如果保存了收据号但没有保存商品明细，或者总价和明细对不上，系统就不可信。因此推荐 SQL 更稳。

I would choose an SQL database for this scenario. The data is structured: each customer has purchases, each purchase has a receipt number, a date, a price, and a set of purchased items. These entities can be represented cleanly using relational tables such as Customers, Receipts, ReceiptItems, and Products.

SQL is also suitable because receipt history is transactional data. In plain terms, this means several related records must be saved together correctly: the receipt number, the item rows, the prices, and the final total. We want integrity constraints, reliable joins, and ACID transactions so that totals, receipt numbers, and item records remain consistent.

A NoSQL database could be justified if the main requirement were high-volume clickstream behaviour or flexible event documents, but based on the question as written, the structured shopping records and consistency requirements make SQL the better default choice.

核心答案（先确认概念）：如果题目改成每次点击、搜索、滚动、设备、实验分组都生成一条事件，每条 JSON-like document 字段不同且写入量很大，就可以辩护 MongoDB/NoSQL。

核心答案（先确认概念）：用“保存收据”解释：收据头、商品明细、库存扣减要么一起成功，要么一起失败；不能钱扣了、items 没保存。

为什么这是对的：用“保存一张收据”理解 ACID 是最自然的——因为收据的一笔交易天然涉及多张表（Receipts + ReceiptItems），必须一起成功或一起失败（Atomicity），保存前后收据总价都必须等于明细之和（Consistency），两个员工不能同时改同一张收据导致数据互相踩踏（Isolation），保存成功后即使服务器断电数据也不能丢（Durability）。四个字母各对应收据场景的一个具体问题，比背英文单词好记好写。

题目语境怎么用：如果考试问“解释 ACID”，你就用收据收银的场景串起来：“A（原子性）：一张收据的 header 行、3 条 item 行、库存扣减操作要么全成功要么全回滚，不能出现钱扣了但 item 没保存。C（一致性）：收据的 total 字段必须等于 SUM(item.price * qty)，数据库的 constraint 强制这个规则。I（隔离性）：两个收银员同时操作时，各自的收据保存过程互不干扰。D（持久性）：收据保存成功后，即使数据库进程重启，数据也必须还在。”

卷面写法：用一个收据场景串四个字母：“以创建收据为例——(A)收据头和明细行一起保存或一起回滚；(C)total 必须等于 item 单价乘数量之和，不满足则操作失败；(I)并发操作同一库存时，后一个必须等前一个完成；(D)提交后即使服务器重启数据也不丢失。这就是 SQL 的 ACID 事务保证。”

常见失分点：只列四个英文单词展开而不给任何场景——ACID 脱离具体场景就像背单词表。C 和 A 混淆（原子性是一起成功/失败，一致性是数据满足规则）。I 解释成“一个人操作”而不是“并发操作不互相干扰”。

4. MVC 把数据、页面和控制流程拆开

MVC 是 CS615 最稳定的大题之一。它的核心不是背 Model、View、Controller 三个词，而是能解释一次请求如何从 route 进入 controller，如何调用 model/database，最后如何选择 view 或 JSON response。

先用一次点餐理解 MVC

可以把 MVC 想成餐厅点餐。View 像菜单和上菜窗口，用户通过它看信息、提交选择；Controller 像服务员，接单、判断该找谁处理、最后把结果带回来；Model 像厨房和库存系统，知道菜品、价格、库存，并真正改变数据。这个比喻不完美，但能帮你记住：controller 不应该自己炒菜，也不应该自己长期保存库存。

MVC 部分 职责 Rails 例子 Spring / Express 对应
Model 代表领域数据和数据访问规则。领域数据就是系统真正关心的东西，例如订单、用户、商品。 Order 继承 Active Record，映射到 orders table Spring entity/repository，Mongoose model
View 把数据呈现给用户。渲染的意思是把数据变成用户能看见的 HTML 页面、组件或 JSON 结果。 ERB template 或 JSON view Thymeleaf, React UI, JSON response
Controller 接收请求、读取参数、调用 model/service、选择 response。它负责协调，不应该塞满业务规则。 OrdersController#create Spring controller, Express route handler/controller

25 分 MVC 题的答案骨架

MVC separates a web application into Model, View, and Controller. The Model represents the application data and business entities, and usually interacts with the database. The View is responsible for presenting information to the user, such as HTML pages or JSON consumed by a frontend. The Controller receives HTTP requests, coordinates the work, and chooses which response to return.

For example, in a Rails shop application, a route such as POST /orders maps to OrdersController#create. The controller builds an Order model from form parameters, asks the model or a service to validate and save it, and then redirects to an order page or renders the form again. The view displays the result to the user.

The same pattern adapts to other environments. In Spring Boot, controllers use annotations such as @GetMapping, models are entities/repositories, and views may be Thymeleaf templates or JSON responses. In Express, routes map URLs to handlers, controllers coordinate request logic, and Mongoose or repository classes handle data.

Sample Rails controller 题：为什么违反 MVC 和 SRP

Sample Q3 的 controller 同时做折扣计算、保存订单、发邮件、记录 analytics、redirect/render。问题是 controller 太胖：它不只是控制流程，还包含业务规则和副作用，违反 Single Responsibility Principle。

问题 为什么不好 重构方向
折扣逻辑在 controller 业务规则难复用、难单元测试 移到 DiscountService 或 Strategy pattern
邮件和 analytics 直接调用 请求流程耦合外部副作用 移到 job/event/observer after order created
controller 承担太多职责 违反 SRP，难维护 controller 只负责参数、调用服务、选择 response

重构后，折扣逻辑可以用普通 unit test 独立测试，不需要构造完整 controller request；controller test 只检查它是否调用服务并处理成功/失败分支。

内容理解题（先做 3 题）

核心答案（先确认概念）：Browser -> route -> controller -> model/database -> view/response。

为什么这是对的：MVC 25 分 essay 题（如 2025 Summer Q2）的评分标准第一条就是“描述了完整的 request flow”。如果只写 Model/View/Controller 三个定义，分数上限很低——因为 MVC 不是一个静态结构，而是一个动态流程：请求怎么进来、经过谁、谁负责调用谁、最后怎么出去。Browser→Route→Controller→Model/Database→View/Response 这条线是 MVC 题的脊梁骨，缺了它，三个定义就是三块孤立的砖。

题目语境怎么用：如果问“MVC 为什么适合 Web 开发”，你先说“Web 应用天然有请求/响应这个流程，MVC 的 Controller 正好对应 HTTP 请求的入口”。然后展开：测试角度——“你可以单独测试 Order 模型的验证逻辑（金额不能为负、商品不能为空），不需要启动整个服务器”。并行角度——“UI 设计师改 View 模板不会碰 Controller 逻辑”。替换角度——“如果以后前端从服务器渲染换成 React SPA，Controller 的 API 保持不变，只需改 View 返回 JSON 即可”。

卷面写法：“MVC 的工程好处来自 separation of concerns：①可测试性：每层可独立单元测试（Model 测验证规则、Controller 测调用序列、View 测渲染输出）；②可维护性：改业务规则只需动 Service/Model 层，改界面只需动 View；③并行开发：前端和后端开发者可以同时开工，约定好 API 格式即可；④技术演进：各层可以独立换技术（如 View 从 ERB 换 React，Model 从 ActiveRecord 换 DataMapper），其他层不受影响。”

常见失分点：只写 separation of concerns 不展开。把 MVC 的好处写成“代码更整洁”这种主观描述。不提 MVC 在考试中最重要的视角——测试和并行开发。

题目语境怎么用：写答案时按“问题→改进”的因果链展开：“HTTP/1.0 的问题：每个资源（图片、CSS、JS）都需要独立的 TCP 连接，一个页面几十个资源就要几十次连接握手，开销巨大。→HTTP/1.1 的改进：默认 keep-alive，一个连接上可以发送多个请求，但请求是串行的——前面一个慢，后面全堵住（HOL blocking）。→HTTP/2 的改进：multiplexing，多个 stream 在一个连接上交错传输，请求和响应可以并发。→HTTP/3 的改进：QUIC over UDP，TCP 层的丢包重传不再阻塞所有 stream。”

常见失分点：罗列版本号和年份，不写“为什么上一代不够好”。不提 HOL blocking 这个概念——这是整个 HTTP 演进题的核心关键词。不说明每一代改进的具体机制（如 multiplexing 和 persistent connection 的区别）。

5. Node.js / Express / React 的高频问答

为什么用 Node.js 做 Web 项目

Node.js 的意思是：让 JavaScript 不只在浏览器里运行，也能在服务器上运行。它的优势来自 JavaScript runtime 和事件驱动 I/O。Runtime 可以理解为“代码运行的环境”；I/O 是 input/output，指读数据库、读文件、发网络请求这类等待外部结果的工作。Node 很适合这类等待很多、计算不重的 Web 服务。

它适合实时应用、API gateway、聊天、dashboard、快速原型和 JavaScript 全栈开发。答案里也要承认边界：CPU 密集型计算不是 Node 的天然强项。CPU 密集型指大量数学计算、图片视频处理这类一直占用处理器的任务，可能需要 worker、队列或其他服务。

优点 解释
Non-blocking I/O等待数据库、文件或网络时，不会让整个服务器停在那里干等。服务器可以先处理其他请求。
Event-driven事件驱动：事情完成时再回来处理。例如数据库查完后，再执行对应 callback 或 promise。
Same language across stack前后端都可使用 JavaScript/TypeScript，团队共享模型和工具。
NPM ecosystemExpress、Mongoose、JWT、testing、build tools 等生态丰富。

Callback 和代码输出题

Callback 是“先交代好，等事情完成后再执行”的函数。比如你设置一个定时器，相当于告诉 Node：几毫秒后请调用这个函数。同步代码会马上执行；定时器和 I/O 回调要等当前同步代码执行完后，再由 event loop 安排执行。Event loop 可以理解为 Node 的排队系统：它不断检查哪些异步任务已经完成，然后把对应 callback 拿出来执行。

因此 2023 的嵌套 setTimeout 代码输出是：

first

fifth

second

third

fourth

原因是 console.log("first") 和 console.log("fifth") 是同步代码，先执行。第一个 timeout 到期后打印 second，然后才注册内部 timeout；内部 timeout 再依次打印 third 和 fourth。

Chaining、routes、controllers、models

概念 答题解释
Chaining 把多个操作连续写在一起，例如 promise 的 .then().catch()，或 API 方法返回对象本身以继续调用。它让异步步骤或配置步骤更紧凑，但过度 chaining 会降低可读性。

React 和 vanilla HTML/CSS/JS 的比较

React 是用于构建用户界面的 JavaScript library。Library 是一组别人写好的工具，你按它的方式调用。React 的核心想法是把 UI 拆成 components。Component 就是一块可复用界面，例如按钮、导航栏、订单卡片。Props 是父组件传给子组件的数据；state 是组件自己记住、会变化的数据，例如搜索框内容或购物车数量。

SPA 是 Single Page Application，意思是用户看起来在一个网页里完成很多切换，页面主要通过 JavaScript 更新内容，而不是每点一次都整页刷新。React 很适合 SPA、dashboard、动态表单和状态变化频繁的应用。Vanilla HTML/CSS/JS 更轻、更直接，适合简单页面、小型交互和不需要复杂状态管理的站点。

内容理解题（先做 3 题）

核心答案（先确认概念）：先输出同步代码，再根据 event loop 和异步回调进入队列的顺序判断后续输出。

为什么这是对的：Node 的代码输出题不是在考你“猜输出”，而是在考你理解 event loop。Event loop 的核心规则是：所有同步代码按顺序执行完→检查 microtask 队列（Promise.then/catch）→检查 macrotask 队列（setTimeout/setInterval）→重复。所以面对一段混合了同步 console.log 和异步 setTimeout 的代码，你必须先找出所有同步调用（它们一定先输出），再按延迟时间排异步回调的顺序。如果靠“感觉”做，遇到嵌套 setTimeout 一定会乱。

题目语境怎么用：以 2023 年那道嵌套 setTimeout 题为例：先标出同步代码——console.log('first') 和 console.log('fifth') →这两行一定最先输出（1→first, 2→fifth）。然后看异步：外层 setTimeout(..., 0) 在同步代码后执行，输出 second 并注册内层 setTimeout(..., 0)。内层在下一轮执行，输出 third 和 fourth。所以最终顺序：first, fifth, second, third, fourth。

卷面写法：第一步列出所有同步代码并标注输出顺序。第二步列出所有异步调用及其延迟。第三步按 event loop 规则排列异步执行的先后。最后写出完整输出序列并配一句解释：“同步代码优先，setTimeout 回调按延迟时间排队，延迟小的先执行，但都在所有同步代码之后”。

常见失分点：以为 setTimeout(0) 是“同步执行”或“立即执行”——它只是插队到当前同步任务之后。搞混嵌套 setTimeout 内外的执行轮次。不知道同步代码跑完后 event loop 才开始处理回调。

核心答案（先确认概念）：从左到右读，每一步返回一个对象或结果，供下一步继续调用。

为什么这是对的：CS615 考 chaining 时经常出一段代码让你解释执行顺序或输出。Chaining 的本质是“每一步的返回值作为下一步的调用对象”——所以必须从左到右读，不能跳步骤。比如 Order.findById(15).populate('items').exec() 这个链：先执行 findById 返回一个 Query 对象→链上的 populate 调这个 Query 的 populate 方法，返回 Query 对象→exec 执行查询并返回 Promise。如果你从中间开始读或者不知道每步返回什么，你就说不清最后输出什么。

题目语境怎么用：如果看到 Model.find().where('total').gt(100).sort('-date').limit(10).exec() 这样的链式代码：第一步 find() 返回 Query 对象；第二步 where() 添加过滤条件（total > 100），返回 Query；第三步 sort() 设置排序（按 date 降序），返回 Query；第四步 limit(10) 限制条数，返回 Query；第五步 exec() 执行查询，返回 Promise。每一步都在前一步返回的对象上调用方法——这就是 chaining。

卷面写法：“Chaining 就是从左到右依次调用，每步返回一个对象（通常是 this），下一步在这个对象上继续调用。读法：拆开每一步，写清楚每步的输入是什么（前一步的返回值）、输出是什么（传给下一步的对象）。过度 chaining 的问题是可读性差——调试时很难在中间插入 log 或断点。Promise 的 .then().catch() 也是 chaining：每个 .then 返回一个新 Promise。”

常见失分点：跳过中间步骤直接猜最终结果。不知道每步返回的是什么类型（有的返回 this，有的返回新对象，有的返回 Promise）。把 chaining 和 nesting（嵌套回调）混为一谈。

Accessibility 和 CSS

Accessible website 指不同能力、设备和辅助技术用户都能使用的网站。辅助技术包括屏幕阅读器、键盘导航、放大显示等。CSS 不能替代语义化 HTML，但它能显著影响可读性、焦点、响应式布局和动画敏感性。语义化 HTML 指用正确标签表达内容意义，例如标题用 heading，按钮用 button，而不是所有东西都用 div。

CSS 做法 为什么提升 accessibility
足够颜色对比度低视力用户能读清文本，避免浅灰字和低对比按钮。
清晰的 :focus 样式键盘用户能知道当前焦点在哪里。
响应式布局和可缩放字体移动端和放大页面时内容不重叠、不丢失。
不要只靠颜色传达信息色盲用户仍能通过文字、图标、边框理解状态。

SEO：white-hat 与 black-hat

SEO 是 Search Engine Optimization，也就是让网站更容易被搜索引擎理解，并更可能出现在合适搜索结果里。White-hat SEO 改善真实内容和用户体验；black-hat SEO 试图欺骗搜索引擎，例如堆砌关键词、隐藏文字、买垃圾链接，可能导致排名惩罚。

White-hat 建议 答题展开
高质量原创内容围绕用户问题提供清晰、有用、更新的内容。
语义化 HTML 和 metadata合理使用 title、description、headings、alt text。
性能和移动端体验快速加载、响应式设计、减少阻塞资源。
自然链接和站点结构清晰 URL、内部链接、sitemap，避免 keyword stuffing 和隐藏文本。
