Section 1：几乎必出（≥90%）

CS608 有四个固定考题位，以下四题的预测基于 2023 Summer、2025 Summer 的交替出题模式和教材核心知识点布局。每题 25 marks，内部子题分值已标注。

Q1: Climate.determine() BVA（25 marks）

预测：Climate.determine(int temp, int humidity)。Spec: temp 范围 -130..60，<16 需 HEAT；humidity 范围 0..100，>60 需 AC；超出范围 → ERROR。

子题：(a) [5] 为什么穷举测试不可行？ (b) [5] Value lines + input/output partitions。 (c) [15] BVA 测试用例（不含 error values）：TCI 表、selected equivalence values、test cases。

(a) 模型答案：穷举测试不可行

变量可能取值数计算
temp191-130 到 60 共 191 个整数
humidity1010 到 100 共 101 个整数
组合总数19,291191 × 101

不可行原因分两层：

  1. Test design time 不可行：近 2 万个组合，每个都需人工写出 expected result，工作量不可能在有限时间内完成。
  2. Test execution time 不可行：即使设计完毕，执行近 2 万个测试用例所需的运行时间远超合理范围。必须使用 systematic methods（如 BVA、EP、DT）来选择代表性测试。

Q2: decideWrite() JaCoCo White-box（25 marks）

预测：Filestore.decideWrite(boolean enabled, boolean exists, boolean overwrite)。JaCoCo 报告：line 28 yellow、line 32 yellow、line 33 red。

子题：(a) [15] Additional tests 实现 full BRANCH coverage。 (b) [5] 解释 "1 of 4 branches missed"（short-circuit）。 (c) [5] Black-box vs white-box 四点区别。

教师批注校准：不要把 Q2 写成一段解释。先列 Conditions/TCI/Test Cases 三张表；TCI 表只放 line number 和 test case 编号，不抄 source code；测试表列名用 Expected Results。

(a) 模型答案：Additional Tests for Full Branch Coverage

Coverage gaps 分析：

  • Line 28 yellow：if (enabled && exists) — 其中一个 branch 方向未被覆盖（例如 enabled&&exists 为 true 的分支已被测，但对应 false 的分支未走）。
  • Line 32 yellow：else if (overwrite) — 同理，某个 branch 方向缺失。
  • Line 33 red：该行从未被执行（statement coverage 0%），位于 else 链末端。

TCI Table：

TCI IDCoverage Item
TCI1enabled && exists → true branch (line 28)
TCI2enabled && exists → false branch (line 28)
TCI3!enabled && overwrite → true branch (line 32)
TCI4!enabled && overwrite → false branch → line 33

Additional Tests：

TestCovered TCIInputExpected ResultExplanation
AT1TCI1decideWrite(true, true, false)从 spec 推导enabled&&exists 为 true，进入 if-block
AT2TCI2decideWrite(true, false, false)从 spec 推导enabled&&exists 为 false，走 false 分支到 line 32
AT3TCI3decideWrite(false, false, true)从 spec 推导!enabled && overwrite 为 true，进入 else-if-block
AT4TCI4decideWrite(false, false, false)false所有条件为 false，运行到 line 33 执行 return false，使红线变绿

注意：每题 expected result 仍必须从 specification 推导，不能因为"这是补 coverage 的"就只写 input。每个 additional test 都要有 TCI + Input + Expected + Explanation 四要素。

(b) 模型答案：Short-circuit & "1 of 4 branches missed"

&& 在 bytecode 层面产生多个 conditional jump 指令，JaCoCo 按字节码插桩计数：

  1. Branch 1：!enabled → 跳转到 L1（不评估 exists）
  2. Branch 2：enabled && !exists → 跳转到 L1
  3. Branch 3：enabled && exists → 进入 true 体
  4. Branch 4：控制流从 L1 继续（隐式 branch）

如果现有测试只击中了部分 branch（例如 enabled=false 时 exists 从未被 evaluated），JaCoCo 就报 "1 of 4 branches missed"。补测的方法：提供让 missing branch 被触发的输入，使得被短路的子条件得到 evaluation。

(c) 模型答案：Black-box vs White-box 四点对比

维度Black-boxWhite-box
1. 发现的错误类型规格理解错误、缺失功能、边界/组合遗漏未执行代码中的逻辑错误、遗漏分支 bug
2. 代码变更影响不受内部重构影响，只要规格不变测试仍有效代码改一行可能导致测试失效，需维护
3. 能否先于代码编写可以——从 spec 即可设计测试（支持 TDD）必须已有代码才能分析覆盖和执行测试
4. Coverage 可测量性基于 spec 覆盖，较难量化（需人工判断）基于代码覆盖，工具可直接出数字（如 JaCoCo 百分比）

Q3: Numbers Class Context（25 marks）

预测：Numbers 类有 static method isNeg(int x) 和 instance methods setValue(int v)、isNegative()、getResult()。

子题：(a) [10] 比较 static 与 instance method 的测试写法。 (b) [15] Braking.decide() 在 class context 中的 EP 测试。

教师批注校准：Class context 题练 2018、2020-2025；state-based 题补 2016、2017。Jan 2017 Q3(b) 若已有 isShipped()/isReturned() 就用它们识别状态；Q3(c) 才要求 full signatures。

(a) 模型答案：Static vs Instance Testing

Static Method (isNeg)Instance Method (isNegative)
调用方式Numbers.isNeg(-1) — 直接一行调用需四步序列创建对象并设状态再调用
测试步骤1 步：调用 static method 并 assert 返回值4 步：new → setValue → call → observe/assert
行为依赖仅依赖传入参数（pure function）依赖对象内部 state（value 字段），由之前 setValue 设定
实例代码assertEquals(true, Numbers.isNeg(-1))Numbers n = new Numbers(); n.setValue(-1); n.checkValue(); assertTrue(n.isNegative())

核心区别：static method 是纯函数——输入直接决定输出，不涉及对象状态；instance method 是有状态方法——结果取决于对象字段的当前值，必须先通过序列调用将对象置于目标状态。

(b) 模型答案：Automated Test Pseudo-code

Random rand = new Random();

// R1: enabled=true, nettPower > 0, expect true
for (int i = 0; i < 100; i++) {
    int np = rand.nextInt(Integer.MAX_VALUE - 1) + 1;
    assertTrue("R1 failed for np=" + np, PV.exportPower(true, np));
}

// R2: enabled=true, nettPower ≤ 0, expect false
for (int i = 0; i < 100; i++) {
    int np = rand.nextInt(Integer.MAX_VALUE) * (rand.nextBoolean() ? -1 : 1);
    if (np > 0) np = 0;  // ensure ≤ 0
    assertFalse("R2 failed", PV.exportPower(true, np));
}

// R3: enabled=false, nettPower > 0, expect false
for (int i = 0; i < 100; i++) {
    int np = rand.nextInt(Integer.MAX_VALUE - 1) + 1;
    assertFalse("R3 failed", PV.exportPower(false, np));
}

// R4: enabled=false, nettPower ≤ 0, expect false
for (int i = 0; i < 100; i++) {
    int np = rand.nextInt(Integer.MAX_VALUE) * (rand.nextBoolean() ? -1 : 1);
    if (np > 0) np = 0;
    assertFalse("R4 failed", PV.exportPower(false, np));
}

(c) 模型答案：三大问题及解决方案

问题描述解决方案
1. Test Data随机生成什么范围的数据？不能完全无界随机根据 DT rules 限制每个 rule 的 nettPower 生成区间（R1: 正数, R2: ≤0），boolean cause 固定不变
2. Oracle如何判断随机输入的输出是否正确？不能调用被测函数自身DT 表作为 oracle——每条 rule 的 expected effect 预先确定（R1→true, R2-R4→false），不依赖 exportPower() 自身输出
3. Completion何时停止随机测试？跑多少算够？Coverage-based completion：每条 DT rule 命中 100 次后停止，而非固定总循环次数

isSquare() Oracle + whatSpeed() Random EP

触发条件：Q4 考 oracle 设计或随机 EP，而非 PV.exportPower。

isSquare(int x) Oracle

题目：对随机 x 判断是否为完全平方数，如何设计 oracle？

答案（不能用被测函数自身当 oracle）：

int x = rand.nextInt(0, 1000);
int r = (int) Math.sqrt(x);
boolean expected = (r * r == x) || ((r + 1) * (r + 1) == x);
assertEquals(expected, isSquare(x));

原理：用 Math.sqrt(x)（独立于被测函数的数学库）计算平方根，取整后反算验证 r² == x。同时检查 (r+1)² 以处理浮点截断问题（sqrt(25) 可能返回 4.999999）。此 oracle 完全不依赖被测 isSquare() 函数的输出。

AI-Generated Tests（可能的新考点）

触发条件：近年 AI 工具流行，教材可能新增 AI 辅助测试设计的内容。

方面内容
Advantages(1) 更快识别边界值和分区——AI 从 spec 文本中自动提取数值阈值。(2) 自动生成 coverage-gap tests——AI 分析 JaCoCo 报告并生成补测用例。(3) 大量重复劳动可自动化。
Risks(1) 可能 misinterpret specification——AI 不懂业务语义，误读条件导致错误 expected result。(2) Oracle 不可靠——AI 的 expected result 可能是"编"的而非从 spec 推导。(3) 遗漏罕见 edge case——AI 倾向于覆盖常见路径，不保证边界完备性。
Human Review 必须检查(1) Expected result 正确性——每条 AI 生成的 expected result 必须人工对照 spec 验证。(2) 边界完整性——检查所有阈值是否都被覆盖（just below/on/above）。(3) DT rule 覆盖率——确认 decision table 每条 feasible rule 都有对应 test case。

Section 3：如果换考法（50–69%）

以下情况在 2025 Autumn 样本中出现过，如果出题人轮换考法，以下预测提供快速切换的答案框架。

If Q1 changes to Braking.evaluate()（2025 Autumn 风格）

替换逻辑：同样的 BVA 结构，但 spec 换了对象。battLevel 和 bPressure 各有阈值。

快速适配步骤：

  1. 从 spec 提取两个变量的所有数值阈值（如 battLevel=20、bPressure=50）。
  2. 为每个变量画 value line 并标注 partitions（有效区间 + 错误区间）。
  3. 对每个阈值取 just below / on / just above 三个值。
  4. 按 one fault assumption 组合：测 battLevel 边界时 bPressure 取 nominal；反之亦然。
  5. 写 TCI table + Selected Values + Test Cases（格式同 Climate.determine）。
  6. 每个 test case 的 expected result 必须从 spec 的对应区间的输出 enum 值推出。

关键：无论 spec 换什么函数，BVA 表格结构不变。核心能力是：从规格文字中找到数字阈值，画出 value line，取 just below/on/above。

If Q2 changes to Statement Coverage Only（2025 Autumn 风格）

替换逻辑：题目只要求补红色 line（unexecuted statements），不要求补黄色 branch。

简化后的答题流程：

  1. 只看 JaCoCo 报告中的红色行（statement coverage = 0%）。
  2. 每个红色行追溯代码：什么条件能让控制流到达该行？
  3. 为每个红色行设计一个 input，使该条件成立。
  4. 写 expected result（依然从 spec 推出，不能省略）。
  5. 说明执行路径——input 如何让控制流经过该红线。

判分门槛更低：SC-only 题目比 BC 简单，因为不需要分析 short-circuit 和 bytecode-level branch。但 expected result 仍然不能省略——coverage 不等于 correctness。

If Q4 changes to MTBF Estimation（2025 Autumn Q4c 风格）

替换逻辑：不考 PV.exportPower 的随机 test case 设计，而考用 random testing 估计 Mean Time Between Failures (MTBF) 并讨论风险管理。

模型答案结构

1. 用随机测试估计 MTBF：

随机生成符合操作 profile 的输入序列（模拟实际用户行为模式），记录每次 failure 之间的时间/操作次数。MTBF = 总运行时间 / failure 次数。随机统计抽样能逼近真实故障率，前提是输入分布反映真实使用场景。

2. 风险管理与投资-利润曲线：

概念解释
Investment-profit curve测试投入（时间/人力/工具）与发现 bug 的价值之间的边际递减关系。初期投入回报高，后期每发现一个 bug 的成本急剧上升。
Risk management 应用在曲线上选择"最优停止点"——继续测试的预期价值 < 测试成本时停止。用 MTBF 作为停止标准之一（MTBF 达到目标值即停止）。
Random statistical sampling随机采样逼近群体特征。如果随机输入分布与操作 profile 匹配，则统计出的 MTBF 近似于真实 MTBF。采样越大，置信区间越窄。

3. 卷面写法提示：

  • 先定义 MTBF = 总运行时间 / failure 次数。
  • 说明 random testing 如何生成有代表性的输入序列。
  • 画出投资-利润曲线（文字描述：X 轴测试投入，Y 轴累计发现 bug 价值，曲线从陡升变为平缓）。
  • 结论：当边际收益低于边际成本时停止测试，MTBF 达标可作为停止条件之一。
