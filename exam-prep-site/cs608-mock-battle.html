Section 1：几乎必出（≥90%）

CS608 有四个固定考题位，以下四题的预测基于 2023 Summer、2025 Summer 的交替出题模式和教材核心知识点布局。每题 25 marks，内部子题分值已标注。

Q1: Climate.determine() BVA（25 marks）

预测：Climate.determine(int temp, int humidity)。Spec: temp 范围 -130..60，<16 需 HEAT；humidity 范围 0..100，>60 需 AC；超出范围 → ERROR。

子题：(a) [5] 为什么穷举测试不可行？ (b) [5] Value lines + input/output partitions。 (c) [15] BVA 测试用例（不含 error values）：TCI 表、selected equivalence values、test cases。

先降坡：这题到底让你做什么

这不是让你随便挑几个温度和湿度测试。题目给了两个整数输入，每个输入都有有效范围边界和业务阈值，BVA（边界值分析，Boundary Value Analysis）要你把这些危险位置系统地挑出来。

1. 先找数字

temp 的关键数字是 -130、16、60；humidity 的关键数字是 0、60、100。没有数字就没有 BVA。

2. 再判断含义

-130 和 60 是 temp 的合法端点；16 是 HEAT 开关阈值。0 和 100 是 humidity 合法端点；60 是 AC 阈值。

3. 每个边界取具体值

例如 16 的附近不是写 "near 16"，而是写 15、16、17。60 的湿度阈值则是 59、60、61。

4. 预期输出从 spec 推

temp=15, humidity=50：temp<16 所以 HEAT；humidity≤60 所以不 AC；Expected Result 是 HEAT_ONLY。

卷面思路：先用一句话说明 exhaustive testing 要 191×101 个组合，所以不可行；再画两条 value line；最后用 one fault assumption（单故障假设，每次只让一个参数靠近边界）生成测试表，并额外补 HEAT_AND_AC 的组合输出。

先搞懂这件事在干嘛

宝宝级解释：想象你是一个质检员，你要测试一台空调遥控器。遥控器上有一个温度旋钮和一个湿度旋钮。说明书说：温度低于16度要开暖气，湿度高于60%要开除湿。你要怎么测？测一次？把所有可能的温度和湿度组合都试一遍？那有一万多种组合，天都黑了。BVA 的思路是：程序员最容易在"刚好在边界上"写错代码。比如说明书写"低于16度开暖气"——程序员可能写成"低于等于16度"。所以你要重点测15度、16度、17度这三个值。15是"刚好低于"，16是"刚好等于"，17是"刚好高于"。这三个值抓住了边界上最容易出错的三个位置。

考场视角：现在把刚才的直觉翻译成考试要交的东西：你要画两条 value line（数值线，温度和湿度各一条），标出范围端点和阈值；然后列三张表——TCI 表（测试覆盖项表，我要覆盖哪些测试目标）、selected values 表（每个目标用哪个具体数字）、test cases 表（输入什么，期望输出什么）。

判分要点
Q1 的核心动作：先画 value line 找到边界和阈值，再为每个边界选 just below/on/above 三个具体值。

One fault assumption 的意思是：每次只让一个参数靠近边界，另一个参数固定在普通有效值，这样失败时知道是哪条边界导致。

(a) 模型答案：穷举测试不可行

用人话讲就是

你有一个函数。它有两个输入。温度和湿度。如果你想每一种组合都试。就得测191×101=19291次。两万次。你写得完吗。跑得完吗。就算跑完了。每一组的"应该输出什么"。你都得事先算好。这根本不现实。所以必须用聪明办法。只挑最可能出错的那几个组合。去测它们。这就是BVA和EP（等价类划分，Equivalence Partitioning）的意义。

考试时你这么写：先列每个变量的可能取值数。再算组合总数。然后从两个角度说不可行。一是设计时间不够（design time）。二是执行时间不够（execution time）。最后提一句替代方案——systematic methods。比如BVA、EP、DT（决策表，Decision Table）。

为什么这样写能拿分：这题5分。老师看三样东西。你算对了组合数（191×101=19291）。你说了设计和执行两个层面。你提到了替代方法。缺一个扣1-2分。注意——不要只写"太多组合"。必须写出具体数字19291。

变量可能取值数计算
temp191-130 到 60 共 191 个整数
humidity1010 到 100 共 101 个整数
组合总数19,291191 × 101

不可行原因分两层：

  1. Test design time 不可行：近 2 万个组合，每个都需人工写出 expected result，工作量不可能在有限时间内完成。
  2. Test execution time 不可行：即使设计完毕，执行近 2 万个测试用例所需的运行时间远超合理范围。必须使用 systematic methods（如 BVA、EP、DT）来选择代表性测试。

(b) 模型答案：Value Lines & Partitions

用人话讲就是

你要画两条数轴。一条表示温度。一条表示湿度。每条数轴上。把关键数字标出来。温度轴上有三个关键点。-130是合法最低温。16是暖气开关线。60是合法最高温。湿度轴上有三个关键点。0是合法最低湿。60是除湿开关线。100是合法最高湿。每个关键点把数轴切成几个区。每个区代表一种程序行为。画完这两条数轴。你就知道测试的重点在哪里了。

考试时你这么写：先画temp的value line。写-130——16——60。再画humidity的value line。写0——60——100。然后每个变量列一张分区表。表里有四列：分区名、范围、行为。注意——每个变量还要列出超出合法范围的ERROR区。虽然题目说"不含error values"。但(b)问要你展示input/output partitions。ERROR区属于input partition。必须写出来。

为什么这样写能拿分：(b)问只要5分。但它是(c)问的基础。value line画对了。分区表才不会漏。常见错误有两种。一是漏掉合法端点（-130、60、0、100）。二是输出分区表漏了HEAT_AND_AC那行。注意——输出分区（output partitions）必须列出四种组合。因为两个条件各有两个状态。2×2=4种输出。

(c) 模型答案：BVA 测试用例

用人话讲就是

现在你要写三张表。第一张叫TCI表（测试覆盖项，Test Coverage Items）。它回答"我要测哪些目标"。比如"temp刚好低于16"是一个目标。"humidity刚好高于60"是另一个目标。第二张叫选定值表（Selected Values）。它回答"每个目标我用什么数字去测"。同时它体现一个规则。叫单故障假设（one fault assumption）。意思是每次只让一个变量去碰边界。另一个变量保持安全值。第三张叫测试用例表（Test Cases）。它回答"具体怎么调函数。期望输出是什么"。

考试时你这么写：TCI表有14行。前10行是10个input边界目标（BV1到BV10）。后4行是4个output目标（OUT1到OUT4）。Selected Values表只有两行。一行说明测temp边界时humidity固定为50。另一行说明测humidity边界时temp固定为20。Test Cases表有11行。TC1到TC10遵循one fault assumption。TC11故意打破它。同时让temp=15和humidity=61。专门覆盖HEAT_AND_AC。

为什么这样写能拿分：15分的大题。判分看三样。TCI表有没有覆盖input和output两类目标。Selected Values表的nominal值选得对不对。Test Cases的Expected Result是向spec推导的还是瞎猜的。漏掉output TCI（OUT1-OUT4）直接扣5分。漏掉TC11扣2-3分。注意——test cases表里的Test Call列要写成函数调用的形式。比如determine(-130, 50)。不要只写数字。

用人话讲就是：为什么有TC11

单故障假设有盲区。它只让你每次动一个参数。但有些bug必须两个参数同时越过阈值才触发。比如暖气坏了。只有"温度低"加"湿度高"才会同时触发。单变量BVA测不到这种情况。所以你要额外加一条TC11。同时把temp放到15（低于16）。把humidity放到61（高于60）。这条用例专门覆盖HEAT_AND_AC。打破one fault assumption是允许的。但只在为了覆盖output partition时这样做。

判分关键：不要漏掉 temp 的有效最小值 (-130) 和有效最大值 (60) 两个端点；不要漏掉湿度端点 0 和 100；TC11 必须给出以覆盖所有四种输出。

Q2: decideWrite() JaCoCo White-box（25 marks）

预测：Filestore.decideWrite(boolean enabled, boolean exists, boolean overwrite)。JaCoCo 报告：line 28 yellow、line 32 yellow、line 33 red。

子题：(a) [15] Additional tests 实现 full BRANCH coverage。 (b) [5] 解释 "1 of 4 branches missed"（short-circuit）。 (c) [5] Black-box vs white-box 四点区别。

先降坡：白盒题先读报告，不先背定义

JaCoCo 报告已经告诉你哪里没覆盖。yellow 不是"这行错了"，而是"这行执行过，但分支方向没全走到"；red 是"这行完全没执行"。所以 additional tests 的任务是补控制流缺口。

Line 28 yellow

说明 if (enabled && exists) 的某个方向缺失。要用输入让 true 和 false 两种结果都出现。

Line 32 yellow

说明 else if (overwrite) 的某个方向缺失。要让程序真的走到 line 32，再分别让 overwrite 为 true / false。

Line 33 red

说明末端 return 从未执行。典型补法是让前面所有条件失败，例如 false,false,false。

Short-circuit 为什么会考

enabled=false 时 exists 不会被评估；因此同一个源码 if 在 bytecode 里可能有多个 branch。

卷面思路：不要写成散文。先列 Conditions table，再列 TCI table，最后列 Test Cases，且每条测试必须有 input 和 Expected Results。

教师批注校准：不要把 Q2 写成一段解释。先列 Conditions/TCI/Test Cases 三张表；TCI 表只放 line number 和 test case 编号，不抄 source code；测试表列名用 Expected Results。

先搞懂这件事在干嘛

宝宝级解释：JaCoCo 就像一个代码染色器。你跑了一遍测试后，它把代码染成三种颜色：绿色（跑了）、黄色（跑了一半，比如 if 条件只测了 true 没测 false）、红色（完全没跑）。你的任务是看报告，找出黄色和红色的行，然后设计新的输入让它们变绿。红色行最简单——写一个能让程序走到那行的输入就行。黄色行需要动脑筋——你得搞清楚是条件的哪一侧没走到。

考场视角：考场上的白盒题不是让你背覆盖率定义。卷面上你要做三件事：①先看 JaCoCo 报告，分析每行 yellow / red 是哪条分支没走到；②列三张表——Conditions table（行号和条件）、TCI table（要覆盖的测试目标）、Test Cases table（输入和预期输出）；③最后用四点对比表说清楚黑盒和白盒的区别（发现的错不一样、要不要看代码、能不能先写、覆盖能不能量化）。

判分要点
Q2 的核心动作：不是解释覆盖率概念，而是根据 JaCoCo 颜色补全未覆盖的分支。先读颜色，再写 Conditions / TCI / Test Cases 三张表。

(a) 模型答案：Additional Tests for Full Branch Coverage

用人话讲就是

你面前有一份代码染色报告。第28行是黄色。第32行是黄色。第33行是红色。第28行的条件是"开关开了而且文件存在"。如果之前只测了true的情况。那false的情况还是黄的。第32行的条件是"开关没开。但覆盖开关开了"。同样缺了一个方向。第33行是红的。说明从没被跑到。你需要给三个布尔参数。分别是enabled、exists、overwrite。一组一组地试。直到所有行都变绿。

考试时你这么写：第一步。列出三行各自缺什么。用文字说清楚。第二步。写TCI表（测试覆盖项表）。每行一个TCI编号和覆盖目标。比如TCI1覆盖第28行true分支。TCI4覆盖第33行的执行。第三步。写Test Cases表。每条测试有四个要素。TCI编号、输入参数、期望输出、为什么这条能补上缺口。

为什么这样写能拿分：15分。三分之一判给coverage gap分析。三分之一判给TCI表结构。三分之一判给test cases的完整性和expected result。注意——Expected Result必须从spec推导。不能因为"这是补覆盖的"就不写。不写Expected Result那行直接0分。

Coverage gaps 分析：

  • Line 28 yellow：if (enabled && exists) — 其中一个 branch 方向未被覆盖（例如 enabled&&exists 为 true 的分支已被测，但对应 false 的分支未走）。
  • Line 32 yellow：else if (overwrite) — 同理，某个 branch 方向缺失。
  • Line 33 red：该行从未被执行（statement coverage 0%），位于 else 链末端。

(b) 模型答案：Short-circuit & "1 of 4 branches missed"

用人话讲就是

代码里写的是enabled && exists。看起来只有一个if。但Java编译成机器码（bytecode）后。它变成了多个跳转指令。想象你走一条路。第一个路口判断"enabled是true吗"。如果不是。直接绕开。连"exists"都不看。这就是短路（short-circuit）。JaCoCo按机器码计数。所以源码里一个if。在报告里可能显示4个分支。你只测了其中3个。它就报"4个分支里漏了1个"。

考试时你这么写：列出四路分支。第一路enabled是false。跳过exists。第二路enabled是true但exists是false。第三路两个都是true。第四路是隐式分支。再说明怎么补。用不同输入让每一路都被执行。

为什么这样写能拿分：5分题。老师看你是否理解"源码一个if≠bytecode一个branch"。你列出四路分支就拿到主要分。再用一行话说明short-circuit的含义。

&& 在 bytecode 层面产生多个 conditional jump 指令，JaCoCo 按字节码插桩计数：

  1. Branch 1：!enabled → 跳转到 L1（不评估 exists）
  2. Branch 2：enabled && !exists → 跳转到 L1
  3. Branch 3：enabled && exists → 进入 true 体
  4. Branch 4：控制流从 L1 继续（隐式 branch）

如果现有测试只击中了部分 branch（例如 enabled=false 时 exists 从未被 evaluated），JaCoCo 就报 "1 of 4 branches missed"。补测的方法：提供让 missing branch 被触发的输入，使得被短路的子条件得到 evaluation。

(c) 模型答案：Black-box vs White-box 四点对比

用人话讲就是

黑盒测试（BB）就像你不用打开手机壳。只看说明书就测功能。白盒测试（WB）就像你打开手机壳。看着电路板找哪里可能断线。黑盒能发现"说明书写错了"这种问题。白盒能发现"某行代码从来没跑过"这种问题。黑盒可以写代码前就写好。白盒必须等代码写完了才能测。黑盒很难说"我测了百分之多少"。白盒有工具告诉你具体覆盖率数字。

考试时你这么写：列四点对比表。每点包括黑盒和白盒两列。四点分别是：发现的错误类型、代码变更影响、能否先于代码编写、coverage可测量性。表后附一句：两者互补。不能互相替代。

为什么这样写能拿分：5分题。四点各1分。最后一句"两者互补"再1分。每点必须有具体的对比内容。不能只写"黑盒不看代码白盒看代码"——太泛了。注意——用词要区分：黑盒叫spec-based。白盒叫code-based。

维度Black-boxWhite-box
1. 发现的错误类型规格理解错误、缺失功能、边界/组合遗漏未执行代码中的逻辑错误、遗漏分支 bug
2. 代码变更影响不受内部重构影响，只要规格不变测试仍有效代码改一行可能导致测试失效，需维护
3. 能否先于代码编写可以——从 spec 即可设计测试（支持 TDD）必须已有代码才能分析覆盖和执行测试
4. Coverage 可测量性基于 spec 覆盖，较难量化（需人工判断）基于代码覆盖，工具可直接出数字（如 JaCoCo 百分比）

(c) 模型答案：三大问题及解决方案

用人话讲就是

随机测试有三个坑。第一个坑是数据问题（data）。你不能满世界乱生数字。因为大部分数字测不到新东西。你必须按规则限定生成范围。第二个坑是裁判问题（oracle）。计算机不知道怎么判断"对还是错"。你不能让它调用被测函数自己判断自己。你必须从规格说明或决策表里预先写出正确答案。第三个坑是停止问题（completion）。你不知道测多久算够。你得定一个覆盖目标。比如每条规则被命中100次。

考试时你这么写：三列表。问题列、描述列、解决方案列。每行的解决方案必须具体到本题。不能写泛泛的"用spec判断"。要写"R1→true, R2-R4→false"。因为阅卷人要看你会不会把抽象概念落实到具体函数上。

为什么这样写能拿分：6分。三个问题各2分。每个问题的解决方案必须是针对PV.exportPower的具体方案。不是泛泛的定义。如果解决方案里没有提到具体函数名和具体规则编号。会扣分。注意——oracle的解决方案中。必须写明"不依赖被测函数自身"。这是核心考点。

boilerSetting() Decision Table（Q1 变体）

触发条件：如果 Q1 不考 BVA 而考 Decision Table。

Spec：boilerSetting(int t, boolean isOn)。输出 NONE / LOW / HIGH。

先搞懂：BVA 和 Decision Table 什么时候用谁

宝宝级解释：BVA 和 DT 的区别就像两种不同的测试策略。BVA 适合"一个参数的不同区间"——比如温度低于16度开暖气、高于60度关暖气，边界是单个数字，你测那个数字的前后左右就行。DT 适合"多个条件组合起来决定结果"——比如温度高不高 而且 开关有没有开，这两个条件组合起来才决定到底加热还是不加热。boilerSetting 就是典型的 DT 场景：温度有三个区间（极低/低温/高温），开关有两个状态（开/关）。你不可能单独测温度或单独测开关——"开关关了温度再低也没用"这种逻辑只有 DT 能覆盖。

考场翻译：拿到 spec 先看有没有 boolean 开关（isOn）配合数值区间。如果有，八成是 DT。做法三步：①列出所有 causes（条件）和 effects（输出）；②算出全部组合 2^4=16，标记哪些组合不可能同时成立（比如温度不可能既低于1度又高于25度），划掉这些 infeasible（不可行的）；③剩下的 feasible rules 每条做一个 test case，选具体数值塞进去。

用人话讲就是：causes/effects/infeasible 的完整走法

你要搞懂DT的三个核心步骤。第一步。列出原因和结果。原因就是你输入的各个条件。比如"温度低于1度吗"是一个原因。结果就是程序产出的行动。比如"不加热"是一个结果。第二步。找出哪些原因不能同时成立。比如温度不可能同时低于1度又高于25度。这种叫互斥条件（mutually exclusive）。把含有互斥条件的规则列划掉。标上INFEASIBLE。第三步。剩下可用的规则。每条选一个具体温度值。搭配开关状态。写出期望输出。比如R1是"极低温+开启→LOW"。你挑t=-50。isOn=true。期望LOW。

isSquare() Oracle + whatSpeed() Random EP

触发条件：Q4 考 oracle 设计或随机 EP，而非 PV.exportPower。

先降坡：isSquare() 题先别急着写代码

isSquare(x) 的功能听起来很简单：如果 x 是完全平方数就返回 true，否则返回 false。真正的考点不是平方根，而是全自动随机测试为什么会卡在三个问题上。你可以把它想成一个测试机器人：机器人能乱生成数字，但它不知道该生成哪些数字、不知道标准答案，也不知道什么时候该停。

1. Data problem：机器人会漏掉稀有输入

在很大的 int 范围里，完全平方数很少。纯随机可能生成 5、-88、10243 这种非平方数很多次，却一次都没生成 4、9、8281。这样 false path 被跑烂了，true path 仍然没测到。

2. Oracle problem：机器人不知道标准答案

如果随机得到 8281，脚本要写 assertEquals(expected, isSquare(8281))。问题是 expected 从哪里来？不能直接调用被测函数自己，也不能靠猜。

3. Completion problem：机器人不知道什么时候测完

跑 100000 次没失败，不代表测完。它可能从没测到 0、负数、最大整数、最大可平方边界、溢出附近这些关键点。

4. 结构化解法：EP + oracle + quota

把输入分区：negative、zero、positive non-square、positive square、large boundary。每个分区随机抽样，并给每个分区设完成数量，而不是全范围盲抽。

一句话总答案：fully automated random testing 的难点不是"能不能随机生成输入"，而是能不能保证生成到有价值的输入、能不能独立判断 expected result、能不能定义合理的停止条件。

用人话讲就是：为什么不能直接调isSquare自己当裁判

假设你要测一个判断平方数的函数。你写了一个测试脚本。随机生成了数字8281。你把这数字同时喂给被测函数和被当作裁判的函数。如果裁判和被测量是同一个函数。那不管函数代码错成什么样。裁判都会说"对"。就等于自己给自己批卷子。永远100分。真正的oracle（裁判机制）必须独立。它要从数学定义出发。用其他方法验证。比如用数学库计算平方根。再反向验证。这样就算被测函数有bug。裁判能独立发现。

三大问题的考场写法：先讲翻车，再讲修法

用人话讲就是：数据问题

你随机抽数字。在-21亿到+21亿的范围里。完全平方数只有大约46340个。占全部整数的十万分之一。你抽1000次。大概率一次平方数都抽不到。这意味着isSquare的true分支从来没跑过。你测了1000次其实只测了false分支。这叫数据问题。解法不是继续盲抽。而是主动制造平方数。比如随机选k。然后算x=k*k。保证x一定是平方数。

用人话讲就是：裁判问题

你生成了8281。怎么判断isSquare(8281)应该返回true还是false。你不能调用isSquare自己来判断。因为你测的就是它。如果它有bug。它会说false。然后你拿这个false当"正确答案"。test就永远pass。这叫裁判问题。解法是用独立的数学方法。Math.sqrt算出平方根。取整后反算。如果r*r等于x。那x就是平方数。这个判断完全不依赖被测函数。

用人话讲就是：停止问题

你跑了100000次都没报错。是不是就测完了。不一定。可能你从没测过x=0。也可能从没测过负数。还可能没测过46340²这种边界大数。这叫停止问题。解法不是跑更多次。而是提前定义好"每个输入类型至少被测了N次"。比如负数100次。零至少1次。平方数100次。非平方数100次。边界值全部到位。全部达标了才算测完。

Test Data Problem 纯随机在巨大整数范围里大多抽到非平方数，true path 可能长期不执行。 Pure random generation may fail to hit sparse input classes. For isSquare(), perfect squares are rare in a large integer range, so many random tests may only exercise the false branch. 按 EP 分区生成：负数、0、正平方数、正非平方数、边界值；平方数分区用 x=k*k 主动制造。
Test Oracle Problem 脚本抽到 8281 时，不会天然知道 expected 是 true。 The automated script can generate inputs, but it does not inherently know the correct expected output. An independent oracle is required to decide whether each random input is a square. 用独立数学性质做 oracle：r=floor(sqrt(x)) 后检查 r*r==x 或 (r+1)*(r+1)==x。
Test Completion Problem 跑 100000 次没失败，也可能没测到 0、负数、MAX、溢出边界。 Random testing has no built-in stopping criterion. isSquare(int x) Oracle

题目：对随机 x 判断是否为完全平方数，如何设计 oracle？

答案（不能用被测函数自身当 oracle）：

int x = rand.nextInt(0, 1000);
int r = (int) Math.sqrt(x);
boolean expected = (r * r == x) || ((r + 1) * (r + 1) == x);
assertEquals(expected, isSquare(x));

原理：用 Math.sqrt(x)（独立于被测函数的数学库）计算平方根，取整后反算验证 r² == x。同时检查 (r+1)² 以处理浮点截断问题（sqrt(25) 可能返回 4.999999）。此 oracle 完全不依赖被测 isSquare() 函数的输出。

为什么这比"再写一个 isSquare2"更好：考试里不要说"我再实现一遍同样的 algorithm"。更稳的说法是：oracle 来自独立的数学定义，完全平方数满足 ∃k. k*k=x；测试代码用库函数 sqrt 找到最接近的 k，再用乘法反查。这和被测函数的实现路径分离，能独立产生 expected result。

边界提醒：若 x<0，expected 必须是 false；若 x=0，expected 通常是 true，因为 0=0²；若接近 Integer.MAX_VALUE，要避免 (r+1)*(r+1) 溢出，真实代码里可用 long 做乘法。

替换逻辑：不考 PV.exportPower 的随机 test case 设计，而考用 random testing 估计 Mean Time Between Failures (MTBF) 并讨论风险管理。

用人话讲就是

MTBF就是"平均多久坏一次"。比如你随机测试一个系统。跑了100小时。坏了5次。那MTBF就是20小时。你要用随机测试来估算MTBF。随机生成符合用户真实使用习惯的输入。记录每次坏掉的时间间隔。然后算平均。MTBF越大说明系统越稳定。但测试投入不是无限的。你测到一定程度。再测下去花的钱比能发现的bug的价值还大。这时候就该停了。这就是投资-利润曲线告诉你的道理。边际收益递减。到某一点就该收手。

注意——MTBF题不要求你列DT规则或写伪代码。它更偏概念和论述。重点是说清楚三点。MTBF怎么算。随机测试怎么帮我们估算MTBF。什么时候该停（投资-利润曲线的拐点）。

模型答案结构

1. 用随机测试估计 MTBF：

随机生成符合操作 profile 的输入序列（模拟实际用户行为模式），记录每次 failure 之间的时间/操作次数。MTBF = 总运行时间 / failure 次数。随机统计抽样能逼近真实故障率，前提是输入分布反映真实使用场景。

2. 风险管理与投资-利润曲线：

概念解释
Investment-profit curve测试投入（时间/人力/工具）与发现 bug 的价值之间的边际递减关系。初期投入回报高，后期每发现一个 bug 的成本急剧上升。
Risk management 应用在曲线上选择"最优停止点"——继续测试的预期价值 < 测试成本时停止。用 MTBF 作为停止标准之一（MTBF 达到目标值即停止）。
Random statistical sampling随机采样逼近群体特征。如果随机输入分布与操作 profile 匹配，则统计出的 MTBF 近似于真实 MTBF。采样越大，置信区间越窄。

3. 卷面写法提示：

  • 先定义 MTBF = 总运行时间 / failure 次数。
  • 说明 random testing 如何生成有代表性的输入序列。
  • 画出投资-利润曲线（文字描述：X 轴测试投入，Y 轴累计发现 bug 价值，曲线从陡升变为平缓）。
  • 结论：当边际收益低于边际成本时停止测试，MTBF 达标可作为停止条件之一。
