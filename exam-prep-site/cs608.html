内容理解题（先做 3 题）

核心答案（先确认概念）：因为它们校准的是“怎么得分”，不是单独的新知识点。

为什么这是对的：CS608 很多分不是丢在概念不会，而是丢在答案形状不符合题目要求。例如你知道 line 29 要用 h<0 覆盖，但如果把整段代码抄进 TCI table、把 expected 写成随口的 output、没有 line number，判卷人很难按 marking scheme 找到你覆盖了什么。教师批注把这些隐性规则变成显性规则：白盒题用表格、术语用 Expected Results、TCI 和 test case 分开、OO 题先看是否要求 signatures。

题目语境怎么用：做任何 past paper 前先问：这题要求的交付物是什么？是 Conditions table、TCI table、Test Cases table，还是 signatures，还是调用序列？先定交付物，再填内容。

卷面写法：把批注转成检查清单：表格列名是否对；是否引用 line number；expected result 是否从 Return Value/spec 推导；题目没问 signatures 时是否避免跑题。

常见失分点：把教师批注当“老师偏好”而不是判分格式。实际上这些批注对应的是 marking visibility：老师必须能快速看到你满足了题目每个要求。

核心答案（先确认概念）：三张表回答三个不同问题：为什么能到达、要覆盖什么、用哪个测试覆盖。

为什么这是对的：Conditions table 是逻辑分析，说明某条红线或 missed branch 的进入条件。例如 line 37 要执行，前面的 h<0、h<2、h<5、h<13 都必须为 false，而 h<25 为 true，所以条件可合并成 13<=h<25。TCI table 是覆盖承诺，说明 SC1 对应 line 37。Test Cases table 才是具体数据，说明用 h=19 覆盖 SC1，Expected Results 是 MODERATE。混在一起会让分析、目标和数据互相遮住。

题目语境怎么用：看到 JaCoCo 红线/黄线，先不要急着给输入。先写“要到达这行，条件必须是什么”。条件想清楚后，具体 input 就只是从区间里选一个代表值。

卷面写法：Conditions table：C1 line 29 h<0；C2 line 37 13<=h<25；C3 line 41 60<=h<140。TCI table：SC1 line 29 T1；SC2 line 37 T2；SC3 line 41 T3。Test Cases table：T1 -10 ERROR；T2 19 MODERATE；T3 100 HIGH。

常见失分点：TCI 表里塞 source code、Test Cases 表没有 Expected Results、或者只有输入没有解释它为什么能到达目标行。

核心答案（先确认概念）：题目问 identify state 时优先用已有 isShipped()/isReturned()；题目明确要求 develop full signatures 时才写完整签名。

为什么这是对的：state-based 或 OO class context 题不是见到状态就自动写 signatures。Jan 2017 的批注说明了题目粒度差异：Q3(b) 只问如何识别状态，所以如果类已经给了 isShipped()、isReturned() 这种 public query method，答案应围绕这些方法说明如何判断对象处于 shipped/returned 状态。Q3(c) 的语境不同：它假设这些 query methods 未定义，所以才要求你补 full signatures，让状态可被测试观察。也就是说，signatures 是解决“没有可观察状态 API”的工具，不是所有状态题的默认答案。

题目语境怎么用：读题时圈关键词：identify、use、develop、signature、not defined。如果题目说 isShipped/isReturned are available，就用它们；如果题目说 they are not defined 且 asks for full signatures，再写 boolean isShipped()、boolean isReturned() 这类方法签名。

卷面写法：Q3(b) 写：“Use isShipped() and isReturned() as observable state queries after invoking the relevant methods.” Q3(c) 写：“Add signatures such as public boolean isShipped() and public boolean isReturned() so tests can observe the state.”

常见失分点：题目没问 signatures 却把主要篇幅写成 API 设计；或者题目明确说没有 isXXX 方法时，仍然假设它们存在。

考试题目题（再做 5 题）

标准答案（先写结论）：line 29 用 h=-10 期望 ERROR；line 37 用 h=19 期望 MODERATE；line 41 用 h=100 期望 HIGH。

为什么这是对的：题面给出的红线是未执行语句。line 29 是 return ERROR，它只在第一层条件 h<0 为 true 时执行，所以 -10 是最直接的代表值。line 37 是 return MODERATE，前面的 ERROR/CALM/SMOOTH/SLIGHT 都必须跳过，到 h<25 才成立，因此合法区间是 13<=h<25，选 19 在区间中间，避免边界歧义。line 41 是 return HIGH，要先跳过 ROUGH，满足 60<=h<140，选 100 是清楚的中间值。

题目语境怎么用：这是 full statement coverage，所以目标是让每条红色 return 语句至少执行一次。黄色 line 28、36、40 是你追溯条件的位置，不是最终 expected result 的列。

卷面写法：Test Cases 表写三行：T1 input h=-10, Expected Results ERROR, covers SC1 line 29；T2 input h=19, Expected Results MODERATE, covers SC2 line 37；T3 input h=100, Expected Results HIGH, covers SC3 line 41。

常见失分点：把 line 41 的 HIGH 区间写成 h>140。题面说 h>=140 是 PHENOMENAL，所以 HIGH 必须是 60<=h<140。

标准答案（先写结论）：列用 TCI、Line Number、Test Case；不要把 source code 放进 TCI table。

为什么这是对的：TCI table 是覆盖目标到测试编号的映射，不是代码摘抄区。老师批注特别指出：只需要 line number，不需要把每行 source code 写进去。原因很实际：题目和代码已经给出，判卷人要看的不是你能抄代码，而是你能把每个 coverage item 绑定到某个追加测试。把 source code 塞进 TCI table 会让表格变长、变乱，还容易掩盖缺少 Test Case 编号的问题。

题目语境怎么用：Waves.categorise 的 TCI table 可以写：SC1 | 29 | T1；SC2 | 37 | T2；SC3 | 41 | T3。条件本身放在 Conditions table，具体输入放在 Test Cases table。

卷面写法：三表分工：Conditions table 用 ID/Line Number/Condition；TCI table 用 TCI/Line Number/Test Case；Test Cases table 用 Test Case/Input/Expected Results。

常见失分点：TCI 表写成“if h<0 return ERROR”这种代码复述；或者用 Expected Output 这个非 IEEE 术语，而不是 Expected Results。

标准答案（先写结论）：Q3(b) 尽量用已有 isShipped()/isReturned() 识别状态；Q3(c) 在这些方法未定义时才写 full signatures，而且只要 signatures，不要 test cases。

为什么这是对的：老师批注意味着这两小题的任务边界不同。Q3(b) 的重点是状态识别：对象执行某些操作后，如何通过 public query method 判断它是否处于 shipped 或 returned 状态。Q3(c) 的重点是 API 补足：如果没有这些 query method，测试无法观察状态，所以你要提出方法签名。它没有要求你继续写 test cases。把 Q3(c) 写成完整测试用例会浪费时间，也可能漏掉“modified signatures as asked for”的关键得分点。

题目语境怎么用：先看题干是否说 “signatures”。有这个词，再写 public boolean isShipped() 等；没有这个词，就把注意力放在用现有方法观察状态。

卷面写法：Q3(b)：After calling ship(), call isShipped() and expect true; after returnItem(), call isReturned() and expect true。Q3(c)：Add signatures public boolean isShipped(), public boolean isReturned() to make these states observable。

常见失分点：Q3(c) 写 test cases 但没写 modified signatures；Q3(b) 忽略已给的 query methods，直接发明新 API。

标准答案（先写结论）：不会作为 2026 examination paper 的题目；它作为额外材料帮助理解 white-box structural testing 的最强形式。

为什么这是对的：老师明确回答 “No”。这给复习优先级画了边界：你不需要为 All Paths Testing 单独准备一套 past-paper 答题模板，也不需要把大量时间花在枚举所有路径。它的价值是帮助你理解覆盖标准强弱关系：statement coverage 看行，branch coverage 看分支，path-level thinking 更强但成本更高。这个理解可以反过来加强 Q2 中“为什么 BC 比 SC 更强、为什么 full path 不现实”的解释。

题目语境怎么用：如果 Q2 问 white-box testing 比较，All Paths Testing 可作为背景一句话：it is stronger but usually impractical because path count explodes。不要把它写成主要答案。

卷面写法：主答案仍围绕 SC/BC/JaCoCo/additional tests。AllPathTesting 只在比较覆盖强度时作为辅助解释，不展开成独立大题。

常见失分点：看到 “strongest white-box structural testing” 就误以为必考，把复习时间从 SC/BC 和 JaCoCo 题挪走。

标准答案（先写结论）：Testing in class context 看 2018、2020、2021、2022、2023、2024、2025；state-based testing 看 2016、2017。

为什么这是对的：这条批注直接给了 Q3 的练习地图。Class context 题训练的是 create-set-call-observe-assert 调用序列、accessor/mutator 识别、static vs instance 区分。State-based testing 题训练的是对象状态、状态查询、signatures 或 transition 的表达。两类题都属于 OO，但考查动作不同：前者更像“怎么写测试步骤”，后者更像“怎么观察和命名状态”。如果混着练，容易把所有 Q3 都写成一个模板，结果遇到 signatures 或 state identification 就跑偏。

题目语境怎么用：第一轮先练 2023-2025 的 class context，因为它和当前题型页最贴近；第二轮补 2016/2017 state-based，专门训练 isXXX、state query、signature 边界。

卷面写法：Class context：对象创建 + setter + method under test + getter/return + assert。State-based：状态名 + 可观察 query/signature + transition/action + expected state。

常见失分点：只练近三年，把 2016/2017 的 state-based 忽略；或者 state-based 题里只写调用序列，不说明如何判断对象已经进入目标状态。

1. 软件测试基础词汇

测试不是“随便试几个输入”。考试里的测试设计要回答：为什么选这些测试、它们覆盖了什么、期望输出是什么、是否还有遗漏。

词 从零理解 考试里怎么用
Specification规格说明。它告诉你方法应该做什么，而不是代码实际怎么写。黑盒测试从 specification 出发。
Test case一次具体测试：输入是什么，期望输出是什么。表格中写 TC1、TC2...
Expected result根据规格推出来的正确结果。不能只写输入，必须写 expected。

White-box看代码结构和覆盖率。Statement coverage、branch coverage、JaCoCo。

内容理解题（先做 3 题）

核心答案（先确认概念）：Test data 是输入值；test case 还包括执行动作和 expected result。

为什么这是对的：Test data 只是孤立的数字，你不知道它要测什么、如何判断结果。Test case 必须包含 expected result，因为测试的本质是对比——把实际输出和期望输出比较才能判定通过/失败。比如给你 (15, 50) 两个数，没有 expected 就没有任何意义；加上 "expected HEAT_ONLY" 并知道它是测 "temp just below 16" 的边界，才算一个可执行的测试。表格里每个 TC 要写的不是"我试过这个值"，而是"我证实了这个行为"。

题目语境怎么用：无论 BVA、DT、white-box 还是 random 题，任何只列输入值的行都不及格——你只写了 test data 而非 test case。判卷人看到只有输入没有 expected result + TCI 的行，直接判定不完整。

卷面写法：卷面每个 test case 写成：输入值 + 执行动作（如调用 determine(15,50)）+ expected result + 覆盖的 TCI 编号。例如：TC1: call determine(15,50) → expected HEAT_ONLY, covers TCI-1 (temp just below 16)。

常见失分点：把 test data 当成 test case 交给考官。漏写 expected result 会让整行无效。考卷里没有 expected 的那一行不会得分。

核心答案（先确认概念）：从 specification 或 oracle 来，不能从程序实际输出倒推。

为什么这是对的：如果你把程序当前输出当作 expected result，你测的不是"程序是否符合规格"，而是"程序是否一直输出相同结果"——这等于把 bug 固化。Expected result 必须从外部独立来源推导。例如测 Climate.determine(15, 50)，expected HEAT_ONLY 是根据规格中"temp < 16 且 humidity ≤ 60 时仅供暖"推导出来的，不是跑一遍程序看返回什么。

题目语境怎么用：Climate 类的 expected 来自规格中 temp/humidity 各区间对应的枚举输出；PV.exportPower 的 expected 来自 Decision Table 中每条 rule 的 effects；isSquare 的 expected 来自数学恒等式 r*r==x。

卷面写法：每个 test case 的 expected result 后面多加一句来源说明：如"根据规格 rule R2：temp<16, humidity≤60 → HEAT_ONLY"。random testing 中 oracle 也用同样方式——先定 expected，再在对应分区内随机选值。

常见失分点：用被测方法的实际返回值当 expected 写回卷面——这是循环论证，测试失去意义。expected 写得太模糊如"应该返回正确的值"也不合格，必须写具体枚举或数值。

核心答案（先确认概念）：一个抽象测试目标，例如某个边界、分区、decision rule 或 branch。

为什么这是对的：Coverage item (TCI) 回答的是"我这个测试要覆盖什么"——它是测试的标的而非手段。例如 "temp = 16 on boundary" 是一个 TCI，而 "call determine(16, 50)" 是覆盖它的 test case。考试要求每个 test case 至少覆盖一个新 TCI，否则就是重复覆盖浪费。没有 TCI 概念，你的测试就是随机撒网，无法证明"该测的都测了"。

题目语境怎么用：BVA 题中 TCI 来自每个输入的边界值 just below/on/above；DT 题中 TCI 来自每条 feasible rule；white-box 题中 TCI 来自红色未执行的 statement 或黄色未走到的 branch。

卷面写法：先列 TCI 表（编号 + 描述：如 TCI-1: temp just below 16），再列 Test Cases 表并在每行写 covered TCI 编号。最后总结一句"所有 TCI 均被覆盖"。每个 TCI 必须被至少一个 test case 覆盖是判分关键。

常见失分点：把 TCI 和 test case 混在一起写成同一张表。或者列了 TCI 但没有 test case 来覆盖它们。在 BVA 题里把 error partition 的 TCI 混入有效值 test case 表。

考试题目题（再做 5 题）

标准答案（先写结论）：Test design time 和 test execution time；Java array 最大长度让输入组合巨大。

为什么这是对的：Exhaustive testing 不可行不仅是因为跑不完（execution time），更因为设计不完（design time）——每个输入组合都要人工写出 expected result。maxv 的输入是 int[]，假设数组长度 n=10 且每个元素范围 int 全空间，组合数是 (2^32)^10，不要说跑完，连列出 1% 的 expected result 都不可能。真题常要求同时提到两种时间才给满分，只提 execution time 会扣分。

题目语境怎么用：2023 Summer Q1(a) 直接问"why exhaustive testing infeasible"，答案必须包含 design time + execution time 双重论证。可以用 Java array 最大长度或 int 取值空间来算爆炸量级。

卷面写法：先写结论"两种时间都不可行"，再分两点展开：(1) design time：输入组合数量爆炸，每个组合都要人工设计 expected result；(2) execution time：即使能设计出来，实际运行所有组合也需要天文数字的时间。

常见失分点：只提"输入太多"而没有区分 design 和 execution 两个维度。或者只从 execution time 角度说"跑不完"，漏了 design time 维度被扣一半分。

标准答案（先写结论）：TCI 是覆盖目标，例如 temp just below 16；Test Case 是具体调用，例如 determine(15, 50) expected HEAT_ONLY。

为什么这是对的：Climate.determine 的规格定义了 temp=16 和 humidity=60 两个阈值，BVA 要求测 each threshold 的 just below/on/above。TCI 说"我要测 temp just below 16"，这是抽象目标；Test Case 说"我用 determine(15,50) 来达成这个目标"，这是具体执行。如果混成一张表，判卷人无法确认你是不是清楚每个测试覆盖了哪个目标——重复覆盖和遗漏都看不出来。分开两张表就是为了让 coverage 可审计。

题目语境怎么用：2025 Summer Q1(c) 直接要求回答 TCI 和 Test Case 的区别。举例用 Climate 题中的 temp=16 边界：TCI 写"temp on 16 boundary"，Test Case 写"temperature=16, humidity=50, expected HEAT_ONLY"。

卷面写法：TCI 表：编号 + 覆盖目标描述（如 TCI-2: temp on 16）。Test Cases 表：TC编号 + 具体输入 + expected result + covered TCI。两张表必须分开，不能合并。

常见失分点：把 TCI 当 test case 写（如"TCI-1: determine(15,50)"），或把 test case 当 TCI 写（如"TC1: temp just below 16"）。这两者层级不同，混淆了说明没有理解测试设计的基本框架。

标准答案（先写结论）：根据 Decision Table 判断 enabled/nettPower 对应 true/false，而不是随机猜结果。

为什么这是对的：PV.exportPower 的规格有两层逻辑：enabled 决定系统是否激活，nettPower 决定是否有电力可输出。随机测试不能"随机断言"——你必须先根据规格建 Decision Table：enabled=true 且 nettPower>0 时 expected true，enabled=false 或 nettPower≤0 时 expected false。随机只用于在符合条件的数据区间内生成数据值，而 expected result 由规格 rules 预先确定。这就是 oracle：一个不依赖被测代码的独立判断标准。

题目语境怎么用：2025 Summer Q4 问 random testing 的 oracle for PV.exportPower。答案要点：oracle = 根据 spec 建立的 Decision Table 中每条 rule 的 effects。

卷面写法：先列出 4 条 feasible rules 作为 oracle 来源；然后在 Random Test Cases 表中每行写 fixed boolean causes + random numeric range + expected result（来自对应 rule）。伪代码中写 assert(actual == expectedFromDT)。

常见失分点：说"oracle 就是多跑几次看结果对不对"——这等于没有 oracle。或者把被测方法自身的输出当 oracle，陷入循环验证。没有写明"expected 来自 DT rule"是常见扣分点。

标准答案（先写结论）：给一个能走到 line 33 的 concrete input，写 expected result，并说明它补 statement coverage。

为什么这是对的：JaCoCo 红色的行表示该语句从未被执行过——意味着之前的测试没有一个输入能到达这里。要补 coverage，你需要反向追踪控制流：line 33 在哪个 if/else 分支里？它的前置条件是什么？例如 line 33 在 if (a && b) 的 true 分支中，你就要给一个让 a 和 b 同时为 true 的输入。仅仅说"加一个测试"不够，必须写具体输入值和对应的 expected result，因为新增测试不只要执行到代码，还要验证它正确。

题目语境怎么用：2025 decideWrite 题中 line 33 红色，先在 JaCoCo 报告中确认它属于哪个条件分支，再设计输入使程序沿该路径执行。例如如果 line 33 是 buffer overflow path，需要给超长字符串作为输入。

卷面写法：写 coverage item（statement at line 33 not executed）→ 写具体输入值及如何调用 → 写 expected result → 说明此 test 补了该 statement。格式：Additional test AT1: input=..., expected=..., covers previously unexecuted line 33。

常见失分点：只写"增加一个测试覆盖 line 33"而没有给具体输入值和 expected result。或者给了输入但没有说明为什么这个输入能走到 line 33（前置条件分析缺失）。

标准答案（先写结论）：短路条件编译成 bytecode-level branches；additional test 要让之前没 evaluated 的子条件被 evaluated。

为什么这是对的：Java 源码中 if (a && b) 看起来像 1 个条件 2 个分支，但编译到 bytecode 后变为 4 个分支路径：a=false 时走一条（b 不被 evaluated），a=true 且 b=false 走第二条，a=true 且 b=true 走第三条，加上隐式控制流第四条。JaCoCo 统计的是 bytecode 层面的 branch，所以显示 "4 branches"。要补满这 4 个分支的测试，你需要设计让 a 单独为 false 的案例（此时 b 不会被 evaluate），以及 a 为 true 后 b 分别为 false/true 的案例。

题目语境怎么用：2025 optimized 题中 line 51 有复合条件，JaCoCo 报告标出 branch 覆盖不完整。回答必须提到 bytecode-level branch coverage，解释短路求值导致源码层面看不出来的分支。

卷面写法：先指出 line 51 的复合条件在 bytecode 层拆成 4 个 branch → 说明当前哪些 branch missed → 给出覆盖每个 missed branch 的具体输入 → 写 expected result。

常见失分点：只从源码层面分析分支，不提 bytecode-level 或 JaCoCo 的工作机制。或者把 short-circuit 条件和普通 if-else 混为一谈。漏了"有些子条件从未被 evaluated"这个关键点。

2. Q1：黑盒测试用规格挑代表性输入

Q1 通常先问 exhaustive testing 为什么不可行，再要求 BVA 或 Decision Table。Exhaustive testing 是把所有输入组合全测一遍；它通常不可行，因为测试数量爆炸，而且每个测试还要设计 expected result。

BVA：边界附近最容易出错

Boundary Value Analysis 的直觉是程序员常把边界写错，例如 < 和 <=。所以 BVA 不测所有值，而是测每个合法分区边缘的值。题目如果说 without errors，测试用例不要放非法输入，但分析表仍可说明错误分区。

BVA 表格 作用 例子
Value line / partitions把每个输入切成规格认为不同的区间。humidity: 0..60 不需要 AC，61..100 需要 AC。
TCI table列出每个要覆盖的边界值和输出。60、61、AC_ONLY。
Selected values写明每个分区可用的具体值。代表值 50，边界值 60、61。
Test Cases每个测试输入、expected、covered TCI。TC3: temp=16, humidity=61, expected=AC_ONLY。

DT：多个条件组合起来决定输出

Decision Table 用于测组合。Cause 是输入条件，例如 isOn 或 t<1；effect 是输出，例如 HIGH。每个可行 rule 都是一个 TCI，也通常对应一个 test case。

DT 答题顺序：定义 causes 和 effects；列 candidate combinations；划掉 infeasible combinations 并解释；把 feasible combinations 写成 decision table；每个 rule 写入 TCI table；最后写 selected data values 和 test cases。

Q1 的详细样章见 CS608 Q1 关于 BVA / DT 和黑盒测试。

内容理解题（先做 3 题）

核心答案（先确认概念）：它主要从 specification 生成代表性输入，不依赖代码内部结构。

为什么这是对的：Q1 所有子问题（exhaustive、BVA、DT）的共同起点是读规格说明书，而不是读代码。BVA 从规格里的"temp < 16"找边界值，DT 从规格里的条件组合推 rules——没有一步需要看实现代码。这意味着测试可以在代码还没写完时就设计，且测试不随内部重构而失效（只要规格不变）。这也是为什么 Q2（白盒）在 Q1 之后——先用黑盒验证行为正确，再用白盒补遗漏代码路径。

题目语境怎么用：Climate.determine 题的"temp<16→HEAT_ONLY"来自规格原文，不是来自 if 语句。boilerSetting 题的 DT 条件来自规格的区间划分。遇到"justify your approach"时要明确指出"based on specification, not code"。

卷面写法：每个 test case 的 expected result 旁标注来源："per specification, rule X"。如果是 BVA 要注明边界值来自规格的哪个阈值；如果是 DT 要注明每条 rule 来自哪几个条件的组合。

常见失分点：把 black-box 和 white-box 混淆，例如在 Q1 里讨论代码行覆盖。或者 expected result 不自觉地从代码实现反推而非从规格推导，这在考试中会被判定为方法错误。

核心答案（先确认概念）：Boundary、minimum、maximum、threshold、less than、at least、between。

为什么这是对的：BVA 的底层假设是程序员在边界条件上最容易出错——写 < 还是 <=、off-by-one 错误、数组越界等问题都集中在阈值附近。因此规格里每出现 "less than 16" 就要取 15（below）和 16（on），出现 "at least 60" 就要取 60（on）和 61（above）。这些关键词就是 BVA 的触发器：看到它们就知道要画 value line 分区间、定边界值。

题目语境怎么用：Climate 题中 temp=16（less than 16 vs >=16）和 humidity=60（<=60 vs >60）。扫一遍规格标出所有数字阈值和比较词，每个都是一个边界点。

卷面写法：先列出所有阈值关键词及其来源，再为每个阈值画 value line，然后写 TCI 表（just below/on/above 各一个），最后 Selected Values 表从中各挑一个具体值。

常见失分点：以为 BVA 就是把边界附近所有值列一遍而不画区间。或者对一个阈值只取了 below 和 on 漏了 above，三个点必须齐全。把 valid 和 error 的值混进同一张有效测试表也是常见错误。

核心答案（先确认概念）：Multiple conditions、combinations、causes、effects、rules、infeasible。

为什么这是对的：当一个方法的输出不是由单个输入决定的，而是由多个条件共同决定时，BVA 不够用——你必须用 DT 穷举条件组合。例如 boilerSetting 的输出 NONE/LOW/HIGH 取决于"温度在哪个区间"AND"isOn 是否 true"，单独测每个条件不能保证所有组合被覆盖。关键词 "multiple conditions" 意味着有 >=2 个独立输入条件，"infeasible" 意味着有些组合在逻辑上不可能。

题目语境怎么用：boilerSetting 题有 4 个温度条件 + 1 个 isOn 条件；PV.exportPower 有 enabled + nettPower>0。先列出所有独立 cause（boolean 条件），再列 effects（可能输出），再填组合表。

卷面写法：标准格式：Causes 行（每个 cause 一行，T/F）→ Effects 行 → Rules 列（每列一个可行组合）→ 为每个 rule 设 TCI → 写 Test Cases。infeasible 的 rule 要标注"infeasible because..."并解释理由。

常见失分点：把 DT 写成普通输入列表（如"输入 50→输出 HIGH"），没有 cause/effect/rule 结构。或者漏标 infeasible rule 被扣分。把所有组合都当 feasible 全列一遍而不检查逻辑自洽。

标准答案（先写结论）：温度/湿度超出规格允许范围时应列 error partitions，并映射到 ERROR 输出。

为什么这是对的：Climate 的规格隐含了输入范围（如温度 0-50、湿度 0-100），超出这些范围的值是非法输入。测试不能假设非法输入不会被调用——实际系统中传感器故障或外部干扰可能产生越界值。把 error partitions 单独列出（如 temp<0、temp>50），每个至少一个 test case 并 expected ERROR，这是测试设计的完整性要求。2025 Summer 明确要求列出这些错误类，漏了会扣分。

为什么这是对的：boilerSetting 的 DT 规则中有"temperature very low"和"temperature very high"这样的抽象条件，但这不能直接当输入——你需要具体数值。题目给的 -50 和 +50 恰好是远低于/高于正常工作区间的值，用作这些 DT rule 的 concrete data value。这意味着命题人帮你做了"选代表值"这一步，你只需把 -50 填入对应 rule 的 test case 输入列。

标准答案（先写结论）：每个 test case 要覆盖新的 input TCI 或 output TCI；已经覆盖的边界不要重复用另一个等价 test case。

为什么这是对的：Duplicate coverage 指的是两个不同 test case 覆盖了完全相同的 TCI 且期望输出一样——这浪费测试资源。例如你已用 determine(15, 50) 覆盖了"temp just below 16"和"humidity≤60 的代表值"两个 TCI，就没必要再加 determine(14, 45) 测同样的东西。避免方法是每新增一个 test case 时检查：它是否至少带来一个之前未被覆盖的 TCI？如果是，有用；如果只是换了个等价值但不增加任何新覆盖，就是 duplicate。

题目语境怎么用：Climate.determine 中 temp 有 3 个边界 TCI（below/on/above 16），humidity 有 3 个（below/on/above 60）。组合设计 test case 时，确保每个 test case 至少覆盖一个之前没测过的 TCI。

卷面写法：在 Test Cases 表最后一列写 "New TCI covered"，如果某行是首次覆盖某个 TCI 就标出来。最后检查：所有 TCI 都在某行的 "New TCI covered" 列中出现过。

常见失分点：列了 8 个 test case 但只覆盖了 3 个不同的 TCI——其余 5 个都是重复等价值。或者没有在表里标注每个 test case 的新增覆盖，导致判卷人无法判断是否重复。

标准答案（先写结论）：用 public setter/constructor 设置 brightness 和 override，调用 decide()，再用 getter 检查 power state。

为什么这是对的：Lighting.decide 是 instance method，输出取决于对象内部状态（brightness、override 等 private field）。你不能直接设 private field，必须通过公开 API：用 setBrightness() 设置亮度、setOverride() 设置覆盖模式，然后调用 decide() 触发决策逻辑，最后用 getPower() 观察结果。这是 in-class context testing 的标准流程，确保测试和实际使用场景一致——用户也是通过公开接口操作对象。

题目语境怎么用：2023 Lighting 题需要先列出 accessor methods：setBrightness(int)、setOverride(boolean)、getPower()。测试流程：new → setBrightness(80) → setOverride(false) → decide() → assert getPower()。

常见失分点：在 test case 里写 "brightness=80" 而不是 "l.setBrightness(80)"——直接赋值暗示访问 private field。或者只调用 decide() 不先设置状态，导致测试依赖对象的默认初始值。

标准答案（先写结论）：enabled=true 且 nettPower>0 时随机取正数 expected true；enabled=false 或 nettPower<=0 expected false。

为什么这是对的：PV.exportPower 有两个输入维度：enabled (boolean) 和 nettPower (int)。Decision Table 产生 4 条可行 rule。转成 random test 时，boolean cause 固定不变，nettPower 在符合该 rule 条件的数值区间内随机生成——如 rule 1 中 nettPower 在 [1, MAX_INT] 随机，rule 2 中在 [MIN_INT, 0] 随机。这样既有随机覆盖又不丢失 expected correctness 判断。

3. Q2：白盒测试用覆盖率补遗漏代码

白盒测试不是从规格开始，而是看代码或覆盖率报告。JaCoCo 是 Java coverage tool，它会告诉你哪些行没执行、哪些分支没走。绿色通常表示覆盖，黄色表示部分覆盖，红色表示没覆盖。

概念 从零理解 考试动作
Statement coverage (SC)每条可执行语句至少执行一次。找红色未执行行，把每个未执行 statement 当 TCI。
Branch coverage (BC)每个条件的 true/false 分支都至少走一次。找黄色 diamond 的 missed branch，设计输入走过去。
Null else代码没有显式 else，但条件 false 时仍有一条“什么都不做”的分支。BC 题要明确写出这种 missed branch。
Short-circuit&& 或 || 中后面的条件可能不会被求值。JaCoCo 可能按 bytecode 统计多个 branches。

Q2 答题流程

  1. 读 coverage report：哪些行红、哪些行黄、diamond 写了几 of 几 branches missed。
  2. 列 unexecuted statements 或 untaken branches。
  3. 反推要走到它们，输入条件必须是什么。
  4. 写 TCI table：每个未覆盖 statement/branch 一个 TCI。
  5. 写 Test Cases table：输入、expected、covered TCI。
  6. 最后说明这些是 additional tests，用来补充已有 black-box tests。

内容理解题（先做 3 题）

核心答案（先确认概念）：Additional tests，并解释它们补了哪些 statements 或 branches。

为什么这是对的：Q2 的命题逻辑是"已有黑盒测试覆盖了一部分代码，现在给你 JaCoCo 报告，让你补上遗漏的"。所以你的输出不是从零写测试，而是 additive——只写能让红色/黄色变绿的新测试。每个 additional test 必须对应 JaCoCo 报告中的一个具体缺口（如 "line 33 statement not covered" 或 "branch false of if-condition at line 51 not taken"），并给出 input + expected。不加解释就无法证明这个测试确实补了缺口。

题目语境怎么用：2025 decideWrite 题给出 JaCoCo 截图，标出红色 line 33 和黄色 line 51。你的任务是为 line 33 写 1 个 additional test，为 line 51 的 missed branch(s) 写 1 个 additional test。

卷面写法：格式：Additional Test AT1 → Coverage Item: statement at line 33 → Input: (具体值) → Expected: (规格推导) → 说明此输入如何走到 line 33。每个 additional test 单独一段。

常见失分点：只写 test case 不写它对应哪个 coverage item。或者把已有的黑盒 test case 重写一遍当 additional test 交，没有新增任何覆盖。

核心答案（先确认概念）：BC 通常更强，因为它要求每个 decision 的 true 和 false 都被覆盖。

为什么这是对的：Statement coverage 只要求每行执行过，但它不关心条件如何取值。例如 if (x > 0) { doA(); }，如果你只测了 x=5，statement coverage 100%（两行都执行了），但 branch coverage 只有 50%——false 分支从未走过。这意味着可能有 bug 藏在 false 分支的路径中未被发现。BC 能暴露 SC 覆盖不到的"条件另一侧"缺陷，所以 BC 是更强的覆盖标准。

题目语境怎么用：2025 decideWrite 题中 JaCoCo 显示 diamond 的 "1 of 2 branches missed" 或 "3 of 4 branches missed"，这要求你找到那个 missed branch 并设计输入走到它。

卷面写法：如果题目问 SC vs BC 比较，写：(1) SC=每行至少执行一次，(2) BC=每个条件的 T/F 都走过，(3) BC 更强因为包含所有 SC 且额外暴露未走分支，(4) 举例说明 SC 100% 但 BC 不到 100% 的场景。

常见失分点：说"SC 和 BC 一样强"——这是错的。或者说"BC 更强所以测 BC 就够了"——忽略了 SC 能发现的一些行级问题。

核心答案（先确认概念）：复合条件可能提前停止，导致某些子条件根本没有被 evaluated。

为什么这是对的：Java 的 && 和 || 有短路语义：if (a && b) 中如果 a 为 false，b 根本不会被 evaluate——直接跳过。这意味着即使你的测试走了这个 if 语句的 false 分支，b 的值从未被"执行"过，JaCoCo 在 bytecode 层面会记录 b 相关的 branch 未被覆盖。这就是为什么源码看上去只有一个 if，JaCoCo 却报了 4 branches——每个子条件的 true/false 各算一个独立分支。

题目语境怎么用：2025 optimized 题 line 51 含 a&&b，JaCoCo 报 4 branches。如果当前测试是 a=false，b 没被 evaluate → b 的 branches 仍未覆盖。需要补 a=true, b=false 和 a=true, b=true 的测试。

卷面写法：说明复合条件的短路层级 → 列出每个子条件的 evaluate/missed 状态 → 为每个 missed evaluation 设计输入 → 写 expected result → 解释为什么这个输入能让被短路的子条件被 evaluate。

常见失分点：只从源码层面数 if-else 分支（如"这个 if 只有 true/false 两个分支"），不提 bytecode 层面因短路造成的额外分支。

4. Q3：类内测试要按对象状态调用方法

Conventional testing 常测 static method：给输入，立刻拿返回值。In-class context testing 测 instance method：先创建对象，通过 setter 设置对象状态，调用方法改变对象，再通过 getter 取结果并断言。

EP in class context

EP 是 Equivalence Partitioning：把输入分成规格认为行为相同的区。Q3 的 EP 题还要先识别 accessor methods。Accessor 就是 get/set 方法，用来读写对象内部属性。

  1. 找 attributes，例如 override、power、needCooling。
  2. 找 accessor methods，例如 setOverride()、getPower()。
  3. 给数值参数画 value line，例如 temperature、dangerLevel、lux。
  4. 列 input/output EP 和 TCI。
  5. 写 test cases，必须包含 method call sequence。

典型 test case 写法：create object；call setter for the relevant state；call the method under test with selected input；call getter to obtain actual result；assert expected equals actual；record covered TCI.

内容理解题（先做 3 题）

核心答案（先确认概念）：对象方法依赖字段和调用历史，必须先让对象进入目标状态。

为什么这是对的：Instance method 不同于 static method——它的输出不是纯函数式的，而是取决于对象当前状态。例如 Braking.decide() 的返回值取决于 dangerLevel、override、emergencyBrake 三个字段的当前值。如果你不先设置这些字段，对象处于默认构造状态（如 override=false, dangerLevel=0），你只能测一种组合。要覆盖所有 EP partition，必须主动用 setter 把对象驱动到每个目标状态再调用被测方法。这就是"class context"的含义：测试要考虑对象的上下文状态。

为什么这是对的：对象的 private field 在测试代码中不可直接访问（也不应该用反射去读），所以你需要 accessor methods（即 getter/setter）来间接操作和观察状态。Setter 用于在测试前把对象调到目标状态（如 b.setDangerLevel(5)），Getter 用于在测试后验证方法是否产生了预期的状态变化（如 b.getEmergencyBrake() 返回 true 说明 decide() 触发了紧急制动）。Q3 的 analysis 阶段第一步就是"identify accessor methods"——找不出 accessor 就无法设计和验证测试。

常见失分点：below/on/above 的逻辑搞反了——比如 temp just below 16 写了 17。或者湿度边界用 60/61/62 而非 59/60/61（below 应该是 threshold-1=59）。

5. Q4：随机测试要解决三件事

Random testing 不是“乱测”。考试里的随机测试通常基于已经人工设计好的 TCI，把具体数据值从固定值改成随机生成，但仍然控制在对应分区内。

问题 从零理解 考试解决方法
Test Data Problem随机数可能都落在同一类，没测到关键分区。为每个 EP/DT rule 写 min/max 或 criteria，用 rand(min,max) 控制范围。
Test Oracle Problem随机输入很多，怎么知道 expected result?先由人工 TCI/分区确定 expected，再在该分区内随机选值。
Test Completion Problem随机测试理论上可以无限跑，何时停?每个 TCI 运行固定次数，或达到覆盖/预算/时间标准。

随机测试表格

常见表格是 Equivalence Value Criteria Table 和 Random EP Test Cases Table。前者写每个分区的随机生成条件，后者写 test id、随机范围、固定 boolean、expected。

Automated random testing outline: define TCI from EP/DT; define random criteria for each TCI; use a data provider to pass min/max or criteria into the test method; generate random values inside those criteria; run the method; compare against the expected result already determined from the TCI; repeat enough times per TCI.

如果题目问 rand(int min, int max)，要说明 inclusive range，并避免越界。Java 中简单 int 可写成 min + random.nextInt(max - min + 1)，但 long 范围更大时要更小心，讲义中提醒过 overflow 和边界问题。

过关问答：先内容理解，再 past/sample paper 题，悬停或点击显示答案

内容理解题（先做 3 题）

核心答案（先确认概念）：仍然要设计数据分布、oracle 和停止条件，否则无法判断结果和覆盖风险。

为什么这是对的：纯随机意味着从整个输入空间均匀随机取——但这会导致测试极度低效：大部分随机值落在"正常"分区，边界值几乎不会被抽到，error partition 可能一个样本都没有。CS608 的 random testing 是"分区导向随机"：先用 EP/DT 划分输入空间，再在每个分区内部随机生成数据。这样既保持随机性的好处（发现未预料的边界附近 bug），又确保每个关键分区都被充分测试。所以它不是"放弃设计纯靠运气"，而是"设计好分区框架后，用随机做数据填充"。

题目语境怎么用：isSquare 题不是从全空间随机，而是按 square/non-square 两个 EP 分区各自随机。PV.exportPower 不是全范围随机 nettPower，而是按 4 条 DT rule 各自的数值约束各自随机。

卷面写法：先写 TCI/EP/DT 分区设计（人工完成），再为每个分区定义随机生成条件（如 rand(1, 10000)），然后写为每个分区跑多少次。这三段缺一不可。

常见失分点：写"随机生成 1000 个输入跑"而不说明在哪些分区生成、每个分区多少、如何判断结果——这就是纯随机，考试中不得分。

核心答案（先确认概念）：测试数量、时间预算、覆盖目标、失败率稳定或风险优先级。

为什么这是对的：随机测试理论上可以无限运行，必须定义什么时候"够"。Completion 回答的是"停止条件"。(1) 按数量：每个 EP partition 跑 100 次或 1000 次——最常用。(2) 按覆盖：所有 TCI 都被覆盖到至少 N 次。(3) 按时间/预算：跑了 10 分钟就停。(4) 按收敛：bug 发现率稳定就停。考试中最稳妥的回答是"每个 partition 固定次数 + 覆盖率检查"，因为这是可量化、可检查的停止标准。

题目语境怎么用：2023 isSquare：为 square partition 和 non-square partition 各生成 1000 个随机数后停止。不要写 "run until enough" 这种不可操作的条件。

卷面写法：在 random testing 伪代码或描述最后写停止条件："Stop after 100 random values have been generated and tested for each TCI"。必须是可量化的条件。

常见失分点：写"randomly stop"或"when feel enough"——不可量化，不得分。或者忘了写 completion 只写了 data 和 oracle 两个问题。随机测试三问题缺一不可。

核心答案（先确认概念）：检查 specification 是否符合、oracle 是否正确、关键边界和风险是否覆盖。

为什么这是对的：AI 可以快速生成大量测试代码，但它不理解你的具体规格和业务逻辑。它可能：(1) 误读了 spec，生成了不对的 expected result；(2) 遗漏了关键边界——AI 习惯取中间值而忽略边界；(3) oracle 不可信——如果 AI 用被测代码本身作为 oracle，测试就失效了。因此人必须检查：每个 expected result 是否真的来自规格、所有边界和 partition 是否都被覆盖、oracle 是否独立于被测代码。AI 是加速器，不是替代品。

题目语境怎么用：讲义 24-using-ai.pdf 专门讨论了 AI-generated tests。如果考试中出现类似问题，回答重点：验证 spec 理解、检查边界覆盖、确认 oracle 独立性。

卷面写法：分点列出 human review checklist：1) Does each expected result match the specification? 2) Are all boundaries/partitions covered? 3) Is the oracle independent of the code under test?

常见失分点：说"AI 生成的测试不需要检查"——这是绝对错误的。或者说"只要 AI 生成的测试能跑就行"——能跑不等于正确。

考试题目题（再做 5 题）

标准答案（先写结论）：每条 rule 固定 boolean cause；例如 rule enabled=true/nettPower>0 时随机取 1..max，expected true；nettPower<=0 时随机取负数或 0，expected false。

为什么这是对的：PV.exportPower 的 4 条 DT rules 转 random test cases 的规则：boolean cause（enabled）保持不变，数值型 cause（nettPower）在该 rule 定义的数值区间内随机。Rule 1: enabled 固定 true，nettPower 用 rand(1, 10000)，expected true。Rule 2: enabled 固定 true，nettPower 用 rand(-10000, 0)，expected false。Rule 3/4: enabled 固定 false。这样保证了每个 rule 至少被一个随机 test case 覆盖，且 oracle 可直接从 DT effects 获取。

题目语境怎么用：2025 Summer Q4 要求 "from DT rules to random test cases"。格式：先写 4 条 DT rules 的原貌，再为每条 rule 设计 1 个 random test case。

卷面写法：Random Test Cases 表：Test ID | DT Rule | enabled (fixed) | nettPower range | Expected | Stop condition。或用伪代码写循环。

常见失分点：把 enabled 也放进了随机范围——这会导致无法确认 expected 对应哪条 rule。或者 4 条 rule 只随机测了 2 条，遗漏了另外 2 条。

为什么这是对的：isSquare 有 2 个主要 EP：square numbers (expected true) 和 non-square numbers (expected false)。Completion 可以定义为每个 partition 跑固定次数（如各 1000 次），确保两个 partition 都被充分测试且运行可预测。也可以定义为"所有 TCI 至少被覆盖 N 次"——这种以覆盖为导向的停止条件更贴近考试要求的 coverage reasoning。关键是条件必须可量化、可检查。

题目语境怎么用：2023 Summer Q4 直接问 completion problem for isSquare。答案：为 square EP 和 non-square EP 各随机生成 1000 个测试值，运行完后停止。

卷面写法：伪代码中明确写停止条件："Stop after testing 1000 random values in square partition AND 1000 random values in non-square partition" 或在循环条件中写 for (int i = 0; i < 1000; i++)。

常见失分点：写"run until you feel confident"——不可量化，不得分。或者只为一个 partition 写了次数，另一个没写。

标准答案（先写结论）：覆盖 temp 的 0..50 partition，同时结合其他 boolean/input EP 和 expected OFF。

为什么这是对的：whatSpeed 的规格将温度输入分成了多个区间，0..50 是其中一个 partition（对应风扇 OFF 行为）。rand(0,50) 的含义是：在这个 partition 内随机生成温度值进行测试。它覆盖的是该温度区间的代表行为——配合其他条件（如 isOn=false），expected result 应该是 OFF。rand(0,50) 不是"测 0 到 50 之间任意一个值"，而是在这个区间内通过多次随机采样来探索是否所有值都符合预期行为。

题目语境怎么用：2023 whatSpeed 的 Random EP Test Cases 表有一行写 rand(0,50)。需要解释它对应 temp partition "0 to 50 inclusive"，结合其他固定条件后 expected OFF。

卷面写法：在 Random EP Test Cases 表中每行：Test ID + fixed conditions + rand range for variable inputs + expected result + covered TCI/partition。rand 范围必须写准确，如 rand(0,50) inclusive。

常见失分点：只写 rand(0,50) 不说明它覆盖哪个 partition 和对应什么 expected。或者 rand 范围搞错了 partition 边界。

标准答案（先写结论）：围绕 temp=16 取 15/16/17，围绕 humidity=60 取 59/60/61，并写 expected enum。

为什么这是对的：Climate.determine 的 BVA 考点是两个阈值 temp=16 和 humidity=60。BVA 规则是每个边界取 just below、on、just above 三个值。Temp 边界：below=15, on=16, above=17。Humidity 边界：below=59, on=60, above=61。这些值不是随便挑的——just below 取 threshold-1，just above 取 threshold+1，因为离阈值最近的非边界值最能暴露 off-by-one 错误（如代码里的 <16 写成 <=16）。然后在 Test Cases 中交叉使用这些值组合出不同的 expected enum。

常见失分点：below/on/above 的逻辑搞反了——比如 temp just below 16 写了 17。或者湿度边界用 60/61/62 而非 59/60/61（below 应该是 threshold-1=59）。

标准答案（先写结论）：它组合温度区间和 isOn，effects 是 NONE/LOW/HIGH，每条可行 rule 要有 test case。

为什么这是对的：boilerSetting 的输出涉及 4 个温度区间和 1 个 boolean isOn 条件的交叉。单独测每个温度区间（BVA）可以验证温度分区本身，但无法验证"温度在高温区间 AND 开关打开"这种组合场景——而实际 bug 往往就藏在条件交互中。DT 是专门处理多条件组合的方法：把每个条件作为 cause，输出作为 effect，穷举所有 cause 组合，排除 infeasible 的，剩下的每条 feasible rule 设计一个 test case。

题目语境怎么用：boilerSetting 的 causes：C1(t<-1), C2(-1≤t<1), C3(1≤t<10), C4(t≥10)——互斥。C5(isOn)。Effects: LOW, HIGH, NONE。Rules 为可行组合。

卷面写法：随机测试通用模板：(1) TCI/EP/DT 分区列表 → (2) 每个分区的 Random Criteria → (3) Expected Result per partition → (4) Completion condition → (5) 伪代码或 Random Test Cases 表。

常见失分点：isSquare 练完 EP 随机后，做 PV.exportPower 时忘了 DT rules 里 boolean cause 要固定——两类随机测试的输入生成逻辑不同：EP 是全范围随机，DT 是部分固定部分随机。

标准答案（先写结论）：短路条件编译成 bytecode-level branches；additional test 要让之前没 evaluated 的子条件被 evaluated。

为什么这是对的：Java 源码中 if (a && b) 看起来像 1 个条件 2 个分支，但编译到 bytecode 后变为 4 个分支路径：a=false 时走一条（b 不被 evaluated），a=true 且 b=false 走第二条，a=true 且 b=true 走第三条，加上隐式控制流第四条。JaCoCo 统计的是 bytecode 层面的 branch，所以显示 "4 branches"。要补满这 4 个分支的测试，你需要设计让 a 单独为 false 的案例（此时 b 不会被 evaluate），以及 a 为 true 后 b 分别为 false/true 的案例。

题目语境怎么用：2025 optimized 题中 line 51 有复合条件，JaCoCo 报告标出 branch 覆盖不完整。回答必须提到 bytecode-level branch coverage，解释短路求值导致源码层面看不出来的分支。

卷面写法：先指出 line 51 的复合条件在 bytecode 层拆成 4 个 branch → 说明当前哪些 branch missed → 给出覆盖每个 missed branch 的具体输入 → 写 expected result。

常见失分点：只从源码层面分析分支，不提 bytecode-level 或 JaCoCo 的工作机制。或者把 short-circuit 条件和普通 if-else 混为一谈。漏了"有些子条件从未被 evaluated"这个关键点。

标准答案（先写结论）：Static 直接 Numbers.isNeg(-1)；instance 要 create object、set value、call method、get result。

为什么这是对的：Numbers 类中 isNeg 可能有两种形态。Static 版本：行为完全由参数决定，调用时不需要对象实例——Numbers.isNeg(-1) 直接返回 true。Instance 版本：isNeg 读取的是对象的内部字段 value（通过 setValue 设置），所以测试必须 Numbers n = new Numbers() → n.setValue(-1) → boolean result = n.isNeg() → assertTrue(result)。两种形态的核心区别在于：static 是无状态单次调用，instance 是有状态且依赖对象上下文的调用——这两种测试格式不能互换。

题目语境怎么用：2025 Numbers 题要求同时展示 static 和 instance 两种测试方式。Static test case：直接类名调用 + 一行 expected。Instance test case：完整的 get/set 调用链。

卷面写法：分开两段写：Static Testing: Numbers.isNeg(-1) → true, Numbers.isNeg(0) → false, Numbers.isNeg(5) → false。Instance Testing: Numbers n = new Numbers(); n.setValue(-1); assertTrue(n.isNeg());。

常见失分点：两种测试用同样的调用格式——比如 instance 版也写成 Numbers.isNeg(-1) 或者 static 版也 new 了一个对象。这暴露了对 static vs instance 语义的不理解。

7. CS608 答案检查清单

  1. 每个测试设计题都要有 TCI table 和 Test Cases table。
  2. BVA 不等于测所有组合；DT 不等于只测边界。
  3. White-box 题要从 coverage report 反推输入条件。
  4. Branch coverage 题不要漏 null else 和 short-circuit branches。
  5. Class context 题必须写完整调用顺序。
  6. Random testing 题必须回答 data、oracle、completion 三个问题。
  7. 最后 review：每个 TCI 被覆盖，每个 test case 至少贡献一个新覆盖项。

过关问答：先内容理解，再 past/sample paper 题，悬停或点击显示答案

内容理解题（先做 3 题）

核心答案（先确认概念）：漏 expected result 或 coverage explanation。只给输入不够。

为什么这是对的：CS608 判分逻辑不是"写了测试就给分"，而是"每个测试是否可验证"。一个测试是否可验证取决于两个要素：expected result（让我知道正确输出是什么）和 coverage explanation（让我知道这个测试覆盖了什么目标）。缺了 expected result，判卷人无法判断你的测试是否通过；缺了 coverage explanation，判卷人无法判断你的测试是否有必要（可能是在重复覆盖）。两道门槛缺任何一道，该 test case 可能直接不得分。

题目语境怎么用：交卷前逐行检查：每个 test case 是否有(1)具体输入(2)执行动作(3)expected result(4)covered TCI。四者缺一则该行不完整。white-box 的 additional test 还要有第(5)项：控制流分析。

卷面写法：给自己做一个 checklist：☐ 每行有 expected result ☐ 每行有 covered TCI ☐ 所有 TCI 都被覆盖 ☐ 没有两个 test case 覆盖完全相同 TCI 集合。交卷前逐项打勾。

常见失分点：写出了漂亮的 TCI 表和输入值，但在 Test Cases 表里忘了写 expected result 列——这个遗漏特别常见且一扣就是大面积失分。

核心答案（先确认概念）：TCI、data values、test cases、expected result 是否一一对应。

为什么这是对的：黑盒题（BVA 或 DT）的答案是一个链条：TCI 表列出所有覆盖目标 → Selected Values 表为每个目标分配具体值 → Test Cases 表用这些值组合出具体测试并附 expected result。检查这个链条是否完整的方法是"反向追踪"：随便挑一个 TCI，能不能在 Test Cases 中找到至少一行覆盖它？随便挑一个 test case，它的 expected result 能不能追溯到 specification？随便挑一个 data value，它是否对应某个 TCI？如果这三问都能答"是"，表格就完整。

题目语境怎么用：Climate BVA：6 个边界 TCI → 检查 6 个是否都在 Test Cases 的 Covered TCI 列中出现过。DT rules：每条 feasible rule 对应一个 TCI → 检查每个 TCI 是否都有 test case。

卷面写法：在答案最后加一行 "Coverage check: TCI-1 to TCI-N all covered by at least one test case. No duplicate coverage. All expected results derived from specification." 向判卷人展示你做了自检。

常见失分点：TCI 表列了 8 个目标但 Test Cases 只写了 5 行——有 3 个 TCI 没有被覆盖。或者 TCI 表少列了某个边界/规则，导致 coverage 不完整。

核心答案（先确认概念）：是否明确说出新增测试覆盖了哪条未覆盖语句或分支。

为什么这是对的：白盒题的核心交付物是 additional tests that improve coverage。判卷人需要看到三个连接：(1) JaCoCo 报告的红色/黄色行号 → (2) 你设计的 input → (3) 这个 input 如何触发之前未执行的语句/分支。连接断了就是无效的 additional test。例如你写 "Additional test: input = 'abc'" 但没有说明 'abc' 能走到 line 33 是因为它触发了哪个 if 的哪个分支——判卷人无法验证。

题目语境怎么用：decideWrite line 33 red → 检查你是否写了：(1) line 33 是哪个 if 的 true body (2) 需要什么条件走到这个 if 的 true 分支 (3) 你的 input 满足这些条件吗 (4) expected result 是什么。四问都在答案中有对应文字才算完整。

卷面写法：每个 additional test 一个段落，包含：Coverage Item (line X, statement/branch) → Control flow analysis → Input → Expected result → 确认覆盖。不能跳步。

常见失分点：写了 input 和 expected 但没有分析 control flow——判卷人看不出你的 input 是怎么"走到"那个红色行的。

考试题目题（再做 5 题）

标准答案（先写结论）：少 TCI、selected equivalence values 或 Test Cases 任一表都会让设计不完整。

为什么这是对的：Climate.determine 是 BVA 题，标准输出至少 3 张表。(1) TCI Table：列出每个边界的覆盖目标——没有这张表，判卷人不知道你打算测哪些东西。(2) Selected Equivalence Values Table：把 TCI 映射到具体数值——没有这张表，TCI 只是空话。(3) Test Cases Table：按具体输入组合写出 test case + expected result + covered TCI——没有这张表等于没有实际测试。三张表各有各的作用，少了任何一张都会导致链条断裂。

题目语境怎么用：Climate BVA 答案必须包含：(1) Input Partitions / Value Line 分析 (2) TCI 表 (3) Selected Values 表 (4) Test Cases 表。即使题目只要求"design test cases"，这四部分也都要有。

卷面写法：不要合表——每张表独立列出，标清楚标题。格式参考 Q1 样章中的 Climate complete example。

常见失分点：把 TCI 和 Test Cases 合成一张表——虽然信息都有了但不符合预期的结构，可能被扣格式分。或者漏了 Selected Values 直接跳到 Test Cases。

标准答案（先写结论）：少解释该 test 触发哪个 untaken branch/statement，不只是给输入。

为什么这是对的：decideWrite 是白盒题，你写的不是"全新测试"而是"补充测试"——每个 additional test 存在的唯一理由就是它覆盖了之前没覆盖的东西。如果你只写"AT1: input='abc', expected='xyz'"，判卷人不知道这个测试和已有的黑盒测试有什么区别。必须加一句如"This test covers the previously unexecuted statement at line 33 by providing an input that makes condition X true." 这句话就是 additional test 的"存在证明"。

题目语境怎么用：decideWrite 至少需要为每个红色/黄色区域写一个 additional test。每个 additional test 必须包含一句以 "This test covers..." 或 "This input triggers..." 开头的解释。

卷面写法：Additional Test 格式：AT1 | Coverage Item: statement at line 33 | Input: (value) | Expected: (value) | Explanation: This input makes condition X at line 30 true, so execution enters the if-block and reaches line 33.

常见失分点：写了 input 和 expected 但没写 "which untaken branch/statement does this cover"——这句话是白盒题的得分关键。

标准答案（先写结论）：少 random value generation criteria 或 oracle/expected result，就不能说明随机测试有效。

为什么这是对的：PV.exportPower random test 需要同时具备三要素才能成立：(1) Random value generation criteria——nettPower 在什么范围内随机？enabled 是固定还是随机？(2) Oracle——如何判断每次随机调用的结果是否正确？（用 DT rules 的 effects）(3) Completion condition——跑多少次停？少了 criteria，随机就是失控的；少了 oracle，测试就是盲目的；少了 completion，测试永远跑不完。考试中这三个问题就是 random testing 三问题的具体化。

题目语境怎么用：PV.exportPower random tests 检查：(1) Criteria: enabled fixed per rule, nettPower rand(min,max) per rule。(2) Oracle: expected from DT rule effects。(3) Completion: e.g. 100 iterations per rule。三者在答案中都要出现。

卷面写法：用三段式结构：(1) Random Data Criteria: [对每个 rule 写 rand 范围] (2) Oracle: expected result = [引用 DT] (3) Completion: stop after [具体条件]。三段缺一不可。

常见失分点：只写了 random criteria 和 completion 但没写 oracle——等于默认"跑完就算测试通过了"，这不算测试。

标准答案（先写结论）：它组合温度区间和 isOn，effects 是 NONE/LOW/HIGH，每条可行 rule 要有 test case。

为什么这是对的：boilerSetting 的输出涉及 4 个温度区间和 1 个 boolean isOn 条件的交叉。单独测每个温度区间（BVA）可以验证温度分区本身，但无法验证"温度在高温区间 AND 开关打开"这种组合场景——而实际 bug 往往就藏在条件交互中。DT 是专门处理多条件组合的方法：把每个条件作为 cause，输出作为 effect，穷举所有 cause 组合，排除 infeasible 的，剩下的每条 feasible rule 设计一个 test case。

题目语境怎么用：boilerSetting 的 causes：C1(t<-1), C2(-1≤t<1), C3(1≤t<10), C4(t≥10)——互斥。C5(isOn)。Effects: LOW, HIGH, NONE。Rules 为可行组合。

卷面写法：Causes 表 + Effects 表 + Rule 矩阵（标出 infeasible）→ TCI → Test Cases。注意互斥的 cause 要标出来，避免 infeasible 组合混入可行表。

常见失分点：把互斥的温度区间当成独立 cause，导致大量 infeasible 组合未标注。或者没有解释"为什么选 DT 而不选 BVA"——直接跳到表格不加 justification 会被扣分。

标准答案（先写结论）：给一个能走到 line 33 的 concrete input，写 expected result，并说明它补 statement coverage。

为什么这是对的：JaCoCo 红色的行表示该语句从未被执行过——意味着之前的测试没有一个输入能到达这里。要补 coverage，你需要反向追踪控制流：line 33 在哪个 if/else 分支里？它的前置条件是什么？例如 line 33 在 if (a && b) 的 true 分支中，你就要给一个让 a 和 b 同时为 true 的输入。仅仅说"加一个测试"不够，必须写具体输入值和对应的 expected result，因为新增测试不只要执行到代码，还要验证它正确。

题目语境怎么用：2025 decideWrite 题中 line 33 红色，先在 JaCoCo 报告中确认它属于哪个条件分支，再设计输入使程序沿该路径执行。例如如果 line 33 是 buffer overflow path，需要给超长字符串作为输入。

卷面写法：写 coverage item（statement at line 33 not executed）→ 写具体输入值及如何调用 → 写 expected result → 说明此 test 补了该 statement。格式：Additional test AT1: input=..., expected=..., covers previously unexecuted line 33。

常见失分点：只写"增加一个测试覆盖 line 33"而没有给具体输入值和 expected result。或者给了输入但没有说明为什么这个输入能走到 line 33（前置条件分析缺失）。
