Section 1：几乎必出（>80% 命中率）

基于 2025 Summer 和 2025 Autumn 真题结构，以下四道题的题型框架是 CS603 最稳定的出题模式。变化通常只在具体程序名（Find → ComputeFib → LinearSearch）或具体场景（BankAccount → Stack → Queue），但子题结构和得分点完全相同。

Q1: Predicate Logic + Hoare Logic — Find Method (25 marks)

预测子题结构：(a)[4] 英语译谓词逻辑（两小题）；(b)[3] tautology / satisfiable / unsatisfiable 定义与例子；(c)[5] 逻辑等价性检查；(d)[9] Find(a:array<int>) 的 Hoare 证明 (partial correctness)；(e)[4] Partial vs total correctness，补充 variant 并解释。

判分要点
Q1 需要完成四个独立的子任务：把英语翻译成逻辑公式、判断公式属于哪一类、做等价变形、用 Hoare logic 的三步法证明循环正确性。

1/4
Every person...∀x(Person(x) → ...)
Somebody...∃x(Person(x) ∧ ...)

判分点：every / 所有 后面跟 ∀ 和 →（implication）；somebody / 存在 后面跟 ∃ 和 ∧（conjunction）。如果你把 “every person is walking” 写成 ∀x(Person(x) ∧ Walking(x))，意思是“宇宙里每个东西都是人而且都在走路”——这显然是错的。

P ∨ ¬Ptautology
P ∧ Qsatisfiable
P ∧ ¬Punsatisfiable

这道题不考背诵定义。卷面上要给一个具体的变量赋值（assignment），用它来证明 satisfiable；给两个 assignment 或逻辑理由说明 tautology / unsat。

init→preserve→exit
range1 ≤ i ≤ a.Length
meaning∀k<i. min≤a[k]
origin∃k<i. min=a[k]

Find 题核心：invariant 不是直接把 postcondition 抄过来——它描述的是循环每轮迭代后已经处理完的那一段满足什么性质。退出时用 i=a.Length 把前缀扩成全数组。

V=a.Length-ii:=i+1V'=V-1
partial correctnesspostcondition if terminates
total correctnesspartial + variant

Total correctness 要多写 termination：variant 非负、每轮严格下降、well-founded。Find 的标准 variant 是 a.Length - i。

Model Answer — (a) 英语译谓词逻辑 [4 marks]

(i) "If John is right, somebody took his pen"

Formula: Right(John) → ∃x Took(x, PenOfJohn)

如果题面要求 domain 限定为人：Right(John) → ∃x (Person(x) ∧ Took(x, PenOfJohn))

Key: if → → (implication)；somebody → ∃ (existential quantifier)。John 和 PenOfJohn 是常量（constant），不需要量化。Predicates 建议：Right(p) 表示 p 是对的；Took(x, item) 表示 x 拿了 item。

Why not ∀x? "somebody" = 存在至少一个人，不是所有人。写成 ∀x 意思全变。

Why not ∧ for the main connective? Right(John) ∧ ∃x ... 表达两个并列事实，不是条件关系。原句是 if-then 结构。

(ii) "Every person is walking and talking"

Formula: ∀x (Person(x) → Walking(x) ∧ Talking(x))

Key: every → ∀ + →；and → ∧。predicates: Person(x), Walking(x), Talking(x)。

Why ∀ with →? x 遍历所有事物，只有 Person(x) 为真时才需检查 Walking 和 Talking。不加 Person(x) 筛选，∀x (Walking(x) ∧ Talking(x)) 要求宇宙中一切都既走又说 — 假得离谱。

致命口诀 (必背)：∀ 一定配 →（全称量词 + implication）；∃ 一定配 ∧（存在量词 + conjunction）。写反了整句话意思全错。∀x (Person(x) ∧ Walking(x)) → "所有东西都是人且都在走路"（假的）。∃x (Person(x) → Walking(x)) → 随手找一只猫不是人，左边 false，整句 true，表达不了"存在一个人在走路"。

Model Answer — (b) Tautology, Satisfiable, Unsatisfiable [3 marks]

术语 (Term)定义公式例子为什么这个例子符合定义
Tautology (永真式)true under ALL assignments — 无论变量取什么值，整句都为真P ∨ ¬PP 取 true: true∨false=true；P 取 false: false∨true=true。全部为真。
Satisfiable (可满足)true under AT LEAST ONE assignment — 存在至少一种赋值为真P ∧ QP=true, Q=true 时为 true；P=true, Q=false 时为 false。至少一种赋值为真。
Unsatisfiable (不可满足)true under NO assignment — 无论怎么赋值都为假P ∧ ¬PP=true: false；P=false: false。无解。

卷面注意事项：(1) 给例子用命题逻辑最简单的公式 — 越简单阅卷人越容易判断对错。(2) satisfiable 的例子不能同时也是 tautology — 每个术语互斥。(3) 每个术语两行：一行定义 + 一行公式和解释。

方法一：真值表（truth table）— 列出所有变量组合，检查两公式是否同列。方法二：等价变换 — 用已知等价式逐步推导。

常用等价式：P → Q ≡ ¬P ∨ Q
De Morgan:¬(P ∧ Q) ≡ ¬P ∨ ¬Q¬(P ∨ Q) ≡ ¬P ∧ ¬Q
Double negation: ¬¬P ≡ P
Distribution: P ∧ (Q ∨ R) ≡ (P∧Q) ∨ (P∧R)

Model Answer — (d) Find 方法 Hoare 证明 (Partial Correctness) [9 marks]

Program:

// returns the minimum value in array a
min := a[0];
i := 1;
while (i < a.Length) {
  if (a[i] < min) { min := a[i]; }
  i := i + 1;
}

Precondition: a.Length > 0 — 确保 a[0] 不越界。这是调用者的义务（design by contract）。

Postcondition:

  • ∀k (0 ≤ k < a.Length → min ≤ a[k]) — min 不大于任意数组元素（min 是最小值）
  • ∃k (0 ≤ k < a.Length ∧ min = a[k]) — min 来自数组（防止 trivial 解：永远返回 -∞）

Loop Invariant:

  1. 1 ≤ i ≤ a.Length — 范围约束（数组访问安全 + 退出推导）
  2. ∀k (0 ≤ k < i → min ≤ a[k]) — 在已扫描前缀 a[0..i) 中，min 是最小值
  3. ∃k (0 ≤ k < i ∧ min = a[k]) — min 来自已扫描前缀

(1) Establishment (建立) — 循环前 invariant 成立：

初始化后 min = a[0], i = 1。已扫描前缀 a[0..1) 只含 a[0]。

- 范围: 1 ≤ 1 ≤ a.Length，由 precondition a.Length > 0 保证。

- 语义: ∀k (0≤k<1 → min≤a[k]) 即 min ≤ a[0]，成立 (min = a[0])。

- 来源: ∃k (0≤k<1 ∧ min=a[k])，取 k=0，成立 (min = a[0])。

(2) Preservation (保持) — 一次迭代后 invariant 仍成立：

假设迭代前 invariant 成立 (i < a.Length)。分两种情况：

- 若 a[i] < min: min := a[i]。新 min = a[i] ≤ 所有旧已扫描元素 (原 min ≤ 它们)，也 ≤ a[i]。所以 ∀k (0≤k<i+1 → min≤a[k]) 成立。

- 若 a[i] ≥ min: min 不变。旧已扫描部分性质保持，加上 a[i] ≥ min → ∀k (0≤k<i+1 → min≤a[k]) 仍成立。

i := i+1: 范围变为 1 ≤ i+1 ≤ a.Length (循环条件 i < a.Length 保证)。来源性质：取新元素 a[i] 或旧 k 值为证。

(3) Exit use (退出推导) — invariant + ¬guard → postcondition：

¬guard: i ≥ a.Length。Invariant 范围: i ≤ a.Length。结合得 i = a.Length。

代入 invariant 语义: ∀k (0≤k<a.Length → min≤a[k]) ∧ ∃k (0≤k<a.Length ∧ min=a[k])，正是 postcondition。QED。

Model Answer — (e) Partial vs Total Correctness [4 marks]

Partial Correctness (部分正确性)Total Correctness (完全正确性)
ProvesIf the program terminates, the result satisfies the postcondition.The program ALWAYS terminates AND the result satisfies the postcondition.
Proof obligationInvariant + 三步法 (establishment, preservation, exit use)Invariant 三步法 + Variant (ranking function) 证明终止
What's missingTermination proof — 循环可能跑 forever，partial correctness 不关心Nothing missing — 完整保证
Variant 要求不需要必须：non-negative (非负) + strictly decreasing each iteration (严格递减) + integer (基于良基序 well-founded ordering)

Variant for Find: V = a.Length - i

(1) Non-negative: 由 invariant i ≤ a.Length 保证 V ≥ 0。

(2) Strictly decreasing: 每次迭代 i := i+1，V' = a.Length - (i+1) = V - 1 < V。

(3) Well-founded: 自然数不能无限递减 → 循环必然终止。

Dafny 写法: decreases a.Length - i。

Model Answer — (a) Ackermann decreases with tuple [6 marks]

Ackermann 函数递归调用：

function Ackermann(m: nat, n: nat): nat
  decreases m, n   // lexicographic tuple
{
  if m == 0 then n + 1
  else if n == 0 then Ackermann(m - 1, 1)         // Branch 1: m decreases
  else Ackermann(m - 1, Ackermann(m, n - 1))       // Branch 2: outer m decreases; inner n decreases
}
decreases 写法Branch 1
Ackermann(m-1, 1)		Branch 2 (inner call)
Ackermann(m, n-1)		是否通过？
decreases mm-1 < m ✅m = m ❌ (不变)❌ Fail
decreases m + n(m-1)+1 = m ❌ (可能不变)m+(n-1) < m+n ✅❌ Fail
decreases m, nm-1 < m → 第一分量 ↓ ✅m=m, n-1<n → 第二分量 ↓ ✅✅ Pass

词典序 (lexicographic ordering) 比较规则： 先比较第一个分量。如果第一个分量严格小于，整个 tuple 就小 — 通过。 如果第一个分量相等，再比较第二个分量。如果第二个分量严格小于，通过。 这恰好覆盖 Ackermann 的所有递归分支：Branch 1 靠 m 减；Branch 2 外层靠 m 减，内层靠 n 减。

Model Answer — (b) 找 Dafny 代码错误 + 给出 invariant 和 variant [5 marks]

常见代码错误 (Code Errors)：

错误类型具体表现如何修复
循环条件写反while (i >= n) 应该递减却用递增条件改为 while (i < n)

Invariant 模板 (任意循环):

  1. 范围 invariant: 0 ≤ i ≤ n 或 1 ≤ i ≤ a.Length — 保证数组访问安全
  2. 语义 invariant: 描述 a[0..i)（已处理部分）满足什么性质，例如 ∀k (0 ≤ k < i → property(a[k], result))
  3. 结果变量 invariant: 描述 result 变量与已处理部分的关系

Variant: decreases n - i 或 decreases a.Length - i。

Model Answer — (c) BankAccount ghost predicate Valid() [8 marks]

BankAccount 类的核心 challenge：Balance 是汇总字段，但要保证它总是等于 deposits 总和减去 withdrawals 总和。用 ghost sequences 记录明细，用 ghost predicate Valid() 把一致性变成可验证的规则。

class BankAccount {
  var Balance: int
  ghost var deposits: seq<int>
  ghost var withdrawals: seq<int>

  ghost predicate Valid() reads this {
    Balance == deposits.total - withdrawals.total && Balance >= 0
  }

  constructor ()
    ensures Valid()
    ensures Balance == 0
    ensures deposits == []
    ensures withdrawals == []
  {
    Balance := 0;
    deposits := [];
    withdrawals := [];
  }

  method Deposit(amount: int)
    requires amount > 0
    modifies deposits
    ensures Valid()
    ensures Balance == old(Balance) + amount
  {
    Balance := Balance + amount;
    deposits := deposits + [amount];
  }

  method Withdraw(amount: int)
    requires amount > 0
    requires amount <= Balance
    modifies withdrawals
    ensures Valid()
    ensures Balance == old(Balance) - amount
  {
    Balance := Balance - amount;
    withdrawals := withdrawals + [amount];
  }
}

关键得分点 (逐条检查)：

  1. Valid() 的核心等式：Balance == deposits.total - withdrawals.total（一致性）+ Balance >= 0（非负性，可选但加了更完整）。
  2. Constructor MUST establish Valid(): 初始 0 == 0 - 0 成立。这是 class invariant 的建立义务。漏了这条 constructor 后对象处于未验证状态。
  3. Every public method MUST re-establish Valid() in its ensures clause. Deposit 和 Withdraw 都要 ensures Valid()。漏任何一个方法 verifier 会报该 method might break the invariant。
  4. Withdraw 的额外 precondition: amount <= Balance 防止余额变负（破坏 Balance >= 0）。如果没有这条，verifier 无法证明 Valid() 中的 Balance >= 0。
  5. modifies clauses: Deposit 只 modifies deposits，Withdraw 只 modifies withdrawals。帮助 verifier 做 framing（知道哪些变量可能变化）。
  6. ghost 关键字: deposits 和 withdrawals 声明为 ghost — 只为证明存在，编译后不占运行时内存。

Model Answer — (d) Deductive verifier 如何检查 class invariants [6 marks]

  1. Translate to VCs (Verification Conditions). Dafny verifier 把方法体、contracts、invariant 翻译成一阶逻辑公式 (first-order logic formulas)。每条 VC 形如：如果 requires 和 invariant 都成立，执行方法体后 ensures 也成立。
  2. Send to Z3 SMT solver. VCs 交给 Microsoft Z3（一个 Satisfiability Modulo Theories solver）。Z3 尝试证明每条 VC 是 valid（在所有整数/数组等背景理论下恒真）。
  3. Check Valid() lifecycle. Constructor 后: verifier 检查 Valid() 是否成立（initial establishment）。每个 public method 的 ensures 列有 Valid() → verifier 检查方法体执行后 Valid() 是否仍成立（preservation）。若某方法有 requires Valid()，verifier 以此为前提。
  4. Report results. 所有 VCs 都 valid → 方法满足 contract。任一 VC invalid → Z3 返回 counterexample，Dafny 报告错误位置和原因。

类比：Dafny verifier 做的事情和 Hoare logic 手工证明完全一样，区别在于 Dafny 靠 Z3 自动推导而不需要人手工推 proof tree。你写的 requires/ensures/invariant 就是 Hoare triple 的 P、Q、I。

Model Answer — (a) Event-B 系统建模 [8 marks]

组件英文内容作用
Context上下文Carrier Sets（载体集合）、Constants（常量）、Axioms（公理）定义系统的静态背景知识，不涉及状态变化
Machine机器Variables（变量）、Invariants（不变量）、Events（事件）定义系统的动态部分：状态 + 状态变迁
Invariant不变量一阶逻辑公式，描述所有合法状态的公共性质约束状态空间 — 任何 event 执行后都不能违反
Event事件含 guards（守卫条件）和 actions（动作）。Guard 为真时 event 可触发定义状态如何变化。每个 event 类似 Hoare triple 中的一个命令
Proof Obligation证明义务自动生成的数学命题必须证明：(1) 每个 event 保持 invariant；(2) refinement 保持抽象行为；(3) guards 不会变弱（refinement 不引入死锁）

Event-B 和 Hoare Logic / Dafny 的关系：三者都是 formal verification 的核心技术。Hoare logic 证明单个程序片段的正确性；Dafny 把 Hoare proof 自动化；Event-B 处理更高层的系统级模型 — 关注"系统整体行为"，而非单行代码。Event-B 常用于需求阶段和架构设计阶段，早于编码。

Model Answer — (b) Data Refinement with coded example [6 marks]

什么是 Data Refinement：把抽象数据类型（例如数学集合 Set）替换为更具体的数据结构（例如数组 Array），同时证明具体实现保持了抽象模型的所有行为。这保证了"实现可以更 detail，但不能违反抽象承诺"。

关键概念：

  • Abstract state (抽象状态): 用数学类型描述，如 contents: set of T
  • Concrete state (具体状态): 用编程语言类型描述，如 arr: array of T; size: int
  • Gluing invariant (胶合不变量): 联系抽象和具体的桥梁。例如 contents = {arr[0], ..., arr[size-1]} — 说明了数组的前 size 个元素恰好构成集合 contents
  • Refinement proof obligation: 对每个 abstract event，存在一个对应的 concrete event 序列，其 net effect 在抽象观察层面等价

Simple coded example — Set refined to Array:

Abstract: add(x) — contents := contents ∪ {x}

Concrete: addConcrete(x) — arr[size] := x; size := size + 1，前提 size < arr.Length (guard)

Gluing invariant: contents = set of arr[0..size) — 保证 addConcrete 的效果等于 add 的效果（集合中多了一个元素 x）。

Model Answer — (c) FRET — Formal Requirements Elicitation Tool [5 marks]

FRET 做什么：将自然语言需求（natural language requirements）转换为时间逻辑公式（temporal logic — LTL 或 CTL）。用户输入 controlled natural language（受控自然语言，如 "When X happens, the system shall eventually do Y"），FRET 解析语义结构 → 自动生成如 G(request → F response) 的 LTL 公式。

FRET 的工作流：

  1. Parse. 用户用模板化的自然语言写需求（trigger、condition、response、timing）。
  2. Semantic mapping. FRET 把关键词映射为 temporal logic 结构：when → condition；shall → always；eventually → F；within → bounded temporal operator。
  3. Output. 生成 LTL 或 FRET-ish temporal logic formula。该输出可直接导入 Spin / NuSMV 做 model checking。

为什么重要：从需求到模型检查的最薄环节一直是"把需求正确表达成公式"。FRET 降低了这个门槛 — 不用手工写 LTL，减少了人为翻译错误。

Model Answer — (d) CDCL SAT Solver — algorithm + advantage [6 marks]

CDCL = Conflict-Driven Clause Learning（冲突驱动的子句学习算法）。

Algorithm steps:

  1. Decide (决策). 选一个未赋值的布尔变量，任意赋值 (true/false)。记录决策级别 (decision level)。
  2. BCP — Boolean Constraint Propagation (布尔约束传播). 扫描所有 clause：若某 clause 只剩一个未赋值 literal，强制赋能使该 clause 成立的值。反复传播直到无 unit clause 或发现冲突。
  3. Conflict Analysis (冲突分析). 若 BCP 导致某 clause 所有 literal 为 false → 冲突。分析冲突原因：回溯 implication graph，找出哪些 decisions 导致了冲突 → 学到一条新 clause（learned clause）— 表示"这组赋值组合不能同时出现"。
  4. Backjump (非时序回溯). 不简单退回上一个 decision level，而是跳回真正导致冲突的那个决策级别（可能回调多层）。
  5. Restart (可选). 定期清空赋值但保留 learned clauses，给 solver 探索新顺序的机会。
  6. Termination. 所有变量赋值后无冲突 → SAT（return model）。回溯到 level 0 仍冲突 → UNSAT（return empty）。

CDCL vs Stochastic Search (e.g. WalkSAT) — 本质优势：

方面Stochastic Search (WalkSAT)CDCL
搜索策略随机翻转一个变量的赋值；偶尔随机重启系统化搜索 + 冲突分析 + 学习 clause
记忆无记忆 — 不会记录之前走过哪些死胡同有记忆 — learned clauses 是永久的剪枝规则
重复冲突可能无限次重复相同的冲突模式learned clause 一劳永逸地剪掉该模式
完整性Incomplete — 可能永远找不到解即使解存在Complete — 一定能在有限步内判定 SAT/UNSAT
适用场景大规模随机公式（random k-SAT）有时更快结构化工业问题 (hardware verification, scheduling) — 快几个数量级

CDCL vs DPLL: DPLL 是 CDCL 的前身。DPLL 也做 Decide + BCP + Backtrack，但 backtrack 是时序的（只退一层），没有 conflict analysis 和 clause learning。在含对称冲突模式的公式上，DPLL 可能指数级慢于 CDCL。

Model Answer — (a) Spin 验证 temporal properties 的流程 [7 marks]

  1. Write Promela model. 用 Promela 语言描述系统：processes（进程）、channels（通道，用于进程通信）、variables（变量）。每个 process 是一段描述行为的代码。例如两个并发的进程交替执行、通过 channel 发消息。
  2. Specify LTL property. 把要验证的性质写成 LTL formula。例如 G !(p1_critical && p2_critical) 表示两个进程永远不会同时进入临界区。
  3. Build state space. Spin 穷举所有可能的交错执行 (interleavings)，构建完整的 state graph（Kripke structure）。每个状态 = 所有进程的 program counter + 所有变量的值。
  4. Negate LTL → Büchi automaton. 将 LTL formula 取反 (negation)，转换成 Büchi automaton — 一种接受无限字 (infinite words) 的有限状态自动机。取反的原理：如果原性质在所有路径上成立，那么取反后的自动机应该接受不到任何路径。
  5. Synchronous product. 计算 state graph × Büchi automaton 的同步积。积的每个状态 = (系统状态, 自动机状态)。如果积中存在 accepting cycle（可接受循环）→ 原性质被违反 → 继续找 counterexample。
  6. Counterexample output. Spin 输出一条从初始状态出发的有限路径 trace + 一个可无限重复的 cycle，展示了性质在哪个时刻被违反。你可以单步执行这个 trace 来 debug。
  7. Safety vs Fairness. Safety: 检查坏状态是否可达。Fairness: 只考虑所有 enabled process 最终都会被执行到的调度（排除 starvation 导致的假性 violation）。

Model Answer — (c) 应对 State Explosion 的两种方法 [4 marks]

1. Abstraction (抽象). 用更小的 abstract model 代替 concrete model，忽略验证目标不关心的细节。例如把整数变量替换为 {negative, zero, positive} 三个抽象值。关键保证：如果 abstract model 满足性质，concrete model 一定满足（conservative / sound abstraction）。常见的如 predicate abstraction、Cegar (CounterExample-Guided Abstraction Refinement)。
2. Partial Order Reduction (偏序归约). 识别独立的并发操作：如果两个操作交换执行顺序不影响最终状态，只探索其中一种顺序。在 N 个并发进程各自执行 M 步的场景中，朴素探索有 (N·M)!/(M!)N 种交错；POR 可以降到多项式级。
3. Symmetry Reduction (对称性归约). 如果系统有 N 个同构的组件（如多台同型号服务器），利用对称性只探索一个代表性组件的状态，其余推到对称等价类。
4. Bounded Model Checking (有界模型检查). 只检查长度 ≤ k 的执行路径，不构建完整状态空间。用 SAT/SMT solver 逐步检查不同 k 值。如果找到反例就报告；找不到则逐步增大 k。不能完全证明无 bug（除非 k 达到 completeness threshold）。

Model Answer — (d) 写 LTL Formulae [8 marks]

必背 Temporal Operators:

Operator全称含义读法
G φGlobally / Alwaysφ 在当前及之后的所有状态都成立"henceforth φ"
F φFinally / Eventuallyφ 在现在或未来某个状态成立"eventually φ"
X φNextφ 在下一个状态成立"next φ"
φ U ψStrong Untilφ 一直成立直到 ψ 成立，且 ψ 必须最终成立"φ until ψ"
φ W ψWeak Untilφ 一直成立直到 ψ 成立，但 ψ 可以不成立（此时 φ 永远成立）"φ unless ψ"

8 条自然语言 → LTL Formula (含解释):

#自然语言LTL Formula分析
1"The light is always on." (灯始终亮)G light_onG: 所有状态。Simple safety — 单一命题在每时每刻都成立。
2"If a door opens, it will eventually close." (门开了最终会关)G (door_open → F door_closed)G: 所有时刻。→: 条件。F: 最终。Liveness — 每次触发都最终得到响应。
3"Train never enters crossing while gate is up." (栏杆抬起时火车绝不进入道口)G ¬(train_in_crossing ∧ gate_up)G + ¬: 在所有状态，坏组合永不出现。经典 Safety property。
4"Car must stop before brake releases." (刹车松开之前车必须停)¬brake_released W car_stoppedW (weak until): 在刹车松开之前，车一直处于停止状态。如果刹车一直不松开，car_stopped 永远成立也没问题。
5"After request, server stays busy until response." (请求后服务器忙直到响应)G (request → (server_busy U response_sent))U (strong until): server_busy 一直成立直到 response_sent 成立。response_sent 必须最终成立 (U 的 right operand 终须满足)。Liveness。
6"The light alternates red and green forever." (红绿灯交替)G (red → X green) ∧ G (green → X red)X (next): 红后下一个状态必绿；绿后下一个状态必红。两方向都覆盖保证无限交替。
7"High temperature → eventually cool down." (高温最终会降温)G (high_temp → F cool_down)同第 2 条模板。每次高温都必须最终降温。Liveness。
8"User stays logged in from authentication onward." (认证后一直登录)G (authenticate → G logged_in)嵌套 G: 一旦认证发生，从那个时刻起往后的所有状态都 logged_in。

Safety vs Liveness 区分口诀：

Safety = "something bad never happens" → G ¬bad 或 W (weak until)

Liveness = "something good eventually happens" → G (trigger → F effect) 或 U (strong until)

卷面写法：每条公式写三行：(1) 自然语言描述；(2) LTL formula；(3) 逐 operator 解释 + Safety/Liveness 标签。

Q1 Variant: ComputeFib(n) Hoare Proof

Same template as Find. ComputeFib 计算第 n 个斐波那契数。

a := 0; b := 1; i := 1;
while (i < n) {
  tmp := a + b;
  a := b;
  b := tmp;
  i := i + 1;
}

Precondition: n ≥ 1（或 n ≥ 0 根据 fib 定义调整）

Invariant: a = fib(i-1) ∧ b = fib(i) ∧ 1 ≤ i ≤ n（范围 + 语义合为一体）

Postcondition: b = fib(n)

Variant: n - i（非负、严格递减）

Proof 三步法：(1) Establishment: i=1, a=0=fib(0), b=1=fib(1) → invariant holds。(2) Preservation: tmp = a+b = fib(i-1)+fib(i) = fib(i+1); a:=fib(i); b:=fib(i+1); i:=i+1 → invariant for new i。(3) Exit: i=n → b=fib(n)。

Q2 Variant: CopyArray / LinearSearch Dafny

两种最常见的 Dafny 数组循环题。

CopyArray:

method CopyArray(src: array<int>, dst: array<int>)
  requires src.Length == dst.Length
  ensures forall i :: 0 <= i < src.Length ==> dst[i] == src[i]
{
  var i := 0;
  while (i < src.Length)
    invariant 0 <= i <= src.Length
    invariant forall k :: 0 <= k < i ==> dst[k] == src[k]
    decreases src.Length - i
  {
    dst[i] := src[i];
    i := i + 1;
  }
}

LinearSearch:

method LinearSearch(a: array<int>, key: int) returns (pos: int)
  ensures pos == -1 || (0 <= pos < a.Length && a[pos] == key)
  ensures pos == -1 <==> forall k :: 0 <= k < a.Length ==> a[k] != key
{
  var i := 0;
  while (i < a.Length)
    invariant 0 <= i <= a.Length
    invariant forall k :: 0 <= k < i ==> a[k] != key
    decreases a.Length - i
  {
    if (a[i] == key) { return i; }
    i := i + 1;
  }
  return -1;
}

Q3 Variant: CDCL vs DPLL — 为什么 DPLL 在某些公式上指数爆炸

DPLL: Decide → BCP → 若冲突则时序回溯（chronological backtracking，只退一层）→ 尝试另一分支。没有 clause learning 机制。

为什么 DPLL 在某些公式上指数慢：考虑一个含大量对称冲突路径的公式。DPLL 每次回溯只退一层，相当于顺序试探每条死胡同。每次重新走到同一个冲突模式时，DPLL 无记忆 — 必须重新推理一遍相同的 BCP 链。导致搜索树呈指数级大小。

CDCL 为什么快：第一次冲突后，conflict analysis 产生一条 learned clause 精确刻画了冲突原因。后续搜索中，这条 clause 在 BCP 阶段就能提前触发冲突，无需深层探索。效果相当于"一刀剪掉搜索树的整个子树"。

卷面写法：对比表格：(1) Backtrack 方式：时序 vs 非时序；(2) 学习能力：无 vs clause learning；(3) 重复冲突：会 vs 不会；(4) 工业公式性能：差 vs 极好。

Q4 Variant: CTL instead of LTL

CTL (Computation Tree Logic) 比 LTL 多了 path quantifiers（路径量词）：

CTL 符号含义LTL 等价?
A"for All paths" — 从该状态出发的所有执行路径LTL 默认量化所有路径

AG φIn All paths, Globally φ≈ G φ

AG(request → AF response)All paths: every request eventually gets response≈ G(request → F response)

CTL expressible but NOT LTL: EG p — "there exists a path where p always holds"。这在 LTL 中不可表达，因为 LTL 假定对所有路径量化。考试中常见问："Give an example expressible in CTL but not LTL" — 答案就是 EG p 或 AG EF reset（在所有路径上总能回到 reset 状态 — 这其实是可表达的，更标准的不可表达例子是 AF AG p 等。

简单不可表达例子：EG p — "存在一条路径 p 始终成立" — LTL 中写不出对应的公式，因为 LTL 是 path formula，没有 path quantifier。

Section 3：如果换考法（30–50% 可能）

若 Q1(e) 换 Hoare problem

同一套模板适用任何 while-loop 程序。核心不变：

  1. 写 invariant。三步公式：范围 (0 ≤ i ≤ n) + 已处理部分性质 (∀k < i: ...) + 结果变量关系 (result = f(processed))。
  2. Establishment。循环第一次开始前 invariant 成立 — 通常依赖初始化代码。
  3. Preservation。一次迭代后 invariant 仍成立 — 需要 case analysis（if-then 的分支出不同修订路径）。
  4. Exit use。¬guard + invariant → postcondition — 通常把 i = n 代入 invariant 的语义部分。
  5. Total correctness: 追加 variant — non-negative、strictly decreasing integer expression。常见 variant: n - i、a.Length - i、remaining elements count。

这个模板覆盖：array copy、array sum、linear search、find max/min、binary search、fibonacci 等所有可迭代程序。

若 Q3 换 Neural Network Verification / AI4SE

这类题的核心不是"神经网络很复杂"，而是把 network 的行为编码成约束，再用 solver 搜索违反性质的输入。答题时一定要围绕 property、constraints、counterexample 三个词。
