0. 形式化验证到底在证明什么

Testing 用有限例子找 bug；formal verification 试图证明一类输入或一类行为都满足某个性质。这里的“性质”就是系统必须一直保持或最终达到的规则。

普通说法形式化说法为什么要形式化
余额不能变成负数。balance ≥ 0可以让工具或证明检查每条路径是否保持。
排序后数组仍包含原来的元素。输出是输入的 permutation。只检查“有序”不够，还要检查没丢元素。
请求最终会得到回应。temporal property: eventually response。这种性质关心时间和路径，不只是程序结束时的值。

四类正确性问题

  1. 前后条件。如果开始时满足 precondition，程序结束后是否满足 postcondition。
  2. 循环性质。循环每次迭代后是否保持 invariant，循环是否最终终止。
  3. 状态系统性质。系统在多个状态之间跳转时，是否会到达坏状态，是否最终到达好状态。
  4. 逻辑可满足性。一组约束是否存在某种赋值让它们同时成立。

过关小题：“程序不会崩溃”太模糊。更可证明的写法是：对所有满足 0 ≤ i < a.Length 的输入，访问 a[i] 不越界；或者方法返回时 result ≥ 0。

内容理解题（先做 3 题）

核心答案（先确认概念）：Testing 用有限例子找错误——你测了 100 个输入没问题，不代表第 101 个也没问题。Formal verification 试图证明一类输入或行为都满足性质——从数学上保证所有输入都不会出错。

为什么这是对的：用排序打个比方。Testing 相当于你手动试了 5 个数组跑排序，输出都递增，于是你觉得"大概没问题"。但你无法确定有没有某个你没试过的数组会让它崩溃。Formal verification 相当于你写了一个数学证明：不管输入是什么数组，算法结束后一定有序，并且没有丢失任何元素。这就是 testing 给 confidence、verification 给 guarantee 的区别。

题目语境怎么用：2025 Summer Q1 让你用 Dafny 证明 Find 方法——这就是 formal verification。2025 Summer Q4 让你用 Spin 检查所有可能路径——也是 formal verification。考场上遇到"explain the difference"类题，先说 testing 的局限（有限例子），再说 verification 的目标（证明所有情况），然后给一个具体例子。

卷面写法：分两段写。第一段写 testing：runs concrete inputs, checks actual vs expected, finds bugs but cannot prove absence。第二段写 verification：writes mathematical properties, proves for all inputs/states。最后给一句话例子：testing sorts 5 arrays; verification proves sort is correct for any array。

常见失分点：考试里最容易混淆的是：写了 Dafny annotation 不等于做了 testing，反过来跑了一堆测试不等于做了 formal verification。两个概念不要互相替代。

核心答案（先确认概念）：自然语言容易含糊；形式化后才能被证明、检查或交给工具分析。

为什么这是对的：比如你说"余额不能变负数"，听起来很清楚，但写代码时有各种边界：取款时余额刚好够吗？手续费扣完变负数算吗？利息计算中途变负数再回到正数可以吗？用自然语言写出来的要求，不同人会做出不同的解读。写成 balance >= 0 之后，就没有歧义了——任何时候余额小于 0 就是违反性质。工具（如 Dafny verifier、Z3、Spin）也只能检查精确公式，它们读不懂"大概不能出问题"这种话。

题目语境怎么用：2025 Summer Q1(d) 的 Find 方法把 postcondition 写成 ensures forall i :: min <= a[i]，而不是写"保证 min 是最小值"——因为后者可能被误解为"min 来自于数组"（实际上还需要另一个性质：min 是数组里某个元素）。考场上写 specification 时，多问自己：这句话有没有隐藏歧义？工具能不能逐字检查？

卷面写法：先给出自然语言版本（让人看懂），再给形式化版本（让工具能检查），最后解释为什么形式化版本没有歧义。例如"数组不为空"写成 requires a.Length > 0，并说明 Length 属性已经排除了 null 和空数组两种误解。

常见失分点：只写自然语言不写公式、公式写对了但覆盖范围不全（比如只说了有序没说元素集合不变）、把不同的性质混在同一个公式里导致工具报错时你自己也看不懂。

核心答案（先确认概念）：写成具体条件，例如数组下标始终在范围内，或返回值始终满足某个 postcondition。

为什么这是对的："不会崩溃"有太多种可能：除以零崩溃、数组越界崩溃、空指针崩溃、栈溢出崩溃。每种崩溃的原因和预防方式都不同。改成可证明性质时，要把这个模糊要求拆成一组具体条件：数组越界写成 0 <= i < a.Length 作为循环 invariant；除零写成 precondition denominator != 0；空指针写成 precondition obj != null。这样每类崩溃对应一条可独立检查的条件，工具才能帮你验证。

题目语境怎么用：这道题训练 CS603 的核心思维：把模糊的"程序应该安全"拆成 Dafny 能检查的 requires、ensures、invariant。2025 Summer Q1 Find 题中，precondition a.Length > 0 就是在防止空数组导致 a[0] 越界崩溃。

卷面写法：列出具体崩溃类型，每种对应一条可检查条件，并解释为什么该条件能防止那种崩溃。不要笼统写"程序应该健壮"，要用公式。

常见失分点：漏掉某种崩溃（比如只处理了数组越界没处理除零），或公式覆盖不足（比如 requires a != null 没防住空数组里的索引越界）。

考试题目题（再做 5 题）

标准答案（先写结论）：返回的 min 不大于数组中任何元素——这是 Find 方法 partial correctness 的核心 postcondition。

为什么这是对的：你想找数组最小值，至少需要保证两条：1) 返回的值确实比所有元素都小或等于（即 min <= a[i]）；2) 返回的值确实来自数组（即 exists j :: min == a[j]）。这一条 ensures 只保证了第一条。在 2025 真题中，Find 方法需要同时写下这两条 postcondition 才完整。缺了第二条，你可以写一个方法永远返回 -999，也满足 min <= a[i]，但显然不是数组最小值。

题目语境怎么用：2025 Summer Q1(d) 完整题目给了 Find 方法代码，让你补充 Dafny annotation。你先找方法需要满足什么：循环 invariant 说 min 是已扫描前缀中的最小值，退出循环后扫描完整个数组，postcondition 就应该说 min 是整个数组中的最小值。

卷面写法：卷面上写两条 postcondition：ensures forall i :: 0 <= i < a.Length ==> min <= a[i] 和 ensures exists j :: 0 <= j < a.Length && min == a[j]。然后解释为什么两者缺一不可。

常见失分点：只写了"最小值"性质但没写"来自数组"性质；或者把 forall 写成了 exists（含义完全不同）；忘了写索引范围 0 <= i < a.Length 导致量词范围无界。

标准答案（先写结论）：这是 class invariant（或 ghost predicate Valid()）中声明的账户一致性性质：余额必须等于存入总额减去取出总额。

为什么这是对的：BankAccount 类有两个内部集合 deposits 和 withdrawals 记录所有交易明细，还有一个 Balance 字段显示当前余额。如果没有这个 invariant，随着存钱取钱操作增多，余额字段可能因为编程错误（比如某次取钱忘了更新 Balance）和交易记录脱节。这个等式把"余额是否准确"变成一个机器可检查的性质：每次 deposit 和 withdraw 方法执行后，verifier 都会检查等式是否仍然成立；不成立就报错。

题目语境怎么用：2025 Summer Q2(c) 完整题目给了 BankAccount 类框架（有 deposits、withdrawals 两个 ghost 序列和 Balance 字段），让你设计 Valid() predicate 和每个方法的 contract。你首先要想到的就是这条等式，因为它是类最核心的不变性质。

卷面写法：定义一个 ghost predicate Valid() reads this { Balance == deposits.total - withdrawals.total }。然后在 constructor 的 ensures 里写 ensures Valid()（建立初始一致性），在 deposit/withdraw 的 ensures 里也写 ensures Valid()（保持一致性）。

常见失分点：只写了 invariant 等式但没在 constructor 的 ensures 中建立它（初始状态也要满足）；或者在 deposit/withdraw 中改了 Balance 但忘了同时更新对应的 deposits/withdrawals 序列，导致等式被破坏。

标准答案（先写结论）：Safety property 写成"坏状态永远不可达"。例如交通灯系统写 G !(NS_green && EW_green)（南北和东西方向永远不会同时绿灯）；银行系统写 G (balance >= 0)（余额永远不会为负）。

为什么这是对的：Safety 的核心直觉是"something bad never happens"。在 model checking 里，你先把所有可能的状态列出来，然后用 temporal logic 公式描述"坏状态"是什么，最后工具检查是否所有执行路径都能避开它。注意 safety 检查的是"每时每刻"，所以用 G（globally / always）。如果工具找到一条能到达坏状态的路径，它就会输出这条 counterexample 给你看。

题目语境怎么用：2025 Summer Q4(a) 给了你一个并发系统的状态模型（比如两个 process 争资源），让你写 safety property。你要先找到题目中"什么情况算坏了"——比如两个 process 同时进入 critical section、或变量值超出安全范围——然后用 G !bad_condition 表达。

卷面写法：先定义坏状态（用自然语言说清楚），再写 LTL formula，最后解释 G 的含义（in every state of every path, the condition holds）。如果能给一个具体的 counterexample 路径说明 violation 是什么，更好。

常见失分点：把 safety 和 liveness 搞混——safety 用 G !bad（永不变坏），liveness 用 G (request -> F response)（好事最终发生）。或者公式里漏了 negation 导致表达的是相反的意思。

标准答案（先写结论）：Valid() 写 Balance == deposits.total - withdrawals.total，表明余额必须等于所有存入总和减去所有取出总和。Constructor 要建立 Valid()，每个 public method（deposit、withdraw）要维持 Valid()。

为什么这是对的：BankAccount 类用 ghost 序列 deposits 和 withdrawals 记录每笔交易，Balance 是当前余额。如果代码有 bug——比如某次 deposit 忘了把金额加到 Balance，这个等式就会被破坏。Valid() 把"余额一致性"变成一条可自动检查的规则：Dafny verifier 会在每次方法调用后检查等式是否仍然成立。Constructor 里 ensures Valid() 意味着对象创建时余额是 0，序列是空的，等式成立；deposit 里 ensures Valid() 意味着存钱后等式必须仍然成立——如果代码只更新了 deposits 忘更新 Balance，verifier 就报错。

题目语境怎么用：2025 Summer Q2 的完整 BankAccount 题要求写 Valid() 和每个方法的 contract。你还需要处理 withdrawal 的 precondition：取款金额不能超过当前余额（否则 Balance 变负虽然等式仍成立，但可能违反了另一条业务规则"余额不能为负"）。

卷面写法：ghost predicate Valid() reads this { Balance == deposits.total - withdrawals.total }。Constructor: ensures Valid() && deposits == [] && withdrawals == []。Deposit(amount): requires amount > 0; modifies deposits; ensures Valid()。Withdraw(amount): requires amount > 0 && amount <= Balance; modifies withdrawals; ensures Valid()。

常见失分点：Valid() 只在 ensures 里提了一次但 constructor 忘了初始建立；或者 deposit/withdraw 里改了 Balance 但忘了同时修改 ghost 序列；或者 withdrawal 缺少"取款金额 <= 余额"的 precondition 导致可取负余额。

标准答案（先写结论）：用 G(request -> F response) 表示：在每一条执行路径的每一个状态，如果出现 request，那么最终（现在或未来某个时刻）一定出现 response。这是 liveness property 的标准写法。

为什么这是对的：G 是 globally（每时每刻），-> 是 implication，F 是 finally（最终）。合起来读作：always, if there is a request, eventually there is a response。这和 safety 不同：safety 关心"坏事永不发生"（G !bad），liveness 关心"好事最终会发生"（G (trigger -> F response)）。Spin 会穷举所有可能的交错执行路径检查这个性质。如果找不到一条路径在某次 request 后永远不 response，性质成立。

题目语境怎么用：2025 Summer Q4 大概率会让你读一个并发模型（例如 client-server 交互），然后写 LTL 公式表达某个要求。你要先判断题目要求的是 safety（"不会两个同时..."）还是 liveness（"最终会..."），然后选对 temporal operator：safety 用 G !，liveness 用 G (cause -> F effect) 或直接用 F goal。

卷面写法：先写自然语言描述（"每次请求最终都有响应"），再写 LTL formula（G(request -> F response)），然后逐符号解释：G 表示对所有路径的所有状态，-> 表示如果左边成立右边必须成立，F 表示当前或未来某个状态。最后说明这属于 liveness property。

常见失分点：把 G(request -> F response) 写成了 F(response)（后者只要求至少有一次 response，不要求每次 request 都有 response）；或者把 liveness 当成 safety 写了 G !(!response)（这表达的是"永远不会不响应"，含义模糊且不标准）。

1. Logic：把自然语言改写成可检查的句子

Logic 是 CS603 的语法地基。它让我们把“如果输入有效，那么输出非负”这类话，写成能推理、能检查的公式。

命题逻辑和谓词逻辑

概念白话解释例子

几个考试常见词

Satisfiable存在某种赋值让公式为 true。例如 x>0 ∧ x<3 可满足。
Unsatisfiable没有任何赋值能让公式为 true。例如 x>0 ∧ x<0 不可满足。
Tautology无论变量怎样取值都为 true。例如 P ∨ ¬P。
Model让公式成立的一组赋值。SAT/SMT solver 返回的 model 就是一份可行赋值。

Implication 最容易误读

P ⇒ Q 的意思是：只要 P 成立，Q 必须成立。如果 P 本来不成立，整句话不会被违反。程序证明里经常写“如果 precondition 成立，那么 postcondition 成立”。

从英语到逻辑的台阶

  1. 先找对象。例如数组、下标、余额、状态。
  2. 再找范围。“所有合法下标”对应 ∀ i :: 0 ≤ i < a.Length。
  3. 再找性质。例如 a[i] ≥ 0。
  4. 最后组合。用 implication 表达条件，用 conjunction 表达多个要求。

内容理解题（先做 3 题）

核心答案（先确认概念）：Satisfiable 是至少有一种赋值为真；tautology 是所有赋值都为真。

为什么这是对的：想象你在解方程。x > 0 是可满足的，因为 x=5 就能让它为真；但 x > 0 ∧ x < 0 不可满足，因为一个数不能既大于零又小于零。tautology 像 P ∨ ¬P——不管 P 取什么值整句话都为真，不需要"找"具体解。SAT solver 返回 sat 意味着找到了可满足的赋值，返回 unsat 意味着证明不存在任何赋值。

题目语境怎么用：2025 Summer Q1(b) 直接让你解释这三个术语并各举一例。satisfiable 的例子可以写 x > 0 ∧ x < 10（有解 x=5）；unsatisfiable 写 P ∧ ¬P；tautology 写 P ∨ ¬P。注意 satisfiable 的例子不能是 tautology——前者只要求存在一种赋值，后者要求所有赋值都成立。

卷面写法：每个术语分两行：一行写定义（at least one interpretation / all interpretations / no interpretation），一行写公式例子，再用一句话解释该例子为什么属于这类。格式统一，阅卷人一眼看到三个点都答了。

常见失分点：把 satisfiable 和 tautology 搞混——satisfiable 只要存在一种赋值，不需要所有赋值都成立。tautology 的要求强得多。例子和定义不匹配：比如定义说 satisfiable 但又给了一个 tautology 作为例子。

核心答案（先确认概念）：只要 P 成立，Q 必须成立；如果 P 不成立，implication 不被违反。

为什么这是对的：很多人把 ⇒ 误解为因果或必要条件，但它只是真值表上的一种关系：唯一的反例是 P 为真且 Q 为假。其他三种组合（P 假 Q 真、P 假 Q 假、P 真 Q 真）整句话都为真。"false implies anything" 听起来反直觉但逻辑上成立——你连前提都没满足，就没有违反承诺。在程序验证里，如果 precondition 不成立，方法不用保证 postcondition。

题目语境怎么用：2025 Summer Q1(a) 的 "if John is right then somebody took his pen" 直接对应 Right(John) ⇒ ∃x Took(x, PenOfJohn)。如果 John is right 是假的，整句话自动为真（vacuous truth）。在写 Dafny contract 时，requires ⇒ ensures 的结构也是这个道理。

卷面写法：可以画四行真值表，也可以用自然语言说"唯一的反例是 P true 且 Q false"。重点强调"P 假时整句自动为真"这一点，因为它最容易被质疑。

常见失分点：以为 ⇒ 表示 P 导致 Q（因果），或以为 Q 必须先成立 P 才能成立（方向反了）。⇒ 不是因果操作符，只是对两个布尔值的组合规则。

核心答案（先确认概念）：先找对象和变量，再找范围，最后写性质和连接符。

为什么这是对的：直接逐词翻译英语通常会丢信息或引入歧义。"Every student passed the exam" 如果写成 ∀x Passed(x) 就错了——x 没限制范围，桌子椅子也能满足未通过考试从而让这句话为假。正确写法是 ∀x (Student(x) ⇒ Passed(x))。所以第一步永远是：这段话讨论什么类型的对象？它们的取值范围是什么？确定之后再选量词和连接词。

题目语境怎么用：2025 Summer Q1(a) 是典型英译逻辑题。遇到 "every" 用 ∀ + ⇒，遇到 "somebody" / "exists" 用 ∃ + ∧。这个口诀几乎覆盖所有英译逻辑题：全称量词配 implication，存在量词配 conjunction。

卷面写法：三步走：(1) 列出所有 predicate 及其参数含义；(2) 写出 domain of discourse（讨论的全体对象是什么集合）；(3) 写出完整公式。每一步都要在卷面上体现，让阅卷人看到你的推理过程。

2. Design by Contract：把方法当成一份协议

Contract 是调用者和方法之间的协议。调用者保证 precondition，方法保证 postcondition。Invariant 是对象或循环过程中必须保持的性质。

词白话解释小例子
Precondition调用前必须满足的条件。除法方法要求 denominator 不为 0。
Postcondition方法结束后承诺满足的条件。返回值等于输入数组中的最大值。
Invariant在某段过程前后一直保持的条件。银行账户余额始终非负。
Partial correctness如果程序终止，结果是对的。不证明一定停机。
Total correctness程序会终止，而且结果正确。需要额外证明 termination。

责任边界

如果调用者没有满足 precondition，方法不需要保证 postcondition。比如 sqrt(x) 规定 x≥0，调用者传 -1，那是调用者破坏协议。考试里要分清“方法应该防御所有坏输入”还是“contract 已经排除了坏输入”。

过关小题：方法 max(a) 的 precondition 可以是 a.Length > 0，postcondition 至少要说 result 是数组中某个元素，并且 result 大于等于每个元素。

内容理解题（先做 3 题）

核心答案（先确认概念）：调用者负责满足 precondition；方法在 precondition 成立时负责保证 postcondition。

为什么这是对的：合同（contract）的核心就是分责。就像租房合同里房东负责修水管、租客负责按时交租——谁的责任没履行就是谁的问题。在程序里：调用者如果传了空数组给 max() 方法（违反 requires a.Length > 0），方法返回什么都怪不到方法头上。反过来说，如果调用者传了合法输入，方法就必须兑现 postcondition 的所有承诺。

为什么这是对的：Invariant 是系统中不会改变的"定律"。循环 invariant 描述每次迭代前都成立的条件——比如 Find 循环中"min 是已扫描部分的最小值"每轮开始时成立。对象 invariant 描述对象在任何对外可见状态下都必须满足的条件——比如 BankAccount 的余额始终等于存入总额减去取出总额。Invariant 和 postcondition 的区别：postcondition 只要求方法结束时成立，invariant 要求过程中所有中间点都成立。

3. Hoare Logic：用前后条件证明程序片段

Hoare triple 写作 {P} C {Q}。意思是：如果执行程序片段 C 前 P 成立，并且 C 终止，那么执行后 Q 成立。

基本规则从最小块开始

结构证明怎么想小例子
Assignment倒着看 postcondition，需要赋值前什么条件。执行 x := x + 1 后要 x>0，执行前需要 x+1>0。
Sequence中间找一个 assertion 把两段接起来。C1; C2 先证明 {P}C1{R}，再证明 {R}C2{Q}。
Ifthen 和 else 两条路都要证明。条件 true 时证明一边，false 时证明另一边。
While找 invariant。进入循环前成立，每次迭代后仍成立，退出时加上循环条件为 false 推出 postcondition。

Loop invariant 的三件事

  1. Establishment。循环第一次开始前，invariant 已经成立。
  2. Preservation。假设某次迭代前 invariant 成立，执行循环体后它仍成立。
  3. Exit use。循环结束时，invariant 加上循环条件为 false，能推出 postcondition。

Variant / ranking function 证明终止

Total correctness 还要证明循环不会永远跑。Variant 是一个非负整数表达式，每次循环都严格变小。比如 n - i 每次迭代减少 1，且不能小于 0，就能证明循环终止。

Loop proof 模板：State the invariant. Prove it holds before the loop. Prove one arbitrary iteration preserves it. At loop exit, combine invariant with the negation of the guard to derive the postcondition. For total correctness, give a non-negative variant that decreases on every iteration.

核心答案（先确认概念）：如果执行 C 前 P 成立，并且 C 终止，则执行后 Q 成立。

为什么这是对的：Hoare triple 是程序证明的基本单位。它有三个部分：P 是前置条件（要求调用前成立），C 是一段代码，Q 是后置条件（要求执行后成立）。关键理解：它的含义是"如果 P 成立且 C 终止，那么 Q 成立"，不是"P 导致 C 运行并产生 Q"。这被称为 partial correctness——不证终止。把整个程序串成多个 triple，就是从一个 condition 推到下一个 condition 的链条。

题目语境怎么用：2025 Summer Q1(d) Find 题本质上让你写出 Hoare triple 形式的证明：P = 数组非空，C = Find 方法体，Q = min 是最小值。循环部分把 triple 拆成更小的 triple（每次迭代前后）。

卷面写法：开头写 {P} C {Q}。然后按程序结构分解：赋值用 assignment axiom 倒推；序列找中间 assertion R 连接；while 给 invariant I 并用三个证明义务分解。

常见失分点：把 triple 读成"P implies Q"——triple 有程序在中间，P 和 Q 之间隔着一个命令 C。忘了 triple 默认是 partial correctness，题目问 total correctness 时要加 variant。

核心答案（先确认概念）：Establishment、preservation、exit use。

标准答案（先写结论）：已扫描部分 a[0..i) 中，min 不大于每个已扫描元素，并且 1≤i≤a.Length。

为什么这是对的：Variant（也叫 ranking function）的工作原理是良基序（well-founded ordering）——每个自然数都有有限多个比它小的自然数。如果每次循环 variant 从 5 变到 4 变到 3...最终必然到达 0，而 variant 小于 0 是不可能（非负条件），所以循环必然终止。选 variant 的常见技巧：找循环中单调变化的值——比如 Find 中 i 不断增加，a.Length - i 就不断减小。

题目语境怎么用：2025 Summer Q1(e) Find total correctness：variant 选 a.Length - i。初始时 i=1，variant = a.Length-1（非负）。每次循环 i=i+1，variant 减小 1。当 variant=0 时 i=a.Length，循环条件为 false，退出。Dafny 中用 decreases a.Length - i。

卷面写法：先给出 variant 表达式，然后证明：(1) 每次进入循环体前 variant >= 0（引用 invariant 的范围条件）；(2) 在循环体的每次执行中，variant 的值严格减小；(3) 由于自然数不能无限减小，循环必须终止。

常见失分点：variant 选了但没证明非负——比如 variant 初始为 0 就已经非负了，需要说明；variant 在某些路径下不变（比如 if 分支里没改 variant）；用浮点数或非良基序作为 variant（必须是良基序，最常见的是自然数）。

标准答案（先写结论）：结合 invariant 的已扫描范围，当 i==a.Length 时已扫描部分就是整个数组。

题目语境怎么用：这是 exit use 步骤的标准模板。考试里你必须显式写出：(1) guard 取反得 i >= a.Length；(2) invariant 索引范围得 i <= a.Length；(3) 结合得 i == a.Length；(4) 代入 invariant 语义推出 postcondition。不能跳过中间步骤。

卷面写法：分四步推导。Step 1: guard negation -> i >= a.Length。Step 2: from invariant, i <= a.Length。Step 3: combine -> i == a.Length。Step 4: substitute into invariant -> forall j :: 0 <= j < a.Length ==> min <= a[j]。

常见失分点：跳过推导步骤直接写结论——阅卷人看不出推理链条。忘了 invariant 索引上界是 i <= a.Length（如果写成 i < a.Length 则退出时矛盾）。

标准答案（先写结论）：可选 a.Length - i，每次循环 i := i+1 后严格减少且非负。

为什么这是对的：选择 variant 的原则：找一个每次循环都单调变化的量。(1) 初始时 i=1，variant = a.Length - 1，由于 a.Length > 0（precondition），variant >= 0。(2) 循环体唯一改变 i 的语句是 i := i + 1，variant 严格减小 1。(3) 循环不会无限进行，因为 a.Length - i 是自然数且每次都减小，最终必定到达 0 或退出条件。

题目语境怎么用：2025 Summer Q1(e) 明确要你证明 total correctness。答完 partial correctness 后单独开一段：(i) 提出 variant；(ii) 证明非负；(iii) 证明严格减小；(iv) 结论：循环必然终止。

卷面写法：Variant: V = a.Length - i。(1) V >= 0 because i <= a.Length (from invariant). (2) After i := i + 1, V' = V - 1 < V. (3) Since V is a natural number and strictly decreases, loop must terminate.

常见失分点：variant 选了但不和 invariant 配合——variant 上下界通常来自 invariant 范围条件。没证明 variant 非负。variant 在某一分支下没减小。

标准答案（先写结论）：用 a.Length - i；每次循环 i 增加，variant 严格变小且非负。

为什么这是对的：variant 选 a.Length - i。三个条件：(1) 非负——a.Length - i >= 0 由 invariant 的 i <= a.Length 保证；(2) 每次循环后严格减小——循环体里只有 i := i + 1，variant 减少 1；(3) 自然数不能无限减小——所以循环必然终止。这是线性扫描的标准模式：看到从起点逐一扫描到终点的循环，variant 就是剩余要扫描的元素个数。

题目语境怎么用：Dafny 里写成 decreases a.Length - i。以后遇到任何逐一扫描数组的循环，variant 就是 a.Length - i。

卷面写法：decreases a.Length - i。Proof: (1) Non-negative: from invariant i <= a.Length; (2) Decrease: i := i + 1 so a.Length - (i+1) < a.Length - i; (3) Well-founded: natural numbers cannot decrease infinitely。

常见失分点：写了 decreases 但没解释为什么每次循环都减小。variant 选了但 invariant 没写 i <= a.Length——没有这个 invariant 证明不了 variant 非负。

标准答案（先写结论）：Ack 有两个递归参数，用 decreases m, n 按词典序证明递归调用整体变小。

为什么这是对的：Ackermann 函数递归调用有两种模式：一种 m 变小（m->m-1），另一种 m 不变 n 变小（n->n-1）。单靠一个 decreases 表达式（如 m+n）在某些分支可能不变。词典序 tuple (m, n) 规则：先比第一个分量，m 变小就通过；m 相等再比 n。这覆盖所有 Ackermann 递归路径。

题目语境怎么用：2025 Summer Q2(a) 给 Ackermann 函数骨架，让你补 decreases 并解释。具体指出：Ack(m-1, ...) 中 m 变小，第一个分量下降；Ack(m, n-1) 中 m 不变 n 变小，需比到第二个分量。

写 Dafny 循环 invariant 的台阶

  1. 范围 invariant。先写 0 ≤ i ≤ a.Length 这种最基本边界，否则数组访问很容易证明不了。
  2. 已处理部分 invariant。说明 0..i 这段已经满足什么性质。
  3. 结果变量 invariant。如果有 max、sum、found，说明它和已处理元素的关系。
  4. 退出时能推出 ensures。当 i == a.Length，已处理部分就是整个数组。
  5. 补 decreases。如果 verifier 看不出终止，写 decreases a.Length - i。

题目语境怎么用：2025 Summer Q2(b) 的 M(x,y) 循环可能涉及累加、乘幂或其他迭代计算。你要先明确方法的 ensures 要证明什么，然后逆推循环每轮应满足什么中间关系（invariant），最后找一个递减量（variant）。

卷面写法：分三步：(1) Identify bugs with fixes。(2) Invariant: [公式] with establishment/preservation/exit use。(3) Variant: [表达式], proof of non-negativity and decrease。三段逻辑明确分开。

常见失分点：只找 code error 不改 invariant（虽循环写对了但没给 invariant 证明不了正确性）；invariant 写太弱（比如说 i 在范围内没说 result 和输入关系），推不出 ensures。

标准答案（先写结论）：把对象一致性条件集中成可复用证明义务，方法前后都要维护它。

为什么这是对的：如果没有 Valid()，每个方法的 ensures 里都要重复写 Balance == deposits.total - withdrawals.total。代码一长容易漏。Valid() 把一致性条件封装成 named predicate，所有方法只需写 ensures Valid()。如果以后一致性条件变了（如增加"所有单笔交易金额 > 0"要求），只改 Valid() 一处即可。

题目语境怎么用：2025 Summer Q2(c) BankAccount 题明确让你定义 ghost predicate Valid()。定义后 constructor 要 ensures Valid()（建），deposit 要 ensures Valid()（维），withdraw 也要 ensures Valid()（维）。

卷面写法：ghost predicate Valid() reads this { Balance == deposits.total - withdrawals.total }。每个方法写 ensures Valid()。Constructor 额外 ensures 初始序列为空、Balance 为 0。解释 reads this 含义。

常见失分点：Valid() 定义了但某个 mutating method 忘了 ensures Valid()——Dafny 报该方法可能破坏 Valid()。reads 子句写错。Valid() 引用的 ghost 变量没声明。

标准答案（先写结论）：在已扫描区间 a[0..i) 内，min 不大于每个已扫描元素，并保持 1<=i<=a.Length。

为什么这是对的：Find 方法维护两个信息：当前扫描到哪里（i）和当前找到的最小值（min）。Invariant 必须同时约束两者。(1) i 的范围：1 <= i <= a.Length；(2) min 和已扫描部分的关系：forall j :: 0 <= j < i ==> min <= a[j]；(3) min 来自已扫描部分：exists j :: 0 <= j < i && min == a[j]。三条缺一不可。退出时 i == a.Length 则已扫描部分 = 整个数组，postcondition 直接推出。

题目语境怎么用：考试中 Find/线性搜索/最小值这类循环的 invariant 模板都是：范围 + 结果变量与已处理区间关系 + 结果变量来自已处理区间。把 i 想象成"已经处理到哪儿了"的指针。

卷面写法：三条 invariant：(1) invariant 1 <= i <= a.Length；(2) invariant forall j :: 0 <= j < i ==> min <= a[j]；(3) invariant exists j :: 0 <= j < i && min == a[j]。分别证明 establishment/preservation/exit use。

常见失分点：只写了 min <= a[j] 但没写 min 来自数组——verifier 可选 min = -inf 来满足第一条但不是真实最小值。索引从 0 开始写错导致多包含或漏掉元素。

标准答案（先写结论）：用 a.Length - i；每次循环 i 增加，variant 严格变小且非负。

为什么这是对的：variant 选 a.Length - i。三个条件：(1) 非负——a.Length - i >= 0 由 invariant 的 i <= a.Length 保证；(2) 每次循环后严格减小——循环体里只有 i := i + 1，variant 减少 1；(3) 自然数不能无限减小——所以循环必然终止。这是线性扫描的标准模式：看到从起点逐一扫描到终点的循环，variant 就是剩余要扫描的元素个数。

常见失分点：写了 decreases 但没解释为什么每次循环都减小。variant 选了但 invariant 没写 i <= a.Length——没有这个 invariant 证明不了 variant 非负。

5. Requirements、Refinement 和 Event-B：代码前的形式化

很多错误不是代码写错，而是需求本身含糊。Formalising requirements 的目标是把自然语言需求改成更精确、可检查、少歧义的描述。

FRET 的思路

FRET 关注需求句子的结构，比如在什么条件下、系统必须做什么、何时做、持续多久。它不只是翻译英文，而是逼你补齐触发条件、响应、时间范围和例外情况。

“valid input”valid 的范围不清楚。输入类型、范围、格式。
“normally”例外情况没列。条件、触发、异常路径。

Refinement 是逐步降低抽象层次

Abstract model 描述“系统应该满足什么”，concrete implementation 描述“系统具体怎么做”。Data refinement 要证明具体数据结构的行为保持抽象模型的性质。换句话说，实现可以更具体，但不能违背抽象承诺。

6. Model Checking：把系统看成状态图

Hoare/Dafny 更像证明程序片段的前后条件；model checking 更适合并发、协议、控制系统这类会在状态之间不断跳转的系统。它把系统建成状态图，再检查时间性质是否在所有或某些路径上成立。

Kripke structure 的部件

部件白话解释小例子
State系统某一刻的情况。door=open、door=closed。
Initial state系统开始状态。电梯一开始在 ground floor。
Transition一个状态能跳到另一个状态。按按钮后从 idle 到 moving。
Label某个状态下为真的命题。alarm、request、grant。
Path沿 transition 走出来的一条状态序列。系统未来可能发生的一条路线。

LTL：沿一条路径看时间

符号意思例子
X pnext：下一步 p 成立。按下按钮后下一状态灯亮。
F peventually：将来某时 p 成立。请求最终被服务。
G pglobally：从现在起一直 p 成立。永远不会同时开两扇互斥门。
p U quntil：p 一直成立直到 q 发生。等待状态持续到 grant 发生。

CTL：把路径量词写进公式

CTL 把路径量词和时间操作符配对。A 表示所有路径，E 表示存在一条路径。比如 AG safe 表示所有路径上一直 safe；EF error 表示存在一条路径最终到达 error。

公式人话常见用途
AG p所有路径上，永远 p。Safety：坏事永不发生。
EF p存在一条路径，最终 p。Reachability：某状态是否可达。
AF p所有路径上，最终 p。Liveness：好事一定会发生。
EG p存在一条路径，可以一直 p。是否能永远停留在某类状态。

Spin 和 NuSMV

Spin 常和 Promela、LTL、并发协议验证联系在一起；NuSMV 常用于符号模型检测和 CTL/LTL 性质。考试里重点不是工具命令细节，而是知道模型、性质、验证结果和 counterexample 的关系。

Model checking 答题顺序：说明系统状态是什么，transition 是什么，label 表示什么；把需求写成 LTL 或 CTL；说明 model checker 会探索状态空间；若性质不成立，counterexample 是一条违反性质的路径。

7. SAT、SMT 和 Z3：自动求解逻辑约束

SAT 问布尔公式能不能被某组 true/false 赋值满足。SMT 在 SAT 的基础上加入理论，例如整数、数组、位向量。Z3 是常见 SMT solver。

SAT 基础词

Literal一个变量或它的否定，例如 x 或 ¬x。
Clause多个 literal 的 OR，例如 x ∨ ¬y ∨ z。
CNF多个 clauses 的 AND，例如 (x∨y) ∧ (¬x∨z)。
Model让公式为 true 的赋值，例如 x=true, y=false。

DPLL / CDCL 的直觉

DPLL 可以理解成系统地猜变量真假，配合简化和回溯。CDCL 在冲突时学习新的 clause，避免以后重复走同样死路。考试通常不需要手写完整 solver，但要知道它们是 SAT solver 背后的搜索和剪枝思想。

SMT 比 SAT 多懂“理论”

SAT 只知道布尔真假，不知道 x+1>x 这种整数性质。SMT solver 懂一些背景理论，所以可以处理程序验证里常见的整数、数组、函数、等式约束。Dafny 背后也会把验证条件交给 SMT solver。

核心答案（先确认概念）：都在证明前后条件、循环不变量和终止性，只是 Dafny 让 verifier 帮忙检查。

卷面写法：先声明两者的共同理论基础（Hoare logic），再解释 Dafny 如何自动化这个理论。可以给同一个 Find 例子：Hoare 版本手工推导 vs Dafny 版本 annotation。

常见失分点：把 Dafny 当成完全不同的东西——忘记了 Dafny 就是 Hoare logic 的工具实现。Hoare 证明了但不提 invariant 对应的 establishment/preservation/exit use。

核心答案（先确认概念）：先定义状态和 transition，再写性质，最后解释 model checker 探索状态空间和 counterexample。

为什么这是对的：标准答题顺序：(1) 定义系统状态——什么构成一个状态（各变量取值组合）；(2) 定义 transitions——什么操作导致状态跳转；(3) 定义 labels——每个状态下哪些命题为真；(4) 把需求写成 LTL 或 CTL 公式；(5) 说明 model checker 会探索状态空间找反例；(6) 如果性质不成立，counterexample 是一条违反性质的路径。

标准答案（先写结论）：用 G(request -> F response) 表示每次 request 后最终出现 response。

为什么这是对的：用 G(request -> F response) 表示：在每一条执行路径的每一个状态，如果出现 request，那么最终一定出现 response。G 是 globally（每时每刻），-> 是 implication，F 是 finally（最终）。这属于 liveness property。和 safety 不同：safety 关心坏事永不发生，liveness 关心好事最终发生。

题目语境怎么用：2025 Summer Q4 大概率让你读并发模型然后写 LTL 公式。先判断是 safety 还是 liveness，再选对 temporal operator：safety 用 G ! ，liveness 用 G (cause -> F effect) 或直接用 F goal。

卷面写法：先写自然语言（每次请求最终都有响应），再写 LTL formula G(request -> F response)，逐符号解释：G 所有路径所有状态，-> 左边成立右边须成立，F 当前或未来。最后说明属 liveness property。

常见失分点：把 G(request -> F response) 写成 F(response)（后者只要求至少一次 response）。把 liveness 当 safety 写 G !(!response)（含义模糊不标准）。

标准答案（先写结论）：CDCL 在冲突后学习 clause，避免重复同类失败；stochastic search 更多靠随机移动，缺少这种系统学习。

为什么这是对的：想象你在走迷宫。Stochastic search（随机搜索）像闭着眼睛乱走——某条路撞墙了下次可能又走同一条。CDCL 像会做笔记的人：每次撞墙（conflict），它分析"为什么撞了"，然后写一条新规则（learned clause），以后再也不走那条路。learned clause 本质是简洁的命题逻辑句子，排除了导致冲突的那组赋值。随着搜索推进，learned clauses 越积越多，搜索空间越来越小。

题目语境怎么用：2025 Summer Q3(d) 问 CDCL 优势，你要对比两种策略的核心机制：stochastic 靠随机游走+偶尔重启碰运气；CDCL 靠冲突分析+学习 clauses+非时序回溯（backjump 到真正导致冲突的决策点）。

卷面写法：对比格式：Stochastic: random walks, local moves, restarts, no memory of failures。CDCL: conflict analysis, clause learning, non-chronological backtracking, learned clauses prune search space。总结：CDCL 从失败中学习，stochastic 不学习。

常见失分点：只说 CDCL "更好"但不解释学习机制；把 CDCL 和 DPLL 搞混（DPLL 是基础回溯框架，CDCL 在 DPLL 上加 conflict learning）；不提 learned clause 或 backjump。

7. SAT、SMT 和 Z3：自动求解逻辑约束

SAT 问布尔公式能不能被某组 true/false 赋值满足。SMT 在 SAT 的基础上加入理论，例如整数、数组、位向量。Z3 是常见 SMT solver。

SAT 基础词

Literal一个变量或它的否定，例如 x 或 ¬x。
Clause多个 literal 的 OR，例如 x ∨ ¬y ∨ z。
CNF多个 clauses 的 AND，例如 (x∨y) ∧ (¬x∨z)。
Model让公式为 true 的赋值，例如 x=true, y=false。

DPLL / CDCL 的直觉

DPLL 可以理解成系统地猜变量真假，配合简化和回溯。CDCL 在冲突时学习新的 clause，避免以后重复走同样死路。考试通常不需要手写完整 solver，但要知道它们是 SAT solver 背后的搜索和剪枝思想。

SMT 比 SAT 多懂“理论”

SAT 只知道布尔真假，不知道 x+1>x 这种整数性质。SMT solver 懂一些背景理论，所以可以处理程序验证里常见的整数、数组、函数、等式约束。Dafny 背后也会把验证条件交给 SMT solver。

内容理解题（先做 3 题）

核心答案（先确认概念）：问一个布尔公式是否存在 true/false 赋值让它成立。

为什么这是对的：SAT 问的是一个布尔公式是否可被某组 true/false 赋值满足。例如 (x OR y) AND (NOT x OR z) 可满足——设 x=true,y=false,z=true 即成立。若没有任何赋值能满足就说 unsat。SAT 是计算机科学最基础 NP-complete 问题，但现代 CDCL solver 在实践中非常高效。

题目语境怎么用：2025 Summer Q1(b) 让你解释 satisfiable/unsatisfiable/tautology——直接对应 SAT 基本概念。2025 Summer Q3(d) 考 CDCL solver。

核心答案（先确认概念）：SMT 还懂整数、数组、等式等背景理论，因此更接近程序验证中的约束。

为什么这是对的：SAT 只知布尔变量真假，不知数字、数组、函数的性质。SMT 在 SAT 基础上加入"背景理论"——整数算术（知道 x+1 > x 永远为真）、数组理论（读写公理）、位向量等。Dafny verifier 背后就是把验证条件转成 SMT formula 交给 Z3 解。所以 SMT 更接近真实程序验证——程序约束不只布尔逻辑还有整数范围、数组索引。

为什么这是对的：想象你在走迷宫。Stochastic search（随机搜索）像闭着眼睛乱走——某条路撞墙了下次可能又走同一条。CDCL 像会做笔记的人：每次撞墙（conflict），它分析"为什么撞了"，然后写一条新规则（learned clause），以后再也不走那条路。learned clause 本质是简洁的命题逻辑句子，排除了导致冲突的那组赋值。随着搜索推进，learned clauses 越积越多，搜索空间越来越小。

卷面写法：对比格式：Stochastic: random walks, local moves, restarts, no memory of failures。CDCL: conflict analysis, clause learning, non-chronological backtracking, learned clauses prune search space。总结：CDCL 从失败中学习，stochastic 不学习。
