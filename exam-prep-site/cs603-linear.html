CS603 教材线性精讲：从逻辑句子学到 Dafny 和模型检测

CS603 的核心是把“程序应该正确”变成可以检查的数学句子。先学怎样写性质，再学怎样证明程序片段满足性质，再学 Dafny 怎样让工具帮你证明，最后学状态系统、时间性质和 SAT/SMT 自动求解。

性质 Logic Hoare Dafny LTL / CTL SAT / SMT / Z3 考试台阶

0. 形式化验证到底在证明什么

Testing 用有限例子找 bug；formal verification 试图证明一类输入或一类行为都满足某个性质。这里的“性质”就是系统必须一直保持或最终达到的规则。

普通说法形式化说法为什么要形式化
余额不能变成负数。balance ≥ 0可以让工具或证明检查每条路径是否保持。
排序后数组仍包含原来的元素。输出是输入的 permutation。只检查“有序”不够，还要检查没丢元素。
请求最终会得到回应。temporal property: eventually response。这种性质关心时间和路径，不只是程序结束时的值。

四类正确性问题

  1. 前后条件。如果开始时满足 precondition，程序结束后是否满足 postcondition。
  2. 循环性质。循环每次迭代后是否保持 invariant，循环是否最终终止。
  3. 状态系统性质。系统在多个状态之间跳转时，是否会到达坏状态，是否最终到达好状态。
  4. 逻辑可满足性。一组约束是否存在某种赋值让它们同时成立。

过关小题：“程序不会崩溃”太模糊。更可证明的写法是：对所有满足 0 ≤ i < a.Length 的输入，访问 a[i] 不越界；或者方法返回时 result ≥ 0。

对应材料：1. ModuleOverview.pdf、Software-Fails-Watch.pdf、FormalMethodsinIndustry.pdf、Design by Contract.pdf、Programming by Contract.pdf。

1. Logic：把自然语言改写成可检查的句子

Logic 是 CS603 的语法地基。它让我们把“如果输入有效，那么输出非负”这类话，写成能推理、能检查的公式。

命题逻辑和谓词逻辑

概念白话解释例子
Proposition有真假值的完整句子。x > 0 在给定 x 后有真假。
Connective连接真假句子的符号。∧ and、∨ or、¬ not、⇒ implies。
Predicate带变量的性质。Even(n) 表示 n 是偶数。
Quantifier说明变量范围。∀ for all，∃ exists。

几个考试常见词

Satisfiable存在某种赋值让公式为 true。例如 x>0 ∧ x<3 可满足。
Unsatisfiable没有任何赋值能让公式为 true。例如 x>0 ∧ x<0 不可满足。
Tautology无论变量怎样取值都为 true。例如 P ∨ ¬P。
Model让公式成立的一组赋值。SAT/SMT solver 返回的 model 就是一份可行赋值。

Implication 最容易误读

P ⇒ Q 的意思是：只要 P 成立，Q 必须成立。如果 P 本来不成立，整句话不会被违反。程序证明里经常写“如果 precondition 成立，那么 postcondition 成立”。

从英语到逻辑的台阶

  1. 先找对象。例如数组、下标、余额、状态。
  2. 再找范围。“所有合法下标”对应 ∀ i :: 0 ≤ i < a.Length。
  3. 再找性质。例如 a[i] ≥ 0。
  4. 最后组合。用 implication 表达条件，用 conjunction 表达多个要求。

对应材料：1_logic_603.pdf、labZ3_PropLogic.pdf、Z3Exercises.pdf。

2. Design by Contract：把方法当成一份协议

Contract 是调用者和方法之间的协议。调用者保证 precondition，方法保证 postcondition。Invariant 是对象或循环过程中必须保持的性质。

词白话解释小例子
Precondition调用前必须满足的条件。除法方法要求 denominator 不为 0。
Postcondition方法结束后承诺满足的条件。返回值等于输入数组中的最大值。
Invariant在某段过程前后一直保持的条件。银行账户余额始终非负。
Partial correctness如果程序终止，结果是对的。不证明一定停机。
Total correctness程序会终止，而且结果正确。需要额外证明 termination。

责任边界

如果调用者没有满足 precondition，方法不需要保证 postcondition。比如 sqrt(x) 规定 x≥0，调用者传 -1，那是调用者破坏协议。考试里要分清“方法应该防御所有坏输入”还是“contract 已经排除了坏输入”。

过关小题：方法 max(a) 的 precondition 可以是 a.Length > 0，postcondition 至少要说 result 是数组中某个元素，并且 result 大于等于每个元素。

3. Hoare Logic：用前后条件证明程序片段

Hoare triple 写作 {P} C {Q}。意思是：如果执行程序片段 C 前 P 成立，并且 C 终止，那么执行后 Q 成立。

基本规则从最小块开始

结构证明怎么想小例子
Assignment倒着看 postcondition，需要赋值前什么条件。执行 x := x + 1 后要 x>0，执行前需要 x+1>0。
Sequence中间找一个 assertion 把两段接起来。C1; C2 先证明 {P}C1{R}，再证明 {R}C2{Q}。
Ifthen 和 else 两条路都要证明。条件 true 时证明一边，false 时证明另一边。
While找 invariant。进入循环前成立，每次迭代后仍成立，退出时加上循环条件为 false 推出 postcondition。

Loop invariant 的三件事

  1. Establishment。循环第一次开始前，invariant 已经成立。
  2. Preservation。假设某次迭代前 invariant 成立，执行循环体后它仍成立。
  3. Exit use。循环结束时，invariant 加上循环条件为 false，能推出 postcondition。

Variant / ranking function 证明终止

Total correctness 还要证明循环不会永远跑。Variant 是一个非负整数表达式，每次循环都严格变小。比如 n - i 每次迭代减少 1，且不能小于 0，就能证明循环终止。

Loop proof 模板：State the invariant. Prove it holds before the loop. Prove one arbitrary iteration preserves it. At loop exit, combine invariant with the negation of the guard to derive the postcondition. For total correctness, give a non-negative variant that decreases on every iteration.

对应材料：2_hoarelogic_RM.pdf、Hoare Logic Summary Sheet.pdf、Exercises from Lecture Hoare Logic if and assign.pdf、Exercises from Lecture_ Hoare Logic Loop.pdf、Hoare Logic Lab B.pdf、Hoare_LabA-1.pdf。

4. Dafny：把证明写进程序，让 verifier 检查

Dafny 是一种带验证器的语言。你不仅写程序，还写 contract、assertion、invariant 和 termination 信息。Verifier 会尝试证明这些性质，而不是只运行几个测试例子。

Dafny 关键词

关键词白话解释考试里怎么用
requiresprecondition，调用前要满足。写输入范围、非空数组、非负数。
ensurespostcondition，方法结束后要满足。写返回值和输入之间的关系。
assert要求 verifier 在此处证明某句为 true。帮助检查中间推理，也帮助定位证明断点。
invariant循环每次开始时都成立的性质。数组循环要写范围和已处理部分的性质。
decreases证明递归或循环会终止的下降量。递归参数变小，或循环剩余长度变小。
ghost只为证明存在，运行时不需要的变量或方法。记录数学信息，帮助 verifier，不影响程序输出。

写 Dafny 循环 invariant 的台阶

  1. 范围 invariant。先写 0 ≤ i ≤ a.Length 这种最基本边界，否则数组访问很容易证明不了。
  2. 已处理部分 invariant。说明 0..i 这段已经满足什么性质。
  3. 结果变量 invariant。如果有 max、sum、found，说明它和已处理元素的关系。
  4. 退出时能推出 ensures。当 i == a.Length，已处理部分就是整个数组。
  5. 补 decreases。如果 verifier 看不出终止，写 decreases a.Length - i。

Arrays、recursion、classes 分别难在哪里

主题难点小台阶
Recursion要证明每次调用更接近 base case。写清 base case，递归调用参数变小，用 decreases。
LoopsVerifier 不会自动知道循环做了什么。用 invariant 描述“已经完成的部分”。
Arrays要同时证明下标合法和元素性质。范围 invariant + forall 描述已处理区域。
Objects对象状态要长期保持一致。写 class invariant 或 ghost predicate，constructor 和 mutator 都要维护它。

Dafny 解题顺序：先读 requires/ensures，确定要证明什么；再找循环或递归；写范围 invariant；写已处理部分 invariant；必要时加 assert 帮 verifier 看到中间事实；最后检查 termination。

对应材料：1.Dafny_Intro.pdf、2.Dafny.pdf、3.RecursionandTermination.pdf、4.DafnyLoops.pdf、5.Iterative.pdf、Arrays I.pdf、7.ArraysII_short.pdf、LinearSearch.pdf、8.ObjectI2023.pdf。

5. Requirements、Refinement 和 Event-B：代码前的形式化

很多错误不是代码写错，而是需求本身含糊。Formalising requirements 的目标是把自然语言需求改成更精确、可检查、少歧义的描述。

FRET 的思路

FRET 关注需求句子的结构，比如在什么条件下、系统必须做什么、何时做、持续多久。它不只是翻译英文，而是逼你补齐触发条件、响应、时间范围和例外情况。

需求弱点为什么危险形式化时补什么
“quickly”没人知道多快算快。具体时间界限，例如 within 2 seconds。
“valid input”valid 的范围不清楚。输入类型、范围、格式。
“normally”例外情况没列。条件、触发、异常路径。

Refinement 是逐步降低抽象层次

Abstract model 描述“系统应该满足什么”，concrete implementation 描述“系统具体怎么做”。Data refinement 要证明具体数据结构的行为保持抽象模型的性质。换句话说，实现可以更具体，但不能违背抽象承诺。

对应材料：FormalisingRequirements.pdf、2026_Slides_Formalise.pdf、EventB-Summary-refcard.pdf、2024_Slides_FMAS.pdf。

6. Model Checking：把系统看成状态图

Hoare/Dafny 更像证明程序片段的前后条件；model checking 更适合并发、协议、控制系统这类会在状态之间不断跳转的系统。它把系统建成状态图，再检查时间性质是否在所有或某些路径上成立。

Kripke structure 的部件

部件白话解释小例子
State系统某一刻的情况。door=open、door=closed。
Initial state系统开始状态。电梯一开始在 ground floor。
Transition一个状态能跳到另一个状态。按按钮后从 idle 到 moving。
Label某个状态下为真的命题。alarm、request、grant。
Path沿 transition 走出来的一条状态序列。系统未来可能发生的一条路线。

LTL：沿一条路径看时间

符号意思例子
X pnext：下一步 p 成立。按下按钮后下一状态灯亮。
F peventually：将来某时 p 成立。请求最终被服务。
G pglobally：从现在起一直 p 成立。永远不会同时开两扇互斥门。
p U quntil：p 一直成立直到 q 发生。等待状态持续到 grant 发生。

CTL：把路径量词写进公式

CTL 把路径量词和时间操作符配对。A 表示所有路径，E 表示存在一条路径。比如 AG safe 表示所有路径上一直 safe；EF error 表示存在一条路径最终到达 error。

公式人话常见用途
AG p所有路径上，永远 p。Safety：坏事永不发生。
EF p存在一条路径，最终 p。Reachability：某状态是否可达。
AF p所有路径上，最终 p。Liveness：好事一定会发生。
EG p存在一条路径，可以一直 p。是否能永远停留在某类状态。

Spin 和 NuSMV

Spin 常和 Promela、LTL、并发协议验证联系在一起；NuSMV 常用于符号模型检测和 CTL/LTL 性质。考试里重点不是工具命令细节，而是知道模型、性质、验证结果和 counterexample 的关系。

Model checking 答题顺序：说明系统状态是什么，transition 是什么，label 表示什么；把需求写成 LTL 或 CTL；说明 model checker 会探索状态空间；若性质不成立，counterexample 是一条违反性质的路径。

对应材料：MCLecture1CS603.pdf、MCLecture2CS603.pdf、MCLecture3CS603.pdf、MCLecture4CS603.pdf、21-ltl_rm.pdf、22-ctl_rm.pdf、temporal-examples_2023.pdf、MCL41VerifyingWithSPIN_cs603.pdf、Verifying with Spin (TL).pdf、SpinTutorial_fromSpinroot.pdf。

7. SAT、SMT 和 Z3：自动求解逻辑约束

SAT 问布尔公式能不能被某组 true/false 赋值满足。SMT 在 SAT 的基础上加入理论，例如整数、数组、位向量。Z3 是常见 SMT solver。

SAT 基础词

Literal一个变量或它的否定，例如 x 或 ¬x。
Clause多个 literal 的 OR，例如 x ∨ ¬y ∨ z。
CNF多个 clauses 的 AND，例如 (x∨y) ∧ (¬x∨z)。
Model让公式为 true 的赋值，例如 x=true, y=false。

DPLL / CDCL 的直觉

DPLL 可以理解成系统地猜变量真假，配合简化和回溯。CDCL 在冲突时学习新的 clause，避免以后重复走同样死路。考试通常不需要手写完整 solver，但要知道它们是 SAT solver 背后的搜索和剪枝思想。

SMT 比 SAT 多懂“理论”

SAT 只知道布尔真假，不知道 x+1>x 这种整数性质。SMT solver 懂一些背景理论，所以可以处理程序验证里常见的整数、数组、函数、等式约束。Dafny 背后也会把验证条件交给 SMT solver。

Z3 的基本交互

动作意思结果
声明变量告诉 solver 有哪些对象和类型。例如整数 x、布尔 p。
加入 constraints把必须满足的条件放进去。例如 x > 0 and x < 3。
check-sat问是否有解。sat、unsat 或 unknown。
get-model如果 sat，取一组具体赋值。例如 x = 1。
