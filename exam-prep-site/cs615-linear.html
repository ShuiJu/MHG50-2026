CS615 linear course

CS615 教材线性精讲：从一次请求反推整门 Web 工程

CS615 的教材不是在随机介绍 Rails、Spring、Node、React 和 AI，而是在反复训练同一件事：一个 Web 系统怎样分层、怎样读写数据、怎样暴露 API、怎样保护和部署。复习时先建立这条主线，再把每份讲义放到主线上的对应位置。

0. 从教材反推学习流程

从本地 CS615 材料看，课程不是先教某一个框架的语法，而是先给 Web 系统的骨架，再用不同技术栈反复实现同一套骨架。现在的学习流程应该按“概念地基 -> 后端框架 -> 前后端 API -> 工程质量 -> 交付运维 -> sample 新题型”走。

审视后的结论

  1. 不能先从 React 或 AI 开始。它们很显眼，但没有 request、API、数据库、分层这些地基，React 只是页面语法，AI agents 也只会变成空泛议论文。
  2. Rails 和 Spring 不是两门孤立技术。它们都在训练“request 进 controller，业务规则放合适位置，数据访问有边界”。区别是 Rails 更强调 convention，Spring 更强调显式分层和 dependency injection。
  3. WSFE 是桥。它把后端 API、MongoDB/Mongoose、Express 和 React 连起来，所以应该放在 Rails/Spring 之后读，而不是只当 JavaScript 笔记。
  4. Security 和 CICDOBS 要回贴到主线。安全不是单独背 XSS；CI/CD 也不是单独背 pipeline。它们都服务于“这个 Web app 如何安全、可靠、可维护地运行”。

内容理解题（先做 3 题）

核心答案（先确认概念）：React 和 AI 很显眼，但它们依赖 request、API、数据库和分层架构这些地基。先学它们会变成背名词。

为什么这是对的：React 只是浏览器端组件库，必须通过 fetch/axios 向后端 API 要数据，后端 API 必须连数据库才能持久化。没学过 request→controller→service→database 这条线，写 React 组件时不知道 useEffect(() => { fetch('/api/books') }) 背后服务器在做什么分层和校验。让 AI agent 生成代码的人如果不理解三层架构，就判断不了 agent 是否把业务规则写进了不该写的地方。先建立分层地基，再学上层工具，才能把工具放对位置。

题目语境怎么用：遇到 2025 Q1 full-stack 题，题目要求解释"用户点查看收据"的完整流程——必须从 browser→server→database 一路画下去，React 只负责 browser 那一段渲染。只学过 React 会卡在"服务器怎么查数据库、怎么验证权限"。

卷面写法：先画 browser→route→controller→service→database→response 主线，再把 React 或 AI agent 放进对应位置（React 在 presentation 层负责渲染和交互，AI agent 在 SDLC 中协助生成但需要人审架构）。

常见失分点：把"full-stack"等同于"会 React + Node"，但说不清每一层在 request 链条里的具体职责。评分看职责边界，不是技术名词数量。

核心答案（先确认概念）：它们是同一条 Web request 主线的两种实现：Rails 训练 MVC 和 convention，Spring 训练显式分层、REST 和 dependency injection。

为什么这是对的：Rails 用 scaffold 命令三行搭出能跑 CRUD 的 app——URL 进 routes.rb→controller action→Active Record model→数据库→view 渲染，全程靠约定自动串联。Spring 把同样的事拆得更清楚：@RestController 只管 HTTP、@Service 只管业务、@Repository 只管数据访问，三者通过 @Autowired 连接。一个直观一个显式，训练的是同一个骨架。学 Spring 的显式分层后再回头看 Rails controller，就能准确判断哪些代码块该抽到 service object。

题目语境怎么用：Sample Q3 的 Rails controller 题要求指出 controller 写了太多业务逻辑——正好是 Spring 里 @Service 层该做的事。两个框架互相参照，比孤立学一个更能看清职责边界。

卷面写法：比较时不要说"Rails 简单 Spring 复杂"。要说 Rails 靠目录约定把代码自动串起来适合快速开发，Spring 靠显式注解和依赖注入让每层职责边界无歧义，适合大型团队和长期维护。

常见失分点：只罗列功能清单（Active Record、JPA）但不解释它们各自怎样组织 request→controller→service→database 这条线的分工。

核心答案（先确认概念）：它把 Express/Mongoose 后端和 React 前端连接起来，让前面的 API、数据库和后面的页面交互合成完整 Web app。

为什么这是对的：前面的 DB_WS_C 教了 SQL 的表和 ACID，ArchitecturalPatterns 教了三层架构，Rails/Spring 教了后端 MVC 和 REST。WSFE 接手后用完全不同的技术栈（JavaScript 替代 Ruby/Java，MongoDB 替代 PostgreSQL）实现同一套分层骨架——Express 的 route+controller 对应 Rails 的 route+controller，Mongoose model 对应 Active Record，React 调用之前 Rails/Spring 已经暴露的 REST API 并渲染成可交互页面。它证明架构思想和技术栈是两回事——换技术栈不影响分层骨架。

题目语境怎么用：2023 Summer Q2 同时考 Node event loop 输出和 Express 分层，2025 Q3 考 React 的 component/props/state——这些题都在 WSFE 范围。答题时用前面 Rails/Spring 学的分层概念来组织答案，而不是散点罗列。

卷面写法：答 Node/Express/React 相关题时，主动把技术放回三层架构：React 是 presentation 层、Express route+controller 是 HTTP 入口、Mongoose model 是 data access 层。解释这个分层如何让前后端独立开发和测试。

常见失分点：把 WSFE 当 JavaScript 语法速成——学完只记得 Express 的 req.params 和 React 的 useState 写法，说不出这些技术在三层架构的哪个位置、换一个框架会不会影响架构。

考试题目题（再做 5 题）

标准答案（先写结论）：因为题目直接要求把 monolith 和 microservices 映射到 Presentation/Application/Data 三层。

为什么这是对的：FinTrack 的 Approach A 用 React 做前端、Spring Boot 做后端、PostgreSQL 做数据库——恰好是 presentation/application/data 的教科书对应。没先学 3-tier 就看不出 Approach B 的 microservices 本质上是在 application tier 内部横向拆，但每个 service 内部仍然有 logic 层和 data 层。3-tier 是两种方案共同的底盘，不先建立这个底盘，比较 monolith 和 microservices 就变成空谈"哪个更灵活"而没有具体维度。FinTrack 的实时股价给 application tier 施加高吞吐压力，交易历史给 data tier 施加一致性和事务压力——把这些压力放进三层框架，才能分别判断每一层该选什么技术。

题目语境怎么用：FinTrack 题面里实时股价和交易历史是两种完全不同的数据压力：前者需要缓存/推送/高频刷新，后者需要 ACID 事务和精确对账。答题时把这两种压力分别映射到不同层，而不是笼统说"用 microservices"。

卷面写法：先用三层框架描述两种方案，再按 scalability、data consistency、deployment 逐维度比较。每个维度要落回 FinTrack 的具体场景（价格服务扩展、交易记录一致性、小团队运维成本）。

常见失分点：只说"microservices 更 scalable"但不解释 FinTrack 的哪一层、什么数据需要独立扩展。实时股价靠缓存和异步推送解决，交易历史靠数据库事务保证——扩展压力完全不同。

标准答案（先写结论）：因为 full-stack 和 shopping habits database choice 都要求先懂 server/database 分工、SQL/NoSQL 和 transaction。

为什么这是对的：2025 Q1(a) 要求向 intern 解释 full-stack——必须说清浏览器请求到数据库查询的完整数据流，不只是三个名词。Q1(b) 直接要求选 SQL 或 NoSQL——真正考的是你能不能从题面数据（items、price、date、unique receipt number）推导出它天然适合关系型表结构：Customers→Receipts→ReceiptItems，需要 foreign key、unique constraint、transaction 防止半张收据或总价不一致。如果没学过 ACID，甚至说不清为什么 receipt 总价必须和 items 明细加总相等——这不是"看起来整洁"，是金融数据的写错代价很高。

题目语境怎么用：receipt/shopping habits 场景可拆成 Customers、Receipts、ReceiptItems；receipt number 要唯一，items 与 total 要对上。clickstream/document 场景才更强调字段灵活、追加写入和横向扩展。

卷面写法：先给出选择，再列题面实体和关系，再解释 key、join、constraint、transaction 是收据系统的硬需求。最后补一句：如果题目改成高量点击事件（字段不固定、写入极大、关系查询少），MongoDB 才更合适。

常见失分点：不要只写 SQL=table、NoSQL=document。也不要看到 shopping/data 就自动选 NoSQL；要看题面到底是收据记录还是高量行为事件。

标准答案（先写结论）：因为 CI/CD 和 observability 要服务于生产 Web app 的交付、回滚、质量和故障排查。

为什么这是对的：Sample Q4(a) 让你各用一句话定义，但 Q4(b) 接着让你用 production bug 场景解释二者如何配合。真正的考试逻辑是：代码 push 后 CI 自动跑 lint+单元测试+集成测试阻止坏代码进主干，CD 把通过测试的代码部署到 production。上线后 error rate 突然飙升，observability 的 trace 顺着 request ID 定位到是哪个 service 的哪次 deploy 引入的 bug，metric 显示是哪个时间段开始异常的。这两套机制如果不能连起来——比如只知道"有 pipeline"但不知道 pipeline 失败后 trace 怎么帮助定位——就拿不全分。FinTrack 微服务架构下，一次部署更新了价格服务但忘记更新交易服务，CI 集成测试本可在 staging 发现不兼容，如果有漏网之鱼上线，observability 的 trace 会显示请求在价格服务正常、在交易服务 500。

题目语境怎么用：用一条 production bug 故事线串起来：commit→CI test fail/prevent→CD deploy→log alert→trace pinpoint→metric confirm→rollback。每一步解释为什么这个环节存在。

卷面写法：定义两者时不要孤立写。用 production bug 时间线把 CI/CD（上线前防护）和 observability（上线后定位）串成闭环。每一步标注是 CI/CD 还是 observability 的职责。

常见失分点：把 CI/CD 写成"自动部署"，observability 写成"监控告警"，两者没有任何交集。评分看你能不能解释一套完整的"写出代码→上线→出事→定位→修好"闭环。

标准答案（先写结论）：不要拼接 '...'+userInput；用 parameterized query/prepared statement、输入验证和最小数据库权限。

为什么这是对的：SQL injection 的根本原因是程序把用户输入和 SQL 指令混在一根字符串里。当代码写 "SELECT * FROM users WHERE name = '" + userInput + "'" 时，攻击者输入 ' OR '1'='1' -- 就把 WHERE 条件改成永远为真。Prepared statement 的原理是把 SQL 结构和参数分开传——数据库先编译好 SELECT * FROM users WHERE name = ?，再把用户输入当字符串值传入，不管输入里有什么特殊字符都不会被当 SQL 执行。这就是"数据和代码分离"——值就是值，SQL 就是 SQL。输入验证和最小数据库权限是 defense-in-depth——就算 prepared statement 被绕过，还有另外两层防线。

题目语境怎么用：SQL injection 用登录框、搜索框或表单输入说明拼接 SQL 的风险；XSS 用评论、用户名或 profile 字段说明输出到页面时为何要转义。

卷面写法：三段式：定义攻击（用户输入被当成 SQL 执行）、贴题例子（登录框输入 ' OR 1=1 -- 跳过密码检查）、防御措施（prepared statement 让输入只作为 value，ORM 安全 API 避免手拼，最小权限限制损害）。

常见失分点：只写 sanitize 不够。sanitize 是猫捉老鼠，prepared statement 是"把老鼠挡在门外"——从根本分离代码和数据，才是正确答案的核心。

标准答案（先写结论）：用用户查看 receipt：browser 发 GET /receipts/7，server 查数据库并执行业务规则，database 返回收据和 items。

为什么这是对的：这道题考的不是背 full-stack 定义，而是把"用户点了一个按钮"拆成一次具体的、有技术细节的 HTTP 往返。高分例子必须具备三个要素：(1) 真实 URL——GET /receipts/7 而不是模糊的"用户请求数据"；(2) 真实业务校验——服务器验证这个 receipt 属于当前用户、检查 items 价格和数量、计算 total；(3) 真实数据操作——SELECT receipts JOIN receipt_items WHERE receipt_id=7 返回含 items 列表的 receipt 对象。三个要素缺一个，例子就还是空泛的"前端请求后端返回"。

题目语境怎么用：receipt/shopping habits 场景可拆成 Customers、Receipts、ReceiptItems；receipt number 要唯一，items 与 total 要对上。举例时体现查询权限检查（这张收据是否属于当前用户）和数据结构（一对多关系的 JOIN）。

卷面写法：先写具体场景（哪个用户、哪张收据、什么动作），再按 request→route→controller→model→database→response 的顺序拆解每一步在干什么，每一步写一句"为什么需要这步"。

常见失分点：写"browser request server"这样太抽象。评分看你能不能把"用户点了一个按钮"拆成一次完整的 HTTP 往返和数据库查询。

本节过关门槛

1. 一次网页请求经过哪些地方

先想象一个普通动作：用户打开购物网站，搜索商品，然后点“加入购物车”。浏览器不会自己完成所有事情。它会向服务器发送 request，服务器处理业务规则和数据库，再把 response 发回来。

最小请求链条

  1. 用户操作浏览器。浏览器显示 HTML/CSS，运行 JavaScript，收集点击、输入、选择。
  2. 浏览器发 HTTP request。Request 里有 URL、method、headers，有时还有 body。URL 说明找谁，method 说明想做什么，body 放表单或 JSON。
  3. 服务器用 route 找处理代码。Route 是 URL 和 controller action 的对应规则。
  4. Controller 协调业务。Controller 读取参数，调用 service/model/repository，检查权限和输入。
  5. 数据库保存或查询长期数据。订单、用户、商品、收据不会只放在内存变量里。
  6. 服务器返回 response。Response 可能是 HTML 页面，也可能是 JSON，让 React 或浏览器继续渲染。

基础词先落地

Client / Browser用户实际操作的一端。浏览器负责显示页面、运行前端 JavaScript、发 request。
Server / Backend处理请求的一端。它运行 Rails、Spring、Express 等框架，执行规则并返回结果。
Database长期保存数据的系统。服务器重启后，用户、订单、收据仍然存在。
API给程序调用的入口。前端不需要知道后端内部代码，只按 API 传入和取回数据。
JSON键值对文本格式，适合浏览器和服务器交换数据，例如 {"id":15,"total":90}。
Business logic业务规则，例如库存是否够、用户是否能下单、折扣怎样计算。

最小完整链条：Browser sends GET /products?q=book. Router maps the URL to a controller action. The controller calls business logic and data access code. The database returns matching records. The server returns HTML or JSON. The browser renders the result.

内容理解题（先做 3 题）

核心答案（先确认概念）：Browser 发 request，router 找 controller，controller 调业务和数据访问，database 返回数据，server 返回 HTML 或 JSON。

为什么这是对的：每步都有必须存在的理由。浏览器不能直接连数据库——因为数据库没有用户认证和业务校验能力，任何拿到连接串的人都能绕过价格检查、权限验证。Router 不能代替 controller——因为 route 只是 URL 匹配规则，controller 才是"接到请求后做什么"的统一决策点。如果跳过 controller 直接让 route 调数据库，就失去了"统一验证输入、检查权限、记录审计日志"的集中控制点。这六步每一步被跳过或合并，都会导致某个职责的缺失。

题目语境怎么用：2025 Q1(a) 要求向 intern 解释 full-stack——如果只说 browser→server→database 三个词，intern 不明白 server 里到底发生了什么。必须把 server 内部拆成 route→controller→business logic→data access，解释每段做了什么、为什么不能省略。

卷面写法：画链条时每步写一句"为什么存在"：Route——URL 到代码的映射让 API 可预测；Controller——统一入口做权限和参数检查；Business Logic——规则集中一处；Data Access——隔离具体数据库类型。

常见失分点：只列六个名词不解释职责边界。评分看能否说清"如果这步被跳过或合并到邻步会发生什么"。

核心答案（先确认概念）：GET 通常读取资源，POST 通常创建资源或提交数据。考试要把 method 和 CRUD 意图连起来。

为什么这是对的：GET 应该是幂等的（safe method）——无论调多少次 GET /receipts/7 都应返回同一张收据，不产生新数据、不改变系统状态。POST 则非幂等——连续两次 POST /orders 会创建两份订单。这就是为什么浏览器在刷新 POST 结果页面时弹出"可能重复提交"警告，但刷新 GET 页面没有。考试里如果把 method 和 CRUD 对应错了——比如用 GET 去创建订单——说明混淆了 safe method（只读不写）和非 safe method 的语义，相当于把"查看"和"下单"的动作搞混。

题目语境怎么用：设计 REST API 题里，如果题目说"用户查看收据列表"，应设计 GET /receipts；如果题目说"用户提交新收据"，应设计 POST /receipts。不能因为"反正都是 HTTP 请求"就混用。

卷面写法：先写 HTTP method→CRUD 映射（GET=Read, POST=Create, PUT/PATCH=Update, DELETE=Delete），再针对题目场景选 method 并解释为什么。例如"查看收据是读操作不改变服务器状态，所以用 GET"。

常见失分点：只说 GET 读 POST 写，不解释"读不能有副作用（safe method）、写通常非幂等"这个更深层语义，也不解释为什么 REST API 要遵循 HTTP method 语义。

核心答案（先确认概念）：前端按约定 URL 和 JSON 调用，后端隐藏内部实现。双方只要遵守接口，就可以分别开发和替换。

为什么这是对的：API 就像插座和插头的关系：只要双方同意"这个 URL 接受什么格式的请求、返回什么格式的 JSON"，前后端就可以完全并行工作。前端用 mock server 模拟 GET /api/products 返回假数据开发页面，后端用 Postman 直接测试 POST /api/orders 验证业务逻辑——不需要等前端做好表单。更关键的是，如果将来后端从 Rails 换成 Spring，只要 URL 和 JSON 格式不变，前端一行代码不用改。这就是 API 作为"契约"的价值：把"怎么实现"封装在接口后面，调用方只关心接口不关心内部。

题目语境怎么用：如果题目问"为什么微服务架构能让不同团队独立开发不同服务"，核心答案就是 API 契约——每个 service 暴露的 REST API 只要保持稳定，调用方就不关心 service 内部用 Java 还是 Node、数据库是 PostgreSQL 还是 MongoDB。

卷面写法：先定义 API 合同（URL + HTTP method + request/response format），再解释"黑盒"优势：调用方只知接口不碰内部实现，因此可以独立更换技术栈、独立测试、独立部署。

常见失分点：只说"API 连接前后端"但不解释"契约约束双方"。没有契约概念，换后端实现时前端就崩了——那就不叫分工，叫耦合。

考试题目题（再做 5 题）

标准答案（先写结论）：Browser 发 GET /receipts/{id}，server route 到 controller，controller 查 database，server 返回 HTML/JSON，browser 渲染。

为什么这是对的：拆链条的关键是每一步要有真实的技术动作。具体来说：(1) 浏览器构造 HTTP request：method=GET, URL=/receipts/7，headers 带 session cookie 证明身份；(2) 服务器 route 层匹配 URL pattern /receipts/:id 交给 ReceiptsController#show；(3) Controller 从 session 验证当前用户是否有权查看这张 receipt，再调用 Receipt.find(7) 和 ReceiptItem.where(receipt_id: 7)；(4) ORM 把它们翻译成 SQL 的 SELECT+JOIN；(5) 服务器把 receipt 对象和 items 数组组装成 JSON response；(6) 浏览器接收后 React 组件把 JSON 放进 state 并渲染收据详情。六步缺哪一步链条就不完整。

题目语境怎么用：receipt/shopping habits 场景可拆成 Customers、Receipts、ReceiptItems；链条里要体现权限检查（这张收据是否属于当前用户）和数据结构（一对多关系的 JOIN）。

卷面写法：每步写出"哪个组件做了什么、为什么不能跳过"：比如"Controller 必须检查用户身份——不能相信前端传过来的 receipt_id，因为攻击者可以修改 URL 查看别人的收据"。不是罗列步骤，而是解释每一步的安全或业务必要性。

常见失分点：只写 browser→server→database→browser 四个词，不展开 server 内部做了什么、不说明数据在每一步的格式变化（URL params→controller 变量→SQL queries→JSON→React state）。

标准答案（先写结论）：需要考虑频繁更新、缓存/streaming、服务吞吐和前端状态刷新，而不是只做一次普通页面加载。

为什么这是对的：普通"查看收据"是一次性 request-response：用户点击→服务器查一次数据库→返回结果→结束。但实时股价每秒都在变，如果还用同样的模式，浏览器每秒要发一次请求，10000 个用户同时看同一支股票就是每秒 10000 次数据库查询——数据库直接被打垮。因此 Application tier 必须引入缓存（把最新价格放在 Redis，数据库只做持久化）、推送机制（WebSocket/SSE 替代 polling，服务器主动推价格变化）、前端优化（只更新价格数字不重渲染整个组件树）。三个技术分别解决数据库压力、网络延迟、前端渲染性能。

题目语境怎么用：FinTrack 题面的"real-time"不是装饰词，它在工程上直接对应三种压力：读压力（高频查询需缓存层）、写压力（价格源每秒更新数千条）、推送压力（客户端需近实时更新）。答题时把这三种压力逐一映射到具体技术方案。

卷面写法：先区分"普通页面加载"和"实时数据推送"的根本差异——前者一次性、后者持续流。再分别说明：数据层怎么缓存（Redis）、传输层怎么推送（WebSocket）、前端怎么局部更新（只 re-render 变化字段）。

常见失分点：只写"用 WebSocket"或"加缓存"但不解释具体解决什么问题。WebSocket 解决"服务器主动推"，Redis 解决"避免每次查数据库"——两个技术的 target problem 完全不同。

标准答案（先写结论）：后端要检查身份、课程容量、权限和数据一致性，不能让前端直接写数据库。

为什么这是对的：前端代码在浏览器里运行——浏览器归用户控制，用户可以用 DevTools 修改任何 JavaScript 变量和 HTML 表单。如果让前端直接写数据库：(1) 学生可以把"选课数量上限"从 5 改成 999；(2) 可以在控制台直接调数据库写入 API 绕过选课时间限制；(3) 不存在"两个学生同时抢最后一个名额"的并发控制。后端 controller 是唯一可信执行环境：从 session token 确认用户身份（不信前端传的 user_id），查数据库验证先修课条件和容量，用 transaction 和行锁确保最后一个名额不被两个人同时抢到。

题目语境怎么用：课程注册场景有三个不可信边界：用户身份（不能信前端传的"我是谁"）、业务约束（不能信"我确实满足先修课条件"）、并发（不能信"当前只剩一个位置时只有我在操作"）。答题时把三个边界分别映射到后端的 auth→validation→transaction。

卷面写法：先写"浏览器代码不可信——用户可以用 DevTools 任意修改"。再分三段：身份——后端从 session/JWT 验证；业务规则——后端查先修课和容量；数据安全——用 transaction 防超选和重复注册。

常见失分点：只说"前端验证可以绕过"但不给具体绕过方式（改 HTML maxlength、禁用 JS、curl 直接发 POST）。评分看能不能具体解释"绕过"的技术手段。

标准答案（先写结论）：若交易历史一致性重要、团队小，可推荐 modular monolith + PostgreSQL；价格流压力大时再拆服务。

为什么这是对的：空泛的推荐是"用微服务因为它可扩展"。具体推荐必须回答三个问题：(1) 扩展谁的什么——实时股价需要高吞吐（靠缓存和异步推送解决，不需要拆独立服务），交易历史需要 ACID 事务（拆成独立服务反而引入分布式事务的额外复杂度）；(2) 团队多大——小团队承担不起微服务的容器编排、服务发现、分布式日志、网络故障处理成本，modular monolith 在代码层面分模块但部署一个整体，维护成本低得多；(3) 什么条件触发演进——当股价 QPS 增长到单实例扛不住、且缓存命中率降低时，才考虑拆成独立服务并单独扩容。

题目语境怎么用：FinTrack 题面给了三条关键信息：real-time price（高频读）、transaction history（一致性写）、small-medium team（运维资源有限）。推荐时扣住这三条：先保证交易数据不错，用缓存层吸收实时压力，等团队规模或流量增长到瓶颈再拆服务。

卷面写法：先写推荐，再按题面三条压力分别论证：交易一致性→monolith 事务简单可靠；实时读压力→Redis 缓存可解决不需要拆服务；小团队→微服务运维负担超收益。最后写触发演进的条件。

常见失分点：不要只写"scalable"或"handles logic"。必须说明是哪种数据压力、职责边界或失败风险让这个选择合理。

标准答案（先写结论）：指出 controller 同时算折扣、存订单、发邮件、记 analytics，违反 SRP，应拆 service、model/repository、job/mailer。

为什么这是对的：指出 controller 问题不能只说"它写了太多代码"。必须逐一标出哪些行属于不同职责并解释为什么混在一起会出事：(1) 折扣计算——业务规则变化时被迫改 controller，但 controller 职责是 HTTP 协调；(2) 存订单——放在 controller 里让测试时必须 mock 整个 HTTP 请求才能验证存库行为；(3) 发邮件——邮件发送可能失败或耗时几秒，放在 action 里让用户等邮件发完才看到"订单成功"；(4) 记 analytics——与核心业务无关的横切关注点，analytics 挂了会影响订单提交。这就是 SRP 的真实含义：每个变化原因只应影响一个类。

题目语境怎么用：Sample Q3 代码里一个 action 调用了三四类不同的东西。答题时分别标记对应不同重构目标：业务计算→service object、数据保存→model/repository、外部通知→mailer/job、横切记录→callback/observer。

卷面写法：先列出 controller 里所有职责（每个一行），每行写"当前在哪里、应该在哪里、为什么应该在别处"。然后给出重构后每个新类的一句话定义，最后说明重构后 controller 和 service 各自的测试策略。

常见失分点：只说"违反 SRP"不给具体代码行对应的职责拆分。评分看能否说清"这段折扣代码移到 service 后，黑五折扣变了只改 service 这一个文件而不是满项目找"。

2. 三层架构、MVC 和 framework 的关系

CS615 很多题都在考 separation of concerns，也就是把不同责任拆开。三层架构从系统层面拆：presentation、business logic、data access。MVC 从 Web framework 代码结构上拆：model、view、controller。

三层架构先建立边界

层负责什么不该负责什么常见技术
Presentation layer显示页面、收集输入、给用户反馈。不直接连数据库，不保存核心业务规则。HTML、CSS、JavaScript、React。
Business logic layer验证输入、处理规则、协调服务。不写页面样式，不把 SQL 到处散落。Rails controller/service、Spring service、Express controller。
Data access layer读写数据库，保证数据可保存和可取回。不决定页面长相，不处理界面状态。Active Record、JPA repository、Mongoose model。

MVC 放在三层里怎么理解

MVC 部分职责三层视角考试关键词
Model代表数据和接近数据的规则。靠近 data access / domain layer。data, validation, Active Record, entity。
View显示用户看到的页面或界面片段。presentation layer。HTML, template, rendering, UI。
Controller接 request，读参数，调用 model/service，选择 response。business/application coordination。route, action, params, response。

Pattern 和 framework 不要混

Architectural pattern 是组织系统的思想，例如 3-tier、MVC、microservices。Framework 是帮你落地这些思想的工具，例如 Rails、Spring Boot、Express。考试里看到 pattern，要讲职责和 trade-off；看到 framework，要讲它怎样组织代码和请求流程。

过关小题：为什么浏览器不该直接连数据库？因为用户可以绕过服务器的权限、价格、库存、验证规则。后端中间层负责统一检查身份、验证输入、运行业务逻辑，并限制数据库权限。

对应教材：DB_WS_C.pdf、ArchitecturalPatterns.pdf。这两份先读，因为它们决定后面 Rails、Spring、Node/Express、React 应该放在哪个位置。

过关问答：先内容理解，再 past/sample paper 题，悬停或点击显示答案

内容理解题（先做 3 题）

核心答案（先确认概念）：Presentation 负责界面，business logic 负责规则，data access 负责读写数据库。不要让浏览器直接越过后端接触数据库。

为什么这是对的：如果把三层压成一层——比如在 HTML 里内嵌 SQL 查询——你会同时失去三个东西：(1) 安全——任何拿到页面的人都能看到数据库结构甚至通过修改 SQL 偷数据；(2) 可维护性——哪天改折扣规则要翻遍所有嵌了折扣逻辑的页面；(3) 可测试性——无法单独验证"价格计算是否正确"而不启动整个页面环境。三层架构的本质是把"展示给用户看""决定该做什么""记住数据"隔离开，让每层的修改不影响其他层。浏览器不能跳过后端直接连数据库，正是因为 presentation 层没有资格做权限判断和业务校验。

题目语境怎么用：如果题目给你系统架构图问"哪里违反了三层边界"，找例如"React 组件里 fetch 直接连 MongoDB"这种跳过 application 层的跨层调用，并解释为什么这是安全漏洞。

卷面写法：先定义三层各自责任范围，再画一条线说"线上不可越界"——浏览器→后端→数据库是单向依赖，不能反向也不能跳跃。最后给一个具体穿越例子（前端直接写 SQL）并解释危害。

常见失分点：只知道三层名字说不清"为什么不能穿层"。穿层的危害不是抽象设计原则，而是具体的：绕过权限检查、SQL injection、业务规则散落、改规则时要改所有前端页面。

核心答案（先确认概念）：三层是系统层面的分工；MVC 是 Web framework 内部组织 request、数据和页面的方式。它们都在服务 separation of concerns。

为什么这是对的：三层是从"整个系统物理部署"角度看——presentation 可能在浏览器（React），logic 在服务器（Spring/Rails），data 在数据库服务器（PostgreSQL）。MVC 全在服务器代码里——Controller 接收 HTTP 请求，Model 封装数据访问和校验，View 生成 HTML。考试里有人写"MVC 的 Model 是 presentation 层"就错了——Model 靠近 data 层，View 才是 presentation 层。正确对照：MVC 的 V→三层 presentation，M→三层 data，C→三层 logic。三层画横线——上面 presentation、中间 logic、下面 data；MVC 画竖框在中间那层里。

题目语境怎么用：题目让画分层图时，三层画横线（物理部署边界），MVC 画竖框在中间 logic 层（代码组织方式）。考试中 3-tier 题讲物理部署分工，MVC 题讲代码组织分工。

卷面写法：先写"MVC 运行在 application/logic 层内，三层是更大的系统边界。"画对照：View→presentation，Controller→logic（协调入口），Model→data access。最后总结两层模型的关系。

常见失分点：把 MVC 的 Model 说成 presentation 或 logic 层，把 View 说成 data 层。这是概念混淆的最常见表现——得分点在于正确对照两个模型。

核心答案（先确认概念）：Pattern 是组织思想，例如 MVC、3-tier；framework 是落地工具，例如 Rails、Spring、Express。

为什么这是对的：这个区分关系到考试里你能不能正确回答"讨论 pattern"还是"讨论 framework"。当题目问"MVC 的职责分工"时，你要讨论 Controller/Model/View 该做什么不该做什么，不要扯到 Rails 的 render 方法。当题目给你一段 Rails 代码让你审查时，你要讨论 OrdersController 这个具体类做了什么不该做的事，可以用 MVC pattern 的概念批判它，但答案要落脚到 framework 层的重构建议（抽 service object、用 mailer）。Pattern 回答"为什么这样设计"，framework 回答"怎么这样实现"。

题目语境怎么用：Sample Q1 问 3-tier pattern 下的架构映射——此时讲"presentation/application/data 各管什么"，不需要讲"Spring Boot 用 @RestController"。但 Sample Q3 让审查 Rails controller 代码——必须讲"create action 里 discount calculation 不该放 controller"，这是 framework-specific 审查。

卷面写法：遇到 pattern 题，先定义 pattern 再讲它解决什么问题，不引入具体框架名。遇到 framework 题，先定义框架再讲它的具体机制（convention over configuration、Active Record）。

常见失分点：pattern 题里写"Spring Boot 的 @Autowired 实现了 MVC"跑题了；framework 题里只写"MVC 好"不写具体框架机制也跑题。

考试题目题（再做 5 题）

标准答案（先写结论）：React frontend 是 presentation；Spring Boot backend 是 application/logic；PostgreSQL 是 data layer。

为什么这是对的：这个映射不是随意贴标签。React 是 presentation 层因为它只负责两件事——把数据渲染成图表和列表，收集用户输入发给后端。它不存持久化数据、不做需防作弊的业务校验。Spring Boot 是 application 层因为所有不可信操作都在这里：验证 token、检查风控规则、计算收益、调外部股价 API、写入数据库。PostgreSQL 是 data 层因为它只负责完整存数据、支持 transaction 保证交易不错账。三者通过明确接口通信——React→Spring 走 REST API（HTTP/JSON），Spring→PostgreSQL 走 JDBC/JPA（SQL）。每层物理上可独立部署、独立扩容互不干扰。

题目语境怎么用：FinTrack 的 Approach A 是教科书式 3-tier 部署：React 跑在浏览器/CDN，Spring Boot 跑在云服务器，PostgreSQL 跑在独立数据库实例。每层可独立扩容——前端加 CDN 缓存，后端加实例，数据库做读写分离。

卷面写法：画三层横线，每层写：名字、具体技术、在这层做了什么（具体到 FinTrack 的交易查询和股价展示）、与邻层怎么通信。每层下方补一句"不管什么"强调边界。

标准答案（先写结论）：B 把 application tier 拆成多个 Node services，服务之间通过 API/网络通信；A 的 logic 多在一个 monolith 内部调用。

Transaction 和 ACID 用付款记

Transaction 是一组必须一起成功或一起失败的操作。ACID 是事务可靠性的四个要求，适合订单、付款、库存、收据这些不能乱来的场景。

NoSQL 和 document database

NoSQL 不是“没有结构”。Document database 例如 MongoDB 把一条记录存成 JSON-like document，字段可以更灵活，相关信息可以嵌在一起。它常适合日志、用户行为事件、灵活内容、海量写入。Mongoose 这种 ODM 仍然可以定义 schema 和 validation。

SQL / NoSQL 判断台阶

  1. 先画题目数据。2025 shopping habits 题可以画成 Customers、Receipts、ReceiptItems；一张收据对应多条商品明细。
  2. 用例子证明 SQL。收据号必须唯一，总价必须等于商品明细加总，保存时不能出现“有收据号但没有 items”的半成品，所以需要 primary key、foreign key、join 和 transaction。
  3. 再给 NoSQL 的合理反例。如果题目改成记录 {userId, page, eventType, device, timestamp} 这类点击流，每条事件字段可能不同、写入量很大、关系查询少，MongoDB/document database 才更自然。
  4. 最后写 trade-off。SQL 牺牲一些灵活性换一致性和关系查询；NoSQL 牺牲一些关系约束换灵活 schema 和高量事件写入。

5. Spring REST：显式分层和 REST resource

Spring Boot 是 Java 生态的 Web framework。它常把 controller、service、repository 分开写，再通过 dependency injection 连接。和 Rails 相比，Spring 更显式，适合解释企业后端为什么要分层。

REST 不只是返回 JSON

REST 是一种设计 Web API 的风格。它把系统里的东西看成 resources，例如 users、orders、products。客户端用 URL 定位 resource，用 HTTP method 表达动作，用 representation 传输状态，常见 representation 是 JSON。

REST 词白话解释例子
Resource系统中可被定位和操作的东西。/orders/15 表示第 15 个订单。
HTTP verbs用 method 表示动作。GET 查，POST 新建，PATCH 改，DELETE 删。
Stateless每个 request 自己带足服务器判断所需信息。服务器不靠“记得上一请求”来理解当前请求。
Representationresource 的传输形式。同一订单可以用 JSON 返回，也可以 HTML 显示。

Spring request flow 台阶

  1. @RestController 接请求。Controller 读取 path variable、query parameter 或 JSON body。
  2. Controller 调 service。Controller 不应塞满业务规则，它负责 HTTP 边界。
  3. Service 执行业务逻辑。例如验证订单状态、检查权限、计算价格。
  4. Repository 访问数据库。数据库细节集中在 data access 层。
  5. 返回 response。结果可以变成 JSON、status code 或错误信息。

Dependency injection 用替换能力理解

Dependency 是一个类运行时需要的其他对象。例如 OrderService 需要 OrderRepository。Dependency injection 是让 framework 把需要的对象传进来。这样测试时可以换 fake repository，维护时可以替换实现，类之间耦合更低。

Forms and security 为什么跟 Spring 放一起

表单是用户输入进入系统的入口。用户可以输入正常姓名，也可以输入超长字符串、脚本、SQL 片段或伪造请求。后端必须验证类型、长度、范围和权限，不能只相信前端页面限制。

核心答案（先确认概念）：把对象看成 resources，用 URL 定位，用 HTTP verbs 表达动作，并保持 stateless request。

为什么这是对的：REST 的核心不是"返回 JSON"，很多非 REST 的 API 也返回 JSON（比如把所有操作放在 POST /api 里用 body 里的 action 字段区分）。REST 的真正约束是四条：(1) Resource-based URL——/orders/15 定位第 15 号订单，不是 /getOrder?orderId=15；(2) HTTP verb 对应动作——GET 读、POST 建、PUT/PATCH 改、DELETE 删，不把动词放 URL；(3) Stateless——服务器不靠"记住上一请求"来理解当前请求，每次 request 带足所有判断所需信息（如 token）；(4) Representation——同一 resource 可以有多种表示（服务器可以返回 JSON 给前端、返回 HTML 给浏览器）。这四条合起来让 API 可预测、可缓存、可独立扩展。

常见失分点：把 REST 等同于"返回 JSON 的 API"。很多 SOAP、GraphQL 也返回 JSON 但不是 REST。核心区分是 resource-based URL + HTTP verb + stateless。

核心答案（先确认概念）：Controller 管 HTTP 边界，service 管业务规则，repository 管数据库访问。

为什么这是对的：Spring 的这三个注解创造了一种"强制自律"的代码结构。@RestController 类里的方法只做三件事：读取 HTTP 请求中的参数（path variable、query param、request body）、调用一个或多个 service 方法、把返回值包装成 ResponseEntity。@Service 类里的方法只做业务判断——查库存、算价格、验证权限——不碰 HTTP（不知道方法是被 GET 还是 POST 触发的），不直接拼 SQL。@Repository 接口只定义数据访问——findById、save、findAll——不包含任何"如果怎样则怎样"的业务判断。这种强制分层让单元测试变得极其简单：测 service 时 mock repository 返回假数据，测 controller 时 mock service 返回假结果。

题目语境怎么用：如果题目给你 Spring 代码让你审查，检查三件事：controller 里有没有 if/else 业务判断（应放 service）、service 里有没有拼 SQL 字符串（应放 repository）、repository 里有没有 HTTP response 构造（应放 controller）。

卷面写法：分别定义三个注解的职责边界，每行一个例子：@RestController→读 HTTP 参数调 service 返回 ResponseEntity；@Service→业务规则和流程编排不碰 HTTP 和 SQL；@Repository→数据访问接口不包含业务判断。

常见失分点：把 service 和 repository 的功能混在一起写"service 查数据库"，或者把 controller 和 service 混在一起写"controller 算折扣"。Spring 题的核心就是证明你理解这三个注解的职责边界。

核心答案（先确认概念）：它降低耦合，让依赖可替换，尤其方便测试时换 fake repository 或替代实现。

为什么这是对的：不用 DI（dependency injection）时，OrderService 内部自己 new 一个 OrderRepository——然后你想测试 OrderService 时，OrderRepository 真会去连数据库，你的单元测试变成集成测试。用了 DI 后，OrderService 的构造函数接收一个 OrderRepository 接口，Spring 在运行时注入真实实现（JPA repository），你在测试时注入一个返回假数据的 fake——OrderService 的代码一行不改，测试环境换了一套依赖。这就是 DI 的核心价值：代码和它的依赖解耦。考试里 DI 通常和"为什么 Spring 分层有利于测试"一起考——分层让依赖关系单向（controller→service→repository），DI 让每层可单独替换。

题目语境怎么用：如果题目问"为什么 Spring Boot 的架构容易做单元测试"，答案是三层分层（controller/service/repository）加 DI——测试 service 时只需 mock repository，测试 controller 时只需 mock service，不需要启动整个 Spring 容器。

卷面写法：先解释 DI 的反面——自己 new 依赖导致代码和具体实现焊接在一起。再解释 DI 的解决——依赖从外部传入，运行时真实现、测试时 fake。最后接回 Spring 的三层架构，说明 DI 如何让每一层可单独测试。

常见失分点：把 DI 解释成"@Autowired 注解的用法"——那是语法，不是原理。考试要的是"为什么依赖从外面传入比自己在里面创建更好"的推理。

内容理解题（先做 3 题）

核心答案（先确认概念）：先标同步代码，它们立即按顺序执行；再看 setTimeout、Promise、数据库请求等异步回调进入队列的时机。

为什么这是对的：Node 的 event loop 有严格的执行顺序：同步代码→微任务（Promise.then/catch、process.nextTick）→宏任务（setTimeout、setInterval、I/O callback）。输出题的核心陷阱在于 Promise 的 .then 比 setTimeout(0) 先执行——因为微任务队列在当前宏任务结束后立即清空，而 setTimeout 在下一个宏任务周期。解题三步法：(1) 用笔标出所有同步语句——这些按代码顺序立即输出；(2) 圈出所有 Promise 和 async/await——它们的 .then 在微任务队列，同步代码执行完后立即按入队顺序输出；(3) 框出 setTimeout/setImmediate——它们在宏任务队列，等微任务全清空后才输出。

题目语境怎么用：2023 Summer Q2(iv) 典型输出题：console.log('first'); setTimeout(() => console.log('timeout'), 0); console.log('fifth');。解法：first 是同步立即输出；fifth 是同步立即输出；setTimeout 回调进宏任务队列，等同步+微任务全完后输出 timeout。

核心答案（先确认概念）：Route 决定 URL 进谁，middleware 做通用前后处理，controller 处理 request 并返回 response。

为什么这是对的：三者构成 Express 的请求处理流水线。Route 是入口匹配器——app.get('/api/books/:id', ...) 把 GET + /api/books/5 交给对应 handler。Middleware 是管道上的过滤器——app.use(express.json()) 在所有 handler 之前把 body 从字符串变成 JS 对象；authMiddleware 在到达 controller 前检查 JWT token 并拒绝无效请求；errorHandler 在所有 route 之后捕获异常统一返回错误格式。Controller 是末端的业务处理——读取 req.params.id、调 Book.findById、检查结果是否为空、返回 res.json(book) 或 res.status(404)。三者串联成一条完整链路：request→middleware chain→route match→controller→response。

题目语境怎么用：2023 Summer Q2 考 Express 分层时，不要只写"Express 有 route、middleware、controller"，要写出它们在一次请求中的执行顺序和各自检查了什么。

常见失分点：把 middleware 和 controller 混在一起说"都是处理请求的"。核心区别：middleware 是链式通用处理器（所有请求都经过），controller 是特定路由的最终处理器。

核心答案（先确认概念）：不能只写更快。要写 component、props、state 如何组织复杂 UI，以及 SEO、initial load、tooling 和 accessibility 的代价。

为什么这是对的：React 的"快"不是 DOM 操作快——原生 DOM 操作也快。React 的价值在组织层面：(1) Component 把 UI 拆成独立可复用的小块——ProductCard 可以在搜索结果页、推荐栏、购物车里复用；(2) Props 让数据单向流动——父组件传数据给子组件，数据来源可追踪；(3) State 让组件管理自己的局部变化——搜索框的输入、loading 状态、错误信息各自独立不互相污染。代价是：SEO 需要 SSR/Next.js 额外处理因为搜索引擎不执行 JS、initial load 需要下载 React bundle 可能比纯 HTML 慢、tooling 复杂（webpack/vite/babel）、键盘导航和 focus 管理需要开发者手动实现。

题目语境怎么用：2025 Q3(b) 让你反驳"永远不该用 React"，你需要同时写 React 适合什么（复杂交互、动态 UI、组件复用）和代价（SEO 需额外方案、tooling 学习成本），不能只说"React 好用"。

考试题目题（再做 5 题）

标准答案（先写结论）：先输出 first，再输出 fifth，因为同步代码先执行，timeout callback 后进队列。

为什么这是对的：Node 的 event loop 执行顺序：当前同步代码必须先全部跑完，才会去检查异步队列。console.log('first') 是同步语句，立即执行输出 first。setTimeout(..., 0) 把回调函数注册进宏任务（macrotask）队列，但当前同步代码还没跑完，所以回调排队等待。console.log('fifth') 是同步语句，紧接着执行输出 fifth。三条同步代码全部执行完毕后，event loop 才开始检查宏任务队列，取出 setTimeout 的回调执行。输出顺序：first→fifth→timeout。关键理解：setTimeout 的 0 毫秒不是"立即执行"，而是"0 毫秒后把这个回调放进队列"，队列前面如果有任务就必须排队等。

题目语境怎么用：这类题的考察重点是你能否区分同步代码和异步回调的执行时机。如果有 Promise，Promise.then 是微任务，会在当前宏任务结束后、下一个宏任务开始前执行——所以 Promise.then 比 setTimeout(0) 先输出。

卷面写法：输出结果后画时序表：行号→代码→类型（同步/宏任务/微任务）→执行时机→输出内容。用箭头标出执行流的先后关系。

7. 可访问性、SEO 和安全：功能跑通之后还要能被信任

Web 系统不是功能跑通就结束。真实网站要面对不同能力用户、搜索引擎、恶意输入、权限边界和长期维护。CS615 历年题很喜欢把这些质量要求拆出来考。

Accessibility

Accessibility 是让更多用户能完成任务。键盘用户、屏幕阅读器用户、低视力用户、色弱用户、临时受伤的用户，都可能依赖语义化 HTML、清楚焦点、足够颜色对比和正确表单标签。

常写点：Use semantic HTML; provide alt text for meaningful images; associate labels with form inputs; ensure keyboard navigation; show visible focus states; maintain colour contrast; do not rely on colour alone for errors.

SEO

SEO 是 Search Engine Optimization。White-hat SEO 强调真实内容、清楚标题、语义结构、可抓取链接、合理性能。Black-hat SEO 是欺骗搜索引擎，例如堆关键词、隐藏文字、垃圾外链，长期风险很高。

SQL injection后端把用户输入拼进 SQL。Parameterized queries、ORM 安全 API、输入验证、最小数据库权限。

标准答案（先写结论）：指出 controller 同时算折扣、存订单、发邮件、记 analytics，违反 SRP，应拆 service、model/repository、job/mailer。

为什么这是对的：指出问题不能只说"太长了"。必须逐一标出每类代码属于什么职责、为什么不该在 controller 里、会造成什么具体后果。折扣计算→业务规则，改折扣政策时被迫改 controller（change reason 错误）；存订单→数据访问，测试时必须 mock HTTP request 才能验证存库行为（test fragility）；发邮件→外部副作用，邮件服务挂了会导致订单提交失败（cascading failure）；记 analytics→横切关注点，与核心业务无关但耦合在同一个方法里（耦合污染）。重构方向：折扣→OrderPricingService，存库→model/repository，邮件→OrderMailer（background job），analytics→after_action callback 或 event bus。

题目语境怎么用：Sample Q3 代码里一个 action 调了三四类不同东西。按职责分类标记，每个类别给重构方向。

卷面写法：先列出所有发现的职责，每行"当前在 controller、应该在 X、因为 Y"。再给出重构后类结构，最后说明重构后测试怎么简化。

常见失分点：只说"违反 SRP"不给代码行对应的拆分。评分看能不能说清"这段移到 service 后改折扣规则只需改一个文件"。

版本解决什么问题考试关键词
HTTP/1.0连接复用弱，多个资源常带来多次连接开销。one request per connection, overhead。

CI/CD 和 Observability

  1. CI 防止坏代码进入主干。每次提交后自动安装依赖、构建、lint、运行测试。
  2. CD 让发布可重复。通过 staging、approval、deployment、rollback 降低人工发布错误。
  3. Logs 记录事件。例如某个 request 报错、某个用户动作发生。
  4. Metrics 记录数字趋势。例如 error rate、latency、CPU、request count。
  5. Traces 串起一次请求路径。特别适合 microservices，能看到 request 穿过哪些服务。

AI Agents 和 Vibe Coding

AI agents 可以生成代码、测试、重构建议、部署脚本、文档，但它们也可能误解需求、编造 API、引入安全漏洞。好答案要把 agent 放进 SDLC：需求和 acceptance criteria 由人把关，代码进入 PR review，测试、lint、安全扫描、人工架构判断仍然需要。

内容理解题（先做 3 题）

核心答案（先确认概念）：每一代怎样减少连接开销、阻塞和等待，让多个 request/response 更高效。

为什么这是对的：HTTP 版本之间的迭代不是"数字越大越快"那么简单，每一代解决的是上一代的特定瓶颈。HTTP/1.0——每次请求都要新建 TCP 连接（三次握手→请求→响应→四次挥手），一个网页如果有 20 个图片/CSS/JS 文件，就要 20 次连接，TCP 握手开销极大。HTTP/1.1——加了 persistent connection（keep-alive），一个 TCP 连接可以复用多次请求，但它是串行的——请求 A 的响应必须等请求 B 前面的请求全部完成（head-of-line blocking）。HTTP/2——引入 multiplexing，一个 TCP 连接上同时跑多个 stream，请求和响应可以交错，彻底解决队头阻塞。HTTP/3——换底层传输协议为 QUIC/UDP，解决 TCP 层面的队头阻塞（一个丢包阻塞整个连接），同时 0-RTT 握手减少连接建立时间。

题目语境怎么用：如果题目问"FinTrack 的实时股价页面（大量小图标和实时数据）适合用哪个 HTTP 版本"，选 HTTP/2 或 HTTP/3——因为 multiplexing 让多个资产并行加载不互相阻塞。

卷面写法：按版本分别写"解决了上一代的什么问题"，每代写瓶颈→方案→效果。例如 HTTP/1.1→瓶颈是每次请求新建连接→方案是 keep-alive→效果是复用连接但仍有队头阻塞。

常见失分点：只写版本号和特性列表（HTTP/2 有 multiplexing、HTTP/3 有 QUIC），不解释每一步"为什么需要这个特性、上一代没有它时会发生什么问题"。

核心答案（先确认概念）：它允许部署自定义 Web app，同时平台管理大量 runtime、部署和伸缩细节，小团队负担更低。

为什么这是对的：课程订阅工具是一个典型的自定义 Web app——需要用自己的数据库 schema 存课程、学生、选课记录，需要自己的业务逻辑（先修课检查、容量限制、时间冲突检测），不是现成 SaaS 能覆盖的场景。但同时这个团队可能是个小团队（甚至学生项目组），没有专门的 DevOps 人员去管理服务器。PaaS 恰好卡在中间：它给你一个"把代码推上去就能跑"的环境（管理了 OS 补丁、Java/Node runtime、数据库实例的备份和更新），但你的应用代码完全自定义（不受 SaaS 模板限制）。相比之下 IaaS 需要团队自己管 OS 和 runtime（对小团队负担大），SaaS 不允许自定义数据库和业务逻辑（不能满足课程注册的独特性）。

题目语境怎么用：2023 course subscriptions 题——题干通常描述一个小团队需要开发学生选课系统。答题时比较 IaaS/PaaS/SaaS 三者的责任边界，然后说"因为需要自定义应用逻辑但团队运维资源有限，PaaS 是最佳平衡"。

卷面写法：先列表比较 IaaS/PaaS/SaaS 的"你管什么 vs 平台管什么"，然后用课程订阅工具的具体需求（自定义 DB schema、权限系统、选课规则）去匹配，说明为什么 PaaS 在中间最合适。

常见失分点：只说"PaaS 好用"不解释为什么 IaaS 对这个小团队太重（要管 OS 补丁和 Java runtime）、SaaS 对自定义需求太死（没法自定义数据库和业务逻辑）。

核心答案（先确认概念）：Monitoring 告诉你是不是坏了；observability 用 logs、metrics、traces 帮你查为什么坏了。

为什么这是对的：Monitoring 通常是预设告警——"CPU 超过 90% 发短信""error rate 超过 5% 打电话"。它告诉你系统异常了，但不告诉你异常的原因。Observability 是调查能力——当你收到 monitoring 的告警后：(1) Metrics 帮你缩小时间范围——"error rate 在 14:32 突然从 0.1% 跳到 8%，那正是新 deploy 的时间"；(2) Traces 帮你定位到具体 service 和 method——"顺着刚才报错的 request ID，trace 显示请求在 TradeService 的 calculateFee() 方法耗了 30 秒后 timeout"；(3) Logs 帮你看到上下文——"查看 calculateFee() 那一刻的日志，发现传入的 transaction amount 是 null，因为价格服务的 JSON 字段名从 amount 改成了 totalAmount"。三者串成一条调查链：monitoring 发现异常→metrics 锁定时间→traces 锁定位置→logs 锁定原因。

题目语境怎么用：Sample Q4(b) 用 production bug 场景要求区分二者——部署后 error rate 升高，monitoring 告诉你"出事了"，observability 帮你查"出什么事了、在哪里、为什么"。

卷面写法：先定义 monitoring（预设条件+告警），再定义 observability（log/metric/trace 三信号调查系统内部状态），然后用一个 production bug 故事线展示二者如何配合。

常见失分点：把 observability 说成"高级 monitoring"——这是概念错误。Monitoring 是被动等告警，observability 是主动探查系统内部状态的能力。二者是互补关系不是升级关系。

考试题目题（再做 5 题）

标准答案（先写结论）：CI/CD 是自动构建、测试、交付/部署流程；Observability 是通过 logs、metrics、traces 理解生产系统内部状态。

为什么这是对的：Sample Q4(a) 的"一句话定义"是陷阱题——看似简单但评分看你能否在定义里包含"目的"。CI/CD 的定义必须包含"自动化"和"目的（防止坏代码上线/让发布可重复）"：CI/CD is the automated pipeline that builds, tests, and deploys code changes to production, aiming to catch defects early and make releases repeatable. Observability 的定义必须包含"三信号"和"目的（理解内部状态）"：Observability is the ability to understand a system's internal state through logs, metrics, and traces, enabling engineers to diagnose issues without predefining what could go wrong. 定义里缺失目的或缺失具体手段（三信号）都扣分。

题目语境怎么用：Q4(a) 之后的 Q4(b) 会紧接着让你用 production bug 场景展开。所以 (a) 的定义要为 (b) 的展开做铺垫——定义里提到的手段（logs/metrics/traces）在 (b) 里会具体用上。

卷面写法：每句话包含三个要素：是什么（pipeline/ability）、用什么（build/test/deploy 或 logs/metrics/traces）、为什么（catch defects/make releases repeatable 或 diagnose without predefined alerts）。

常见失分点：定义只写"是什么"不写"目的是什么"。考试定义题通常考察你能否在最短篇幅内传达核心价值而非表面描述。

标准答案（先写结论）：Monitoring 告诉你 error rate 升高；observability 帮你用 trace/log 找到哪个 service、哪次 deploy、哪条 request 出问题。

为什么这是对的：用 FinTrack 的具体 bug 场景串起来：新 deploy 后 monitoring 的 alert 响了——"TradeService error rate 从 0.1% 跳升到 8%"。工程师开始用 observability 调查：(1) Metric dashboard 显示错误率在 deploy 时刻 14:32 开始飙升——锁定了是这次 deploy 引入的；(2) Distributed trace 顺着一个出错的 request ID 追下去——请求在 StockPriceService 正常返回 {symbol:"AAPL", price:150.25}，但在 TradeService 的 calculateFee() 方法里 price 字段取到的是 undefined——因为 StockPriceService 新版本把 JSON 字段从 price 改成了 currentPrice；(3) Log 搜索 calculateFee 那一刻的日志——confirm 了 TypeError: Cannot read property 'price' of undefined。这个故事里 monitoring 回答"出事了"，observability 回答"谁改了字段名、在哪个 service、影响了多少请求"。

题目语境怎么用：考到这类题时，不要只列概念解释——用一个具体的、涉及多个微服务的 production bug 故事线把 monitoring 和 observability 的职责分界展示出来。

卷面写法：故事结构：deploy→monitoring alert→metric narrows time→trace pinpoints service→log reveals root cause→rollback/fix。每步标注是 monitoring 还是 observability。

常见失分点：把 monitoring 和 observability 说成同一个东西的不同叫法。必须区分：monitoring=预设告警（已知的未知），observability=探查能力（未知的未知）。

标准答案（先写结论）：维护性下降、debug 困难、技术债隐藏，团队可能无法审查架构边界和安全影响。

为什么这是对的：AI agent 生成的代码有一个致命问题——它可能"正确但不透明"。Agent 可以把一个 monolith 拆成 5 个 microservices，每个 service 的 API 都能正常调用，但团队不知道它为什么这样拆、service 之间的通信协议为什么选 gRPC 而不是 REST、数据库是按什么原则切分的。三个月后系统出 bug 时：(1) 没人理解 agent 的拆分逻辑——debug 只能靠试；(2) Agent 可能引入了安全漏洞——比如在跨 service 通信时没有加认证，因为原始 monolith 的代码是进程内调用不需要认证；(3) Agent 可能过度优化——把不相关的功能拆进同一个 service 只是因为"代码看起来相似"，结果一个 service 承担了两种完全不同的 change reason。这就是为什么考试强调 agent 的输出必须经过人工 code review 和架构审查——不是不信任 AI，而是一旦代码上线，维护它的是人，人必须能理解它。

题目语境怎么用：Sample Q2 的 AI agents 题通常要求你讨论 agent 在 SDLC 中的合适角色——辅助生成需要人审核，而不是端到端替代人做架构决策。

卷面写法：分三段：(1) agent 能做什么——代码生成、测试生成、重构建议；(2) agent 不能替代什么——架构决策（为什么这样分层）、安全审查（跨 service 认证）、团队 ownership（出 bug 谁修）；(3) 最佳实践——agent 生成→人工 PR review→CI 自动测试→人工架构审查→merge。

常见失分点：只说"AI 不完美"——太笼统。必须写出"不透明代码在 debug、安全审查、长期维护"三个维度的具体后果。

标准答案（先写结论）：不要拼接 '...'+userInput；用 parameterized query/prepared statement、输入验证和最小数据库权限。

为什么这是对的：这道题在整个课程中出现多次，因为它是 CS615 安全章的核心考题。重复出现的补题验证一个原则：SQL injection 的防御必须解释"为什么不是 sanitize"——因为 sanitize 是黑名单思维（列举所有可能的恶意字符并过滤），攻击者总能找到新绕法。Prepared statement 是白名单思维——只有预先编译好的 SQL 模板是"允许的代码"，所有外部输入都只能是"数据值"。这就是 defense 从"过滤坏东西"进化为"把数据关在代码外面"——前者是永无止境的猫鼠游戏，后者是协议层面的安全保证。

题目语境怎么用：SQL injection 用登录框、搜索框或表单输入说明拼接 SQL 的风险；XSS 用评论、用户名或 profile 字段说明输出到页面时为何要转义。

卷面写法：卷面写三段：定义攻击；给贴题例子；写防御并解释理由。prepared statements/parameterized queries 让输入只作为 value，ORM 安全 API 避免手拼 SQL，server-side validation 和最小权限负责兜底。

常见失分点：只写 sanitize 不够。sanitize 没有说明上下文，也不能替代参数化查询、输出转义和最小数据库权限。

标准答案（先写结论）：用用户查看 receipt：browser 发 GET /receipts/7，server 查数据库并执行业务规则，database 返回收据和 items。

为什么这是对的：2025 Q1(a) 要求向一个 intern 解释 full-stack。这题的隐含考点是"你能不能用非工程师能理解的语言，同时保持技术准确性"。好的例子：(1) 从用户角度描述——"你登录购物网站，点了'我的收据'，然后点 #7 号收据"；(2) 逐步解释背后发生了什么——"浏览器把你点的那个按钮翻译成一个网络请求，就像写信告诉服务器'请把 #7 号收据的内容发给我'；(3) 用比喻解释验证——"服务器的门卫先检查这封信是不是你发的（登录验证），然后去档案室（数据库）找那张收据；(4) 解释为什么要几步——"门卫不能让你直接进档案室随便翻，因为你会看到别人的收据"。这个例子的核心是：技术上准确（request→auth→query→response），但表达上 intern 能听懂。

题目语境怎么用：receipt/shopping habits 场景可拆成 Customers、Receipts、ReceiptItems；举例时体现权限检查（非主人不能看）和数据结构（收据头带多条明细）。

常见失分点：不要只写 SQL=table、NoSQL=document。也不要看到 shopping/data 就自动选 NoSQL；要看题面到底是收据记录还是高量行为事件。

内容理解题（先做 3 题）

核心答案（先确认概念）：先定义两个方案，再按维度比较 trade-off，最后落回题目场景给 recommendation。

为什么这是对的：CS615 的比较题（SQL vs NoSQL、monolith vs microservices、IaaS vs PaaS vs SaaS、React vs vanilla）万变不离其宗——考官不是要你"证明其中一个更好"，而是要你展示"你理解每个方案在什么条件下更合适"。固定四段：(1) 各用一句话定义两个方案——不是背课本定义，而是从题目场景提取出来的定义（比如"monolith 在此 FinTrack 场景下指 React+Spring Boot+PostgreSQL 部署为一个整体"）；(2) 选 2-3 个维度比较——维度必须是题目场景相关的（不是通用的 scalability/maintainability 空词），比如 FinTrack 的比较维度是 transaction consistency、real-time data push pressure、team size；(3) 每个维度写 A 好还是 B 好并给一个具体的技术理由——不是"A 更 scalable"而是"B 的 price service 可独立扩容到 10 个实例而 A 需要全量扩容造成资源浪费"；(4) 收尾 recommendation 扣回题目约束——"因为题面要求 transaction history 严格一致且团队是 small-medium，先选 monolith + PostgreSQL，当 QPS 超阈值再拆 price service"。

题目语境怎么用：拿到比较题后先圈出题面里所有约束条件——"real-time""strict consistency""small team"——这些词不是装饰，是考官给你列的必须覆盖的维度。

卷面写法：四段式：定义→维度比较→维度内 A vs B→recommendation 扣题。每个维度 1-2 句，不要展开成散文。全题控制在 150-200 词为佳。

常见失分点：定义段只写"MVC 是 model view controller"不解释它在这个场景的位置。维度选通用词而不是题面衍生词。最后 recommendation 和前面分析脱节——分析了半天 scalability 最后推荐了 data consistency 最优的选项但没解释为什么 scalability 让步了。

核心答案（先确认概念）：考试要的是职责和理由。Rails、Spring、Express、React 要放回 request flow、分层、数据和维护性里解释。

为什么这是对的：CS615 的评分标准不是"你提到了多少技术名词"，而是"你展示了多少对职责边界的理解"。一个答案里写"系统用 React 做前端、Spring Boot 做后端、PostgreSQL 做数据库"只有 1 分（识别技术）。同样这个答案如果改成"React 负责在浏览器收集用户搜索输入并渲染产品列表，它不存数据也不做权限判断；Spring Boot 的 controller 接请求后调 service 检查折扣规则和库存，service 通过 repository 查 PostgreSQL 的 products 表——所有业务规则集中在 service 层，数据库只负责可靠存储和事务保证"，这就是 4-5 分（每个技术解释了它在三层架构的哪一层、做了什么、为什么不越界）。考试的核心是你能否把技术放回职责——React 因为什么所以该在哪层、Spring 因为什么所以该管什么。

题目语境怎么用：任何题答案里出现技术名时，跟在后面的一句话解释"这个技术在这个场景做什么、为什么选它而不是另一个"。不要裸写技术名。

卷面写法：每个技术名后面跟一句职责说明。格式："[技术名]负责[具体做什么]，因为它[为什么适合这个职责]，它不[不该做什么]"。

常见失分点：答案里写了 8 个框架名但每个只有名字没有职责解释——结果 1 分。另一个答案只用 3 个技术但每个技术解释了职责边界和选择理由——结果 5 分。

核心答案（先确认概念）：Browser -> route -> controller -> service/model -> database -> response。这条线能串起绝大多数 CS615 题。

为什么这是对的：这条线是 CS615 的"万能答题底盘"。试试看：full-stack 题→在这条线上找到每个技术的位置（前端是 browser、API 是 route+controller、业务是 service、持久化是 database）；MVC 题→找到 controller(model/view) 在线上的位置；SQL/NoSQL 题→找到 database 那段，分析数据形状和一致性压力来自线的上一段（service 的查询模式）还是下一段（持久化需求）；security 题→找到 controller 和 database 之间的薄弱点——用户输入在哪段进入系统、在哪段被当成代码执行；CI/CD/observability 题→找到"代码→deploy→production"这条延长线。如果考前只能记一样东西，记这条线——每道题先把题目场景在这条线上"跑一遍"，然后看题目问的是线的前段、中段还是后段。

题目语境怎么用：遇到任何不能立刻下笔的题，先在草稿纸上画这条线，然后把题面的技术名词（React、Spring、MongoDB、Redis）放到线的对应位置。问自己"题目在问这条线的哪一段？那段出了什么问题？"

卷面写法：草稿上画：Browser(用户)→Route(URL match)→Controller(HTTP协调)→Service(业务规则)→Model/Repository(数据访问)→Database(持久化)→Response(HTML/JSON)→Browser(渲染)。每个箭头旁边写数据格式变化（URL params→对象→SQL→JSON→state）。

常见失分点：题面复杂时直接跳到"应该用哪个框架"的结论，没有先用这条线分析题面压力在哪一段。结果是 recommend 了一个和题面压力无关的方案。

考试题目题（再做 5 题）

标准答案（先写结论）：练 2025 Summer Q1：intern full-stack explanation + shopping habits SQL/NoSQL choice。

为什么这是对的：2025 Q1 是 CS615 最基础的考试题——它考两样最核心的东西：能否用简单语言解释 full-stack（request 链条），能否从题面数据推导数据库选择（SQL/NoSQL 判断）。这两样覆盖了线性页第 1 节和第 3 节的内容，而且评分标准相对温和（给 intern 解释不需要高级技术术语）。如果这道题你写不顺——比如 full-stack 解释里说不出"为什么浏览器不能直接连数据库"，或者 SQL 选择里只说"因为是结构化数据"而说不出"receipt number 要 unique、items 和 total 要对上"——说明地基（request flow 和 data constraints）还没打牢，后面更复杂的架构题会更吃力。反之这道题写顺了，说明你已经能用 request 链条和数据库约束这两个核心工具去分析其他任何 Web 工程题。

题目语境怎么用：receipt/shopping habits 场景可拆成 Customers、Receipts、ReceiptItems；receipt number 要唯一，items 与 total 要对上。练题时先自己写出完整答案再对照线性页的答案看哪里漏了。

卷面写法：先给选择，再列题面对象和关系，再解释 key、join、constraint、transaction 或 flexible document schema 为什么有用，最后补 trade-off。

常见失分点：不要只写 SQL=table、NoSQL=document。也不要看到 shopping/data 就自动选 NoSQL；要看题面到底是收据记录还是高量行为事件。

标准答案（先写结论）：练 Sample Q1 FinTrack：3-tier monolith vs microservices，并比较 scalability、resilience、deployment。

为什么这是对的：Sample Q1 是 CS615 最综合的架构题——它要求你把 3-tier、framework、数据库选择、team size 约束全部糅进一个答案里。练这道题的最佳顺序：(1) 先画 Approach A 和 B 的架构图——把 React、Spring Boot/Node services、PostgreSQL/MongoDB 放在三层横线上；(2) 列出比较维度——不能是通用词，必须从题面提取：transaction consistency（交易历史必须准确）、real-time data push（股价需要推送）、team size（small-medium 团队运维能力有限）；(3) 逐维度比较——每个维度写 A 好还是 B 好并给具体技术理由；(4) 给 recommendation 并解释什么条件下会改变。如果这道题你只写了"microservices 更 scalable 更 flexible"而没有具体的 FinTrack 场景映射（股价→缓存层、交易→ACID、小团队→运维成本），说明你还在背答案阶段。

题目语境怎么用：FinTrack 的三条关键约束是 real-time price、transaction history strict consistency、small-medium team。把每条约束映射到具体的技术决策上才算完成了这道题。

卷面写法：画图→列维度→逐维度比较→recommendation。每个维度用题面原词（不要换成自己的词），每个比较用技术术语（不要用 vague 形容词）。

常见失分点：维度用"scalability""flexibility"这些通用词而不是题面衍生的具体维度（如"real-time price push throughput""transaction history consistency guarantee"）。

标准答案（先写结论）：练 Sample Q3 OrdersController：指出 controller 过胖、违反 SRP，并提出 service/job/model 重构。

为什么这是对的：Sample Q3 是考察从"能读代码"到"能审查代码"的质变。能读代码的人会说"这个 create action 做了四件事：读参数、算折扣、存订单、发邮件"——这是翻译不是审查。能审查代码的人会说：(1) 折扣计算属于业务规则，应该放在 OrderPricingService——因为它只依赖输入参数不依赖 HTTP session；(2) 存订单时如果想做复杂校验应该用 model 的 callback 或 service 层而非 controller；(3) 发邮件属于外部副作用，应该放在 background job 或 mailer——因为邮件可能失败且耗时数秒；(4) 记 analytics 是横切关注点，应该用 observer 或 after_action——因为它和订单创建的核心流程无关。练这道题的最低标准：你能把 controller 里的每一行代码归类到一个"它应该在的层"并解释为什么。

题目语境怎么用：Sample Q3 给的代码里通常 5-6 个"坏味道"——你至少找出 4 个并给出重构方向才及格。找的时候按"这行代码会因为什么原因被改？"—如果两个不同行会因为不同原因被改，它们不该在同一个类。

卷面写法：先列出 controller 里发现的所有职责（每个一行），每行写"当前在哪里、应该在哪里、因为什么 change reason 不同"。再给重构后类结构，最后说明测试策略变化。

常见失分点：只指出问题不给出重构方向——code review 题不只是"批评"，更重要的是"建议"。找到坏味道是 50% 的分，给出正确的重构方向是另外 50%。

标准答案（先写结论）：指出 controller 同时算折扣、存订单、发邮件、记 analytics，违反 SRP，应拆 service、model/repository、job/mailer。

为什么这是对的：这道补题反复出现在多个 section 里因为它验证的是 CS615 最核心的技能——用分层和 SRP 的眼光审查代码。无论你学的是 Rails、Spring 还是 Express，审查逻辑完全一样：找出"一段代码在什么情况下需要修改"（change reason），如果 controller 里的某段代码会因为"折扣政策调整"而修改，但 controller 自身会因为"URL 结构改变"而修改——那这两段代码承担了不同的 change reason，不该在同一处。这个审查方法通用于任何框架、任何语言——因为它是关于职责和耦合的工程判断，不是关于某个框架的语法。

题目语境怎么用：Sample Q3 代码里通常 5-6 个"坏味道"——至少找出 4 个并配重构方向才及格。用 change reason 的逻辑去逐一分析——"这行代码会因为什么原因变？"

卷面写法：先列所有发现的职责，每行"在哪→应在哪→为什么（change reason 不同）"。再给重构后类结构。最后说明重构后 controller 和 service 各自的测试策略。

常见失分点：只说"违反 SRP"不给具体代码行对应的拆分。评分看能不能说清"这段移到 service 后改折扣规则只改一个文件"。

标准答案（先写结论）：CI/CD 管 build/test/deploy；observability 用 logs/metrics/traces 定位哪次 deploy、哪个 service、哪条 request 出错。

为什么这是对的：这道补题出现在多个 section 里是因为 CI/CD 和 observability 是 CS615 "工程化"维度的核心——它不是某个具体技术的语法题，而是考你能不能把"写代码→上线→出事→修好"串成一个完整的工程闭环。区分的关键是时间线：CI/CD 在代码提交后到上线前（build→lint→test→staging→deploy），目标是"防止坏代码进生产"；observability 在上线后（logs 记录事件、metrics 统计趋势、traces 串联路径），目标是"坏代码进生产后能快速定位和回滚"。这两者在一个 production bug 场景中的配合——CI 的测试有盲区没拦住 bug → CD 把含 bug 代码 deploy → observability 发现异常 → log/trace/metric 定位 → 修 bug + 补测试（补 CI 缺口）——才构成完整的工程能力。

题目语境怎么用：用 FinTrack 具体例子：CD deploy 价格服务的字段名从 price 改为 currentPrice → CI 的测试没覆盖这个 field name 变化（pass）→ production 中交易服务调价格服务取 price 字段拿到 undefined → observability trace 显示请求在价格正常交易 500 → 定位、回滚、修 bug、补 CI 测试。

卷面写法：先定义两者，再写时间线串联两者，最后用 production bug 故事展示配合。每步标注是 CI/CD 还是 observability 的职责。

常见失分点：不把两者串成故事线。评分看你能不能解释 CI/CD 和 observability 如何在同一 bug 的生命周期里互补——不是两个独立的概念。
