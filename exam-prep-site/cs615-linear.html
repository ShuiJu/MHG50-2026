CS615 linear course

CS615 教材线性精讲：从一次请求反推整门 Web 工程

CS615 的教材不是在随机介绍 Rails、Spring、Node、React 和 AI，而是在反复训练同一件事：一个 Web 系统怎样分层、怎样读写数据、怎样暴露 API、怎样保护和部署。复习时先建立这条主线，再把每份讲义放到主线上的对应位置。

教材反推流程 Request / Response 3-Tier / MVC SQL / NoSQL Rails Spring REST Node / React CI/CD / AI

0. 从教材反推学习流程

从本地 CS615 材料看，课程不是先教某一个框架的语法，而是先给 Web 系统的骨架，再用不同技术栈反复实现同一套骨架。现在的学习流程应该按“概念地基 -> 后端框架 -> 前后端 API -> 工程质量 -> 交付运维 -> sample 新题型”走。

审视后的结论

  1. 不能先从 React 或 AI 开始。它们很显眼，但没有 request、API、数据库、分层这些地基，React 只是页面语法，AI agents 也只会变成空泛议论文。
  2. Rails 和 Spring 不是两门孤立技术。它们都在训练“request 进 controller，业务规则放合适位置，数据访问有边界”。区别是 Rails 更强调 convention，Spring 更强调显式分层和 dependency injection。
  3. WSFE 是桥。它把后端 API、MongoDB/Mongoose、Express 和 React 连起来，所以应该放在 Rails/Spring 之后读，而不是只当 JavaScript 笔记。
  4. Security 和 CICDOBS 要回贴到主线。安全不是单独背 XSS；CI/CD 也不是单独背 pipeline。它们都服务于“这个 Web app 如何安全、可靠、可维护地运行”。

本节过关门槛

1. 一次网页请求经过哪些地方

先想象一个普通动作：用户打开购物网站，搜索商品，然后点“加入购物车”。浏览器不会自己完成所有事情。它会向服务器发送 request，服务器处理业务规则和数据库，再把 response 发回来。

最小请求链条

  1. 用户操作浏览器。浏览器显示 HTML/CSS，运行 JavaScript，收集点击、输入、选择。
  2. 浏览器发 HTTP request。Request 里有 URL、method、headers，有时还有 body。URL 说明找谁，method 说明想做什么，body 放表单或 JSON。
  3. 服务器用 route 找处理代码。Route 是 URL 和 controller action 的对应规则。
  4. Controller 协调业务。Controller 读取参数，调用 service/model/repository，检查权限和输入。
  5. 数据库保存或查询长期数据。订单、用户、商品、收据不会只放在内存变量里。
  6. 服务器返回 response。Response 可能是 HTML 页面，也可能是 JSON，让 React 或浏览器继续渲染。

基础词先落地

Client / Browser用户实际操作的一端。浏览器负责显示页面、运行前端 JavaScript、发 request。
Server / Backend处理请求的一端。它运行 Rails、Spring、Express 等框架，执行规则并返回结果。
Database长期保存数据的系统。服务器重启后，用户、订单、收据仍然存在。
API给程序调用的入口。前端不需要知道后端内部代码，只按 API 传入和取回数据。
JSON键值对文本格式，适合浏览器和服务器交换数据，例如 {"id":15,"total":90}。
Business logic业务规则，例如库存是否够、用户是否能下单、折扣怎样计算。

最小完整链条：Browser sends GET /products?q=book. Router maps the URL to a controller action. The controller calls business logic and data access code. The database returns matching records. The server returns HTML or JSON. The browser renders the result.

过关小题：解释 POST /orders 和 GET /orders/15 的区别：前者通常创建新订单，后者读取已有订单。它们都是 request，但 method 和目的不同。

2. 三层架构、MVC 和 framework 的关系

CS615 很多题都在考 separation of concerns，也就是把不同责任拆开。三层架构从系统层面拆：presentation、business logic、data access。MVC 从 Web framework 代码结构上拆：model、view、controller。

三层架构先建立边界

层负责什么不该负责什么常见技术
Presentation layer显示页面、收集输入、给用户反馈。不直接连数据库，不保存核心业务规则。HTML、CSS、JavaScript、React。
Business logic layer验证输入、处理规则、协调服务。不写页面样式，不把 SQL 到处散落。Rails controller/service、Spring service、Express controller。
Data access layer读写数据库，保证数据可保存和可取回。不决定页面长相，不处理界面状态。Active Record、JPA repository、Mongoose model。

MVC 放在三层里怎么理解

MVC 部分职责三层视角考试关键词
Model代表数据和接近数据的规则。靠近 data access / domain layer。data, validation, Active Record, entity。
View显示用户看到的页面或界面片段。presentation layer。HTML, template, rendering, UI。
Controller接 request，读参数，调用 model/service，选择 response。business/application coordination。route, action, params, response。

Pattern 和 framework 不要混

Architectural pattern 是组织系统的思想，例如 3-tier、MVC、microservices。Framework 是帮你落地这些思想的工具，例如 Rails、Spring Boot、Express。考试里看到 pattern，要讲职责和 trade-off；看到 framework，要讲它怎样组织代码和请求流程。

过关小题：为什么浏览器不该直接连数据库？因为用户可以绕过服务器的权限、价格、库存、验证规则。后端中间层负责统一检查身份、验证输入、运行业务逻辑，并限制数据库权限。

对应教材：DB_WS_C.pdf、ArchitecturalPatterns.pdf。这两份先读，因为它们决定后面 Rails、Spring、Node/Express、React 应该放在哪个位置。

Transaction 和 ACID 用付款记

Transaction 是一组必须一起成功或一起失败的操作。ACID 是事务可靠性的四个要求，适合订单、付款、库存、收据这些不能乱来的场景。

字母意思付款例子
A Atomicity要么全部成功，要么全部取消。钱扣了但订单没生成，不允许。
C Consistency操作前后都符合系统规则。库存不能为负，总价不能和明细矛盾。
I Isolation同时操作不能互相看到半成品。两个人不能同时买到最后一件库存。
D Durability成功后结果真正保存。订单确认后服务器重启，订单仍在。

NoSQL 和 document database

NoSQL 不是“没有结构”。Document database 例如 MongoDB 把一条记录存成 JSON-like document，字段可以更灵活，相关信息可以嵌在一起。它常适合日志、用户行为事件、灵活内容、海量写入。Mongoose 这种 ODM 仍然可以定义 schema 和 validation。

SQL / NoSQL 判断台阶

  1. 先画题目数据。2025 shopping habits 题可以画成 Customers、Receipts、ReceiptItems；一张收据对应多条商品明细。
  2. 用例子证明 SQL。收据号必须唯一，总价必须等于商品明细加总，保存时不能出现“有收据号但没有 items”的半成品，所以需要 primary key、foreign key、join 和 transaction。
  3. 再给 NoSQL 的合理反例。如果题目改成记录 {userId, page, eventType, device, timestamp} 这类点击流，每条事件字段可能不同、写入量很大、关系查询少，MongoDB/document database 才更自然。
  4. 最后写 trade-off。SQL 牺牲一些灵活性换一致性和关系查询；NoSQL 牺牲一些关系约束换灵活 schema 和高量事件写入。

购物习惯题答案骨架：I would choose SQL because the question is about purchase records, not just vague behaviour. A customer can have many receipts, and each receipt has a unique receipt number, date, total price, and several item rows. These fit relational tables such as Customers, Receipts, and ReceiptItems.

SQL is safer here because the receipt data must stay consistent: the receipt number should be unique, the total should match the item prices, and the receipt header and item rows should be saved together in one transaction. NoSQL would become easier to justify if the requirement changed to high-volume clickstream events, where each event is an independent JSON-like document with flexible fields.

Rails request flow 台阶

  1. 浏览器访问 URL。例如 GET /books/1。
  2. Router 匹配 route。它决定进入 BooksController#show。
  3. Controller 读取 params。例如从 URL 里拿到 id=1。
  4. Controller 调用 model 或 service。例如 Book.find(params[:id])。
  5. Model 通过 Active Record 读写数据库。代码对象和数据库记录被 ORM 翻译。
  6. Controller render view 或 redirect。最终返回 HTML 或跳到另一个页面。

5. Spring REST：显式分层和 REST resource

Spring Boot 是 Java 生态的 Web framework。它常把 controller、service、repository 分开写，再通过 dependency injection 连接。和 Rails 相比，Spring 更显式，适合解释企业后端为什么要分层。

REST 不只是返回 JSON

REST 是一种设计 Web API 的风格。它把系统里的东西看成 resources，例如 users、orders、products。客户端用 URL 定位 resource，用 HTTP method 表达动作，用 representation 传输状态，常见 representation 是 JSON。

REST 词白话解释例子
Resource系统中可被定位和操作的东西。/orders/15 表示第 15 个订单。
HTTP verbs用 method 表示动作。GET 查，POST 新建，PATCH 改，DELETE 删。
Stateless每个 request 自己带足服务器判断所需信息。服务器不靠“记得上一请求”来理解当前请求。
Representationresource 的传输形式。同一订单可以用 JSON 返回，也可以 HTML 显示。

Spring request flow 台阶

  1. @RestController 接请求。Controller 读取 path variable、query parameter 或 JSON body。
  2. Controller 调 service。Controller 不应塞满业务规则，它负责 HTTP 边界。
  3. Service 执行业务逻辑。例如验证订单状态、检查权限、计算价格。
  4. Repository 访问数据库。数据库细节集中在 data access 层。
  5. 返回 response。结果可以变成 JSON、status code 或错误信息。

Dependency injection 用替换能力理解

Dependency 是一个类运行时需要的其他对象。例如 OrderService 需要 OrderRepository。Dependency injection 是让 framework 把需要的对象传进来。这样测试时可以换 fake repository，维护时可以替换实现，类之间耦合更低。

Forms and security 为什么跟 Spring 放一起

表单是用户输入进入系统的入口。用户可以输入正常姓名，也可以输入超长字符串、脚本、SQL 片段或伪造请求。后端必须验证类型、长度、范围和权限，不能只相信前端页面限制。

Node 和 event loop

Node.js 让 JavaScript 可以跑在服务器上。I/O 指输入输出，例如读文件、查数据库、等网络响应。这些动作经常需要等待。Node 的 event loop 负责在同步代码跑完后，按队列处理异步回调。

  1. 先标同步代码。普通 console.log 立刻按顺序执行。
  2. 再标异步动作。setTimeout、database query、file read、Promise callback 都可能稍后执行。
  3. 同步跑完后看队列。回调等事件完成后进入队列，再由 event loop 执行。
  4. 输出题写解释。不要只列答案，要说为什么某行先输出。

React 和 API 的台阶

  1. 组件加载。页面先显示初始 UI 或 loading。
  2. 调用 API。用 Fetch 或 Axios 请求后端，例如 GET /api/products。
  3. 处理成功状态。把返回 JSON 放进 state，组件重新渲染。

7. 可访问性、SEO 和安全：功能跑通之后还要能被信任

Web 系统不是功能跑通就结束。真实网站要面对不同能力用户、搜索引擎、恶意输入、权限边界和长期维护。CS615 历年题很喜欢把这些质量要求拆出来考。

Accessibility

Accessibility 是让更多用户能完成任务。键盘用户、屏幕阅读器用户、低视力用户、色弱用户、临时受伤的用户，都可能依赖语义化 HTML、清楚焦点、足够颜色对比和正确表单标签。

常写点：Use semantic HTML; provide alt text for meaningful images; associate labels with form inputs; ensure keyboard navigation; show visible focus states; maintain colour contrast; do not rely on colour alone for errors.

SEO

SEO 是 Search Engine Optimization。White-hat SEO 强调真实内容、清楚标题、语义结构、可抓取链接、合理性能。Black-hat SEO 是欺骗搜索引擎，例如堆关键词、隐藏文字、垃圾外链，长期风险很高。

SQL injection后端把用户输入拼进 SQL。Parameterized queries、ORM 安全 API、输入验证、最小数据库权限。
XSS用户输入被当成 HTML/JavaScript 输出。Output encoding、sanitization、CSP、避免危险 HTML 插入。
CSRF用户已登录时，被诱导发出非本人意图的请求。CSRF token、SameSite cookie、重要操作二次确认。
Broken access control前端隐藏按钮，但后端没有检查权限。每个敏感 API 都在服务器端检查身份和权限。

8. HTTP、Cloud、CI/CD、Observability 和 AI Agents

CS615 后半部分把 Web app 从“能写出来”推到“能上线、能演进、能排查问题”。这些内容要回贴到主线：request 如何传输，应用在哪里运行，代码如何发布，出事如何定位，AI 如何进入 SDLC。

HTTP 版本看传输效率

版本解决什么问题考试关键词
HTTP/1.0连接复用弱，多个资源常带来多次连接开销。one request per connection, overhead。
HTTP/1.1persistent connections 改善复用，但仍可能阻塞。keep-alive, pipelining, head-of-line blocking。
HTTP/2一个 TCP 连接上 multiplex 多个 streams。multiplexing, binary framing, streams。
HTTP/3基于 QUIC/UDP，改善连接建立和丢包时表现。QUIC, UDP, reduced HOL blocking。

Cloud model 看你想管理多少东西

模型你主要管理适合场景
IaaS虚拟机、操作系统、runtime、部署、补丁。需要底层控制，团队有运维能力。
PaaS代码和配置，平台处理大部分运行环境。小团队部署自定义 Web app，课程订阅工具常推荐。
SaaS主要使用现成软件功能。邮件、CRM、文档协作等现成应用。

CI/CD 和 Observability

  1. CI 防止坏代码进入主干。每次提交后自动安装依赖、构建、lint、运行测试。
  2. CD 让发布可重复。通过 staging、approval、deployment、rollback 降低人工发布错误。
  3. Logs 记录事件。例如某个 request 报错、某个用户动作发生。
  4. Metrics 记录数字趋势。例如 error rate、latency、CPU、request count。
  5. Traces 串起一次请求路径。特别适合 microservices，能看到 request 穿过哪些服务。

AI Agents 和 Vibe Coding

AI agents 可以生成代码、测试、重构建议、部署脚本、文档，但它们也可能误解需求、编造 API、引入安全漏洞。好答案要把 agent 放进 SDLC：需求和 acceptance criteria 由人把关，代码进入 PR review，测试、lint、安全扫描、人工架构判断仍然需要。
