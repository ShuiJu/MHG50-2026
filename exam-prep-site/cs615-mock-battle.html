Q1. FinTrack 架构比较 [40 marks]

题面核心：FinTrack 是实时金融 dashboard，需要高频 stock/crypto 更新、复杂的用户自定义过滤、严格一致的 transaction history。Approach A 是 React + Spring Boot + PostgreSQL 的 3-tier monolith，单个垂直扩容实例；Approach B 是 React SPA + 多个 Node.js microservices + NoSQL document stores。

配分：(a) map both approaches to 3-tier and identify application-tier communication flow differences [10]；(b) compare scalability/performance, resilience, development velocity/deployment [15]；(c) recommend one architecture and justify [15]。

Q1(a) 三层映射和通信流 [10]

标准答案先写结论：两种方案都能映射到 Presentation, Application/Logic, Data 三层；差异不在“有没有三层”，而在 Application Tier 是一个 monolithic backend 还是多个通过网络通信的 services。

层Approach A: 3-tier monolithApproach B: microservices
PresentationReact frontend。负责 dashboard UI、filters、charts、form input。Decoupled React SPA。通过 REST/GraphQL/WebSocket 调 API gateway 或各 service。
Application / Logic一个 Spring Boot backend。Controller、service、security、filtering logic、transaction-history logic 都在同一个 deployable application 内。多个 Node.js services，例如 price-service、portfolio-service、filter-service、transaction-service。每个 service 负责一小块业务能力。
DataPostgreSQL。关系表、foreign keys、transactions、SQL joins 支撑 transaction history 和 user-specific filtering。NoSQL document stores，通常每个 service 拥有自己的 data store，schema 更灵活但跨服务 join/transaction 更难。

Application Tier 通信差异：Approach A 里 controller 调 service、service 调 repository 是进程内方法调用，速度快，事务边界清楚，日志和调试集中。Approach B 里一次 dashboard request 可能经过 API gateway、price-service、portfolio-service、filter-service 和 transaction-service；这些调用走 HTTP/gRPC/message queue，引入 network latency、timeouts、retries、service discovery 和 distributed tracing。

卷面例子：用户打开 dashboard 时，Approach A 是 React -> Spring DashboardController -> DashboardService -> PostgreSQL -> JSON response。Approach B 是 React SPA -> API Gateway -> price-service + portfolio-service + transaction-service -> separate NoSQL stores，再由 gateway 或 frontend 聚合结果。

Q1(b) 三个维度比较 [15]

1. Scalability and Performance

Approach AApproach BFinTrack 判断
垂直扩容简单：升级 CPU/RAM 即可。单体内部方法调用快，SQL join/filter 在一个数据库内完成。可以独立扩容热点服务，例如 price-service 单独加实例处理高频行情，transaction-service 不必跟着扩容。实时价格更新支持 B；复杂用户过滤和 transaction history 支持 A，因为 SQL 查询和事务更自然。
缺点是整体扩容粒度粗。只有 price updates 热，也要复制整个 backend 或升级整台机器。缺点是网络调用多，跨 service 聚合 dashboard 增加 latency；NoSQL 跨集合/跨服务查询比 SQL join 更笨重。如果瓶颈主要是 stock/crypto feed，B 更有弹性；如果瓶颈是交易历史查询准确性，A 更稳。

2. System Resilience

Approach AApproach BFinTrack 判断
故障模式简单：一个 app、一个 DB，容易定位和回滚。但单体进程或单机挂掉会影响整个系统。故障隔离更好：price-service 挂了，transaction history 理论上还能查。每个 service 可独立部署和恢复。FinTrack 可以接受价格短暂延迟，但 transaction history 错账不可接受。A 的强一致和简单事务边界是重大优势。
PostgreSQL ACID transaction 能保证交易记录、账户变更、审计日志同进同退。跨服务一致性要靠 saga/eventual consistency/compensating action，出错面更大。严格数据一致性是题面明示条件，不能只因 microservices resilience 好就忽略一致性风险。

3. Development Velocity and Deployment

Approach AApproach BFinTrack 判断
初期更快：一个 repo、一个 build、一个 deployable artifact。团队不用维护多个 service contract。长期团队变大后更快：不同小队可分别开发 price、portfolio、transaction service，独立部署。Startup 早期通常更需要快速验证产品和保持系统可理解，A 更合适。
缺点是 codebase 长大后可能变成 big ball of mud，部署任何小改动都要部署整个 app。缺点是 DevOps 成本高：CI/CD、container、service discovery、observability、API versioning 都要搭好。B 的速度收益只有在团队足够大、service 边界稳定时才会超过成本。

Q1(c) 推荐方案 [15]

推荐：For the initial version of FinTrack, choose Approach A, but implement it as a modular monolith and design the price-update module so it can later be extracted into a separate service.

为什么不是直接选 B：题面里最危险的数据不是实时价格，而是 transaction history。价格可以延迟、缓存、重试；交易历史如果出现半写入、重复写入、总额和明细不一致，会直接破坏用户信任和审计能力。PostgreSQL 的 transactions、unique constraints、foreign keys 和 SQL joins 更适合这类强一致金融记录。

为什么 A 仍能处理实时压力：Approach A 不等于“什么优化都不能做”。可以在 Spring Boot 内部使用 Redis cache、WebSocket/SSE push、background worker 拉行情、read replica 做查询扩展。先用这些低复杂度手段解决 high-frequency updates；只有当 price feed 成为明确瓶颈时，再把 price-service 抽出来。

必须写 trade-off：A 的代价是独立扩容能力弱，未来用户暴增时可能需要拆服务；B 的代价是现在就承担分布式系统复杂度。基于 startup、strict consistency、complex filtering 这三个条件，A 的简单性和数据可靠性更重要。

高分收束句：This is not a rejection of microservices. It is a staged architecture decision: start with a modular monolith to preserve consistency and delivery speed, observe real bottlenecks, then extract the high-frequency price component when scale and team maturity justify the operational cost.

Q2. AI Agents in the SDLC [30 marks]

题面核心：AI Agents 能执行多步骤 workflow，比如创建登录页、写 unit tests、部署到 staging。题目要求分析它们进入大型 e-commerce 平台 SDLC 的优势/风险，讨论 agent 把 legacy monolith 重构成 microservices 时的 opaque code 风险，并评价“AI Agents will replace junior web developers”。

配分：(a) 3 advantages + 3 risks [10]；(b) opaque optimized code 对 maintainability/technical debt 的影响 + oversight strategy [10]；(c) 批判性评价 junior developers 是否会被替代 [10]。

Q2(a) 三个优势和三个风险 [10]

优势 1：多步骤自动化提高 throughput。传统 developer-led workflow 需要人手动切换任务：写页面、写 API、写测试、改 CI、部署 staging。AI Agent 可以把这些串成一个 workflow，快速生成 login page、form validation、API call、unit test 和 deployment script。对 e-commerce 平台，常见 CRUD 页面、admin dashboard、测试样板代码可以更快完成。

优势 2：把重复工程任务标准化。Agent 可以按团队 template 生成 consistent code，例如所有 React form 都使用同一套 validation pattern，所有 API endpoint 都带 authentication middleware 和 error handling。传统手写容易因不同开发者习惯造成风格不一致。

优势 3：缩短 feedback loop。Agent 可在生成代码后立即运行 tests、lint、type check，并根据失败信息修复一轮。这让 developer 更快看到可运行原型，把时间投入到支付安全、库存一致性、性能瓶颈等高价值判断上。

风险 1：生成看似正确但不安全或不符合需求的代码。例如 agent 生成 login page 时把 JWT 存在 localStorage，或忘记 CSRF protection。代码能跑不代表符合 e-commerce 的安全要求。

风险 2：架构层面的误判会放大 technical debt。Agent 可能为追求局部优化生成复杂抽象，或把订单、支付、库存拆成过细 microservices，导致团队后来要维护大量 API contracts、distributed transactions 和 tracing。

风险 3：可解释性、责任和数据风险。Agent 可能生成 opaque code、引入许可证不明的片段，或把业务数据/密钥暴露给外部工具。大型平台还要考虑 audit trail：谁批准了这次改动，谁负责生产事故。

Q2(b) Opaque optimized code 和 oversight strategy [10]

对 maintainability 的影响：高度优化但不透明的 code 会让系统短期看似更快，长期更难维护。开发者无法解释某个 service boundary 为什么这样拆、某个 cache invalidation 为什么这样写、某段异步 retry 为什么不会重复扣款。结果是 bug 难定位、onboarding 变慢、测试难写、变更风险上升。

对 technical debt 的影响：Legacy monolith 拆 microservices 本身已经会引入 network failures、eventual consistency、distributed tracing、deployment orchestration。如果 agent 又生成难懂的优化代码，团队等于同时背上两层债：原 monolith 的历史债没有真正消失，新 microservices 的运维债又增加了。

平衡策略：

  1. Human architecture gate. 人先定义 bounded contexts 和 service boundaries，例如 catalog、cart、order、payment，而不是让 agent 自己按文件夹机械拆。
  2. Strangler pattern. 不一次性重写整个 monolith。先抽一个低风险模块，通过 facade/API gateway 逐步替换，保留可回滚路径。
  3. Mandatory review and ADR. Agent 生成的关键设计必须配 Architecture Decision Record，说明为什么这样拆、替代方案是什么、trade-off 是什么。
  4. Tests before trust. 对每个新 service 要有 contract tests、integration tests、security tests 和 performance smoke tests。AI 生成代码只有通过 CI 才能进入主干。
  5. Observability from day one. 每个 service 必须有 structured logs、metrics、trace IDs。否则拆分后出错只会更难定位。

卷面收束：Use AI agents as accelerators, not autonomous architects. They can propose code and migrations, but humans must own service boundaries, correctness, security and long-term maintainability.

Q2(c) 是否会取代 junior developers [10]

立场：The statement is too strong. AI agents will change junior developers' work, but they will not remove the need for junior developers.

为什么不会完全替代：Junior developers 不只是敲样板代码。他们也学习 domain rules、debug real production issues、理解用户需求、写 tests、做 code review、维护文档、和团队沟通。AI Agent 可以生成初稿，但它不知道真实业务优先级，也不能承担事故责任。

角色会怎样变化：Junior 的工作会从“从空白文件写第一版”转向“prompt, review, verify, debug”。他们需要能判断 agent 输出是否破坏 MVC/SRP、是否漏掉 input validation、是否引入 SQL injection/XSS、是否缺少 tests。换句话说，implementation skill 仍然重要，但 verification skill 变得更重要。

平衡评价：AI 会减少低价值重复劳动，让 junior 更快接触真实系统；但如果团队只让 junior 接受 agent 输出而不训练他们理解架构，长期会削弱工程判断。最好的组织会把 AI 当 pair-programmer 和 tutor，同时保留 human mentorship、review 和 ownership。

高分收束句：AI agents may reduce the amount of boilerplate assigned to junior developers, but the developer role shifts toward orchestration, verification and accountability rather than disappearing.

Q3. Rails Controller, MVC Flow and SRP [30 marks]

题面核心：给出 OrdersController#create，其中 controller 创建 order、计算折扣、保存、发邮件、记 analytics、redirect/render。要求指出 MVC flow 和 Single Responsibility Principle 问题，提出重构方案，并说明重构后 discount logic 的 unit tests 如何改变。

配分：(a) identify MVC/SRP violations [10]；(b) refactor and mention applicable patterns [14]；(c) testing impact [6]。

题面代码的核心坏味道：

class OrdersController < ApplicationController
  def create
    @order = Order.new(order_params)

    if @order.total > 1000 && @order.user.premium?
      @order.total *= 0.9
    elsif @order.total > 500
      @order.total -= 50
    end

    if @order.save
      Mailer.send_receipt(@order).deliver_now
      Analytics.track("order_created", @order)
      redirect_to @order, notice: "Success"
    else
      render :new
    end
  end
end

Q3(a) 违规识别 [10]

  1. Discount calculation is business logic inside the controller. @order.total > 1000、premium user、10% discount、minus 50 都是 pricing rules。Controller 的职责应是处理 HTTP request/response，不应拥有价格策略。
  2. Controller knows too much about the model. 它直接读取 @order.total 和 @order.user.premium?，说明 controller 被迫理解订单内部结构和用户等级规则。
  3. Synchronous email is a side effect in the request path. deliver_now 会让用户等待邮件发送；邮件服务失败可能导致订单流程异常。应该用 background job。
  4. Analytics is a cross-cutting concern. Analytics.track 与创建订单的核心行为不同，应该通过 domain event、observer 或 job 解耦。
  5. SRP violation. 这个 action 有多个变化原因：折扣规则变、邮件模板变、analytics schema 变、保存逻辑变、redirect/render 行为变。一个方法同时承担太多责任。

卷面一句话：The controller is no longer a coordinator. It has become a place where HTTP handling, pricing policy, persistence and side effects are mixed together.

Q3(b) 重构方案 [14]

原问题移到哪里可提 pattern
折扣规则DiscountPolicy 或 OrderPricingServiceStrategy / Policy object。不同折扣规则可独立替换。
创建订单流程CreateOrderService 或 command objectService object / Command。封装 use case。
发邮件OrderReceiptJobBackground Job。异步执行，可 retry。
AnalyticsOrderCreated domain event 的 subscriberObserver / Pub-Sub。横切关注点解耦。

重构后 controller 应该变薄：

def create
  result = CreateOrderService.call(order_params, current_user)

  if result.success?
    redirect_to result.order, notice: "Success"
  else
    @order = result.order
    render :new, status: :unprocessable_entity
  end
end

Pricing policy 示例：

class DiscountPolicy
  def self.apply(order)
    if order.total > 1000 && order.user.premium?
      order.total * 0.9
    elsif order.total > 500
      order.total - 50
    else
      order.total
    end
  end
end

CreateOrderService 负责 orchestration：build order, apply discount, save order in a transaction, publish OrderCreated event, enqueue receipt job. Controller 只关心 success/failure 和 response。

高分提醒：不要把所有代码简单塞进 model 里然后说“fat model”。更好的答案是把 pricing policy、workflow orchestration、side effects 分开，因为它们变化原因不同。

Q3(c) Unit tests 如何改变 [6]

重构前：要测试折扣逻辑，必须通过 controller request 构造 order params、user、session、database save，还可能触发 email 和 analytics。测试慢、脆弱，失败时不清楚是折扣错、数据库错、邮件错还是 redirect 错。

重构后：discount logic 是纯业务对象，可以直接测：

it "gives 10 percent discount to premium orders over 1000" do
  order = build_order(total: 1200, premium: true)
  expect(DiscountPolicy.apply(order)).to eq(1080)
end

测试分层：DiscountPolicy unit tests 覆盖价格规则；CreateOrderService integration tests 覆盖保存和事件发布；controller tests 只验证 successful result redirects、failed result renders :new。这样 tests 更快、更稳定、更能定位问题。

Q4. CI/CD and Observability [30 marks]

题面核心：现代 Web development 中，delivery speed（CI/CD）和理解 production 行为（Observability）是关键成功因素。题目问定义、区分 Observability 和 Monitoring、分析二者如何共同保证质量可靠性、以及组织实施完整 observability stack 时的 trade-offs/challenges。

配分：(a) define CI/CD and Observability [6]；(b) distinguish Observability from Monitoring [4]；(c) analyze how they work together [10]；(d) strategic trade-offs and challenges [10]。

Q4(a) 定义 [6]

Continuous Integration: Developers frequently merge code into a shared main branch. Each merge triggers automated build, linting, unit tests and integration tests. The goal is to find integration problems early rather than letting many branches drift apart.

Continuous Delivery/Deployment: Once code passes CI, it is packaged and promoted through environments such as staging and production using an automated pipeline. Continuous delivery may keep a manual approval gate; continuous deployment pushes automatically when checks pass.

Observability: The ability to understand a running system's internal state from its external outputs, especially logs, metrics and traces. It lets engineers investigate why a specific request failed, which service was slow, or which deployment changed behavior.

Q4(b) Observability vs Monitoring [4]

MonitoringObservability
Tracks known indicators against predefined thresholds, such as CPU > 90%, error rate > 2%, disk almost full.Supports open-ended debugging of unknown problems using logs, metrics and traces.
Answers “Is the system healthy?”Answers “Why is this user request slow or failing?”
Often alert-oriented.Investigation-oriented and correlation-oriented.

一句话：Monitoring tells you something is wrong; observability helps you find out why.

Q4(c) CI/CD 和 Observability 如何配合 [10]

  1. Before deployment, CI reduces defect injection. 每次 commit 先跑 lint、unit tests、integration tests、security checks，阻止明显坏代码进入 main branch。
  2. During deployment, CD makes release repeatable. Pipeline 用同样步骤部署 staging 和 production，减少手工操作差异。Canary release 或 blue/green deployment 让新版本先影响少量用户。
  3. After deployment, observability validates real behavior. Metrics 比较 deploy 前后的 error rate、latency、throughput；logs 提供错误上下文；traces 显示一次 request 横跨哪些 services。
  4. Feedback loop closes the quality cycle. 如果 observability 发现 production bug，团队把失败场景加入 regression test；下一次 CI 会防止同类 bug 再上线。

具体例子：一个 checkout service 新版本上线后，metrics 显示 p95 latency 从 300ms 升到 2s。Trace 显示请求卡在 payment-service 调 tax-service 的 HTTP call。Logs 显示新版本缺少 timeout，导致线程等待。团队回滚该 deploy，添加 timeout 和 fallback，写 integration test 模拟 tax-service 延迟，再由 CI/CD 发布修复。这个流程把 CI/CD 的交付控制和 observability 的生产诊断连成闭环。

Q4(d) Trade-offs and challenges [10]

挑战为什么重要应对策略
Speed vs safety自动部署提高速度，但错误也可能更快进 production。Use staged rollout, canary, feature flags, automatic rollback based on metrics.
Flaky tests不稳定测试会让团队不信任 CI，最后绕过 pipeline。Keep tests deterministic, quarantine flaky tests, maintain test data carefully.
Observability costLogs/traces/metrics 数据量巨大，存储和查询成本高。Sampling, retention policies, structured logging, only high-value spans.
Alert fatigue太多低价值告警会让工程师忽视真正事故。Alert on user impact and SLOs, not every noisy internal metric.
Privacy and securityLogs 可能泄露 emails、tokens、payment data。Redaction, access control, encryption, avoid logging secrets.
Organizational changeCI/CD and observability require developers to own production, not throw code over the wall.Shared ownership, runbooks, incident reviews, deployment discipline.

高分收束：The strategic trade-off is not simply tool cost. CI/CD and observability change how an organization releases, detects, diagnoses and learns from failures. The benefit is faster and safer delivery, but only if the team invests in tests, telemetry quality, privacy controls and operational discipline.
