0. 先看清楚游戏规则

考试形式很简单：3 小时，Q1 必答，Q2/Q3/Q4 至少选两题。Q1 值 40 分，Q2/Q3/Q4 各 30 分。总分 100。

既然样卷已经告诉我们就是这个形式，那复习就应该完全围着这四道题走。以前学的 MVC essay、event loop、JWT、TypeScript 这些还是会用上，但它们现在是背景知识，不是主角。

时间怎么分配？Q1 给足 65 到 75 分钟，因为分值最高、要写的内容也最多。剩下三题各 45 到 50 分钟。写不完没关系——关键是每题的核心论点都要点到。

卷面总原则：CS615 不奖励只背名词。每一段都按这个节奏走：先给结论 → 解释技术机制 → 贴回题面场景 → 点出 trade-off。比较题两边都要写，最后必须推荐一个方案，同时说清楚你的前提是什么。

Q1. FinTrack 架构比较 [40 marks]

题目在问什么：FinTrack 是一个实时金融 dashboard。它要展示股票和加密货币的实时价格（高频更新）、支持用户自己定义复杂的筛选条件、还要严格记录每一笔交易历史——不能丢、不能错、不能对不上。现在给了两种技术方案让你比较。

方案 A：React 前端 + Spring Boot 后端 + PostgreSQL 数据库。所有东西打包成一个应用，运行在一台机器上（可以升级配置，但始终是一个整体）。

方案 B：React SPA 前端 + 多个 Node.js 微服务 + NoSQL 文档数据库。后端拆成了好几个独立的小服务，各管各的事，各用各的数据库。

配分：(a) 把两种方案映射到三层架构，分析应用层通信差异 [10]；(b) 从 scalability/performance、resilience、development velocity/deployment 三个维度比较 [15]；(c) 推荐一个方案并论证 [15]。

Q1(a) 三层映射和通信流的差异 [10]

先想清楚一件事：所谓"三层架构"只是逻辑上的划分——展示层管界面，应用层管业务逻辑，数据层管存储。任何 Web 系统都能这么分。所以两种方案都能映射到三层，区别不在这，而在应用层是一个整体还是拆成了多个服务。

方案 A 就像一个完整的工具箱。所有工具都在一个箱子里——前端 React 管界面，后端 Spring Boot 一个进程处理所有业务逻辑（实时价格、用户过滤、交易记录全在一起），数据库用 PostgreSQL 管所有数据。用户打开 dashboard，流程很直：

React → Spring Boot Controller → Service → Repository → PostgreSQL → 返回 JSON

每一步都是进程内的方法调用，速度快，出错了也好查——日志在一个地方，调试不用跨服务。

方案 B 像把工具分到了不同的抽屉里。每个抽屉独立存在——price-service 管价格、portfolio-service 管持仓、filter-service 管筛选、transaction-service 管交易记录。每个 service 有自己的 NoSQL 数据库。用户打开 dashboard，流程变成：

React SPA → API Gateway → price-service + portfolio-service + transaction-service → 各自查各自的 NoSQL → Gateway（或前端）汇总结果

这里的每一步都是网络调用。网络有延迟、有超时、有丢包。出问题了，你得靠分布式追踪才能搞清楚是哪个 service 挂了。

层方案 A（单体）方案 B（微服务）
PresentationReact 前端，负责仪表盘 UI、图表、筛选表单React SPA，通过 REST/GraphQL/WebSocket 调 API Gateway 或各 service
Application一个 Spring Boot 后端，controller、service、security、过滤逻辑、交易逻辑都在同一个部署单元里多个 Node.js 服务，每个服务管一小块业务能力，通过网络互相调用
DataPostgreSQL 关系数据库，外键、事务、SQL join 支撑交易记录和复杂过滤NoSQL 文档数据库，每个 service 有自己的 data store，跨服务 join/事务困难

Q1(b) 三个维度逐一比较 [15]

维度一：扩容和性能。

方案 A 扩容靠垂直升级——CPU 不够了换更好的 CPU，内存不够了加内存。简单，但也有个尴尬的地方：可能只有"实时价格推送"这个模块特别忙，但你没办法只给这一个模块加资源，得把整个后端一起升级。

方案 B 可以水平扩容——哪个 service 忙就给哪个 service 加实例。如果 price-service 扛不住高频行情，就给它单独加几个实例，transaction-service 不用跟着动。但代价是：跨多个 service 拼一个 dashboard 页面时，网络延迟是累加的。而且 NoSQL 里做跨集合查询（类似"找出所有满足用户自定义筛选条件的交易记录"）比 SQL join 笨重得多。

维度二：出错了谁扛得住。

方案 A 的故障模式很简单：一个程序、一个数据库。出问题容易定位，容易回滚。但单体进程或单机挂了，整个系统都没了。

方案 B 的故障隔离更好：price-service 挂了，transaction history 理论上还能查。但麻烦在数据一致性上。一笔交易可能要改好几个 service 的数据——PostgreSQL 的 ACID 事务能保证要么全成功要么全失败；NoSQL 跨服务一致性得靠 saga、补偿事务、最终一致性，出错面大多了。

对 FinTrack 来说，价格可以延迟几秒，用户察觉不到；但交易记录一旦出错——半写入、重复扣款、总额和明细对不上——用户信任就崩了。所以方案 A 的强一致事务是很大的优势。

维度三：开发速度和部署。

方案 A 初期更快：一个代码仓库、一次构建、一个部署包。团队小的时候不用纠结 service contract、API 版本、服务发现这些事。

方案 B 长期看有优势：不同小队可以各管各的 service，独立发布，互不阻塞。但这是有前提的——你得先把 CI/CD、容器编排、服务发现、API 版本管理、分布式追踪都搭好。这些都是前期成本，在 startup 阶段可能不划算。

维度方案 A 优势方案 B 优势FinTrack 的关键判断
扩容/性能进程内调用快，SQL join/filter 在一个 DB 内完成热点服务独立扩容，不浪费资源价格推送支持 B，但交易查询和复杂过滤支持 A
容错ACID 事务保证交易记录强一致，故障易定位故障隔离好，一个服务挂了不影响其他交易记录一致性 > 价格可用性，A 更稳
开发速度单一 repo/build/deploy，早期迭代快团队大了以后可并行开发、独立部署Startup 阶段快速验证 > 长期并行效率

Q1(c) 推荐方案并论证 [15]

我的推荐：先选方案 A，但把代码写"松"一点——做成模块化单体（modular monolith），为以后可能拆成微服务留后路。

为什么不是直接选 B？因为这道题里最危险的数据不是实时价格，而是交易记录。价格可以延迟、可以缓存、可以重试——用户可能根本注意不到。但交易记录一旦出问题——半写入、重复扣款、总额和明细对不上——用户信任就没了，审计也过不了。PostgreSQL 的事务、唯一约束、外键、SQL join 就是为这种场景设计的。

但选 A 不等于放弃性能。你可以在 Spring Boot 内部用好几种低复杂度手段来应对高频更新：用 Redis 做价格缓存、用 WebSocket 或 SSE 推送实时数据、用后台 worker 定时拉取行情、用只读副库分担查询压力。这些手段投入小、见效快。只有当 price feed 真的变成明确瓶颈了，再把价格模块抽成独立的 service。

必须写的 trade-off：选 A 牺牲了独立扩容的弹性——将来用户暴增时可能需要拆服务。选 B 牺牲了数据一致性的简洁保障——现在就背上分布式系统的复杂度。对于 startup 阶段的 FinTrack——团队小、需要快速验证产品、数据一致性要求极高——方案 A 的简单性和可靠性比方案 B 的弹性更重要。

收束句：这不是拒绝微服务，而是分阶段走的架构决策。先用模块化单体保住一致性和交付速度，在实际运行中观察真正的瓶颈，等到规模和团队成熟度值得承担运维成本时，再把高频价格模块单独抽出来。

Q2. AI Agents 进入软件开发生命周期 [30 marks]

题目在问什么：AI Agent 不只是聊天——它能自动执行多步骤的任务。比如你说"给电商平台加一个登录页"，它不只是生成代码，还能自动写表单验证、API 对接、单元测试、甚至部署到 staging 环境。题目要求你分析这种能力进入大型 e-commerce 平台 SDLC 的优势和风险，讨论 Agent 把 legacy monolith 重构成微服务时产生的"看不懂的优化代码"（opaque code）问题，最后评价"AI 会取代初级开发者"这个说法。

配分：(a) 3 个优势 + 3 个风险 [10]；(b) opaque code 对 maintainability/technical debt 的影响 + 管控策略 [10]；(c) 批判性评价初级开发者是否会被取代 [10]。

Q2(a) 三个优势、三个风险 [10]

先说好处。

优势一：吞吐量上去了。一个大电商平台有几十个后台管理页面——商品管理、订单列表、优惠券配置、用户管理——每一个都是类似的 CRUD 套路。传统做法是开发者手动切换：写页面 → 写 API → 写校验 → 写测试 → 改 CI。AI Agent 可以把这些串成一个自动化流程，快速出第一版。人的时间省下来想真正难的事情：支付安全怎么做？库存扣减怎么保证不出错？秒杀场景怎么扛？

优势二：代码风格统一了。每个开发者有自己的习惯——有人用这种 validation 模式，有人用那种。Agent 可以按团队定好的模板生成代码，保证所有表单校验、所有 API 错误处理都是同一个套路。对几十人的大团队来说，这种一致性本身就是质量。

优势三：反馈变快了。Agent 生成代码后能立刻跑测试、lint、类型检查，失败了还能自己改一轮。你不用等半天才能看到一个能跑的原型——几分钟就有了。这让开发者能更快地判断方向对不对，不用在"这个方案能不能跑通"上耗太久。

再说风险——每一个都和好处一样真实。

风险一：代码能跑，但不安全。Agent 可能会把 JWT 存到 localStorage 里——能跑啊，功能看起来也对，但遇到 XSS 攻击就完了。或者忘了加 CSRF 保护、忘了做输入校验。电商平台的登录和支付页面——安全不是可选配置，是底线。

风险二：架构层面的错误代价巨大。Agent 为了"优化"可能会搞出过度复杂的抽象。比如在拆 legacy monolith 的时候，把订单、支付、库存拆成十几个微服务——每个都"解耦"了，每个都"独立"了，但团队要维护十几套 API 协议、处理分布式事务、排查跨服务 bug。这种 architectural tech debt 要还很久。

风险三：责任说不清楚。Agent 生成的代码谁负责？出了生产事故找谁？它还可能生成一些来路不明的代码片段，或者不小心把业务数据发到外部 API。大平台的合规团队一定会问：audit trail 在哪？谁批准了这次变更？

Q2(b) Opaque code：为什么看不懂的代码比写错的代码更可怕 [10]

Opaque code 就是那种"高度优化但你根本看不懂"的代码。正常开发者写代码的时候，你大概能猜到他的思路——为什么用这个数据结构、为什么拆成这个函数。但 AI 生成的优化代码可能完全是黑箱：你知道它跑得快，但不知道为什么快，更不知道什么情况下会出错。

这有多可怕？想象一个场景：AI 帮你把 legacy monolith 拆成了微服务。它写了一堆 service 之间的调用逻辑、缓存策略、异步重试机制。代码跑起来了，性能还不错。三个月后，用户投诉说订单被重复扣款了。你翻代码想找原因，发现某个 retry 逻辑写得极其晦涩——你看不懂它在什么条件下会重试，也判断不了重试是不是幂等的。这就是 opaque code 真正的杀伤力：出事了你修不了，因为你看不懂。

对 maintainability 的影响：新同事入职，看这段代码完全懵了。老同事也解释不清——因为不是人写的。bug 修不了，新功能不敢加，因为你不知道改了这里会不会崩那里。系统从"难维护"变成了"不可维护"。

对 technical debt 的影响：legacy monolith 本身就有历史债。拆微服务本身又加了一层运维债——网络故障、最终一致性、分布式追踪。如果拆的过程中又引入看不懂的优化代码，团队等于背着三层债在跑。原 monolith 的历史债没有真正消失，新微服务的运维债又来了，再加上 AI 生成的"黑箱债"——这就是技术破产的路线。

怎么办？五条底线策略：

  1. 人先定边界，AI 再执行。不要让 AI 自己决定怎么拆服务。人先把 bounded context 划清楚——商品、购物车、订单、支付——然后让 AI 在既定边界内干活。架构决策必须是人做的。
  2. 用 Strangler Pattern 渐进替换。别一次性重写整个 monolith。先抽一个低风险模块，通过 facade 或 API gateway 逐步替换旧代码，保留随时回滚的能力。
  3. 强制写 ADR。AI 生成的关键设计必须配 Architecture Decision Record：为什么这样拆？替代方案是什么？trade-off 是什么？如果解释不清楚，说明这个设计本身就有问题。
  4. 测试不过不准合。对每个新 service 要有 contract test、integration test、security test 和性能冒烟测试。AI 生成的代码只有全部通过 CI pipeline 才能进入主干。
  5. 从第一天就做可观测性。每个 service 必须有 structured log、metrics、trace ID。不然拆完之后出问题你连哪里坏了都不知道。

收束句：把 AI Agent 当加速器，别当自动驾驶。它可以生成代码和迁移脚本，但人必须掌控 service boundary、正确性、安全性和长期的可维护性。

Q2(c) AI 会取代初级开发者吗？[10]

我的立场：这个说法太绝对了。AI 会改变初级开发者的工作，但不会让这个岗位消失。

为什么不会完全取代？因为初级开发者做的事远不止敲样板代码。他们在学习业务规则、排查生产事故、理解用户需求、写测试、做 code review、和团队沟通。AI Agent 可以生成代码初稿，但它不知道真实的业务优先级——是修这个 bug 重要还是上新功能重要？它也不能为生产事故负责。责任，最终永远在人身上。

但角色确实在变。以前初级开发者的工作可能是"从空白文件写第一版"。以后会变成"写 prompt、审核输出、验证正确性、修 AI 没发现的 bug"。他们需要能回答这些问题：AI 生成的代码有没有破坏 MVC 分层？有没有漏掉输入校验？有没有引入 SQL 注入或 XSS？实现能力依然重要，但验证能力和判断能力变得比以前更关键了。

一个平衡的结论：AI 会吃掉低价值的重复劳动，让初级开发者更快接触真正的系统问题。但如果团队只让他们接收 AI 输出而不训练他们理解架构、做 code review、分析 trade-off，长期会严重削弱工程判断力。最好的组织会把 AI 当成结对编程的伙伴和导师，同时保留人类 mentorship、严格 review 和明确的 ownership——工具变了，但培养人的逻辑没有变。

Q3. Rails Controller 代码审查：MVC 和 SRP [30 marks]

题目在问什么：给你一段真实的 Rails controller 代码——OrdersController#create 方法。表面上看它能跑：创建订单、算折扣、保存、发邮件、记 analytics、跳转页面。但仔细看，一个方法干了五件事。题目要求你指出 MVC 和 Single Responsibility Principle 的违规，提出重构方案，并说明重构后折扣逻辑的单元测试会发生什么变化。

配分：(a) 识别 MVC/SRP 违规 [10]；(b) 重构并提 design patterns [14]；(c) 测试的变化 [6]。

先看题面代码——问题都在这几行里：

class OrdersController < ApplicationController
  def create
    @order = Order.new(order_params)

    if @order.total > 1000 && @order.user.premium?
      @order.total *= 0.9
    elsif @order.total > 500
      @order.total -= 50
    end

    if @order.save
      Mailer.send_receipt(@order).deliver_now
      Analytics.track("order_created", @order)
      redirect_to @order, notice: "Success"
    else
      render :new
    end
  end
end

Q3(a) 违规识别——如果是 code review，我会这样说 [10]

  1. 折扣计算写在了 controller 里。这一段 if @order.total > 1000 && @order.user.premium? 是定价规则——纯业务逻辑。Controller 的职责应该是"接收 HTTP 请求、调合适的服务、返回 HTTP 响应"，不应该拥有价格策略。哪天折扣规则变了（比如加了"新用户首单 8 折"），你要改 controller——但 controller 的 HTTP 处理逻辑一点没变啊。
  2. Controller 对 model 内部知道得太多了。它直接读 @order.total、调 @order.user.premium?。这意味着 controller 被迫理解了订单的内部结构和用户等级规则。将来 model 重构了，controller 也得跟着改——两个本应独立的层耦合了。
  3. 同步发邮件是隐患。deliver_now 意味着用户要等邮件发完才能看到成功页面。如果邮件服务挂了或者慢了，用户明明已经下单成功，却看到错误页面——合理吗？邮件应该用后台任务异步处理。
  4. Analytics 是横切关注点。记 analytics 和创建订单是两个完全不同的职责。有一天 analytics 的 API 变了，你又要改 controller——但 create order 的逻辑没变。这就是 SRP 违规的典型症状：一个类有不止一个变化的原因。
  5. SRP 的大问题。这个 action 至少有三个变化原因：折扣规则变、邮件模板变、analytics schema 变。加上保存逻辑和 redirect/render 行为——一个方法承担了太多责任，任何一个方向的变化都会迫你修改这个方法。

一句话总结：这个 controller 已经不是"协调者"了。它变成了一个 HTTP 处理、定价策略、持久化、副作用全搅在一起的杂物堆。

Q3(b) 重构：把每种责任分给专门的对象 [14]

重构思路很简单：一个类只做一件事。每种责任分给一个专门的对象。

原问题移到哪里用的模式
折扣规则DiscountPolicyStrategy / Policy Object——不同折扣规则可独立替换，互不影响
创建订单流程CreateOrderServiceService Object / Command——封装整个 use case 的编排
发邮件OrderReceiptJobBackground Job——异步执行，失败可重试，不阻塞用户请求
AnalyticsOrderCreated 事件的 subscriberObserver / Pub-Sub——横切关注点和核心逻辑解耦

重构之后，controller 变得极薄——只做它该做的事：接请求、调 service、根据结果决定跳转或渲染。

def create
  result = CreateOrderService.call(order_params, current_user)

  if result.success?
    redirect_to result.order, notice: "Success"
  else
    @order = result.order
    render :new, status: :unprocessable_entity
  end
end

折扣规则单独抽出来，变成一个纯粹的 Ruby 对象：

class DiscountPolicy
  def self.apply(order)
    if order.total > 1000 && order.user.premium?
      order.total * 0.9
    elsif order.total > 500
      order.total - 50
    else
      order.total
    end
  end
end

CreateOrderService 负责整个流程的编排：build order → apply discount → save（在事务里）→ publish OrderCreated event → enqueue receipt job。Controller 只关心一件事：成功还是失败，据此决定 response。

高分提醒：不要把所有代码简单塞进 model 里然后说"fat model, skinny controller"。更好的答案是把 pricing policy、workflow orchestration、side effects 分开——因为它们变化的原因不同，变化的速度不同，修改其中一个不应该影响其他的。

Q3(c) 单元测试会怎么变 [6]

重构前，测试是一个噩梦。想测折扣逻辑？你得通过 controller request 来测——构造 order params、模拟 user session、准备数据库、还得处理 email 和 analytics 的副作用。测试慢、脆弱、失败时分不清是折扣算错了还是数据库连不上还是邮件服务器挂了。

重构后，一切变简单了。折扣逻辑是纯业务对象，没有任何外部依赖：

it "gives 10% discount to premium orders over 1000" do
  order = build_order(total: 1200, premium: true)
  expect(DiscountPolicy.apply(order)).to eq(1080)
end

测试分层清晰了：DiscountPolicy 的 unit test 只测价格规则——快、稳、纯；CreateOrderService 的 integration test 测完整编排流程——保存、事件发布、job 入队；Controller test 只验证一件事——success 时 redirect、failure 时 render :new 并返回 unprocessable_entity。每一层测什么一目了然，出问题也能立刻定位到是哪个层坏了。

Q4. CI/CD 和 Observability [30 marks]

题目在问什么：在现代 Web 开发中，交付速度（靠 CI/CD）和理解生产环境发生了什么（靠 Observability）是两件决定成败的事。题目要求你定义这两个概念、区分 Observability 和 Monitoring、分析它们怎么配合才能保证质量和可靠性、以及推行完整的 observability stack 会面临哪些 trade-off 和组织挑战。

配分：(a) 定义 CI/CD 和 Observability [6]；(b) 区分 Observability 和 Monitoring [4]；(c) 分析它们如何配合 [10]；(d) 战略层面的 trade-off 和挑战 [10]。

Q4(a) 用一句话讲清楚每个概念 [6]

先从一个真实场景开始：你 push 了一行代码，然后发生了什么？

CI（持续集成）就是 push 之后自动发生的那件事：服务器拉取你的代码，跑 lint、跑单元测试、跑集成测试。全过了，代码合进主分支。有任何一个挂了，打回去修。目的很简单：别让每个人的分支越飘越远，把集成问题尽早揪出来。

CD（持续交付/部署）接着 CI 干：测试通过的代码，自动打包，推送到 staging 环境，再推到 production。持续交付可能在发布前保留一个人工审批按钮；持续部署则是全自动——检查过了就直接上线。

Observability（可观测性）解决另一个问题：代码上线了，你怎么知道它跑得好不好？它让你通过系统对外暴露的信号——日志（logs）、指标（metrics）、分布式追踪（traces）——来理解系统内部的真实状态。比如某个用户说"我的订单打不开"，你能顺着 trace ID 看到他的请求经过了哪些服务、哪个步骤卡住了、为什么卡住了。

Q4(b) Observability 和 Monitoring 不是一回事 [4]

Monitoring 是"盯着你知道该盯的东西"。你预先定好指标——CPU 不能超过 90%、错误率不能超过 2%、磁盘不能快满——然后系统帮你盯着。一旦超标，报警。它回答的问题是："系统现在还健康吗？"

Observability 是"调查你不知道的问题"。你无法预先把所有可能的故障模式都列出来。真正的生产事故往往是你没想到的那种。Observability 给你的是探索未知的工具——通过日志、指标和追踪，你可以问："为什么这个用户的请求失败了？到底是哪个环节出了问题？"

Monitoring（监控）Observability（可观测性）
追踪已知指标，和预设阈值比较支持对未知问题的开放式调查
回答"系统健康吗？"回答"为什么这个请求慢了/失败了？"
告警导向——告诉你出事了调查导向——帮你找出为什么出事

一句话区分：Monitoring 告诉你出事了；Observability 帮你找出为什么出事。

Q4(c) CI/CD 和 Observability 怎么配合：一个闭环 [10]

它们不是两件独立的事，而是一个持续运转的闭环。

  1. 上线前——CI 挡住低级错误。每次 commit 先过 lint、单元测试、集成测试、安全检查，烂代码根本进不了主分支。这一步把大部分显而易见的 bug 消灭在开发阶段。
  2. 发布中——CD 让上线可控。Pipeline 用完全相同的步骤部署 staging 和 production，不会出现"我本地能跑啊"这种经典问题。再用 canary release 或 blue-green deployment，让新版本先影响 5% 的用户，跑一段时间没问题再全量。
  3. 上线后——Observability 验证真实效果。对比发布前后的错误率、p95 延迟、吞吐量。如果变差了，trace 能帮你精准定位是哪个 service、哪个调用变慢了，log 能告诉你具体报了什么错。
  4. 把事故变成预防——闭环合上。如果 observability 发现了一个 production bug，把这个失败场景写成 integration test 或 regression test 放进 CI pipeline。下一次任何人提交代码，这个 bug 都不会再次上线。

一个具体的例子把这个闭环走通：

你们上线了新版的 checkout service。几分钟后，metrics 显示 p95 延迟从 300ms 飙升到 2 秒。Trace 显示请求卡在 payment-service 调 tax-service 的 HTTP 调用上。Log 显示新版本漏配了 HTTP timeout，导致线程一直傻等。你马上回滚这个部署。然后做三件事：加上 timeout 和 fallback 逻辑；写一个 integration test 来模拟 tax-service 延迟；通过 CI/CD 发布修复版本。从此以后，任何人改动 checkout 代码，这个 test 都会跑，同一个坑不会再掉进去。

这就是闭环的威力：CI 防患于未然，CD 让发布可靠，Observability 让问题被发现并追溯到根因，测试把教训固化进流水线。

Q4(d) 推行这套体系的 trade-off 和挑战 [10]

CI/CD 和 Observability 听起来很美好，但真推起来到处都是坑。

挑战为什么棘手怎么应对
速度 vs 安全自动部署快，但错误也传播得快。一次坏部署可能直接炸生产。Staged rollout、canary、feature flag、基于指标的自动回滚——出问题立刻刹车。
Flaky tests不稳定的测试会让团队不再信任 CI。一旦习惯"挂了就重跑"，pipeline 形同虚设。保持测试确定性，隔离不稳定的测试单独修复，别让 flaky test 留在主干 pipeline 里。
可观测性成本日志、指标、追踪的数据量是指数增长的。存和查都很贵。采样策略、保留期限、结构化日志。不是所有 span 都值得存，只存高价值的。
告警疲劳太多低价值告警会让工程师麻木，真正的事故被淹没在噪音里。只对用户真正受影响的情况告警——比如 SLO 被突破——而不是每个内部指标抖动都喊人。
隐私和安全日志里可能包含邮箱、token、支付卡号。日志泄露就是数据泄露。脱敏、权限控制、加密、绝对禁止在日志里打印密码和密钥。
组织变革CI/CD 和 observability 要求开发者对生产环境负责——不再是"代码写完扔给运维"。Shared ownership、runbook、事故复盘、上线纪律——这是文化问题，不是工具问题。

每个挑战的背后都是一个选择题：你要在某个方面做出权衡。比如"速度 vs 安全"——你想更快上线，就得接受自动部署带来的风险，然后用 canary 和自动回滚来对冲。不存在"又快又零风险"的方案。

收束句：CI/CD 和 observability 真正改变的从来不是工具本身，而是团队怎么发布、怎么发现问题、怎么从事故中学习。收益是更快更安全的交付——但前提是团队愿意在测试质量、遥测质量、隐私控制和运维纪律上持续投入。这些投入省不了，省了就是假 CI、假 observability。

不是说逐字背——而是脑子里要有清晰的结构，看到题就知道从哪几个方向写。

必须能说清楚检查自己：能不能回答这个问题？

Q1 变体：换业务场景，但仍考架构比较

可能题面它在考什么教材点答题骨架
HealthTrack：病人记录系统，要实时预约提醒、医生 dashboard、严格保护医疗记录。比较 3-tier monolith 和 microservices。 ArchitecturalPatterns.pdf 的 logical vs physical architecture、n-tier、microservices；DB_WS_C.pdf 的 SQL/NoSQL、ACID、CAP。 先映射 Presentation/Application/Data。再说病人记录和审计日志强一致，SQL/ACID 更重要；实时提醒可用 queue/cache 解决，不必一开始拆微服务。推荐 modular monolith，后续可拆 notification-service。
ShopPulse：电商平台要实时库存、个性化推荐、订单支付历史。问 SQL、NoSQL 或 polyglot persistence 怎么选。 关系数据 vs 文档数据；SQL 的 relationships/foreign keys/transactions；NoSQL 的 flexible schema/horizontal scalability。 订单、支付、库存扣减用 SQL，因为 order、payment、inventory 关系强，写错代价高。推荐/浏览日志可用 NoSQL，因为字段变化快、写入量大、允许 eventual consistency。高分点是不要二选一死答，要按数据子域拆。
同样是 FinTrack，但题目把 Approach B 改成 event-driven microservices，用 message queue 同步价格、portfolio 和 alerts。 Architectural patterns 里的 event-driven architecture、network boundaries、asynchronous interactions。 先承认 event-driven 能解耦和吸收高频 price updates；再指出 eventual consistency、消息重复、顺序问题、debug 难度。交易历史仍要强一致，不能让“消息最终会到”替代金融 transaction。
问 logical architecture 和 physical architecture 的区别，并用 Sample 的 Approach A/B 举例。 ArchitecturalPatterns.pdf 明确区分 logical architecture 和 physical architecture。 Logical 是 Presentation/Business Logic/Data Access 这些职责划分；Physical 是它们部署在哪些服务器、容器、cloud services、databases 上。Approach A 逻辑三层可能物理上在一台实例；Approach B 逻辑上仍三层，但 application/data 在多个 services 和 stores 上物理分散。

Q2 变体：不直接问 AI Agents，而问 AI-assisted workflow

可能题面它在考什么教材点答题骨架
团队用 AI 工具从空白开始生成 full-stack app。分析一个可靠 AI-native workflow 应该怎么设计。 VibeCoding.pdf 的 7-step workflow：Analyse, PRD, Tech Plan, Align, AI Plan, Build, Verify。 不要让 AI 直接写完整系统。先分析数据和用户问题，写 PRD，再写技术计划，让 AI 复述理解，分步骤 build，每一步后测试。核心句：AI can build fast, but humans must define requirements, approve architecture and verify each step。
AI 生成了一个 800 行单文件应用，能跑但难维护。讨论 technical debt。 VibeCoding 里的 “The code works. The architecture is yours to maintain forever.” 和 Design before build。 先说可运行不等于可维护。问题：分层缺失、测试困难、下一位开发者读不懂、每次加功能都更乱。策略：拆层、写 tests、建立 module boundaries、让 AI 一次只改一个模块并解释 plan。
AI Agent 接入 MCP/浏览器/数据库/部署工具，可以自动操作外部系统。讨论收益和安全风险。 VibeCoding 的 MCP/tools；Forms and Security 的认证、授权、token/security。 收益：能读取真实数据、跑测试、部署、反馈。风险：权限过大、误操作生产、泄露 API key/用户数据、prompt injection。策略：least privilege、sandbox、approval gates、audit logs、secret redaction、CI/CD checks。
评价 “developer 的价值从会写语法变成会写清楚 specification”。 VibeCoding 强调 clarity of specification、PRD、Tech Plan、Verify。 同意但要平衡。语法负担下降，但工程判断上升：需求拆解、边界设计、测试、debug、安全、review。Junior 不消失，但工作从纯实现转向 orchestration and verification。

Q3 变体：换一段坏代码，仍考 MVC/SRP/Patterns

可能题面它在考什么教材点答题骨架
ProductsController#update 同时检查 admin 权限、上传图片、调整价格、发库存告警、写 analytics。 Rails MVC；REST Architecture with Spring 的 Controller -> Service -> Repository；Design Patterns 的 Observer。 Controller 只该接 request、调 use case、返回 response。权限移到 Policy/Authorisation；图片处理移到 job/storage service；价格规则移到 PricingService/Strategy；库存告警和 analytics 用 event/observer。
ReportsController#export 查询所有订单、循环生成 CSV、发邮件、记录审计日志，页面卡 30 秒。 Active Record batches、background job、thin controller、observability。 问题：controller 做长任务、阻塞 request、一次性读全表、混入邮件和 audit。重构：ReportExportJob 异步执行，ActiveRecord find_each 分批，ReportService 生成文件，完成后发邮件/事件。
折扣规则有很多 if/elsif：premium、student、Black Friday、coupon、first order。问怎么重构。 Design Patterns.pdf 的 Strategy：replace if/else with interchangeable objects。 用 Strategy/Policy object。每个折扣规则一个 strategy，统一 calculate(order) 接口。新增折扣时加新类，不改 controller/use case。测试每个 strategy 的边界条件。
创建订单后要发邮件、扣库存、记日志、推送通知。题问为什么不应写在 create_order 里。 Design Patterns 的 Observer：when something happens, other parts react without tight coupling。 核心操作是 create order；邮件/库存/日志/通知是 reactions。发布 order.created event，多个 subscribers 分别处理。这样新增 notification 不必修改创建订单核心代码。
Rails route 匹配错误：通用 route 放在具体 route 上面，导致请求走错 action。 Rails Routes and Active Record.pdf：routes top-down，most specific routes first。 说明 Rails 从上到下找第一条匹配 route。具体 route 应放在通用 route 前面。答题时把 URL、expected action、actual action、修复后的 route order 写清楚。

Q4 变体：仍围绕安全发布和生产可理解性

可能题面它在考什么教材点答题骨架
新版本 canary 给 5% 用户后 error rate 上升。说明 CI/CD 和 observability 如何自动回滚。 CICDOBS.pdf 的 canary releases、automated rollback、metrics/logs/traces。 CI 先跑 tests/security scan；CD canary 只放 5% 流量；observability 监测 error rate/latency/SLO burn；超过阈值自动 rollback；trace/logs 定位根因；把事故写成 regression test。
公司有 monitoring，但线上问题仍难排查。问还缺什么。 Monitoring tells you when; Observability tells you why。 Monitoring 有 CPU/error threshold，只能说“出事了”。还需要 structured logs、metrics、distributed traces、correlation/request ID，才能回答哪个 deploy、哪个 service、哪条 request 出错。
设计一条 CI pipeline，要求覆盖 unit、integration、E2E、security、performance。 CICDOBS 的 layered testing strategy：goal is confidence in critical paths。 Unit tests 快速验证函数/策略；integration tests 验证 service + DB；E2E 覆盖关键用户路径但少而精；security scans 如 dependency audit；performance smoke tests 验 p95 latency。不要追 100% coverage，要保护 critical path。
实施 full observability stack 的成本和隐私挑战。 CICDOBS 的 observability at scale、logs/metrics/traces、trade-offs。 成本：数据量、存储、查询、采样。隐私：日志可能含 email/token/payment data。组织挑战：alert fatigue、SLO 定义、开发者生产责任。策略：sampling、retention、redaction、access control、SLO-based alerts。
