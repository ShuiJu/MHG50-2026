Section 1：几乎必出（Sample Paper 已经给出信号）

Sample Paper 的四道题分别对应了 microservices 架构比较、Rails controller 代码审查、CI/CD + observability 定义与配合。But Sample 没有覆盖 full-stack 解释和 MVC 设计模式——而这两道题在 2023–2025 真题中年年出现。因此，Section 1 的三道题为最高概率题，每道都必须能闭卷完整写出。

Q1: Full-Stack 架构解释 + 数据库选型（预测 25 分）

题面预测：给你一个业务场景（如 FinTrack 股票追踪、或购物习惯追踪），要求：(a) 解释 frontend/backend/database 三层各自角色，用一个具体请求串起来；(b) 根据场景数据特征推荐 SQL 或 NoSQL，给出三点技术理由 + 一点 trade-off。

标准答案框架 (a) 三层架构解释（12 分）：

用"用户点击查看收据"串起完整数据流：

  1. Browser (React) 捕获用户点击事件，构造 HTTP 请求：GET /api/receipts/15，headers 携带 session cookie 或 JWT token 证明身份。
  2. Route 层 将 URL pattern /api/receipts/:id 匹配到 ReceiptsController#show。
  3. Controller 验证用户身份（从 token 解析 user_id，检查该 receipt 是否属于该用户），调用 ReceiptService.findById(15)。
  4. Service 层 执行核心业务规则：权限校验、折扣计算、数据聚合。不直接写 SQL，而是调用 ReceiptRepository。
  5. Repository / Data Access 层 生成 SQL：SELECT r.*, ri.* FROM receipts r JOIN receipt_items ri ON r.id = ri.receipt_id WHERE r.id = 15，发给 PostgreSQL。
  6. PostgreSQL 执行查询，返回 receipt header + item rows 的结果集。
  7. Server → Browser 将结果集组装成 JSON {"id":15, "date":"2025-01-03", "total":90, "items":[...]} 返回。React 组件将 JSON 放进 state 并渲染收据详情。

三层职责边界（关键句）：

层负责不知道 / 不负责
Presentation（React）展示数据、收集用户输入、交互反馈数据怎么存的、业务规则怎么算的
Application（Spring Boot / Rails / Express）处理业务规则（折扣计算、权限检查）、协调 service 和 repository不直接写 SQL（那是 Repository 的活）、不负责页面样式
Data（PostgreSQL / MongoDB）持久化存储、高效查询、ACID 事务不知道业务规则（不关心打折公式）、不知道页面长什么样

关键句："每层只和相邻层通信——React ↔ REST API ↔ Backend ↔ DB Driver ↔ Database。换掉 React 用 Vue，不影响后端和数据库；换掉 PostgreSQL 用 MySQL，不影响前端的页面和业务规则。"

标准答案框架 (b) 数据库选型（13 分）：

默认选 SQL（PostgreSQL）。三点技术理由，每条贴回题面具体数据：

  1. 数据是结构化交易记录——场景可建模为 Customers → Receipts → ReceiptItems 三张表（一对多关系）。分析购物习惯需要 JOIN 这三张表做 "某用户过去 30 天总消费" 的聚合查询，SQL 的 FOREIGN KEY + JOIN 是原生支持。
  2. receipt_number 需要 UNIQUE constraint——两张收据撞号的后果是财务对账无法确定哪张是真的。SQL 的 UNIQUE constraint 在数据库层面强制执行，写入那一刻就阻止重复，比应用层代码校验可靠。
  3. 一张收据和它的明细需要 ACID transaction 保证同时写入——不能出现"收据头已存但 2 条明细只存了 1 条"的半成品状态。PostgreSQL 的 BEGIN; INSERT INTO receipts ...; INSERT INTO receipt_items ...; COMMIT; 保证原子性。

Trade-off：如果将来需求从"购买记录"变成"点击流/行为日志"（每秒万条写入、字段不固定如 experimentId 随时增减、不需要 JOIN、丢失几条不影响对账），NoSQL（MongoDB）的灵活 schema 和高写入吞吐用更低成本支撑相同业务。现阶段收据数据的写错代价远高于灵活性需求，所以 SQL。"

Q2: MVC 设计模式 + 框架实例（预测 25 分）

题面预测：解释 MVC 设计模式，用 web 应用例子说明 request flow，再用另一个编程环境（如 mobile app）说明 MVC 同样适用，画图。

标准答案框架：

MVC 三部分职责定义：

组件职责Web 例子Mobile 例子
Model管理数据和业务规则。代表系统中"知道什么"的部分。ActiveRecord / Repository + PostgreSQL。例如 Order model 存储订单数据、验证金额不为负。本地 SQLite 数据库 + 网络 API 层。例如 OrderManager 封装数据库读写和 API 调用。
View只负责展示。不包含业务逻辑，不知道数据怎么存的。React 组件 / HTML 模板。例如 OrderSummary 组件用 props 渲染订单列表。Android Activity / iOS ViewController。例如 OrderListActivity 用 RecyclerView 展示订单。
Controller接收用户动作，调用 Model，选择 View。是"协调者"而非执行者。Rails Controller / Spring @RestController。例如 OrdersController#checkout。Activity 的事件处理器。例如 onCheckoutButtonClick()。

Web request flow 六步：

  1. 用户点击"结账"按钮 —— 浏览器发送 POST /orders/checkout，body 含 cart items 和 shipping address。
  2. 路由到 OrdersController#checkout —— URL pattern 匹配到 controller action。
  3. Controller 调 OrderService 验库存算总价 —— Controller 自己不计算，只把 params 传给 service："这是用户要买的东西，帮我验能不能买"。
  4. Service 通过 OrderRepository 写数据库 —— OrderRepository 负责 SQL 翻译，Service 不直接碰数据库。
  5. Controller 选择 response —— 成功则 render json: {order_id: 123, status: "confirmed"}；失败则 render :new with errors。
  6. React 收到 JSON 更新 UI —— 渲染"订单确认"页面，显示订单号和预计送达时间。

关键句："Controller 不应该包含业务逻辑和数据库操作——那是 Model 的活。Controller 的职责是 HTTP 协调：接请求、调对的人、选对的 response。"

Mobile 映射：

  1. 用户点"下单" → Activity 处理点击事件。
  2. Activity 调 OrderManager.create(cartItems) —— 此处在 mobile 里 OrderManager 充当 Controller 的角色（接收用户意图）。
  3. OrderManager 调本地 DB（SQLite）存待发订单 + 远程 API 发 POST 请求。
  4. Activity 更新 RecyclerView（View）展示最新订单列表。

│ (React/HTML  │     │ (协调者)      │     │ (数据+规则)   │
│  Activity)   │     │              │     │              │
└──────────────┘     └──────────────┘     └──────────────┘

Q3: 微服务 vs 单体架构比较（预测 25-40 分，来自 Sample Q1）

题面预测：FinTrack 案例——Approach A 是单体（React + Spring Boot + PostgreSQL），Approach B 是微服务（React SPA + 多个 Node 服务 + MongoDB）。从 scalability、resilience、development velocity 三个维度比较，最后推荐一个方案并说明理由。

标准答案框架：

Scalability（可扩展性）：

维度Approach A（单体）Approach B（微服务）
独立扩容全量扩容——即使只有 price-service 有压力，也必须复制整个 monolith，浪费 CPU/内存。price-service 可独立扩容到 10 个实例应对实时股价推送，trade-history-service 保持 2 个实例即可。精细粒度扩容。
通信成本进程内方法调用（微秒级），零网络延迟，不会因网络抖动失败。服务间通过 HTTP/gRPC 通信（1–50ms），引入网络延迟、超时、重试开销。
结论B 更优：FinTrack 的实时股价对 price-service 有高吞吐压力但 trade-history 没有，独立扩容避免浪费。但 B 的跨服务网络调用增加延迟，需要加缓存（Redis）和 circuit breaker 缓解。

Resilience（故障容错）：

维度Approach A（单体）Approach B（微服务）
故障隔离单点故障——monolith 挂了整个系统不可用。但运维简单：只有一个进程要监控、一个日志文件。price-service 挂了不影响 trade-history 查询（故障隔离）。但引入新的故障模式：网络分区（两个 service 之间的网络断了）、级联超时（A 超时→B 堆积→C 也超时）、分布式事务失败。
数据一致性单体事务 ACID 简单可靠——BEGIN → INSERT receipt → INSERT items → COMMIT 四条 SQL 在一个 DB connection 里完成。跨 service 事务需要分布式 saga 或 2PC，复杂度指数增长。FinTrack 交易历史需要强一致，拆分后一致性担保成本很高。
结论各有优劣：B 有更好的故障隔离但引入分布式故障模式。FinTrack 场景下，如果交易数据的正确性比价格服务的可用性更重要，A 的简单故障模式反而是优势——"宁可价格显示慢一秒，不能交易记录错一笔"。

Development Velocity（开发速度）：

阶段Approach A（单体）Approach B（微服务）
初期A 更快——一个 repo、一套 CI/CD、一次部署。3 人团队不需要协调跨 service 的 API contract，改一行代码立即看效果。B 初期慢——需要决定 service 边界、定义 API contract、搭建容器编排（K8s）、配置服务发现和分布式日志。3 人团队被基础设施消耗大量时间。
长期A 变慢——代码库膨胀后，改一个功能可能影响全局。所有开发者挤在一个 codebase 里，merge conflict 增多。B 长期更快——price 小队和 trade 小队可独立迭代、独立部署、独立选技术栈。一个 service 的 bug 不会 block 整个 release。
结论A 短期更优，B 长期更优。关键看团队规模和项目阶段。"

推荐方案（必须扣回题面约束）：

推荐：选 A（单体 + PostgreSQL），但预留 price-service 拆分接口。

理由：题面约束是①团队小（small-medium team）——微服务的 K8s 运维、分布式 tracing、跨 service API versioning 的维护成本超过收益；②交易历史需强一致——单体的 ACID 事务比分布式 saga 简单得多，错账代价太高；③实时股价有吞吐压力——但可以通过 Redis 缓存 + WebSocket 推送在单体内部解决，不需要拆独立服务。

演进条件：当 price-service 的 QPS 超过单实例承载能力 + Redis 缓存命中率降到阈值以下 + 团队扩展到 10 人以上可以分成两个 squad 时——再拆出独立的 price-service 并逐步迁移。

答题关键：每个维度都给"两边各有什么好、什么坏"，不能说一边全好一边全坏。最后推荐必须扣回题面约束（团队规模、数据一致性需求、实时性压力），而不是泛泛说"微服务好"。

Q4: Node.js + Event Loop 输出题（预测 10-15 分）

题面预测：给一段嵌套 setTimeout / console.log 代码，要求写出输出顺序并解释 event loop 机制。例如：

console.log('A');
setTimeout(() => console.log('B'), 0);
Promise.resolve().then(() => console.log('C'));
console.log('D');

标准答案框架：

输出：A → D → C → B

解释（分层递进）：

  1. 同步代码先全部执行完（call stack 清空） —— console.log('A') 和 console.log('D') 是同步调用，直接压入 call stack 执行后弹出。此时输出 A 和 D。
  2. Microtask queue 优先于 macrotask queue —— Promise.resolve().then(...) 的 callback 进入 microtask queue（Promise jobs）。setTimeout(...) 的 callback 进入 macrotask queue（Timer phase）。Event loop 在每个 macrotask 之间清空所有 microtask，所以 C 先于 B 输出。
  3. setTimeout(fn, 0) 不是"立刻执行" —— 0ms 的意思是"尽快加入 macrotask queue"，但至少等当前同步代码跑完 + microtask queue 清空后，event loop 才会取它执行。所以 B 是最后一个。

完整 event loop 时间线：

t0: call stack → console.log('A') → 输出 A
t1: call stack → setTimeout(..., 0) → callback 进入 macrotask queue
t2: call stack → Promise.resolve().then(...) → callback 进入 microtask queue
t3: call stack → console.log('D') → 输出 D
t4: call stack 清空 → event loop 先取 microtask: console.log('C') → 输出 C
t5: microtask queue 清空 → event loop 取 macrotask: console.log('B') → 输出 B

Q5: CI/CD + Observability（预测 15-25 分，来自 Sample Q4）

题面预测：定义 CI/CD 和 Observability，解释两者如何配合排查 production bug，写一个具体故障排查流程。

标准答案框架：

定义：

概念定义核心能力
CI（Continuous Integration）代码合并到主干后自动 build + test。每次 git push 触发 pipeline：装依赖→lint→单元测试→集成测试→build artifact。任何一步失败就中断，阻止坏代码进主干。防止 integration hell、尽早发现 bug、保证 main branch 始终可部署。
CD（Continuous Delivery/Deployment）通过测试的代码自动部署到 staging，经过 smoke test 后自动/手动推到 production。缩短"代码写完→用户能用"的时间、减少手工部署的人为失误。
Observability通过 logs（每条请求的详细记录）、metrics（聚合数字趋势）、traces（跨 service 的请求路径）让工程师能问"为什么 user A 的请求返回了 500"。回答未知问题——不只是看已知指标有没有超阈值。

Observability ≠ Monitoring：

Monitoring 告诉你 "CPU 90% 了"（已知指标超预设阈值）。Observability 让你问 "为什么用户 A 看收据的请求返回了 500"（未知问题的自由探索）。Monitoring 定义"健康"的边界；Observability 提供诊断"不健康"原因的工具。两者互补但目的不同。

故障排查流程（完整的 production bug 闭环）：

  1. 收到 PagerDuty alert —— observability 的 metrics 检测到 /api/receipts 端点的 p99 latency 从 200ms 跳到 3000ms、error rate 从 0.1% 跳到 5%。
  2. 在 tracing 系统（Jaeger）找到该 trace ID —— 挑一个失败的 trace，看到请求路径：React → API Gateway → receipt-service（200ms, OK）→ price-service（2800ms, Timeout）。bottleneck 定位到 price-service。
  3. 查 price-service 的 logs —— 在对应时间窗口 grep 出 "Redis connection timeout" 的 stack trace。日志说 price-service 在连 Redis 时 30 次尝试均超时。
  4. 查 metrics 确认 —— Redis 连接池的 available_connections 指标在故障时刻从 20 降到 0，connection_wait_time 从 1ms 飙升到 5000ms。证实连接池耗尽。
  5. 修复 —— 增加 Redis 连接池大小从 20 到 50，或为 price-service 加 circuit breaker（Redis 不可用时返回缓存价格而不是超时报错）。
  6. CI/CD pipeline 自动部署修复并跑 regression test —— git push 触发 CI 跑 lint + 单元测试 + 集成测试（staging 上模拟 Redis 不可用的 fallback 行为）。Pass → CD 部署到 production。

关键：不能只写 "logs/metrics/traces"，要写出三者各自提供什么信息、联合使用的场景。

Q6: Rails Controller 代码审查（预测 15-30 分，来自 Sample Q3）

题面预测：给你一段 OrdersController#create 代码（内嵌折扣计算、订单保存、邮件发送、analytics 记录），要求指出 MVC/SRP 违规、提出重构方案、说明重构如何改善测试。

# Sample 中你可能会看到的坏代码结构：
def create
  @order = Order.new(order_params)
  # 折扣计算（业务规则在 controller 里）
  if @order.total > 200
    @order.discount = 20
  elsif @order.total > 500
    @order.discount = 80
  end
  # 直接保存（数据操作在 controller 里）
  @order.save!
  # 发邮件（副作用在 controller 里）
  OrderMailer.confirmation(@order).deliver_now
  # 记 analytics（横切关注点在 controller 里）
  AnalyticsTracker.track("order_created", { order_id: @order.id })
  redirect_to @order
end

标准答案框架：

违规识别（逐一标出）：

  1. 折扣计算（业务规则，应在 Service 层） —— if @order.total > 200 ... 是业务规则。违反 SRP：折扣策略变（比如黑五改成满 300 减 50），必须改 controller。Controller 的职责是 HTTP 协调，不是业务计算。
  2. 直接写数据库（应用 Repository 封装） —— @order.save! 直接在 controller 里调。虽然一行，但只要 controller 知道"保存"这件事，任何持久化方式变更（加缓存层、改分库规则）都可能影响 controller。
  3. 发邮件（副作用，应通过 background job） —— OrderMailer.confirmation(@order).deliver_now 让用户等邮件发完才看到 redirect。邮件服务商挂了会导致订单提交也失败。应该改为 deliver_later 放进 ActiveJob。
  4. 记 analytics（横切关注点，应用 Observer/pub-sub） —— AnalyticsTracker.track(...) 与核心业务无关。Analytics 服务挂了会影响订单提交。应该通过 model callback 或 event bus 解耦。

违反 SRP 的核心原则：create action 有超过一个"会变化的原因"——折扣策略变、邮件模板变、analytics 需求变。任何一个变都要改 controller。SRP 说：一个类应该只有一个变化的原因。

重构方案：

原来在 controller 里移到哪里为什么
折扣计算（if total > 200 ...）app/services/order_pricing_service.rb纯业务规则，输入 order + coupon → 输出 discount。独立单元测试：expect(service.calculate(order, nil)).to eq(20)。
订单保存（@order.save!）app/repositories/order_repository.rb数据访问封装。换 ORM 或加缓存时只改 repository。
发邮件（deliver_now）app/jobs/order_confirmation_job.rb（ActiveJob）异步执行，用户不等邮件。失败可 retry，不影响订单提交。
记 analytics（track）app/models/order.rb（after_create callback）或 EventBus横切关注点。Analytics 挂了不 block 订单。

重构后 controller：

def create
  @order = OrderPricingService.new(order_params).build_and_apply_pricing
  if @order.valid?
    OrderRepository.save!(@order)       # after_create callback 触发: mailer job + analytics event
    redirect_to @order, notice: "Order created"
  else
    render :new, status: :unprocessable_entity
  end
end

测试改善（这是得分关键）：

重构前重构后
测 create action 需要 stub 数据库、邮件、analytics 三层。一个"测 redirect 是否成功"的测试被迫配置全部 mock。Controller 测试：只需 stub OrderPricingService 返回成功/失败，验证 HTTP 层的 redirect / render。不碰数据库和邮件。
测折扣计算必须构造完整 HTTP request（POST 到 /orders 带 form params），mock session 认证，然后解析 response 验证价格。OrderPricingService 单元测试：纯 Ruby 对象，service = OrderPricingService.new(items: [...])，测各种折扣组合——满 200 减 20、满 500 减 80、优惠券过期——不需要 HTTP、数据库、邮件。
改折扣公式 → 所有 controller test 可能因 mock 配置过时而 break。改折扣公式 → 只影响 OrderPricingServiceSpec 中的 3 个 test case。Controller test 不受影响。

Section 3：如果突然换题型——会换成什么但用类似手法

Q7: TypeScript 对工程质量的影响（替代 "React 有什么优点"）

题面预测（如果不再问 React merits）："TypeScript 如何减少 Web 应用的 bug？从类型安全、重构信心、团队协作三个角度回答。给具体代码例子。"

标准答案框架：

答题手法：和 "React 有什么优点" 题一模一样——对比动态类型语言的劣势，每条给具体例子，不是空泛说 "类型安全好"。

1. 类型安全 — 编译时阻止运行时错误：

JavaScript 里：function add(a, b) { return a + b } 可以 add("1", 2) 得到 "12"（字符串拼接而非数学加法），生产环境 bug 靠用户报告才发现。TypeScript 里：

function add(a: number, b: number): number {
  return a + b;
}
add("1", 2); // ❌ Compile error: Argument of type 'string' is not assignable to parameter of type 'number'.

TS 在编译阶段就拦截此类 bug。更复杂的例子：API response 的类型定义让 IDE 自动提示字段名、拼写错误在编译时发现而不是在浏览器 console。

2. 重构信心 — 类型系统充当安全网：

// 重构前
interface User { name: string; email: string; }
function greet(u: User) { return "Hello " + u.name; }

// 重构：把 email 拆成 workEmail + personalEmail
interface User { name: string; workEmail: string; personalEmail: string; }
// ❌ 所有引用了 user.email 的地方自动标红 — 编译器告诉你 37 处需要更新
// JavaScript 里你只能 grep "email" 并祈祷没有误伤

3. 团队协作 — 类型是活的文档：

interface CreateOrderRequest {
  items: { productId: string; quantity: number }[];
  shippingAddress: string;
  couponCode?: string;           // ? = optional — 自文档化
  paymentMethod: "card" | "paypal"; // union type — 限定合法值，新人不会传 "cash" 然后等 runtime error
}

新队员调用此 API 时不需要读文档或问人，IDE 自动补全所有字段并标出类型错误。

总结句（扣回考题）："TypeScript 不消除所有 bug，但把 bug 从'用户在生产环境发现'前移到'开发者在 IDE 里看到红色波浪线'——大幅降低修复成本和用户影响。"

Q8: WebSocket 实时推送的输出顺序（替代 Node.js callback 输出题）

题面预测（如果不再考 setTimeout 输出顺序）："用户 A 编辑股价、用户 B 同时查询、server 通过 WebSocket 推送——写出各方收到消息的顺序并解释为什么。"

标准答案框架：

场景描述：用户 A 在 FinTrack admin panel 修改 AAPL 股价从 $150 到 $152，同时用户 B 正在看 AAPL 详情页。Server 通过 WebSocket 向所有连接推送更新。

时序分析：

  1. 用户 A 的浏览器发送 POST /api/stocks/AAPL { price: 152 }。这是 HTTP request，通过 TCP 连接发到 server。
  2. Server 收到 POST —— controller 验证 A 的 admin 权限，调用 StockService.updatePrice("AAPL", 152)。Service 更新数据库中的 AAPL 记录。
  3. Server 通过 WebSocket broadcast —— wss.broadcast({ type: "PRICE_UPDATE", symbol: "AAPL", price: 152, timestamp: ... }) 发送给所有已连接的 WebSocket 客户端。
  4. 用户 A 收到 POST response —— HTTP response { status: "ok" } 通过同一个 TCP 连接返回。注意：A 的 WebSocket 也同时收到了自己触发的 PRICE_UPDATE 消息（因为 broadcast 包含所有连接）。
  5. 用户 B 收到 PRICE_UPDATE —— 如果 B 和 A 到 server 的网络延迟不同，B 可能比 A 先收到 WebSocket 推送（因为 A 走 HTTP response 路径而 B 始终在 WebSocket 连接上等待）。

输出顺序（考虑 event loop 和网络 IO）：

Server 端执行顺序（同步）：
① 处理 HTTP POST → ② 更新 DB → ③ WebSocket broadcast → ④ 返回 HTTP response

各客户端收到消息的实际顺序（异步，受网络延迟影响）：
A: 先收到 HTTP response → 然后 WebSocket PRICE_UPDATE 到达（TCP 保序）
B: 没有 HTTP response，只收到 WebSocket PRICE_UPDATE
如果 A 的网络延迟 > B:  B 的 WebSocket 消息 可能比 A 的 HTTP response 先到

底层考的仍是 event loop + 异步：WebSocket 的 onmessage callback 和 HTTP fetch().then() 都是异步回调。Server 端 broadcast 是同步调用（推消息到 socket buffer），但网络传输和客户端接收是异步的。原理和 setTimeout vs Promise 一样：谁先进入 callback queue 取决于 IO 完成时间，不取决于代码书写顺序。

Q9: JWT Authentication 流程（替代 accessibility/SEO 题）

题面预测（如果不再单独考 accessibility 和 SEO）："画 JWT authentication 的流程图，说明 access token 和 refresh token 的分工，以及为什么 token 比 session cookie 更适合微服务架构。"

标准答案框架：

JWT Authentication 流程图：

┌─────────┐     ① POST /login          ┌──────────┐
│  Client  │ ──────────────────────────→ │  Server   │
│ (React)  │ ←────────────────────────── │ (Auth)    │
└─────────┘  ② {access_token, refresh}   └──────────┘
     │                                          │
     │ ③ GET /api/orders                        │
     │    Header: Authorization: Bearer <AT>    │
     ├──────────────────────────────────────────→│
     │                                           │ ④ verify signature
     │                                           │    (no DB lookup needed)
     │ ⑤ 200 { orders: [...] }                  │
     │←──────────────────────────────────────────┤
     │                                           │
     │ [15 min later, AT expired]                │
     │ ⑥ POST /refresh                          │
     ├──────────────────────────────────────────→│
     │ ⑦ { access_token: "new..." }             │
     │←──────────────────────────────────────────┤

Access Token 和 Refresh Token 的分工：

属性Access TokenRefresh Token
有效期短（15 分钟）长（7 天）
用途每次 API 请求携带，证明"我是谁、我能做什么"仅用于获取新的 access token，不参与常规 API 调用
暴露风险每次请求都发，网络抓包风险高→短有效期限制损害窗口极少发送（只在 access token 过期时），暴露面小
存储位置前端内存（React state）HttpOnly cookie（JS 无法读取，防 XSS）
是否查数据库否——自包含（header.payload.signature），server 只验证签名是——server 需查 DB 确认此 refresh token 未被 revoke

为什么 JWT 比 session cookie 更适合微服务架构：

  1. 无状态验证 — 每个 service 独立验证 token，不需要查集中式 session store。 Session cookie 要求所有 service 共享 Redis/Central DB 来查 session，token 的验证只需公钥验证签名——price-service 和 trade-service 各自独立验证同一个 JWT，不需要互相通信。在微服务几十个 service 的规模下，节省了 session store 的瓶颈和故障点。
  2. 自包含 claims — payload 里可以放 user_id、roles、permissions。 不需要每次调 /users/me 查权限。service 从 token 直接读出 {sub: "user_42", scope: "admin stocks:write"} 就知道该用户能不能改股价。
  3. 跨语言通用 — 签名验证是标准算法（HMAC-SHA256 / RSA）。 price-service 用 Node、trade-service 用 Java、auth-service 用 Go——三者都能验证同一个 JWT，不需要共享 session 实现。

Trade-off：JWT 不能服务端主动 invalidate（logout 后只要 access token 没过期就还能用），需要用短有效期 + refresh token rotation 缓解。Session cookie 可随时删掉 session 记录立即生效。
