CS608 linear course

CS608 教材线性精讲：从一个测试用例学完整门课

CS608 的主线是：在有限时间里，怎样系统挑出有代表性的测试，而不是凭感觉随便试。整门课从 specification 到 test case，再到黑盒、白盒、对象状态、自动化、随机测试和过程管理，都在回答同一个问题：测试从哪里来，凭什么说它覆盖了风险。

测试用例 EP / BVA / DT SC / BC / JaCoCo Class Context Automation 考试台阶

0. 一个测试用例到底由什么组成

测试不是“运行一下看看”。一个 test case 至少要说明输入是什么、怎样执行、期望结果是什么。没有 expected result 的运行，只能叫试用，不能算严格测试。

测试词白话解释小例子
Specification系统应该做什么的描述。年龄小于 18 返回 child fare，18 及以上返回 adult fare。
Input喂给程序的数据。age=17。
Expected result按 specification 推出的正确结果。期望返回 child fare。
Actual result程序实际跑出来的结果。程序返回 adult fare。
Failure外部可见的不正确行为。17 岁被收成人价。
Fault / defect代码或设计里的错误原因。代码写成 age <= 17 或边界条件写反。

测试设计的最小闭环

  1. 读 specification。先找输入变量、输出结果、规则、边界、例外情况。
  2. 选 test coverage items。TCI 是你打算覆盖的“测试目标”，例如某个输入分区、某个边界、某条决策表规则。
  3. 把 TCI 变成 concrete data。Coverage item 是抽象目标，data value 是具体数值。
  4. 写 test case。说明输入、执行步骤、expected result。
  5. 运行并比较。Actual result 和 expected result 不一致，才说明发现 failure。

过关小题：如果题目只给你一个函数描述，不给代码，优先用黑盒方法；如果题目给你代码和 JaCoCo 报告，优先用白盒方法补 coverage。

对应材料：01-module-overview.pdf 和 02-intro-to-testing.pdf。先抓 specification、test case、expected result、coverage 这些词，后面的 BVA、DT、SC、BC 都是在细化 coverage 从哪里来。

内容理解题（先做 3 题）

核心答案（先确认概念）：因为你只能看到程序跑了，却无法判断 actual result 是否正确。

为什么这是对的：测试的本质是对照规格验证行为。如果你不知道期望结果，程序返回任何值你都只能说"它返回了X"，而不能说"它对"或"它错"。比如跑 fare(17) 得到 child，但如果不知道 spec 规定 age<18 才是 child，就无从判断输出是否与规格一致。没有 expected result 的运行只是观察行为，不是验证行为。

题目语境怎么用：考试中任何"请给出测试"的题目，答案都必须包含输入、执行步骤和 expected result 三要素。缺了 expected result，整道题等于没做——阅卷人无法判断你的测试能否发现错误。判断力来自 expected result 与 actual result 的比较。

卷面写法：用三列表格：Input | Expected Result | Actual Result。Expected Result 必须从 specification 推导，不能写"程序输出即为期望"。引用 spec 里的具体规则（如"根据 spec 第3条，age<18 child"），答案更有说服力。

常见失分点：只给 input 不给 expected result；把 expected result 写成"程序跑出来的就是期望"（循环论证）；用被测程序自己的输出反推期望值——等于让考生给自己打分。

核心答案（先确认概念）：输入、执行方式和 expected result。更完整时还包括测试目标、环境和实际结果记录。

为什么这是对的：test case 是"可独立复现的验证单元"。假设你写"测 age=17"，别人不知道你调哪个函数、期望什么输出。但写成"调用 fare(17)，期望返回 child fare"，任何同学都能复现并判断对错。加上 TCI 是为了可追溯——知道这个 test case 在覆盖哪个风险点。

题目语境怎么用：考试里给出 test case 时，至少包含 Method Name、Input Values、Expected Output 三列。如果涉及对象类（如 Q3），还要加上 Object Initial State，因为 instance method 的行为依赖对象状态。

卷面写法：用结构化表格：列标题 TCI | Input Data | Method Call | Expected Result。不要写成描述性段落——阅卷人找不到关键信息会扣分。TCI 写抽象目标，Data 写具体值，Expected 写可验证的输出。

常见失分点：漏写 expected result；把 TCI 和 test case 混为一谈填在同一格；忘记写被测方法名称和参数；对象测试题漏写对象初始状态。

核心答案（先确认概念）：Failure 是外部可见错误行为；fault 是代码或设计中的错误原因。

为什么这是对的：Failure 是用户或测试人员能看到的症状——程序崩溃、返回值不对。Fault 是代码里的 bug——某行写错了运算符、漏了边界检查。一个 fault 可能始终不产生 failure（如死代码里的 bug），一个 failure 可能由多个 fault 共同导致。用医学类比：failure 是症状（发烧），fault 是病因（细菌感染）。

题目语境怎么用：考试可能描述一段场景问"这是 failure 还是 fault"。如果描述"程序接收 age=17 却返回 adult fare"，这是 failure；如果描述"代码第12行把 age>=18 写成了 age>18"，这是 fault。先判断题面给的是表象还是根因。

卷面写法：分两行写：Failure（外部可见）：程序输出 adult fare 而非 child fare。Fault（代码原因）：line 12 条件误写为 <= 17 导致 17 岁被当作成人。如果题目问修复方案，给出具体代码改动。

常见失分点：把两个词颠倒使用；把 failure 描述成代码行级别的错误；只解释概念不联系题面具体场景；混淆 defect/bug/fault 三个同义词。

考试题目题（再做 5 题）

标准答案（先写结论）：温度边界控制 HEAT，湿度边界控制 AC；两者组合产生 NONE、HEAT_ONLY、AC_ONLY、HEAT_AND_AC。

为什么这是对的：Climate.determine 的 specification 给出两条独立规则：temp<16 触发 HEAT，humidity>60 触发 AC。两个布尔条件独立判断，所以存在 2x2=4 种组合：都不触发 NONE，仅温度触发 HEAT_ONLY，仅湿度触发 AC_ONLY，都触发 HEAT_AND_AC。这种多条件组合模式天然适合 Decision Table 建模。

题目语境怎么用：这道题的 (b) 子题要求先识别 outputs，为后续 (c) BVA 和 (d) DT 打基础。读题时先把 spec 里的每个比较条件单独提取，再列出所有布尔组合对应的输出枚举值。不要跳步直接写边界值。

卷面写法：先写两条独立条件链：cond1=(temp<16) HEAT，cond2=(humidity>60) AC。再列四行组合表并给每个组合写上对应的 enum 输出值 NONE/HEAT_ONLY/AC_ONLY/HEAT_AND_AC。

常见失分点：只列出三种输出漏掉 HEAT_AND_AC；把 temp 和 humidity 当同一个维度的两个区间而非两个独立条件；使用自创的输出名称而非题目提供的枚举值。

标准答案（先写结论）：maxv 的 oracle 应该是数组最大值定义；如果用 actual output 当 expected，就无法发现错误。

为什么这是对的：如果你用被测 maxv() 自己算出的结果作为 expected result，就形成了循环论证：程序算出 5 你说期望也是 5，测试永远通过。但如果程序有 bug 始终返回第一个元素，你的期望也跟着错。正确做法是用独立的 oracle——手动遍历数组找最大值，或用数学定义验证——这些不依赖被测代码。

题目语境怎么用：2023 Q1(a) 直接问 expected result 的来源，你要回答：来自 specification 对最大值的数学定义——"数组中没有任何元素比它大"。这个定义可独立验证，不需要调用被测函数。

卷面写法：先声明 oracle 来源（如"数组最大值的数学定义：对任意元素 a[i]，expected a[i] 且 expected 等于某个元素"），再给具体验证方法（如"遍历数组确认返回值大于等于所有元素"），最后说明为何不能用程序输出。

常见失分点：说"手动算"但不说明怎么算；说"用另一个函数算"但如果那个函数也是被测代码的一部分同样没解决问题；没有点出循环论证这个核心逻辑漏洞。

标准答案（先写结论）：因为 coverage 只说明执行路径，expected result 才能判断路径上的行为是否正确。

为什么这是对的：JaCoCo 告诉你 line 33 被执行了（绿了），但不告诉你执行结果对不对。比如 line 33 是 return x*2 但规格要求 return x+2——coverage 100% 但功能是错的。Expected result 的作用是双保险：不仅代码跑到了，而且跑出来的是正确结果。Coverage 解决覆盖了吗，expected result 解决做对了吗。

题目语境怎么用：在 2025 Q2 白盒题中，你补 additional test 的目的是让红线变绿。但每个新 test case 仍要从 specification 推导 expected result，不能因为"这是补覆盖的"就只给 input。题目表格里 Expected Result 列不能空。

卷面写法：每个 additional test 写四要素：Coverage Item（要补的 line/branch）、Input（能触达该代码的具体值）、Expected Result（从 spec 推出的正确输出）、Explanation（数据如何让控制流走到目标位置）。

常见失分点：只给 input 说"这样能跑到 line X"但没有 expected result；expected result 不是从 spec 推出而是猜的；混淆 coverage improvement 和 correctness verification。

标准答案（先写结论）：Static 直接 Numbers.isNeg(-1)；instance 要 create object、set value、call method、get result。

为什么这是对的：static method 的行为只依赖传入参数，不依赖对象内部状态——调用 Numbers.isNeg(x) 时结果仅由 x 决定。instance method（如 isNegative()）则依赖对象内部存储的 value 字段——必须先创建对象并通过 setValue() 设定该值再调方法。这是 static 和 instance 的根本区别：前者无状态，后者有状态且状态影响行为。

题目语境怎么用：Numbers 题中，isNeg(int x) 有 static 关键字所以直接调用即可。isNegative() 没有 static，必须先 Numbers obj = new Numbers(); obj.setValue(-1); 再调被测方法。读题时先看方法签名里有没有 static。

卷面写法：static 测试写成一行：assertEquals(true, Numbers.isNeg(-1))。instance 测试写成完整序列：Numbers n = new Numbers(); n.setValue(-1); n.checkValue(); assertEquals(true, n.isNegative())。每一步写清楚目的。

常见失分点：把 instance method 当 static 写成 Numbers.isNegative(-1)；调用前忘记 setValue；以为 constructor 会自动设置被测值；把 setValue 和 checkValue 的调用顺序搞反。

标准答案（先写结论）：用 public setter/constructor 设置 brightness 和 override，调用 decide()，再用 getter 检查 power state。

为什么这是对的：Lighting 类有 private 字段 brightness 和 override。decide() 读取这两个字段决定 power state。测试时必须通过 public API——构造函数 new Lighting(50,false) 或 setter setBrightness(int)——设置状态，调用 decide() 触发逻辑，再用 getPowerState() 读取结果。直接访问 private 字段会让测试绑死在内部实现上。

题目语境怎么用：2023 Q3(b) 让你先识别 accessor methods（getter/setter），因为之后每步测试都要通过这些公开方法操作对象。按顺序：构造函数设初始状态 setter 调整 调用被测方法 decide() getter 取结果 assert。

卷面写法：写出完整的方法调用序列并注解每步目的：Lighting l = new Lighting(50, false); l.setBrightness(20); l.decide(); assertEquals(PowerState.ON, l.getPowerState())。每一步之间用注释连接成逻辑链。

常见失分点：在伪代码里写 l.brightness=20（直接访问 private field）；不设 override 就用默认值测；只写 decide() 调用不写前后状态设置和观察步骤。

本节过关门槛

1. Coverage 不是“测得多”，而是“测到该测的东西”

Coverage 的意思是覆盖度。它不是单纯追求 test case 数量多，而是说明哪些风险点已经被代表性测试碰到。CS608 里最关键的能力，就是把题目要求转换成 coverage items。

Coverage 来源看什么会生成什么 TCI
Specification输入范围、输出分类、业务规则。Equivalence partitions、boundaries、decision table rules。
Code语句、分支、条件、短路逻辑。Statement coverage items、branch coverage items。
Object state对象字段、构造状态、方法调用历史。Class context tests。
Random strategy数据分布、oracle、停止条件。Random test criteria。

从 TCI 到 test case 的区别

TCI 是“要覆盖什么”，test case 是“用哪个具体输入去覆盖”。比如 BVA 中 “just below lower boundary” 是 TCI，具体数据 age=17 才是 test data。考试里表格常要求你分清 TCI、Data Values 和 Test Cases。

写表格时的顺序：先列 TCI，例如 age just below 18；再给 Data Value，例如 age=17；最后写 Test Case，例如 call fare(17) and expect child fare。

内容理解题（先做 3 题）

核心答案（先确认概念）：是测到应该测的风险点。TCI 说明测试覆盖目标，test case 才是具体输入和期望。

为什么这是对的：假设 fare(age) 有 100 个可能的 age 值，你随机测 50 个看起来测得多，但如果 50 个全落在 20-60 岁而漏掉了关键边界 18 岁，再多测试也没用。Coverage 的逻辑是：先识别风险点（边界 18、无效输入 -1），再确保每个风险点至少有一个代表测试。TCI 就是风险点清单，test case 是弹药。所以测得多不等于覆盖全。

题目语境怎么用：考试题问请给出 coverage items 时，你的任务是列要测试哪些种类的场景——比如 age 在有效范围内、age 恰好等于下边界、age 为负数（无效）。这些是抽象目标，后面才配具体值。

卷面写法：用表格分列 TCI（抽象目标）和 Data Value（具体值）。TCI 写 just below lower boundary of age，Data Value 写 17。不要让两类信息混在同一格。Expected Result 单独一列，必须从 spec 推导。

常见失分点：把多写几个测试当成 coverage——数量不等于覆盖质量；TCI 列写得太具体直接写数值导致与 Data Value 重复；漏掉无效类别的 TCI；忽略 one fault assumption 做全组合。

核心答案（先确认概念）：TCI 是抽象覆盖目标，例如 just below boundary；data value 是具体值，例如 17。

为什么这是对的：just below boundary of 18 这个 TCI 可以用很多值满足：17、10、甚至 0。但 boundary value analysis 要求选 17——离边界最近的——因为 fault 最可能出现在阈值刚跨过的位置（如代码把 < 写成 <=）。TCI 告诉你测什么类型，data value 告诉你用什么具体数据去触发。选 data value 的原则是最可能暴露 fault，不是随机挑。

题目语境怎么用：Climate 题中 TCI 写 temp just below 16，data value 写 15 而不是 10，因为 15 离边界最近。isSquare 题中 TCI 写 perfect square，data value 选 16 或 25——选代表性强的具体数字。

卷面写法：表格列顺序必须是 TCI Data Value Test Case Expected Result。TCI 用英文短语（如 temp just below 16），Data Value 用具体数字（如 15），Test Case 写完整调用，Expected Result 写输出。不要把 TCI 栏写成数字。

常见失分点：TCI 和 Data Value 写成一模一样；Data Value 选了离边界很远的数；忘记一个 TCI 对应一个 data value 把多个 data values 堆在同一行。

核心答案（先确认概念）：Specification、code、object state 和 random strategy 都能生成不同类型的 coverage items。

为什么这是对的：四种来源对应 CS608 四道大题：spec 生成 EP/BVA/DT 的 TCI（Q1）；代码从 JaCoCo 报告发现未覆盖语句分支（Q2）；对象状态设计不同构造和 setter 组合覆盖类行为路径（Q3）；随机策略按 rule/partition 生成随机数据（Q4）。同一条 spec 可从四个维度产生完全不同的 coverage items。

题目语境怎么用：如果题目只给 spec 不给代码（如 Q1 Climate），coverage 来自 spec 做 EP/BVA。如果题目给代码和 JaCoCo 报告（如 Q2 decideWrite），coverage 来自 code 做 SC/BC 补漏。如果题目给类和字段（如 Q3 Numbers），coverage 来自 object state 设计调用序列。

卷面写法：明确标注选用的 coverage 来源并说明原因：如本题采用 specification-based coverage，因为题面只提供函数规格未提供源代码。然后在 TCI 表格中体现所选方法。

常见失分点：四种来源混用导致答案逻辑混乱；明明有代码和覆盖率报告却只做黑盒分析漏掉白盒补测；在对象题里忽略 state-based coverage 只按函数参数测。

考试题目题（再做 5 题）

标准答案（先写结论）：围绕 temp=16 和 humidity=60，例如 just below/on/above，且不包含 error values。

为什么这是对的：Climate.determine 的 spec 有 temp=16 和 humidity=60 两个数值阈值。BVA 的核心原则是 fault 最容易出现在阈值刚跨过的位置——程序员最可能在这里写错比较运算符。你必须选每个阈值的三点：just below（刚好低于）、on（等于）、just above（刚好高于）。对 temp=16 就是 15/16/17，对 humidity=60 就是 59/60/61。

题目语境怎么用：题目明确说不含 error values，意思是 (c) 子题的 BVA 表格只处理有效范围边界，无效值已在 (b) 子题的 partitions 中处理。你的表格只出现 15/16/17 和 59/60/61，不要出现 -1 或 999 这类越界值。

卷面写法：用两个独立表格或一个表格分两组。如同时测两个变量，按 one fault assumption 一次只让一个变量靠边界、另一个取 nominal（如 temp=15, humidity=50 expected HEAT_ONLY）。每个组合都要写 expected enum。

常见失分点：把 error values 混进 BVA 表；漏掉 on-boundary 值（只测 above 和 below）；两个变量同时取边界值做全组合；expected result 不写具体 enum 值而写描述性文字。

标准答案（先写结论）：红线表示未执行 statement，需要设计输入让该语句执行，形成 statement coverage TCI。

为什么这是对的：JaCoCo 用颜色标记代码行的执行状态：红色 = 该行从未执行过（0% coverage），绿色 = 完全执行，黄色 = 部分执行。红线是最紧急的信号——一段代码逻辑完全没被测试到，里面可能存在任意 bug 你不知道。要消除红线，必须设计一个测试输入让程序的控制流恰好走到那一行。

题目语境怎么用：2023 Q2 Wind.categorise 里看到某行红色时，追溯代码逻辑：什么条件能让程序走到这一行？需要 wind speed 在什么范围？给出具体 wind speed 值和 expected category，说明这个输入会让控制流经过红色行。

卷面写法：写成：Coverage Gap：line X (red, statement not executed)。Additional Test Input：windSpeed = Y。Expected Result：category = Z。Explanation：windSpeed=Y falls into range [A,B) which causes if-condition at line N to be true, directing execution to line X。

常见失分点：只给 input 值不说明为什么能走到红线；把黄色和红色搞混；没有追踪控制流逻辑随便猜一个 input 值；补了 SC 但漏了 BC 的补测。

标准答案（先写结论）：写成具体未走分支，例如 line 28 true/false branch 或 null else branch，而不是只写“yellow line”。

为什么这是对的：JaCoCo 里黄色 diamond 表示一个 decision point（如 if/while）的 branch coverage 不完整——可能 true 分支走了但 false 分支没走。你要明确指出缺的是哪个方向的分支。比如 line 28 的 if 条件，当前测试只覆盖了 true 分支（条件成立时走进了 if 块），你需要给一个让条件不成立的输入来触发 false 分支。

题目语境怎么用：2025 Q2 decideWrite 有多处黄色 diamond。你要逐个分析每个黄色点缺的是什么方向的 branch，给每个黄色点一个独立的 additional test，明确写补 line X true/false branch。

卷面写法：写成：TCI：line 28 false branch (currently missed, only true branch covered)。Input：fileName = null。Expected Result：写出正确输出。Reason：当 fileName 为 null 时 if 条件不成立，执行流走 false branch 到达 else 块。

常见失分点：写 line 28 need branch test 太泛没有说明缺 true 还是 false；一个 test case 用来补多个黄色点但不一定能同时走到；忽略 short-circuit 导致的额外 branch。

标准答案（先写结论）：随机给 x 后，用整数平方根反算验证 r*r==x，不能用被测函数输出当 expected。

为什么这是对的：isSquare(x) 判断整数 x 是否为完全平方数。如果你用被测函数自己的输出当 expected result——它返回 true 我就期待 true——就等于放弃判断力。正确 oracle 是独立验证：对每个随机生成的 x，用 floor(sqrt(x)) 算出整数平方根 r，然后检查 r*r==x。这个逻辑不依赖被测 isSquare 函数，是客观数学判断。

题目语境怎么用：2023 Q4 随机测试题要求写 oracle。你不能写 call isSquare and check result——这就是循环论证。你要用独立于被测函数的验证逻辑，用数学性质（平方根反算）来判断。同样适用于其他随机测试题。

卷面写法：分三段伪代码：1) x = random.nextInt(0, 1000)——数据生成；2) int r = (int)Math.sqrt(x); boolean expected = (r*r==x)——oracle 计算；3) assertEquals(expected, isSquare(x))——断言。注意浮点精度问题。

常见失分点：用被测函数 isSquare() 作为 oracle（自己测自己）；浮点数平方根截断问题（sqrt(25) 可能是 4.999 导致 floor=4 和 4*4=16+25）；只生成平方数不生成非平方数。

标准答案（先写结论）：围绕 temp=16 取 15/16/17，围绕 humidity=60 取 59/60/61，并写 expected enum。

为什么这是对的：temp=16 是 HEAT 触发阈值——低于 16 就开暖气，16及以上不开。测 15（just below HEAT 应触发）、16（on HEAT 不应触发）、17（just above HEAT 不应触发）。humidity=60 是 AC 触发阈值——高于 60 就开空调。测 59（just below AC 不应触发）、60（on AC 不应触发）、61（just above AC 应触发）。选值间隔为 1 因为温度是整数。

题目语境怎么用：按 one fault assumption：测试 temp 边界时 humidity 取 nominal 50（不触发 AC），测试 humidity 边界时 temp 取 nominal 20（不触发 HEAT）。这样能隔离每个变量的边界效应。

卷面写法：表格示例：| TCI | temp | humidity | determine(temp,humidity) | Expected |。行1：temp just below 16 | 15 | 50 | determine(15,50) | HEAT_ONLY。行2：temp on 16 | 16 | 50 | determine(16,50) | NONE。

常见失分点：两个变量同时取边界值（如 temp=15, humidity=61）导致无法判断是哪个变量触发的输出变化；漏掉 on-boundary 值；湿度边界取 58/60/62 而不是 59/60/61（离边界距离不对称）。

2. 黑盒测试：只看规格，不看代码

Black-box testing 把程序当成一个盒子。你知道输入和应该输出什么，但不看内部代码。它的优点是贴近需求，缺点是可能不知道代码内部某个分支没跑到。

EP、BVA、DT 分别解决什么

方法核心问题适合题面产物
Equivalence Partitioning哪些输入应该表现相同？输入可以分成有效/无效类别。Input partitions、output partitions、representative values。
Boundary Value Analysis边界附近会不会写错？有数值范围、阈值、最小最大值。Just below、on、just above boundary 的 values。
Decision Table多个条件组合会不会漏？题面有若干 conditions 和 actions。Causes、effects、rules、test cases。

Equivalence Partitioning 的台阶

  1. 找输入变量。例如 age、temperature、membership type。
  2. 按规格把输入分区。一个 partition 里所有值应该触发相同逻辑。
  3. 每个分区选代表值。有效分区要测，无效分区也要测。
  4. 写 expected result。每个代表值必须能从规格推出期望输出。

BVA 的台阶

  1. 找边界。关键词包括 less than、at least、between、minimum、maximum、threshold。
  2. 确定每个边界两侧。例如边界 18 附近，常看 17、18、19。
  3. 避免盲目全组合。多变量 BVA 通常以 one fault assumption 为基础，一次只让一个变量靠边界，其他变量取 nominal value。
  4. 把边界值接到 expected result。边界本身属于哪边，必须由 specification 决定，不能靠猜。

Decision Table 的台阶

  1. 列 causes。Causes 是条件，例如 user logged in、stock available、payment valid。
  2. 列 effects。Effects 是动作或结果，例如 accept order、show error、request login。
  3. 列 rules。每个 rule 是一组 T/F 条件组合，对应一组 effects。
  4. 删 infeasible combinations。有些组合现实中不可能，例如同时小于 0 又大于 10。
  5. 每条可行 rule 至少一个 test case。用具体输入让该 rule 发生，再写 expected effect。

题型判断：题面是数值范围，优先 BVA；题面是多个布尔条件和动作，优先 DT；题面是类别划分但没有明显边界，优先 EP。Q1 样章见 BVA / DT 黑盒完整拆解。

内容理解题（先做 3 题）

核心答案（先确认概念）：类别用 EP，边界用 BVA，条件组合用 DT。

为什么这是对的：EP 核心假设是同一 partition 里所有值行为相同，所以适合按类别分组——如用户类型（普通/VIP/管理员），每种类型选一个代表。BVA 核心假设是 fault 集中在边界，所以适合有明确数值阈值的场景——如 age<18 要用离边界最近的值。DT 核心是多个条件组合决定行为——如是否登录+是否有库存+是否付款三个条件构成组合。口诀帮你一分钟内判断用哪个方法。

题目语境怎么用：看到 spec 有 less than、at least、between X and Y 优先 BVA。看到 spec 列举几种类型但没有数值边界优先 EP。看到 spec 有多个 yes/no 条件优先 DT。混合场景先用 EP 分区再用 BVA 取边界值。

卷面写法：在答案开头声明方法选择及理由：本题存在明确的数值阈值（temp=16, humidity=60），符合 BVA 适用场景，因此采用 Boundary Value Analysis。然后展示 BVA 表格。不要默默用方法不说明。

常见失分点：有数值边界却用 EP（漏掉关键边界两侧测试）；有多个布尔条件却分别用 BVA 独立测每个变量（忽略组合效应）；在一个题里混用三种方法；用我觉得而不是 spec 特征来选方法。

核心答案（先确认概念）：多变量时常用 one fault assumption，一次只让一个变量靠边界，其余取 nominal value，避免测试爆炸。

为什么这是对的：假设三个变量各有 3 个边界值（just below/on/above），全组合是 3x3x3=27 个测试。但 one fault assumption 认为：大多数 bug 是由单个变量的边界错误引起，多个变量同时出错的概率很低。所以只需对每个变量单独做边界测试，其他变量取 nominal（正常值），从 27 个测试降到 9 个。

题目语境怎么用：Climate.determine 题中 temp 和 humidity 各有边界。全组合是 (15/16/17) x (59/60/61) = 9 个。但遵循 one fault assumption：测 temp 边界时 humidity 固定为 50（nominal），测 humidity 边界时 temp 固定为 20（nominal），总计 3+3=6 个。

卷面写法：在 BVA 表格上方写一句采用 one fault assumption，然后在表格中明确标注哪个变量被测试、哪个被固定为 nominal。如行1：temp=15 (just below), humidity=50 (nominal)。

常见失分点：不做 one fault assumption 而做全组合（浪费测试但考试可能扣分因为没体现方法理解）；没有标注哪个变量取 nominal 以及 nominal 值是多少；把 nominal 值也选在边界附近。

核心答案（先确认概念）：一组 conditions 的真假组合，以及该组合对应的 effects 或 actions。

为什么这是对的：一条 DT rule 完整描述了在什么条件下，系统应该做什么。如 boilerSetting 的一条 rule：t 在 1t<25 范围内 AND isOn=true 效果是 LOW。这条 rule 包含 causes（温度区间+开关状态）和 effects（输出 LOW）。每条 rule 必须从 spec 推导、在现实中可行（feasible）、且至少有一个 test case 覆盖它。

题目语境怎么用：考试要求你从 spec 中提取 causes（条件）和 effects（结果），用真值表交叉出所有理论组合，然后划掉互斥的不可能组合。DT 的核心价值是系统性地暴露条件组合遗漏——单独测每个条件可能都覆盖了，但条件之间的交互没测到。

卷面写法：DT 表格式：条件行列出所有 causes，效果行列所有 effects，每列是一条 rule。不可行 rule 用横线划掉并标注 infeasible。在表下方为每条可行 rule 写一个 test case。

常见失分点：只列条件不列效果（表不完整）；没有划掉 infeasible combinations；把互斥的条件当成独立条件导致 rule 数爆炸；每条 rule 不给 test case 只停在表层面。

为什么这是对的：boilerSetting 的 spec 对温度 t 有三个行为分区：t<1 非常低（特殊处理），1t<25 正常低温范围，t25 高温范围。每个区间触发不同的输出规则。加上 isOn 这个布尔开关，形成 DT 的多条件组合。t 的区间本身互斥——同一时刻 t 只能在一个区间——所以 DT 中这些 cause 需做 infeasible 标记。

标准答案（先写结论）：Decision Table 可能包含互斥条件组合，例如同一 t 不可能同时在两个温度区间。

为什么这是对的：boilerSetting 的 DT 中 causes 是 t 在区间1/区间2/区间3/isOn，同一 t 不可能同时属于两个区间（互斥），所以任何让两个区间同时为 T 的 rule 都是 infeasible。不划掉这些 rule 就会给不存在的场景设计 test case，浪费资源且逻辑不自洽。划掉后从一个大的理论空间缩小到实际有意义的几个 rule。

标准答案（先写结论）：enabled=true 且 nettPower>0 时随机取正数 expected true；enabled=false 或 nettPower<=0 expected false。

标准答案（先写结论）：随机给 x 后，用整数平方根反算验证 r*r==x，不能用被测函数输出当 expected。

为什么这是对的：isSquare(x) 判断完全平方数。oracle 不能是被测函数自身，必须用独立数学性质：对随机 x 用 floor(sqrt(x)) 算整数平方根 r，检查 r*r==x。如果等式成立 expected=true 否则 false。这个验证逻辑完全不依赖 isSquare 函数，即使 isSquare 坏了 oracle 依然正确。这是 oracle problem 的经典解法。

题目语境怎么用：任何随机测试题问到 oracle，都要回答独立于被测函数的验证方法。对 isSquare 是平方根反算，对 maxv 是遍历比较，对 exportPower 是 DT 规则表。关键特征：oracle 不能调用被测函数，必须从 spec 或数学性质独立推导。

卷面写法：伪代码三段式：1) x=random(0,1000) 数据生成；2) int r=(int)Math.sqrt(x); boolean expected=(r*r==x) oracle 计算；3) assertEquals(expected, isSquare(x)) 断言。用整数运算 r*r 比浮点 sqrt 安全。

3. 白盒测试：看代码，补没有跑到的地方

White-box testing 看程序内部结构。它不是替代黑盒，而是在黑盒之后问：这些测试实际跑到了哪些代码？还有哪些语句、分支或条件没有被执行？

SC 和 BC 的差别

Coverage问的问题小例子
Statement Coverage (SC)每条可执行语句有没有至少执行一次？return discount 这一行有没有跑到。
Branch Coverage (BC)每个 decision 的 true 和 false 结果有没有都出现？if (age >= 18) 的 true 和 false 都要测。
Condition / short-circuit detail复合条件里的子条件是否被真正 evaluated？A || B 中若 A true，B 可能不会执行。

读 JaCoCo 的台阶

  1. 先看红色。Red 通常表示语句没执行，先补一个能到达该语句的测试。
  2. 再看黄色。Yellow 常表示某个 branch 只覆盖了一边。
  3. 定位 decision。找到对应的 if、loop 或复合条件。
  4. 构造 additional test。让之前没出现的 true/false 结果发生。
  5. 写为什么它补 coverage。不要只给输入，要说明它触发了哪条未覆盖语句或分支。

短路逻辑不要跳过

Short-circuit evaluation 指复合条件可能提前停止。A && B 中，如果 A false，B 不会被检查；A || B 中，如果 A true，B 不会被检查。这会影响分支和条件覆盖，也会影响某些函数调用是否发生。

白盒题写法：指出当前测试缺了哪条 statement 或 branch；给一个 additional test input；说明执行路径怎样经过该代码；写 expected result；最后说 coverage 从哪里补上。

内容理解题（先做 3 题）

核心答案（先确认概念）：每条可执行语句有没有至少执行一次。

为什么这是对的：Statement Coverage 是最基本的白盒覆盖标准：每一行代码至少跑过一次吗？比如 else 分支里的 return discount，所有测试都让 if 条件为 true 时这行永远没跑到——你完全没验证 else 分支行为。SC 100% 是最低要求，但不保证分支组合和边界条件都覆盖。

题目语境怎么用：JaCoCo 报告里红色行 = 0% SC（从未执行），你要给能到达该行的 input。绿色行 = 100% SC，但绿色不意味着没 bug——只是被执行过。考试通常先识别红色（SC 缺口）再识别黄色（BC 缺口）分两步答。

卷面写法：写成：Current Coverage Gap：line X is red — statement not executed。Additional test：input = Y。Execution Path：input=Y makes condition at line N false, so execution enters else-block and reaches line X。Coverage Achieved：statement coverage for line X。然后写 expected result。

常见失分点：把 SC 和 BC 混淆——SC 只看语句执行没，BC 还看决策的每个方向；以为一行绿色 = 代码正确（coverage 不等于 correctness）；红色行的 TCI 写成补 line X branch（红色是 statement 不是 branch）。

核心答案（先确认概念）：每个 decision 的 true 和 false 结果是否都被测试到。

为什么这是对的：SC=100% 不等于 BC=100%。例如 if(x>10){doA();}doB();——一个测试让 x>10，SC 就 100% 了，但 if 的 false 分支从未被覆盖，条件不成立时的行为完全没验证。BC 要求每个 decision point 的 true 和 false 两个方向至少各执行一次。

题目语境怎么用：JaCoCo 的黄色 diamond 是 BC 不完整的信号——当前测试只走了 decision 的一个方向。你要分析缺的是 true 还是 false，构造让条件朝缺失方向走的 input。注意复合条件在 bytecode 层面可能拆成多个 branch。

卷面写法：写成：TCI：line N true branch (currently missing — only false branch covered via existing tests)。Input：value = Z 使 condition 为 true。Expected Result：写出正确输出。Coverage：achieves branch coverage for line N true direction。

常见失分点：认为 SC 100% 意味着 BC 也 100%；黄色 diamond 只写补 branch test 而不说缺的是 true 还是 false；补 BC 的 test case 没有 expected result；忽略 loop 也是一种 decision point。

核心答案（先确认概念）：某个语句或分支只覆盖了一部分，常需要 additional test 补未走到的 branch。

为什么这是对的：黄色 = partial coverage。两种位置：1) diamond 黄色 → branch coverage 不完整（缺 true 或 false 方向）；2) 行标记黄色 → 该行被部分执行，可能复合条件中某子条件从未被 evaluation（短路跳过）。无论哪种都意味着未测试的代码路径需要 additional test 触发。

题目语境怎么用：黄色 diamond → 找 if/while 等 decision，分析缺哪个方向，给触发缺失方向的 input。黄色行 → 检查是否有复合条件(&&/||)，分析短路导致哪些子条件从未被 evaluation，给让子条件被 evaluator 的 input。不要笼统说加测试。

卷面写法：黄色 diamond：写 line N diamond yellow — false branch uncovered; additional input: X = value that makes condition false。黄色行：写 line M yellow due to short-circuit — condition B never evaluated because A always true; additional input: make A false so B gets evaluated。都要配 expected result。

常见失分点：看到黄色就笼统写增加 branch coverage 测试；把黄色行的短路问题和黄色 diamond 的分支问题混为一谈；给的 input 不能确切触发缺失路径；忘记写 expected result。

考试题目题（再做 5 题）

标准答案（先写结论）：先把黄色写成缺失 branch，把红色写成缺失 statement；例如 line 33 red 就要给能走到该 else/return 的具体输入和 expected result。

为什么这是对的：decideWrite 的 JaCoCo 报告有三种覆盖问题：line 33 红色 → 这行从未执行；line 28 黄色 → if 分支只走一边；line 32 黄色同理。优先级先处理红色（完全未覆盖 = 最高风险）再处理黄色。每个缺口写成独立 coverage item 配独立 input 和 expected result。

题目语境怎么用：先列出现有测试覆盖了哪些路径，再列缺失路径。每个缺失路径写成一个 TCI-input-expected result 三元组。line 33 红色意味着某 else 分支的代码（可能是 return false 或写错误信息）从未被执行。

卷面写法：先列出已有 test cases 和覆盖路径。逐个写 additional test：Test A（补 line 33 SC）：input=X, expected=Y, explanation：input=X makes condition at line 26 false, execution enters else-block reaches line 33。Test B（补 line 28 BC）类似结构。

常见失分点：把三个缺口写成一个 test（一个 input 不可能同时精确覆盖三个不同位置）；不追溯代码逻辑就给定 input（导致 input 走不到目标行）；把 line 33 红当作 branch 问题写成补 line 33 false branch。

标准答案（先写结论）：复合条件经过 short-circuit bytecode 分成多个 branch，不只是源代码层面一个 if 的 true/false。

为什么这是对的：源代码复合条件如 if(A&&B) 在字节码层面 short-circuit 产生 4 个 branch：1)评估 A 2)A 为 false 跳转 3)A 为 true 评估 B 4)B 为 false 跳转 5)B 为 true 进入 if 块。JaCoCo 报告 1 of 4 branches missed 意味着其中一个方向从未触发——通常是被短路跳过的子条件方向。

题目语境怎么用：2025 Q2(b) 明确考对 short-circuit 的理解。不能按源代码表面说一个 if 只有 2 个 branch。要分析复合条件的字节码分支结构，找出缺的是哪个子条件的哪个方向，给让缺失方向触发的 input。

卷面写法：先写出字节码级 4 个 branch：1)A=true evaluate B 2)A=true+B=true enter if 3)A=true+B=false skip if 4)A=false skip if without evaluating B。指出当前覆盖了哪些 branch、缺哪个。给 additional test：input=Y makes A=false, so B never evaluated but the A-false branch is now covered。写 expected result。

常见失分点：按源代码表面的 2 个 branch 分析（会被扣分因为没体现对 short-circuit bytecode 的理解）；不知道短路时 B 可能不被 evaluated 带来的 coverage 影响；给的 input 没有精确触发缺失方向。

标准答案（先写结论）：SC 只要求红色语句执行；BC 还要求每个 decision 的 true/false 分支都走到。

为什么这是对的：SC 最低要求每行代码至少执行一次。Wind.categorise 里如果 return Storm 从未被跑到（红色）SC 就不达标。BC 更高要求：还要每个 if/else 的两种方向都走到。SC 是 BC 的必要条件（BC 100% 则 SC 100%），但反过来不成立。

题目语境怎么用：2023 Q2 给了 Wind.categorise 代码和 JaCoCo 报告。问需要多少 additional tests 达到 full SC 只数红色行。问 full BC 除红色行还要看黄色 diamond 把缺失的 true/false 方向补全。

卷面写法：分别列 SC 缺口和 BC 缺口。SC 缺口：line X red (statement not executed) → 需要 test A。BC 缺口（在 SC 已补基础上）：line Y diamond yellow, false branch missing → 需要 test B。写清楚 full SC 需 M 个 additional tests，full BC 需 N 个。

常见失分点：认为一个 test 就能同时达到 full SC 和 full BC（通常需要多个 test）；BC 缺口数少了——每个 diamond 的两个方向都要检查；把黄色行和黄色 diamond 都当作同一类问题。

标准答案（先写结论）：围绕 temp=16 取 15/16/17，围绕 humidity=60 取 59/60/61，并写 expected enum。

为什么这是对的：temp=16 是 HEAT 触发阈值——低于 16 就开暖气。取 15（just below → 应触发 HEAT）、16（on → 不应触发）、17（just above → 不应触发）分别检验比较运算符有没有写错。humidity=60 同理：59/60/61。选值离边界一步之遥，最能暴露 off-by-one 错误。

题目语境怎么用：按 one fault assumption：测 temp 边界时 humidity 取 nominal（如 50），测 humidity 边界时 temp 取 nominal（如 20）。Expected result 必须从 spec 推出并写成精确 enum 值。

卷面写法：表格：| TCI | Selected Temp | Selected Humidity | Test Case | Expected Result |。行示例：| temp just below 16 | 15 | 50 | determine(15, 50) | HEAT_ONLY |。| temp on 16 | 16 | 50 | determine(16, 50) | NONE |。

常见失分点：两个变量同时取边界值导致无法辨识结果来自哪个变量的边界；漏掉 on-boundary（如只测 15 和 17 不测 16）；expected result 写 NONE 但实际 spec 里该组合应对应 HEAT_ONLY。

标准答案（先写结论）：它组合温度区间和 isOn，effects 是 NONE/LOW/HIGH，每条可行 rule 要有 test case。

为什么这是对的：boilerSetting 的输出由温度 t 落在哪个区间（三个互斥区间之一）和 isOn 是否开启两个条件的组合决定。单独测温度或单独测开关都不能覆盖温度高但开关关这种组合场景。DT 天然适合多条件组合决定行为的模式。

题目语境怎么用：读题时识别：是否有多个独立维度？输出是否由维度组合决定？是 → 用 DT。boilerSetting 有 3 个温度区间 x 2 个开关状态 = 6 种理论组合，因温度区间互斥，共 3x2=6 条 feasible rules。

常见失分点：用 BVA 去分析多条件组合（只测边界不测条件交互）；三个温度区间当独立布尔条件不标互斥；没给每条 feasible rule 配 test case；DT 的 effects 写得不精确。

4. 对象和 class context：方法不是孤立函数

对象方法常依赖对象内部状态。只给一个输入参数不够，还要知道对象创建时字段是什么、之前调用过哪些方法、当前是否处于有效状态。

OO 测试基本词

Class对象的蓝图，定义字段和方法。
Objectclass 的一个具体实例，有自己的字段值。
Attribute / field对象保存的状态，例如 balance、temperature、items。
Method对象能执行的操作，可能读取或修改字段。
Accessor读取字段的方法，例如 getter。
Mutator修改字段的方法，例如 setter 或 add/remove 方法。

类内测试的台阶

  1. 创建对象。调用 constructor，确保初始状态明确。
  2. 设置状态。通过 setter 或方法调用让对象进入目标状态。
  3. 调用被测方法。执行题目要求测试的方法。
  4. 观察结果。用 return value 或 getter 检查外部可见结果。
  5. 写 assert。把 expected result 明确写进测试。

class context 模板：Create an object in a known initial state. Use public methods to place it in the required state. Call the method under test. Use an accessor or return value to observe the result. Assert the expected value derived from the specification.

核心答案（先确认概念）：因为它可能依赖对象字段和之前的方法调用历史，也就是 class context。

为什么这是对的：bankAccount.withdraw(100) 能不能成功不只取决于参数 100，还取决于 balance 字段当前是多少。balance=50 时 withdraw 应失败，balance=200 时应成功。不设对象状态就调方法，你不知道该成功还是失败。Instance method 的行为 = 参数 + 对象状态，忽略状态等于忽略一半行为逻辑。

题目语境怎么用：考到 Q3 class context 题，第一反应：这个方法依赖哪些字段？这些字段初始值是什么？需调哪些 setter 或 constructor 来设置？Numbers 题里 instance method isNegative() 依赖 value 字段，必须先用 setValue 设定。

卷面写法：写出完整序列：Step 1: Create Numbers obj = new Numbers(); value=0。Step 2: Set obj.setValue(-5); value=-5。Step 3: Call obj.checkValue();。Step 4: Observe boolean r = obj.isNegative();。Step 5: Assert assertEquals(true, r)。每步标注 state change。

核心答案（先确认概念）：创建对象，设置状态，调用被测方法，通过 getter 或返回值观察，最后 assert。

为什么这是对的：五步是 OO 测试的最小闭环：Create 提供测试舞台；Set state 设前置条件（像函数测试设参数值）；Call 触发被测行为；Observe 收集输出证据；Assert 对照 spec 判断对错。跳过任何一步链条就断了。

题目语境怎么用：哪怕简单的 Q3 题如 Numbers 也要严格走五步。不要因为只有一个字段就跳步。Create Numbers n=new Numbers(); Set n.setValue(-1); Call n.checkValue(); Observe boolean r=n.isNegative(); Assert assertEquals(true,r)。模板化不容易漏步骤。

卷面写法：用表格或编号写清每步：Step | Code | State After | Purpose。或编号：(1) new Numbers() 创建对象 value=0 (2) setValue(-1) 设 value=-1 (3) checkValue() 触发逻辑 (4) isNegative() 返回 true (5) assert true 验证与 spec 一致。

常见失分点：只有 Call 和 Assert 没有 Create 和 Set（假设对象凭空存在且状态已设好）；Observe 写了 getter 调用但没写 assert；步骤顺序混乱（先 Call 后 Set）；用伪代码而非具体方法名。

核心答案（先确认概念）：那会把测试绑死在内部实现上。测试应尽量通过 public API 设置和观察行为。

为什么这是对的：假设直接写 obj.brightness=20 来设状态。下周开发者把 brightness 改名为 luminance——你的测试编译不过了，尽管类的公开行为完全没变。这叫测试脆弱性。正确做法通过 public API（构造函数或 setter）设状态，只要公开接口不变测试就不受内部重构影响。

题目语境怎么用：考试伪代码中出现 obj.field=value 会被扣分。应写 obj.setField(value) 或通过构造函数传参。Lighting 题正确写法是 new Lighting(50,false) 或 lighting.setBrightness(50)，不是 lighting.brightness=50。

卷面写法：时刻检查伪代码是否通过 public method 访问对象。setting→constructor 参数或 setter。getting→getter 或 return value。triggering→被测 public method。不要出现 .fieldName= 赋值语句。

常见失分点：伪代码直接写 obj.brightness=50（Java 里编译不过也暴露不理解封装）；用 getter 获取后直接赋值改字段；在解释中说可以改 private field 来设置状态（直接扣分）。

Random testing 不是把随机数扔给程序就完事。它仍然需要设计：随机数据从哪里来，怎样判断结果对不对，什么时候停止。

数值计算为什么难测

Floating point 不是实数。小数在二进制里可能无法精确表示，所以比较数值结果时不能总用完全相等。数值测试常用 tolerance，例如实际结果和期望结果相差小于某个阈值就接受。

AI 辅助测试的安全写法

AI 可以帮你生成测试想法、边界值、断言草稿、自动化代码，但它不应该替你决定 oracle。好答案要强调 human review：检查生成的测试是否符合 specification，是否覆盖关键风险，是否引入错误断言。

内容理解题（先做 3 题）

核心答案（先确认概念）：Data problem、oracle problem、completion problem。

为什么这是对的：随机测试三问题是设计随机测试的框架。Data problem：随机输入怎样生成——要覆盖有效无效、边界附近、不同分布，不能完全均匀乱选。Oracle problem：怎样知道输出对不对——需要 reference implementation、invariant、metamorphic relation 或人工规则。Completion problem：什么时候算测够了——可以用测试数量、覆盖目标、失败率稳定、时间预算等条件。

核心答案（先确认概念）：随机输入跑完后，怎样知道输出对不对。可能需要 reference implementation、invariant 或 metamorphic relation。

为什么这是对的：Oracle problem 是随机测试的核心难题：程序跑出结果后，你怎么判断它对不对？如果你用被测程序自己来判定（它说 true 就认为对），就形成循环论证永远发现不了 bug。解决方法：需要 independent means——reference implementation（黄金版本）、mathematical invariant（数学不变量）、metamorphic relation（变体关系）或 manual rules（人工规则表）。

题目语境怎么用：根据题目选 oracle 方法：isSquare 用数学性质（平方根反算）；maxv 用遍历比较；Climate 用 spec 规则表；exportPower 用 DT rule。关键：不能调用被测函数自身来判断。

卷面写法：写明 oracle 类型和具体实现：Oracle type: Mathematical property。Implementation: For each random x, compute r = floor(sqrt(x)), expected = (r*r == x)。然后 assert。

常见失分点：说用 checker function 但不说明 checker 怎么实现的；oracle 调用了被测方法；混淆 expected result（从 spec 来）和 oracle（独立验证机制）；用太复杂的 oracle 引入 oracle bug。

核心答案（先确认概念）：Floating point 可能有表示误差，数值测试常用 tolerance 判断结果是否足够接近。

为什么这是对的：浮点数在计算机里用二进制表示，很多十进制小数无法精确表示（如 0.1 在二进制里是无限循环小数）。所以 0.1+0.2 不等于 0.3——等于 0.30000000000000004。测试浮点运算时如果写 assertEquals(0.3, 0.1+0.2)，测试会失败。解决方法是用 tolerance（容差）：abs(actual-expected) < epsilon。

题目语境怎么用：考到数值计算相关题时（如涉及浮点除法、sqrt、三角函数的测试），expected result 不要写精确相等，要写 approximately equal with tolerance。选择合理的 tolerance 值（如 1e-9 或根据问题域设置）。

卷面写法：assertEquals(expected, actual, 1e-9) —— TestNG 的带 delta 的 assertEquals。如果是整数运算避免用浮点数，如用整数平方根而不是浮点 sqrt。说明为什么选这个 tolerance：基于问题域的精度要求。

常见失分点：浮点比较用 ==（会因表示误差失败）；tolerance 值选得太小（比浮点精度还小）或太大（掩盖实际 bug）；明明可用整数避免浮点却还是用浮点（如平方根检验用整数乘法不用 sqrt）。

卷面写法：写：Rule 1 data conditions: enabled = true (fixed), nettPower = random.nextInt(1, Integer.MAX_VALUE) or a bounded range like [1, 10000]。Test call: assertEquals(true, pv.exportPower()) after setting enabled=true and nettPower to the random positive。

常见失分点：nettPower 生成范围包含 0 或负数（0 不算 >0）；忘记设置 enabled=true；expected 写成 false；没有写 nettPower 的上限范围导致生成不合理值。

标准答案（先写结论）：随机给 x 后，必须有办法判断它是否平方数，例如计算整数平方根再验证。

为什么这是对的：isSquare 的 oracle 不能是 isSquare 自己。必须用独立数学验证：随机生成 x，用整数平方根反算。Math.sqrt(x) 返回 double，floor 取整得到 r，然后做 r*r==x。但浮点 sqrt 可能有精度问题：sqrt(25) 可能是 4.999999...，floor 得到 4，4*4=16!=25 导致 expected 错判。精确方案是同时检查 r*r 和 (r+1)*(r+1) 哪个等于 x。

题目语境怎么用：2023 Q4(a) 直接问你 oracle problem 怎么具体化。你要写出完整的 oracle 伪代码，包括如何处理浮点精度。这是体现你对数值计算理解的关键点。

卷面写法：int r = (int)Math.sqrt(x); boolean expected = (r*r == x) || ((r+1)*(r+1) == x)。这个双重检查解决了浮点 sqrt 的截断不确定性。如果 x 是平方数，要么 r*r==x 要么 (r+1)*(r+1)==x。非平方数则是两者都不等。

常见失分点：只用单次 r*r==x 检查（浮点截断导致漏判）；用 Math.sqrt 的 double 结果做相等比较（精度问题）；oracle 调用了被测 isSquare 函数；没有考虑浮点截断问题的考生会在考试中失分。

标准答案（先写结论）：例如 temp below valid range 或 above valid range，对应 EP1*/EP5*，expected ERROR。

为什么这是对的：whatSpeed 的 spec 有多个温度区间对应不同风扇速度，超出有效范围的温度返回 ERROR。ERROR partitions 是无效输入的 EP 类别：temp 低于最小有效值是一个 ERROR partition，temp 高于最大有效值是另一个。随机测试时这些 partition 要单独生成随机值，expected 都是 ERROR。

题目语境怎么用：2023 Q4(b) 要求写出各 EP 对应的随机生成范围。ERROR partitions 的范围是 (-Inf, minValid) 和 (maxValid, +Inf)。实际生成时取合理范围如 [-100, minValid-1] 和 [maxValid+1, 200]。expected result = ERROR。其他有效 partition 按区间生成。

卷面写法：列出所有分区和对应生成范围表：EP1: temp < 0 → [-100, -1], expected ERROR。EP2: 0<=temp<20 → [0, 19], expected LOW。EP3: 20<=temp<40 → [20, 39], expected MED。EP4: temp>=40 → [40, 100], expected HIGH。EP5: temp > 100 → [101, 200], expected ERROR。

常见失分点：漏掉某一端的 ERROR partition（只测了低于范围的没测高于范围的）；ERROR partition 生成范围没覆盖足够多的值；expected result 写错（ERROR partition 写成某个有效输出）。

标准答案（先写结论）：给一个能走到 line 33 的 concrete input，写 expected result，并说明它补 statement coverage。

为什么这是对的：JaCoCo line 33 红色 = 该行从未执行。decideWrite 里 line 33 可能在 else 块里。要补这个 statement coverage，需要找到什么输入能让控制流走到 else 块到达 line 33。追踪代码逻辑：line 33 前的 condition 要使程序跳过 if 块进入 else。然后给具体输入值、写 expected result、说明补了 SC。

题目语境怎么用：2025 Q2 的 line 33 红色是最直接的 SC 缺口。优先构造能走到 line 33 的 input，因为完全未覆盖风险最高。注意区分 SC（红色）和 BC（黄色）——line 33 红色用 statement coverage TCI，不要写成 branch coverage TCI。

卷面写法：Coverage Gap: line 33 red, statement never executed。Additional test: input = X (specific value that makes condition at line 26/28 false)。Expected Result: Y (from spec)。Coverage: statement coverage for line 33 is now achieved as execution enters else-block。

常见失分点：给 input 不说为什么能走到 line 33；把红色当黄色写成补 branch coverage；input 不能精确触达 line 33（控制流逻辑没追对）；漏写 expected result。

标准答案（先写结论）：短路条件编译成 bytecode-level branches；additional test 要让之前没 evaluated 的子条件被 evaluated。

为什么这是对的：源代码 if(A&&B) 看起来是 2 个 branch（整体 true/false），但在 JVM 字节码层面 short-circuit 产生 4 个跳转：A==false 跳转、A==true 且 B==false 跳转、A==true 且 B==true 进 if 块。JaCoCo 统计字节码级 branch，所以报告 4 branches。1 of 4 missed 意味着这 4 个里面有一个从未触发——通常是被短路的子条件的某个方向。

题目语境怎么用：2025 Q2(b) 考 short-circuit 导致的字节码级 branch 多于源代码级。你要解释是因为 && 的 lazy evaluation 导致编译器生成多个跳转指令。然后 construct 一个 test 让缺失的那个方向被触发。

卷面写法：先列出 4 个 branch：1)A evaluates to true -> evaluate B。2)A true + B true -> enter if-body。3)A true + B false -> skip if-body。4)A false -> skip if-body without evaluating B。指出当前 missing 哪个。additional test: input=Y that makes A false, triggering branch 4。

常见失分点：只按源代码 2 分支解释（不体现字节码理解）；不知道短路导致 B 可能不被 evaluated；给的 input 触发了已有 branch 而非缺失 branch。

内容理解题（先做 3 题）

核心答案（先确认概念）：Q1 黑盒，Q2 白盒，Q3 class context，Q4 random testing。

为什么这是对的：Q1 黑盒考 specification-based testing，产物是 BVA/DT 的 TCI 表格和 test cases。Q2 白盒考 code-based testing，产物是 additional tests 加 coverage explanation。Q3 class context 考 OO 测试，产物是对象创建-设置-调用-观察-assert 的序列。Q4 random testing 考随机策略，产物是 data/oracle/completion 三段式设计。四题覆盖四种 coverage 来源。

题目语境怎么用：先看题号判断考什么：Q1 看有没有数值范围或条件组合决定输出；Q2 看有没有 JaCoCo 报告和代码片段；Q3 看有没有类定义和对象方法；Q4 看有没有随机测试要求或 DT rules。

卷面写法：先判题型，然后套对应模板。Q1：EP/BVA/DT 表格。Q2：Coverage Gap + Additional Test + Explanation。Q3：Create-Set-Call-Observe-Assert 序列。Q4：Data 段 + Oracle 段 + Completion 段。

常见失分点：把四题的方法搞混（如在 Q1 里写 JaCoCo 分析）；每题的产物格式不对（Q2 只给 input 不给 expected result 和 coverage explanation）；不知道四题分别对应哪四种 coverage 来源。

核心答案（先确认概念）：TCI、data values、test cases，以及 expected result。

为什么这是对的：Q1 考的是从 spec 中提取测试需求。产物由抽象到具体递进：先列 TCI（覆盖目标——如 just below boundary of 16）、再选 Data Value（具体值——如 15）、再写 Test Case（完整调用加 expected result）。这个三层结构是 Q1 的评分骨架，缺任何一层都会被扣分。

题目语境怎么用：Climate.determine 题先做 EP 分区（有效/无效），再做 BVA 表格（just below/on/just above），最后配 expected result。三道子题 (a)(b)(c) 层层递进，每层产物不同。

卷面写法：用三层表格：TCI | Selected Value(s) | Test Method Call | Expected Result。TCI 用英文短语描述覆盖目标，Selected Value 写具体数字，Test Method Call 写完整调用，Expected Result 写可验证输出。每层不能空。

常见失分点：跳过 TCI 直接写值（阅卷人不知道你为什么选这个数）；TCI 写得太具体重复了 Data Value；Expected Result 不是从 spec 推导而是猜的；给 test case 不写 expected result。

核心答案（先确认概念）：不能只给输入；还要说明它补了哪条 statement 或 branch coverage。

为什么这是对的：Q2 的核心是读 JaCoCo 报告找覆盖缺口，然后设计 additional test 来补洞。缺口有两种：红色（statement 未执行）和黄色（branch 只覆盖了一边）。每个缺口需要一个独立的 additional test，产物是 input + expected result + coverage explanation。不能只给 input，必须说明这个 input 怎样让控制流到达目标代码、补了哪种 coverage。

题目语境怎么用：看到红色先补 SC，看到黄色再补 BC。红色通常优先级更高（完全未覆盖）。每个 additional test 要说明：触发了哪条未执行语句或未覆盖分支，从 spec 推出 expected result。

卷面写法：用一个表格写：Coverage Gap | Additional Test Input | Expected Result | Coverage Achieved | Execution Path。红色行写 line X red, statement not executed → input=Y → expected=Z → line X now executed via else-branch。黄色行类似。

常见失分点：只给 input 不给 expected result；只写补 branch coverage 不写具体是哪个 branch 的哪个方向；给的 input 跟缺口不匹配（控制流逻辑追错了）；混淆 SC 红色和 BC 黄色。

考试题目题（再做 5 题）

标准答案（先写结论）：练 2025 Summer Q1 Climate.determine partitions + BVA tables。

为什么这是对的：2025 Summer Q1 Climate.determine 是最标准的黑盒入门题。它有明确的数值阈值（temp=16, humidity=60），两个独立布尔条件，输出是枚举值。练习它可以覆盖 EP（分区有效/无效）、BVA（取三点边界值）、DT（条件组合决定输出）全部三种黑盒方法。做完这道题你就掌握了 Q1 的完整套路。

题目语境怎么用：先做 EP：把 temp 和 humidity 各自分成有效/无效 partition，列出 representative values。再做 BVA：在有效范围内围绕 temp=16 取 15/16/17，围绕 humidity=60 取 59/60/61。最后做 DT：两个条件组合的四种输出。

卷面写法：按 Climate 题的 (a)(b)(c) 子题顺序：先列出所有 partitions，再做 BVA 表格并标注 one fault assumption，最后考虑条件组合（可选 DT）。逐步递进不要跳。

常见失分点：跳过 EP 直接做 BVA（题目要求先分区）；BVA 里忘记标注 one fault assumption；把无效值写进 BVA 表（题目说不含 error values）；expected result 写中文描述不写 enum。

标准答案（先写结论）：练 2025 Summer Q2 decideWrite JaCoCo 黄色/红色 additional BC tests。

为什么这是对的：2025 Summer Q2 decideWrite 是最标准的白盒题。它给了代码、JaCoCo 报告（红色/黄色行）、以及已有的部分测试。你要先读懂 JaCoCo 报告识别 SC 和 BC 缺口，再追溯代码逻辑构造能覆盖缺口的 additional test。这道题还考了 short-circuit 导致的字节码级 branch，是白盒题的示范。

题目语境怎么用：先列出已有测试覆盖了哪些路径，再列出红色行（SC 缺口）和黄色 diamond（BC 缺口）。对每个缺口分析代码逻辑：什么 input 能到达该行？优先补红色（line 33 等），再补黄色。最后处理短路问题。

卷面写法：写表格：Gap | Additional Input | Expected Result | Covered | Path。Gap 列写 line 33 red SC not covered，Additional Input 写具体值和类型，Expected Result 从 spec 推出，Path 写数据如何走到的逻辑解释。

常见失分点：不追溯代码逻辑就给定 input（可能走不到目标行）；没解释短路导致的 4 branches；把三个缺口写成一个 test；line 33 红色当成 branch 问题。

标准答案（先写结论）：练 2025 Summer Q4 PV.exportPower random DT tests 和 automated code outline。

为什么这是对的：2025 Summer Q4 PV.exportPower 是最标准的随机测试题。它给了 DT rules，要求你基于 rules 设计随机测试策略。你要解决三个问题：Data（按 DT rule 生成随机输入）、Oracle（DT 表就是 oracle——对照 rule 的 expected effect 判断对错）、Completion（每条 rule 覆盖 N 次或覆盖特定 partition 后停止）。

题目语境怎么用：先把 DT rules 列清楚：rule1 enabled=T, nettPower>0 -> true；rule2 enabled=F -> false；rule3 nettPower<=0 -> false。然后为每条 rule 写随机生成范围、expected result 和循环结构。最后写 automated code outline。

卷面写法：三段式：(1) Data：for each rule, generate random inputs matching conditions。(2) Oracle：compare exportPower() result against rule expected。(3) Completion：stop after each rule tested N=100 times。附 pseudo code。

常见失分点：不按 DT rule 生成数据（盲目随机）；oracle 不来自 DT rule 而来自猜测；completion 条件模糊；只写概念不写 pseudo code。

标准答案（先写结论）：它组合温度区间和 isOn，effects 是 NONE/LOW/HIGH，每条可行 rule 要有 test case。

为什么这是对的：boilerSetting 是 DT 的经典案例。温度 t 有三个互斥区间（<1、[1,25)、>=25）和 isOn 布尔条件，两个维度组合决定三态输出 NONE/LOW/HIGH。单独测温度或单独测开关都不够——必须测组合。DT 表系统地陈列所有组合，划掉互斥不可行的，为每条可行 rule 写 test case。

题目语境怎么用：这道题示范了 DT 的完整步骤：1)识别 causes（温度区间 + 开关状态）；2)识别 effects（三种输出）；3)列所有理论组合；4)划掉互斥 infeasible 组合；5)为 feasible rules 写 test case。

卷面写法：DT 表：causes t<1? | 1<=t<25? | t>=25? | isOn? ；effects NONE | LOW | HIGH。每列一条 rule。互斥的 t 区间同时为 T 的列划掉。然后为每条可行 rule 给具体 t 和 isOn 值和 expected output。

常见失分点：把三个温度区间当独立条件不标互斥；没有划掉 infeasible combinations；只列 DT 不给 test case；effects 写成 maybe LOW 而非精确定值。

标准答案（先写结论）：给一个能走到 line 33 的 concrete input，写 expected result，并说明它补 statement coverage。

为什么这是对的：line 33 红色是 Q2 白盒的典型问题——statement 从未执行。要补这个缺口：1)追溯代码逻辑找到什么条件能让 line 33 执行（通常在 else 块或某个条件不成立时）；2)构造具体 input 满足这个条件；3)从 spec 推 expected result；4)说明 input 如何让控制流到达 line 33。这四个要素缺一不可。

题目语境怎么用：看到红色先定位该行在哪个分支里：是 if-block 还是 else-block？前面的 condition 是什么？当前测试都让 condition 怎么样？然后反推：要让该行执行，condition 应该取什么值？给出对应该值的 input。

卷面写法：Coverage Gap: line 33 red — statement never executed。Additional Test Input: X = specific value that makes the condition false (or true, depending on logic)。Expected Result: Y (from spec)。Explanation: when input = X, condition at line N becomes false, execution enters else-block, reaching line 33 — statement coverage achieved。

常见失分点：不追代码逻辑就猜一个 input；把红色当黄色补 branch；写了 input 没 expected result；没说清楚控制流怎么走到 line 33（阅卷人不知道你为什么选这个 input）。
