CS608 linear course

CS608 教材线性精讲：从一个测试用例学完整门课

CS608 的主线是：在有限时间里，怎样系统挑出有代表性的测试，而不是凭感觉随便试。整门课从 specification 到 test case，再到黑盒、白盒、对象状态、自动化、随机测试和过程管理，都在回答同一个问题：测试从哪里来，凭什么说它覆盖了风险。

测试用例 EP / BVA / DT SC / BC / JaCoCo Class Context Automation 考试台阶

0. 一个测试用例到底由什么组成

测试不是“运行一下看看”。一个 test case 至少要说明输入是什么、怎样执行、期望结果是什么。没有 expected result 的运行，只能叫试用，不能算严格测试。

测试词白话解释小例子
Specification系统应该做什么的描述。年龄小于 18 返回 child fare，18 及以上返回 adult fare。
Input喂给程序的数据。age=17。
Expected result按 specification 推出的正确结果。期望返回 child fare。
Actual result程序实际跑出来的结果。程序返回 adult fare。
Failure外部可见的不正确行为。17 岁被收成人价。
Fault / defect代码或设计里的错误原因。代码写成 age <= 17 或边界条件写反。

测试设计的最小闭环

  1. 读 specification。先找输入变量、输出结果、规则、边界、例外情况。
  2. 选 test coverage items。TCI 是你打算覆盖的“测试目标”，例如某个输入分区、某个边界、某条决策表规则。
  3. 把 TCI 变成 concrete data。Coverage item 是抽象目标，data value 是具体数值。
  4. 写 test case。说明输入、执行步骤、expected result。
  5. 运行并比较。Actual result 和 expected result 不一致，才说明发现 failure。

过关小题：如果题目只给你一个函数描述，不给代码，优先用黑盒方法；如果题目给你代码和 JaCoCo 报告，优先用白盒方法补 coverage。

本节过关门槛

1. Coverage 不是“测得多”，而是“测到该测的东西”

Coverage 的意思是覆盖度。它不是单纯追求 test case 数量多，而是说明哪些风险点已经被代表性测试碰到。CS608 里最关键的能力，就是把题目要求转换成 coverage items。

Coverage 来源看什么会生成什么 TCI
Specification输入范围、输出分类、业务规则。Equivalence partitions、boundaries、decision table rules。
Code语句、分支、条件、短路逻辑。Statement coverage items、branch coverage items。
Object state对象字段、构造状态、方法调用历史。Class context tests。
Random strategy数据分布、oracle、停止条件。Random test criteria。

从 TCI 到 test case 的区别

TCI 是“要覆盖什么”，test case 是“用哪个具体输入去覆盖”。比如 BVA 中 “just below lower boundary” 是 TCI，具体数据 age=17 才是 test data。考试里表格常要求你分清 TCI、Data Values 和 Test Cases。

写表格时的顺序：先列 TCI，例如 age just below 18；再给 Data Value，例如 age=17；最后写 Test Case，例如 call fare(17) and expect child fare。

2. 黑盒测试：只看规格，不看代码

Black-box testing 把程序当成一个盒子。你知道输入和应该输出什么，但不看内部代码。它的优点是贴近需求，缺点是可能不知道代码内部某个分支没跑到。

EP、BVA、DT 分别解决什么

方法核心问题适合题面产物
Equivalence Partitioning哪些输入应该表现相同？输入可以分成有效/无效类别。Input partitions、output partitions、representative values。

Equivalence Partitioning 的台阶

  1. 找输入变量。例如 age、temperature、membership type。
  2. 按规格把输入分区。一个 partition 里所有值应该触发相同逻辑。
  3. 每个分区选代表值。有效分区要测，无效分区也要测。
  4. 写 expected result。每个代表值必须能从规格推出期望输出。

BVA 的台阶

  1. 找边界。关键词包括 less than、at least、between、minimum、maximum、threshold。
  2. 确定每个边界两侧。例如边界 18 附近，常看 17、18、19。
  3. 避免盲目全组合。多变量 BVA 通常以 one fault assumption 为基础，一次只让一个变量靠边界，其他变量取 nominal value。
  4. 把边界值接到 expected result。边界本身属于哪边，必须由 specification 决定，不能靠猜。

Decision Table 的台阶

  1. 列 causes。Causes 是条件，例如 user logged in、stock available、payment valid。
  2. 列 effects。Effects 是动作或结果，例如 accept order、show error、request login。
  3. 列 rules。每个 rule 是一组 T/F 条件组合，对应一组 effects。
  4. 删 infeasible combinations。有些组合现实中不可能，例如同时小于 0 又大于 10。
  5. 每条可行 rule 至少一个 test case。用具体输入让该 rule 发生，再写 expected effect。

题型判断：题面是数值范围，优先 BVA；题面是多个布尔条件和动作，优先 DT；题面是类别划分但没有明显边界，优先 EP。Q1 样章见 BVA / DT 黑盒完整拆解。

3. 白盒测试：看代码，补没有跑到的地方

White-box testing 看程序内部结构。它不是替代黑盒，而是在黑盒之后问：这些测试实际跑到了哪些代码？还有哪些语句、分支或条件没有被执行？

SC 和 BC 的差别

Coverage问的问题小例子
Statement Coverage (SC)每条可执行语句有没有至少执行一次？return discount 这一行有没有跑到。
Branch Coverage (BC)每个 decision 的 true 和 false 结果有没有都出现？if (age >= 18) 的 true 和 false 都要测。
Condition / short-circuit detail复合条件里的子条件是否被真正 evaluated？A || B 中若 A true，B 可能不会执行。

读 JaCoCo 的台阶

  1. 先看红色。Red 通常表示语句没执行，先补一个能到达该语句的测试。
  2. 再看黄色。Yellow 常表示某个 branch 只覆盖了一边。
  3. 定位 decision。找到对应的 if、loop 或复合条件。
  4. 构造 additional test。让之前没出现的 true/false 结果发生。
  5. 写为什么它补 coverage。不要只给输入，要说明它触发了哪条未覆盖语句或分支。

短路逻辑不要跳过

Short-circuit evaluation 指复合条件可能提前停止。A && B 中，如果 A false，B 不会被检查；A || B 中，如果 A true，B 不会被检查。这会影响分支和条件覆盖，也会影响某些函数调用是否发生。

白盒题写法：指出当前测试缺了哪条 statement 或 branch；给一个 additional test input；说明执行路径怎样经过该代码；写 expected result；最后说 coverage 从哪里补上。

OO 测试基本词

Class对象的蓝图，定义字段和方法。
Objectclass 的一个具体实例，有自己的字段值。
Attribute / field对象保存的状态，例如 balance、temperature、items。
Method对象能执行的操作，可能读取或修改字段。

Mutator修改字段的方法，例如 setter 或 add/remove 方法。

类内测试的台阶

  1. 创建对象。调用 constructor，确保初始状态明确。
  2. 设置状态。通过 setter 或方法调用让对象进入目标状态。
  3. 调用被测方法。执行题目要求测试的方法。
  4. 观察结果。用 return value 或 getter 检查外部可见结果。
  5. 写 assert。把 expected result 明确写进测试。

class context 模板：Create an object in a known initial state. Use public methods to place it in the required state. Call the method under test. Use an accessor or return value to observe the result. Assert the expected value derived from the specification.

6. 随机测试、数值问题、AI 和过程

Random testing 不是把随机数扔给程序就完事。它仍然需要设计：随机数据从哪里来，怎样判断结果对不对，什么时候停止。

随机测试三问题

问题含义考试答案角度
Data problem随机输入怎样生成。要覆盖有效/无效、边界附近、不同分布，而不是完全均匀乱选。
Oracle problem怎样知道输出对不对。需要 reference implementation、invariant、metamorphic relation 或人工规则。
Completion problem什么时候算测够了。可以用测试数量、覆盖目标、失败率稳定、时间预算等停止条件。

数值计算为什么难测

Floating point 不是实数。小数在二进制里可能无法精确表示，所以比较数值结果时不能总用完全相等。数值测试常用 tolerance，例如实际结果和期望结果相差小于某个阈值就接受。

AI 辅助测试的安全写法

AI 可以帮你生成测试想法、边界值、断言草稿、自动化代码，但它不应该替你决定 oracle。好答案要强调 human review：检查生成的测试是否符合 specification，是否覆盖关键风险，是否引入错误断言。

标准分析（为什么）：这道小题问的是“考试5. CS608 考试补题：2025 decideWrite line 33 red 要怎么补？”。标准答案里的“给一个能走到 line 33 的 concrete input，写 expected result，并说明它补 statement coverage。”只是结论，真正要拿分的是说明题面为什么支持这个结论。 白盒题要从 coverage 缺口反推测试。红色通常是 statement 未执行，黄色通常是 branch 未全覆盖。

题目语境怎么用：看到 line red/yellow，要问什么输入能到达这条 statement 或走到缺失 branch；short-circuit 条件还可能产生 bytecode-level branch。

卷面写法：卷面写 coverage item、input/call、expected result、补上的 line/branch。不要只给 input。

常见失分点：覆盖率不是正确性的替代品。每个新测试仍要有 specification 推出的 expected result。
