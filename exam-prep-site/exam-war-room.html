考前最后一轮：把题面直接变成可交卷答案。

这一页不是重新学课。它专门训练考场动作：读题后先判断题型、确定要交付的产物、写出第一段、检查有没有漏掉能拿分的表格、公式、证明或例子。

基于 2023-2025 past papers 与 sample papers 题面触发词 最后三轮 整题包 漏分点

1. 读题分流：先判断要交付什么

考题的第一步不是回忆所有知识，而是识别这道题要你交付什么。CS615 常交付解释和取舍；CS605 交付证明；CS608 交付测试表和 expected result；CS603 交付公式、invariant、annotation 或 temporal property。

常见失分点：不要只写“测边界”。只列输入没有 expected result 也不完整；题目说不含 error values 时，别把无效值混入有效边界表。

标准答案（先写结论）：先填目标语言名并假设它 decidable，再写从 HALT 输入 <M,w> 到新 Java program 的 computable mapping。

标准分析（为什么）：这道小题问的是“考试2. 考场读到 2025 CS605 L3/L5 reduction template，第一步填什么？”。标准答案里的“先填目标语言名并假设它 decidable，再写从 HALT 输入 到新 Java program 的 computable mapping。”只是结论，真正要拿分的是说明题面为什么支持这个结论。 HALT 归约题的核心是把题目 property 变成停机问题的开关：如果目标性质可判定，就能判定 M(w) 是否停机，从而矛盾。

题目语境怎么用：构造程序 N 时通常先模拟 M(w)：若停机，就触发 exception、执行指定 line 或破坏某个 throughout 条件；若不停机，就不触发该性质。

卷面写法：卷面四步：假设 decider D 存在；从 有效构造 N；运行 D(N)；证明 M(w) halts iff N 具有题目性质；最后写 contradiction。

常见失分点：不能只写 reduce to HALT。mapping 怎么造、为什么 computable、iff 两个方向都要写。

题目语境怎么用：request -> response 是 liveness；永不 error 是 safety；counterexample 是一条具体执行路径。

2. 最后三轮复习法

最后一轮不要平均用力。按题目产物复习，比按章节标题复习更接近试卷。每一轮都要写一点东西，不能只看。

第一轮：整题识别

拿一份 past/sample paper，只看题面关键词，不写完整答案。给每题标出：解释题、比较题、证明题、测试表、代码覆盖、Dafny、model checking。

第二轮：答案骨架

每题只写 5 行骨架：定义、题面对象、交付产物、例子、最后检查点。目标是速度和方向，不追求完整段落。

第三轮：补高频漏分

回看自己漏掉的内容：CS605 的 iff、CS608 的 expected result、CS603 的 variant、CS615 的 trade-off。只补这些会直接丢分的点。

推进规则：一道题如果 60 秒内说不出“题型 + 交付产物 + 一个具体例子”，就回到对应题型页；如果能说出但写不细，再回线性页补该 section。

常见失分点：不能只写 reduce to HALT。mapping 怎么造、为什么 computable、iff 两个方向都要写。

标准答案（先写结论）：必须接具体 boundary values，例如 just below/on/above，而不是只写 boundary testing is useful。

标准分析（为什么）：这道小题问的是“考试2. CS608 BVA 开头句之后必须接什么？”。标准答案里的“必须接具体 boundary values，例如 just below/on/above，而不是只写 boundary testing is useful。”只是结论，真正要拿分的是说明题面为什么支持这个结论。 BVA 的理由是 fault 常在阈值刚跨过的位置。具体值要服务 just below/on/above，而不是随机挑数。

题目语境怎么用：Climate 题围绕 temp=16、humidity=60；其他 BVA 题也要先找 threshold、minimum、maximum 或 between 的边界。

卷面写法：卷面用表格写 TCI、selected value、test case、expected result。每个 expected result 都必须来自 specification。

常见失分点：不要只写“测边界”。只列输入没有 expected result 也不完整；题目说不含 error values 时，别把无效值混入有效边界表。

标准答案（先写结论）：必须接具体 formula，例如 G(request -> F response) 或 CTL 的 AG safe，并解释 operator。

标准分析（为什么）：这道小题问的是“考试3. CS603 model checking 开头句之后必须接什么？”。标准答案里的“必须接具体 formula，例如 G(request -> F response) 或 CTL 的 AG safe，并解释 operator。”只是结论，真正要拿分的是说明题面为什么支持这个结论。 Model checking 题要同时有模型和性质。Temporal formula 明确描述所有路径或未来状态中要检查的行为。

题目语境怎么用：request -> response 是 liveness；永不 error 是 safety；counterexample 是一条具体执行路径。

卷面写法：卷面写 states/transitions/labels，再写 LTL/CTL formula 并解释 G、F、X、U、A、E 等 operator。

常见失分点：不要只写工具名 Spin/NuSMV。公式方向写反会改变性质。

标准答案（先写结论）：先补 3-tier monolith、microservices、PostgreSQL/NoSQL、scalability/resilience/deployment trade-off。

5. 最常见漏分点

这些不是小错误，是真题里经常让答案从“懂”掉到“不给分”的位置。

科目危险写法改成这样
CS615只列 React、Node、SQL、NoSQL 名词。写每层职责、request flow、数据一致性、安全/部署 trade-off。
CS615Security 只写 sanitize。SQL injection 写 parameterized queries；XSS 写 output encoding、sanitization、CSP、HttpOnly cookies。
CS605Pumping proof 只展示一个切法。写任意合法 split，并用条件限制被 pump 的段在哪里。
CS605Reduction 只写“reduce from HALT”。写构造、computable mapping、iff、contradiction。
CS608Test case 只有输入，没有 expected result。每个 test case 都写 call、inputs、expected output，并说明覆盖哪个 TCI。
CS608White-box 只说“add tests”。指出未覆盖 line/branch，给触发条件和 expected result。
CS603Invariant 只写自然语言“it works”。写范围 invariant、已处理部分性质、退出时如何推出 postcondition。
CS603LTL/CTL 只写工具名。写状态模型、公式、operator 含义和 counterexample 解释。

标准分析（为什么）：这道小题问的是“内容1. 漏分点表最该怎么用？”。标准答案里的“写完答案后用它反向检查：有没有 expected result、iff、variant、trade-off、operator meaning。”只是结论，真正要拿分的是说明题面为什么支持这个结论。 这里的“为什么”来自复习流程本身：先铺概念台阶，才能看懂题型；先做内容题，才能确认概念没空；再做 past/sample 小题，才能确认能交卷。
